## 一、项目概述

### 1.1 项目背景

本项目旨在开发一个公募基金行为教练 Agent，基于有知有行的投资方法论（R=A+B-C、好资产+好价格+长期持有），为投资者提供基金分析、买入决策辅助和行为干预功能。

经过前期研究（有知有行知识体系、E大投资策略、Dayu-Agent 架构设计），从第一性原理审视后，将 MVP 聚焦于 3 个杀手级功能，砍掉了数据成本高、实现难度大的功能，确保 4 周内可交付可用版本。

### 1.2 MVP 目标

开发一个命令行工具，输入基金代码，输出**多章节分析报告**（定性模板 v2），包含：

- **基金年报关键数据自动提取**（费率、换手率、自购变化、投资者收益 vs 基金收益）
- **R=A+B-C 收益归因计算**（区分结构性超额 vs 阶段性超额）
- **基金经理言行一致性检验**（§4 说 vs §8 做交叉验证）
- **投资者获得感分析**（行为损益 = 投资者实际收益 - 基金产品收益）
- **核心风险与否决项**（E大"品种必须不死" + 压力测试）
- **买入前检查清单**（7 个问题，红/黄/绿灯）

### 1.3 不在 MVP 范围内的功能

- 全市场横向比较（数据成本高，有知有行已做；v2 阶段在严选基金池内做）
- 实时行为偏差检测（需要交易记录，改为买入前检查清单）
- 复杂多章节报告的 LLM 生成（MVP 用模板填充，v2 引入 LLM 写作）
- 温度计自建（数据源依赖有知有行爬虫，MVP 使用缓存数据）
- 组合管理功能（v2 阶段）

### 1.4 与 v1.0 的主要变化

| 变化项 | v1.0 | v2.0 | 变化原因 |
|--------|------|------|---------|
| 输出格式 | 一页纸报告 | 8 章定性模板 | 借鉴 Dayu-Agent 模板设计，信息更完整 |
| 分析维度 | 3 个（成本/收益/检查清单） | 7 个（+产品本质/基金经理/获得感/风险否决） | 覆盖"产品-人-钱-风险"全链路 |
| 架构设计 | 无 | 四层架构（UI→Service→Host→Agent） | 借鉴 Dayu-Agent，为后续扩展打基础 |
| 审计机制 | 无 | 三层审计（程序→LLM→证据复核） | 确保分析结果可验证 |
| 超额收益 | 仅计算 A=R-B | 区分结构性 vs 阶段性超额 | 第一性原理：可持续的能力 vs 风格顺风 |
| 检查清单 | 嵌入报告 | 独立模块（温度计触发时使用） | 检查清单是行为干预工具，不是分析工具 |

## 二、功能规格

### 2.1 基金年报 X 光机（核心功能）

**功能描述**

输入基金代码，自动从巨潮资讯网下载最新年报（或季报），提取关键数据，填充定性模板各章节。

**数据提取清单**

| 数据项 | 说明 | 来源章节 | 用于模板章节 |
|--------|------|---------|-------------|
| 基金名称、代码、类型 | 基本身份信息 | §1/§2 | 第 0 章、第 1 章 |
| 投资目标、投资策略 | 产品本质定义 | 招募说明书 | 第 1 章 |
| 业绩比较基准 | Beta 计算基础 | §2/招募说明书 | 第 1 章、第 2 章 |
| 管理费率、托管费率 | 显性成本 | §2 | 第 2 章 |
| 换手率 | 隐性成本计算基础 | §8 | 第 2 章 |
| 净值增长率 vs 业绩基准 | 超额收益来源 | §3 | 第 2 章 |
| 基金收益率 vs 投资者收益率 | 行为损耗核心指标 | §3（2026 新规） | 第 4 章 |
| 份额变动 | 资金流向判断 | §10 | 第 4 章 |
| 基金经理自购金额 | 利益绑定信号 | §9 | 第 3 章 |
| 基金经理投资策略描述 | 言行一致性"说" | §4 | 第 3 章 |
| 前十大重仓股及行业分布 | 言行一致性"做" | §8 | 第 3 章 |
| 持有人结构（机构 vs 个人） | 专业投资者认可度 | §9 | 第 6 章 |

**技术实现**

- PDF 下载：requests 库从巨潮资讯网下载
- PDF 解析：pdfplumber 库提取表格和文本
- 数据清洗：正则表达式匹配关键指标
- 缓存机制：已下载的 PDF 缓存到本地，避免重复下载

### 2.2 R=A+B-C 计算器（核心功能）

**功能描述**

基于基金净值和业绩基准数据，计算收益归因，将收益拆分为 Alpha（超额收益）、Beta（市场收益）、Cost（成本侵蚀）三个部分。

**计算公式**

| 组成部分 | 计算方法 | 说明 |
|---------|---------|------|
| R（总收益） | 基金净值增长率 | 投资者实际获得的收益 |
| B（Beta） | 业绩基准收益率 × 股票仓位 | 市场平均收益部分 |
| A（Alpha） | R - B | 超额收益，需判断结构性 vs 阶段性 |
| C（Cost） | 管理费 + 托管费 + 换手率 × 0.3% | 成本侵蚀，确定性最高 |
| **净超额** | **A - C** | **扣成本后投资者真正多赚了多少** |

**超额收益性质判断（v2.0 新增）**

| 特征 | 结构性超额（可持续） | 阶段性超额（不可持续） |
|------|-------------------|---------------------|
| 多年度为正 | ✅ 通常 4/5 年以上 | ❌ 集中在某 1-2 年 |
| 不同市场环境 | ✅ 牛熊都为正 | ❌ 仅在特定风格顺风时为正 |
| 来源可解释 | ✅ 选股/配置能力 | ❌ 无法解释来源 |
| 与风格因子相关性 | ❌ 低相关性 | ✅ 与特定因子高度相关 |

**数据来源**

- 基金净值序列：天天基金 API 或 akshare
- 业绩基准：基金年报 §2 中披露的业绩比较基准
- 费率数据：从年报提取或使用默认值

### 2.3 基金经理言行一致性检验（v2.0 新增）

**功能描述**

通过年报§4（"说"）和年报§8（"做"）的交叉验证，判断基金经理的投资策略是否清晰、言行是否一致、风格是否稳定。

**检验维度**

| 检验维度 | "说"（§4 管理人报告） | "做"（§8 投资组合报告） | 判定信号 |
|---------|---------------------|---------------------|---------|
| 投资风格 | 宣称的风格定位 | 实际持仓风格 | 🟢/🟡/🔴 |
| 行业偏好 | 看好的行业 | 实际重仓行业 | 🟢/🟡/🔴 |
| 仓位管理 | 宣称的仓位策略 | 实际股票仓位 | 🟢/🟡/🔴 |
| 换手水平 | 宣称的持有周期 | 实际换手率 | 🟢/🟡/🔴 |
| 利益一致 | — | 是否持有本基金 | 🟢/🔴 |

### 2.4 投资者获得感分析（v2.0 新增）

**功能描述**

回答"买了这只基金的人真的赚到钱了吗"。

**核心公式**

```
行为损益 = 投资者实际收益 - 基金产品收益
```

**数据来源**（2026 年新规要求披露）

- 基金产品收益：年报§3 净值增长率
- 投资者实际收益：年报§3 加权平均投资者收益率
- 盈利投资者占比：年报§3
- 份额变动：年报§10（判断追涨/抄底信号）

### 2.5 核心风险与否决项（v2.0 新增）

**功能描述**

借鉴 E大"品种必须不死"理念和 Dayu-Agent 第 9 章"安全阀"设计，提供一票否决机制。

**否决项清单**

| 风险项 | 否决条件 | 数据来源 |
|--------|---------|---------|
| 清盘风险 | 规模 < 5000 万 | 天天基金 |
| 基金经理离职 | 管理本基金时间 < 6 个月 | 年报§2 |
| 风格严重漂移 | 言行一致性检验 🔴 | §4 vs §8 |
| 费率远超同类 | 总成本 > 同类 2 倍中位数 | 年报§2 |
| 跟踪误差过大 | 指数基金跟踪误差 > 2% | 净值 vs 基准 |

**压力测试（借鉴 E大网格策略）**

> "压力测试是最重要的。" —— E大

模拟 -20%/-40%/-60% 三个场景，判断投资者能否承受。

### 2.6 买入前检查清单（独立模块）

**功能描述**

当用户说"我想买 XX 基金"时，自动回答 7 个问题，给出红/黄/绿灯综合建议。

**注意**：检查清单在 v2.0 中从分析报告中独立出来，作为独立模块。分析报告回答"这只基金好不好"，检查清单回答"现在该不该买"（后者需要结合市场温度）。

**7 个检查问题**

| # | 问题 | 判定依据 | 数据来源 |
|---|------|---------|---------|
| 1 | 超额收益能覆盖成本吗？ | A - C > 0 | R=A+B-C 计算器 |
| 2 | 基金经理跟我一条心吗？ | 从业人员持有 > 0 | 年报§9 |
| 3 | 投资者真的赚到钱了吗？ | 盈利投资者占比 > 50% | 年报§3（2026 新规） |
| 4 | 说的和做的一样吗？ | 言行一致性通过 | §4 vs §8 |
| 5 | 这只基金"不死"吗？ | 规模 > 2 亿，成立 > 3 年 | 天天基金 |
| 6 | 当前估值处于什么位置？ | PE/PB 百分位 < 50% | 温度计模块 |
| 7 | 这笔钱 3-4 年内不会用吗？ | 用户自评 | 用户输入 |

**综合判断规则**

| 信号数量 | 判定 | 建议 |
|---------|------|------|
| 🟢 ≥ 5 | 可以买入 | 通过检查，可以执行买入计划 |
| 🟢 3-4 且无 🔴 | 建议观望 | 基金质量尚可，但建议等待更好的时机 |
| 🔴 ≥ 2 | 建议暂缓 | 存在明显问题，建议寻找替代方案 |

**关于有知有行温度计**

检查清单第6题（"当前估值处于什么位置？"）原计划爬取有知有行网站获取温度数据，但：
- 有知有行无官方 API，需网页爬取
- 存在反爬限制和页面结构变更风险

**替代方案（按优先级）**：
1. **AKShare 自行计算**（推荐）：使用 AKShare 获取指数 PE/PB 数据，自行计算历史百分位
2. **优雅降级**：爬取失败时提示"⚠️ 温度数据暂时不可用，请手动确认估值位置"
3. **缓存策略**：本地缓存上次成功获取的温度数据（7天有效期）

MVP 阶段采用方案2+3，v2 阶段实现方案1（自建温度计计算）。

### 4.3 关键设计决策

**PDF 解析策略**

不同基金公司的年报格式存在差异，采用"关键字 + 正则"的策略：

- 在 PDF 文本中搜索关键字（如"管理费率"、"换手率"）
- 在关键字附近提取数值
- 同一数据项可能出现在多个位置，取第一个有效值

**数据缓存策略**

为避免重复下载和解析，采用两级缓存：

- 第一级：PDF 文件缓存到本地 `cache/` 目录
- 第二级：提取的数据缓存到 SQLite 数据库

**温度数据获取策略**

有知有行数据页面无 API，采用爬虫方式获取：

- 每 24 小时最多请求一次（尊重服务器资源）
- 缓存上次获取的温度数据到本地
- 本地无缓存时，使用上次获取的数据

**审计策略（v2.0 新增）**

MVP 阶段实现程序审计（无需 LLM）：

- P1：章节结构完整性检查
- P2：关键字段非空检查
- L1：R=A+B-C 计算闭合性检查
- R1：检查清单规则一致性检查

## 六、风险与应对

| 风险 | 概率 | 应对措施 |
|------|------|---------|
| 巨潮网 PDF 下载失败 | 中 | 使用备用数据源（天天基金网、东方财富） |
| PDF 格式不统一导致解析失败 | 高 | 设计兜底策略：解析失败时显示"数据获取失败"而非崩溃 |
| 有知有行页面结构变更 | 中 | 采用替代方案：AKShare 自行计算 PE/PB 百分位，或提示用户手动确认 |
| 年报披露不及时 | 高 | 优先使用季报数据；无季报时提示用户数据可能不是最新 |
| 2026 新规"投资者收益率"尚未落地 | 中 | 先用份额变动数据估算行为损益，新规落地后切换 |
| 审计规则过严导致频繁阻断 | 低 | MVP 阶段仅启用程序审计（P1/P2/L1/R1），LLM 审计留到 v2 |
