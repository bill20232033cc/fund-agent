## Findings

### F1-MEDIUM-Roster guard 依赖 cell heading_path 且 fallback role 可能取到 heading 文本

- **入口/函数**: `_manager_profile_roster_entry` → `_manager_profile_roster_row_context`
- **文件(行号)**: `fund_agent/fund/processors/fund_disclosure_processor.py:358-414` (context 构建), `:314-318` (fallback role)
- **输入场景**: 表格包含非基金经理行（如研究员、交易员），且 cell.heading_path 继承了包含"基金经理"的更上层 section heading（如"基金管理人及基金经理情况"）
- **实际分支**:
  1. `_manager_profile_roster_row_context` 将 `cell.heading_path` 加入 row_context（行号 409-411）
  2. 若 heading_path 含有"基金经理情况"等文本，`_matches_any_token(row_context, ("基金经理",))` 返回 True（行号 313-314），非经理行通过守卫
  3. 当 role cell 的值为"研究员"（不含"基金经理"）时，行号 315 条件成立，进入 fallback
  4. `_first_non_empty(tuple(text for text in row_context if text and "基金经理" in text))` 取到 heading_path 文本"基金经理情况"作为 role（行号 316-318）
- **预期行为**: 非基金经理行不应通过守卫；若通过，role 不应被设置为 heading 文本
- **实际行为**: 非基金经理行以错误 role（heading 文本）进入 `portfolio_managers`
- **直接证据**: 行号 409-411 无条件将 cell.heading_path 加入 row_context；行号 313-314 对此 context 做 substring 匹配；行号 316-318 对 row_context 全量取首个含"基金经理"文本
- **影响**: 静默产生错误的基金经理记录；下游可能误以为"李四"是基金经理
- **建议改法和验证点**:
  1. row_context 不应包含 cell.heading_path，或至少排除不直接属于该行 cell 值的路径元素
  2. fallback role 不应从 heading path 中取值——若 cell disclosed role 不含"基金经理"，应返回 None 或保持原值
  3. 增加测试：broad heading 表格中非基金经理行（姓名字段有值、职务为"研究员"、heading_path 含"基金经理情况"）应被过滤
- **修复风险（低）**: 修改 row_context 构建逻辑，移除 cell.heading_path；或过滤 row_context 中与 table heading 重复的文本
- **严重程度（中）**

### F2-MEDIUM-Strategy 段落匹配同时检查 body text 产生内容敏感误判风险

- **入口/函数**: `_manager_profile_paragraph_matches_strategy` → `_matches_any_token`
- **文件(行号)**: `fund_agent/fund/processors/fund_disclosure_processor.py:559-583`
- **输入场景**: 某 paragraph 的 heading_path 为（"其他重要事项",），其 body text 包含"本基金将坚持既定的投资策略"
- **实际分支**: `_matches_any_token` 对 `paragraph.text_normalized` 做 substring 匹配，"投资策略" 命中 `_MANAGER_PROFILE_STRATEGY_HEADINGS` 中的 "投资策略" token，返回 True
- **预期行为**: 计划 5.2 节要求 paragraph 位于 strategy/outlook heading **之下**（"under"）；heading_path 应是主判断依据
- **实际行为**: body text 中包含 strategy token 的无关 paragraph 可能被误分类为 strategy text
- **直接证据**: 行号 576-583：`_matches_any_token` 的输入同时包含 `paragraph.heading_path`、`paragraph.text_normalized`、`paragraph.text_raw`
- **影响**: strategy 段落集合可能混入非 strategy section 的正文，导致 `strategy_summary` / `market_outlook` 内容不准确
- **建议改法和验证点**:
  1. 考虑仅用 heading_path 判断段落归属，body text 仅用于取值而非归属判断
  2. 如果 body text 匹配是 fallback for missing heading_path，应显式区分缺失情况
  3. 增加测试：paragraph heading_path 不含 strategy tokens 但 body text 含有时不被抽取
- **修复风险（中）**: 仅用 heading_path 会降低 strategy paragraph 召回率；需确认 FDD 协议中 heading_path 的覆盖率
- **严重程度（中）**

- **broader heading 守卫缺失测试**: 当前无测试覆盖 broad heading（"主要人员情况"）+ 非经理行的组合场景
- **roster conflict 路径无测试**: `_select_manager_profile_portfolio_managers` 中 `conflicted_names` 路径未被 Slice 2 测试直接覆盖
- **`start_date` 为 None 时仍写入 entry**: 计划 4 节要求"disclosed start date text when present"；当前实现始终写入 `"start_date": start_date`（可为 None），与 `end_date` 的显式 `if end_date:` 不对称。无功能影响，但语义不够精确
