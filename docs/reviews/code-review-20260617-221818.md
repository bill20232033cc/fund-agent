### 001-未修复-中-column_header flag can turn non-header row 0 into a derived header

- **入口/函数**: `_match_performance_field` / `_match_portfolio_managers` / `_match_holding_row` -> `_derive_table_context` -> `_is_header_like_row`
- **文件(行号)**: `fund_agent/fund/documents/candidates/template_field_extraction.py:1028`
- **输入场景**: A table row has at least two cells with `column_header=True` but the cell text does not contain known header labels. This can occur in current synthetic builders and can also occur if a parser marks first-row cells structurally without semantic labels.
- **实际分支**: `_is_header_like_row` increments `hits` when `_contains_keyword(text, _TABLE_HEADER_LABELS) or cell.column_header` is true.
- **预期行为**: The accepted plan says derived labels should come from header-like rows whose non-empty cells contain known table-header terms, and should fail closed without deterministic textual header context.
- **实际行为**: A row can be accepted as a header solely because multiple cells have `column_header=True`; `_derive_table_context` then maps arbitrary row text into `derived_column_labels_by_column`.
- **直接证据**: Lines `1041-1046` count `cell.column_header` as a full header hit; line `1012` stores the row text as a derived column label.
- **影响**: False candidate extraction or missed rows when row 0 is a title/data row marked structurally as column header. This violates the no-position-guessing boundary and can make coverage uplift harder to interpret.
- **建议改法和验证点**: Remove `cell.column_header` from `_is_header_like_row` evidence. Add or rely on existing no-header negative coverage to prove structural position alone is insufficient.
- **修复风险（低/中/高）**: 低
