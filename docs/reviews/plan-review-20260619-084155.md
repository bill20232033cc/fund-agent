# Plan Review: FundDisclosureDocument S6-A Candidate Evidence Contract Plan

## Reviewed Target And Scope

- Target plan: `docs/reviews/funddisclosuredocument-s6a-candidate-evidence-contract-plan-20260619.md`
- Scope reviewed: candidate evidence contract, admission/content protocol split, public `EvidenceAnchor` avoidance, field-family status semantics, facade projection boundary, write set, validation.

## Assumptions Tested

- A separate content protocol can avoid overcoupling the admission protocol.
- Candidate-only evidence can be carried on `FundFieldFamilyResult` without satisfying public `partial` / `accepted` status.
- Avoiding `value` and public `EvidenceAnchor` is sufficient to prevent `_active_processor_result_to_bundle()` leakage.
- The implementation write set is narrow enough for a heavy contract slice.
- Tests are specific enough to prevent implementation from reintroducing the blocked S6 plan risks.

## Findings

No material findings.

The plan directly addresses the blockers from `docs/reviews/plan-review-20260619-083944.md`:

- It keeps `FundDisclosureDocumentIntermediate` admission-only.
- It introduces a separate content protocol instead of widening admission.
- It avoids public `EvidenceAnchor` for candidate-only evidence.
- It keeps candidate evidence from satisfying `partial` / `accepted`.
- It keeps facade projection out of scope.
- It defers selector extraction to a later single-family mapping gate.

## Open Questions

- Should `locator_stability` validate against the existing closed locator-stability values, or is non-empty string validation sufficient for this contract slice?
- Should `FundCandidateEvidenceRecord.source_boundary` reuse the existing `FundExtractionSourceBoundary` literal type or remain a narrower `Literal["candidate_only"]` field?
- Should S6-A implementation add a regression test directly against `_field_from_family()` only if a leakage path is discovered, or keep all validation in processor contract tests as planned?

## Residual Risks And Suggested Tracking

- Implementation must not put candidate evidence into `FundFieldFamilyResult.value`; track via a contract test that `candidate_evidence` is separate from `value`.
- Implementation must preserve the existing `partial` anchor invariant; track via the planned negative `partial` test.
- S6-B must not broaden back to six-family keyword extraction without a reviewed mapping table; track in the next S6-B plan gate.
- Documentation sync may expose current design-doc staleness around S5/S6; track any nontrivial design inconsistency as a separate truth-sync residual if it exceeds S6-A scope.

## Final Plan Review Conclusion

`pass-with-risks`

The S6-A plan is code-generation-ready for a contract-only implementation gate. It should not be expanded to selector extraction or facade projection during implementation.
