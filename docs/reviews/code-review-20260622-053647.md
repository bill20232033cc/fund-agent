## Findings

### 001-未修复-[中]-projection V2 结果没有按计划要求以 chapter_id 为第一排序键
- **入口/函数**: `confirm_projection_evidence_v2`
- **文件(行号)**: `fund_agent/fund/evidence_confirm.py:408`, `fund_agent/fund/evidence_confirm.py:430`, `fund_agent/fund/evidence_confirm.py:442`; `tests/fund/test_evidence_confirm.py:895`; `docs/reviews/evidence-confirm-scoring-v2-plan-20260621.md:149`
- **输入场景**: 调用方传入包含多个章节的 `ChapterFactProjection`，且不同章节的 `source_field_id` / `fact_id` 字典序不能自然保持章节顺序。
- **实际分支**: 函数先按 `chapter.chapter_id` 遍历章节，再把所有 `fact_results` 用 `sorted(fact_results, key=lambda item: (item.source_field_id, item.fact_id))` 全局重排。
- **预期行为**: 计划明确要求 `confirm_projection_evidence_v2` 的 fact 结果按 `chapter_id`, `source_field_id`, `fact_id` 稳定排序。
- **实际行为**: V2 result schema 没有保留 `chapter_id` 字段，且最终返回顺序只以 `source_field_id`, `fact_id` 排序；测试也断言了这个二元排序，未覆盖 `chapter_id` 优先级。
- **直接证据**: 计划在 `docs/reviews/evidence-confirm-scoring-v2-plan-20260621.md:149-153` 规定 projection V2 稳定排序键为 `chapter_id`, `source_field_id`, `fact_id`。实现虽然在 `fund_agent/fund/evidence_confirm.py:430-432` 按 chapter 遍历，但 `fund_agent/fund/evidence_confirm.py:442-443` 用二元 key 重新排序；`EvidenceConfirmFactResultV2` 在 `fund_agent/fund/evidence_confirm.py:225-247` 不包含 `chapter_id`，返回后无法由结构化字段恢复计划要求的章节优先顺序。新增测试在 `tests/fund/test_evidence_confirm.py:895-911` 只断言二元排序，正好固化了偏离计划的行为。
- **影响**: V2 public contract 与 accepted plan 不一致；跨章节 projection 的 fact result 顺序可能被来源字段字典序重排，后续审计/桥接流程若按顺序展示或比较结果，会得到非章节主序的输出。这个问题不会破坏单 fact scoring，但会破坏 projection 聚合的可预测 API 语义。
- **建议改法和验证点**: 在 projection 聚合时保留 `(chapter_id, fact_result)` 对并按 `(chapter_id, source_field_id, fact_id)` 生成最终 tuple，或在 V2 fact result schema 中显式加入 `chapter_id` 后按三元 key 排序。补一个多章节测试，构造会被 `source_field_id` 字典序跨章重排的 projection，并断言返回顺序的第一排序键是 `chapter_id`。
