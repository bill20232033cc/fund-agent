# Plan Review - RR-09 A5 Projection Locator Adoption Plan

Reviewed target: `docs/reviews/evidence-confirm-productionization-rr-09-a5-projection-locator-adoption-plan-20260624.md`

Scope: A5 no-live plan for projection locator adoption and R3 `missing_section=3` residual disposition.

Conclusion: `fail`

## Assumptions Tested

- A5 can safely change `_field_from_family()` without expanding public schema.
- Processor field-family anchors can be filtered by `row_locator` `field=` path.
- Empty anchors are safe when no compatible `field=` locator exists.
- The plan remains bounded to FDD/proof-positive projection and does not regress other processor paths.
- R3 missing-section residual remains separate from locator adoption.

## Findings

### PR-001-未修复-高-无 field= locator 时空 anchors 的规则会破坏非 FDD processor family projection

- **位置**: `Implementation Decisions / Legacy compatibility boundary`; `A5-S1: Field-specific processor anchor adoption`
- **问题类型**: 范围漂移 / 架构边界 / 测试缺口 / 不可直接实施
- **当前写法**: Plan says if `_field_from_family()` sees a family result whose anchors have no Processor-style `field=` locator, tests should make this explicit, and the preferred fail-closed behavior for proof-positive processor family fields is empty anchors.
- **反例/失败场景**: `FundDataExtractor._processor_result_to_bundle()` is not limited to FDD source-truth results. It also consumes processor-family outputs from other processor paths. `fund_agent/fund/processors/active_annual.py` maps legacy extractor fields into `return_attribution.v1` and `manager_profile.v1`; those anchors can carry legacy row locators such as `management_fee`, `nav_growth_rate` or `manager_holding`, not semicolon `field=` locators. A global `_field_from_family()` change that empties anchors when no `field=` path exists would turn previously evidenced processor output into `evidence_missing` projection, outside the A5 target.
- **为什么有问题**: A5's stated target is current FDD/Processor row locator adoption for RR-09 residuals, not a behavior change for every processor family projection. The plan's filtering rule is too broad for the actual call path because `_field_from_family()` is shared by all `FundProcessorResult` projections.
- **直接证据**:
  - `fund_agent/fund/data_extractor.py:588-617` assigns `family_result.anchors` in the shared `_field_from_family()` helper.
  - `fund_agent/fund/data_extractor.py:706-789` projects all processor family results through `_field_from_family()`.
  - `fund_agent/fund/processors/active_annual.py:16-23` documents processor mappings from legacy extractor outputs into `return_attribution.v1` and `manager_profile.v1`.
  - Existing legacy extractor anchors use field-name locators rather than Processor `field=` paths, for example `fund_agent/fund/extractors/profile.py:748-755`, `fund_agent/fund/extractors/performance.py:1020-1027`, and `fund_agent/fund/extractors/manager_ownership.py:731-738`.
- **影响**: Implementation agent could make a shared projection change that silently strips valid anchors from non-FDD processor outputs, creating new evidence gaps and unrelated regressions while trying to fix RR-09.
- **建议改法和验证点**:
  - Narrow the plan: field-specific filtering applies only when the family anchor set contains recognized semicolon `field=` Processor locator(s), or when a local helper can classify the family as FDD field-locator capable.
  - If no recognized `field=` locators exist in a family, preserve the existing `family_result.anchors` behavior unless a later gate explicitly authorizes a fail-closed change for that processor path.
  - Add tests proving legacy/no-`field=` processor family fixtures keep current anchors.
  - Add tests proving mixed FDD field-locator families filter to field-compatible anchors.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

## Open Questions

- Does runtime R1-R4 currently use FDD field-locator capable family results, legacy active annual processor results, or a mixed path? The plan correctly requires no-live diagnostics before live re-evidence, but it must not assume all processor families are FDD locator-capable.

## Residual Risks

| Risk | Suggested tracking destination |
| --- | --- |
| Runtime may still have `processor_row_locator_rows=0` after FDD-only filtering because the active path uses legacy locators. | A5-S0 diagnostic artifact and later live/PDF authorization precheck. |
| R3 `missing_section=3` remains unresolved. | A5-S2 or separate R3 missing-section diagnostic gate. |

## Final Conclusion

`fail`

The plan is directionally sound but not yet implementation-ready because its proposed fail-closed filtering rule is too broad for the shared `_field_from_family()` call path. Fix the plan to preserve non-FDD/no-`field=` processor family anchor behavior and add explicit regression tests for that boundary.
