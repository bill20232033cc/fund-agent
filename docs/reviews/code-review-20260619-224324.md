## Findings

### 1-未修复-中-`_validate_source_truth_admission` 对非 content intermediate 静默放行，source-truth 诊断缺口不可见

- **入口/函数**: `FundDisclosureDocumentProcessor.extract()` → `_validate_source_truth_admission()`
- **文件(行号)**: `fund_disclosure_processor.py:704-706`
- **输入场景**: 非 content 的 `FundDisclosureDocumentIntermediate`（无 sections/paragraphs/table_blocks），`candidate_boundary=None`，无 failure，有 provenance，但没有任何 source-truth proof
- **实际分支**: `_content_intermediate_or_none(intermediate)` 返回 `None` → `_validate_source_truth_admission` 直接返回 `None`（无 gap）
- **预期行为**: 按 plan 要求 "Proof-missing or proof-invalid non-candidate inputs return public field families as missing, with no public anchors and no public values"，公开行为满足（field families 确实 missing）。但从诊断角度，plan 定义了 `source_truth_admission_missing` gap 应在 proof 缺失时出现
- **实际行为**: 非 content intermediate 的字段族 gap 为 `field_family_missing`（通用缺省），而非更精确的 `source_truth_admission_missing`。消费者无法区分"确实没有内容"和"有内容但缺 proof"两种情形
- **直接证据**:
  - `fund_disclosure_processor.py:704-706`: `content_intermediate = _content_intermediate_or_none(intermediate); if content_intermediate is None: return None`
  - `fund_disclosure_processor.py:678-679`: `if source_truth_gap_code is None: return field_families` — 无 gap 追加
  - 对比 `fund_disclosure_processor.py:708-711`: content intermediate 无 proof 时正确返回 `"source_truth_admission_missing"`
- **影响**: 诊断缺口。非 content FDD intermediate 即使 candidate_boundary=None 也无从得知 source-truth 校验是否执行过。未来 Slice B 上线后，非 content intermediate 永远无法通过 proof（因其协议不允许携带 proof），但也不会报 source-truth 级别 gap，诊断信息不完整
- **建议改法和验证点**:
  1. 在 plan/controller 层面确认：非 content FDD intermediate 是否应被视为"隐式 proof-missing"并追加 `source_truth_admission_missing` gap
  2. 若确认需要，将 `_content_intermediate_or_none is None` 的 early return 改为返回 `"source_truth_admission_missing"` 而非 `None`
  3. 回归测试：非 content intermediate 的 gap codes 应包含 `source_truth_admission_missing`
- **修复风险（低）**: 仅改变 gap code 附加行为，不改变 public value/anchor/status 的 fail-closed 语义
- **严重程度（中）**: 不影响正确性（字段族仍是 missing），但诊断不可见可能在未来 Slice B 联调时造成排查困难

### 2-未修复-低-`test_source_truth_admission_accepts_repository_loaded_identity_proof` 未验证 candidate evidence 在 proof-positive 状态下共存

- **入口/函数**: `test_source_truth_admission_accepts_repository_loaded_identity_proof`
- **文件(行号)**: `test_fund_disclosure_processor.py:411-449`
- **输入场景**: proof-positive content stub（默认 section/paragraph/table 含"基金简介""基金名称"文本，应触发 candidate evidence）
- **实际分支**: 测试断言 `"source_truth_admission_missing" not in _gap_codes(product)` 和 `"source_truth_admission_invalid" not in _gap_codes(product)`，但未检查 `product.candidate_evidence`
- **预期行为**: proof-positive 状态下 candidate evidence 应仍被收集（作为 S6 diagnostic），不被 source-truth proof 抹掉或提升
- **实际行为**: candidate evidence 大概率存在（stub 默认内容会匹配），但测试未显式验证，形成断言盲区
- **直接证据**: `test_fund_disclosure_processor.py:445-449` 只做 gap code 缺失断言，不做 candidate_evidence 存在断言。对比 `test_source_truth_admission_requires_positive_proof:375` 正确断言了 `assert product.candidate_evidence`
- **影响**: 若未来重构导致 proof-positive 状态下 candidate evidence 被错误清空，此测试不会捕获
- **建议改法和验证点**: 在 `test_source_truth_admission_accepts_repository_loaded_identity_proof` 末尾增加 `assert product.candidate_evidence`
- **修复风险（低）**: 单行断言新增
- **严重程度（低）**: 仅测试覆盖缺口，不反映生产代码缺陷

### 3-未修复-低-`_with_source_truth_admission_gap` 对所有六个字段族统一追加 source-truth gap，跨字段语义略宽

- **入口/函数**: `_field_families_for_intermediate()` → `_with_source_truth_admission_gap()`
- **文件(行号)**: `fund_disclosure_processor.py:678-683`
- **输入场景**: 任意 content FDD intermediate proof 缺失或非法
- **实际分支**: `source_truth_gap_code is not None` → 对所有六个 `field_families` tuple 逐项调用 `_with_source_truth_admission_gap`
- **预期行为**: source-truth proof 缺失应阻止所有字段族产出 public value，六族全加 gap 语义正确
- **实际行为**: 六族各得一条 `source_truth_admission_missing` gap，与各自已有的 `candidate_only_not_source_truth` 或 `field_family_missing` gap 并存。behavior 正确，但六条相同 message 的 gap 在 result 中造成信息冗余
- **直接证据**: `fund_disclosure_processor.py:680-683` 的 tuple comprehension 对所有 family 无差别调用
- **影响**: 无功能影响。仅在 result 可读性和未来 result-level gap 聚合时有轻微冗余
- **建议改法和验证点**: 可考虑将 source-truth admission gap 提升为 result-level 跨字段 gap（`field_family_id=None`），而非逐族重复。需 controller 确认后再改
- **修复风险（低）**: 仅改变 gap 归属层级，不改变 fail-closed 语义
- **严重程度（低）**: 纯结构偏好，不影响正确性

## 逐项验证报告

以下按用户指定的五个审查重点逐项报告验证结论：

### 1. `candidate_boundary=None` 不能成为 source truth proof

**结论：通过。**

- `_validate_source_truth_admission` (line 708-709) 显式检查 `proof is None` → 返回 `source_truth_admission_missing`
- `test_source_truth_admission_requires_positive_proof` 验证：`candidate_boundary=None` + `source_truth_admission=None` → `value == {}`, `anchors == ()`, `"source_truth_admission_missing" in gaps`
- 正向 proof 数据类 `FundDisclosureSourceTruthAdmissionProof` (contracts.py:189-261) 要求四个布尔字段均为 `Literal[True]`，`__post_init__` 强制校验
- `candidate_boundary is None` 是必要条件但非充分条件，plan 要求完全满足

### 2. proof missing/invalid 必须 fail closed 且无 public value/anchors

**结论：通过。**

- proof missing → `_with_source_truth_admission_gap` (line 2794-2800) 强制 `value={}`, `status="missing"`, `extraction_mode="missing"`, `anchors=()`
- proof invalid (identity mismatch) → 同样路径，`test_source_truth_admission_rejects_identity_mismatch` 验证 `fund_code="110011"` mismatch → `value == {}`, `anchors == ()`
- proof 类型非法 (`not isinstance(proof, FundDisclosureSourceTruthAdmissionProof)`) → `"source_truth_admission_invalid"` (line 711-712)
- `candidate_boundary is not None` + proof present → `"source_truth_admission_invalid"` (line 713-714)，双重保险
- `failure_class is not None` + proof present → `"source_truth_admission_invalid"` (line 715-716)
- `source_provenance is None` + proof present → `"source_truth_admission_invalid"` (line 717-718)

### 3. candidate evidence diagnostic 是否被错误抹掉或提升

**结论：通过。**

- `_with_source_truth_admission_gap` (line 2813) 保留 `candidate_evidence=family.candidate_evidence`，不抹除
- proof-missing 测试 (line 375) 断言 `assert product.candidate_evidence` — candidate evidence 存在
- candidate evidence 的 `source_boundary="candidate_only"` 不变，`field_correctness_status="not_proven"` 不变
- candidate evidence 未被复制到 `anchors` 或 `value`，仅保留在 `candidate_evidence` tuple
- candidate-boundary blocked 测试 (line 696-732) 独立验证：`candidate_evidence` 存在但 `value=={}` `anchors==()` `contract_status=="blocked"`

### 4. candidate-boundary blocked、default parsed route、non-active explicit FDD 是否回归

**结论：通过，无回归。**

- **candidate-boundary blocked**: `admit_disclosure_intermediate` (dispatch.py:119-125) 对 `candidate_boundary is not None` 返回 `contract_status="blocked"`。Processor (line 531) 仅在 `contract_status != "blocked"` 时调用 `_validate_source_truth_admission`。`test_extract_admits_candidate_boundary_but_returns_blocked` (line 4593) 和六个 `test_*_selector_preserves_candidate_boundary_blocked_status` 测试全部通过
- **default parsed route**: `FundDisclosureDocumentProcessor.supports()` (line 441-446) 仅匹配 `intermediate_kind == "fund_disclosure_document.v1"`，不拦截 `parsed_annual_report.v1`。`test_supports_rejects_parsed_annual_report_v1` (line 4286) 通过
- **non-active explicit FDD**: `supports()` 要求 `fund_type == "active_fund"`。`test_supports_rejects_non_active_fund_type` (line 4304) 通过

### 5. 是否越权实现 Slice B 或扩 EvidenceSourceKind

**结论：通过，无越权。**

- 无 `product_essence.v1` public value 提取代码。所有字段族仍返回 `value={}`, `status="missing"`
- 无 label map、table/cell rule、paragraph fallback、anchor rule 等 Slice B 逻辑
- `EvidenceSourceKind` 未被修改。新增的 `source_truth_unverified` 是 `FundExtractionSourceBoundary`（内部 gap 边界），非 `EvidenceSourceKind`
- plan 显式授权新增 `FundExtractionGapCode`: `source_truth_admission_missing`, `source_truth_admission_invalid` 和 `FundExtractionSourceBoundary`: `source_truth_unverified`（plan 第 173-183 行）
- 新增 `FundDisclosureSourceTruthAdmissionProof.source_kind` 为 `Literal["annual_report"]`，使用已有 source kind，未扩展
