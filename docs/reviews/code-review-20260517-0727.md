## Findings

### 1-未修复-高-同步阻塞 I/O 在异步调用链中未隔离

- **入口/函数**: `AnnualReportPdfAdapter.load_annual_report` → `self._text_extractor(pdf_path)` / `self._table_extractor(pdf_path)`; `download_annual_report` → `_find_annual_report_id`
- **文件(行号)**:
  - `fund_agent/fund/documents/adapters/annual_report_pdf.py:172-173` — `extract_text` 与 `extract_tables` 在 async 协程内同步调用 `pdfplumber.open()`
  - `fund_agent/fund/pdf/downloader.py:125` — `_find_annual_report_id` 在 async 协程内同步调用 `ak.fund_announcement_report_em()`
- **输入场景**: 任意 `load_annual_report("110011", 2024)` 调用
- **实际分支**: 执行到 `raw_text = self._text_extractor(pdf_path)` 时，当前事件循环线程被 `pdfplumber.open()` 的磁盘 I/O 阻塞；执行到 `report_id = _find_annual_report_id(...)` 时，被 akshare 网络 I/O 阻塞
- **预期行为**: 异步协程内不应执行可能耗时的同步阻塞 I/O；应通过 `asyncio.to_thread` 或 `loop.run_in_executor` 将阻塞操作卸载到线程池
- **实际行为**: pdfplumber 文件读取与 akshare HTTP 查询直接阻塞事件循环，多协程并发时所有其他协程被暂停
- **直接证据**: adapter:172 调用 `self._text_extractor(pdf_path)`，`text_extractor` 默认值为 `extract_text`（parser.py:31），其内部 `pdfplumber.open(pdf_path)` 为同步文件 I/O。`_find_annual_report_id`（downloader.py:32）调用 `ak.fund_announcement_report_em()` 是 akshare 的同步 HTTP 查询。两者均未使用 `run_in_executor`
- **影响**: 在异步 runtime（如 FastAPI、asyncio 应用）中，单次年报加载会暂停整个事件循环数百毫秒至数秒；多个并发请求时延迟线性叠加，退化为串行
- **建议改法和验证点**:
  - `_find_annual_report_id` 需包装为 `await asyncio.to_thread(_find_annual_report_id, fund_code, year)` 或在 `download_annual_report` 内用 `loop.run_in_executor` 调用
  - `extract_text` / `extract_tables` 的阻塞问题属于 parser.py 边界，P1-S1 禁止修改 parser.py，但适配器可通过 `asyncio.to_thread` 包装调用，无需改动 parser.py
  - 验证：用 `pytest-asyncio` 的 `event_loop` fixture + 模拟耗时操作，确认协程不被阻塞
- **修复风险（中）**: 线程池包装需确保 pdfplumber 对象可跨线程安全使用；aksShare 的 pandas DataFrame 操作在 GIL 下安全
- **严重程度（高）**: 静默性能退化，在异步运行时下从并发退化为串行，但不会导致错误结果

### 2-未修复-中-年报年份查找 fallback 静默返回错误年份数据

- **入口/函数**: `_find_annual_report_id` → `download_annual_report` → adapter → repository
- **文件(行号)**: `fund_agent/fund/pdf/downloader.py:41-42`
- **输入场景**: 请求某基金 2020 年年报，但该基金公告列表中无标题含 "2020" 的年度报告（如基金 2021 年才成立），但存在其他年份年报
- **实际分支**: `for` 循环未找到匹配 → 进入 fallback 分支 `if len(annual) > 0: return str(annual.iloc[-1]["报告ID"])`
- **预期行为**: 找不到指定年份时应返回 `None`，由上层抛出 `FileNotFoundError`
- **实际行为**: 返回最新一份年报的 ID（可能是 2024 年），上层 `download_annual_report` 用该 ID 下载并返回。最终 `DocumentKey` 中 `year=2020` 但 `raw_text` 内容是 2024 年年报，年份与内容不匹配
- **直接证据**: downloader.py:35-42，`for` 循环内 `if str(year) in title` 不命中时未 return，流程穿透到 line 41 的 fallback return
- **影响**: 静默数据错配 — 上层后续分析将 2024 年数据当作 2020 年数据消费，无任何异常信号
- **建议改法和验证点**: 移除 fallback 或将其改为仅在 `year` 参数未显式传入时生效；若 year 由调用方显式传入（repository 已做），fallback 不应触发。至少应在 fallback 路径上 emit warning 日志
- **修复风险（低）**: 仅删除或条件化 3 行代码；需同步更新测试覆盖 `None` 返回路径
- **严重程度（中）**: 数据正确性问题，但在 P1-S1 阶段样本基金均有对应年份年报，触发概率取决于调用方传入的年份是否存在于公告列表

### 3-未修复-中-章节置信度硬编码为 1.0 掩盖解析不可靠性

- **入口/函数**: `_build_sections` → `AnnualReportPdfAdapter.load_annual_report`
- **文件(行号)**: `fund_agent/fund/documents/adapters/annual_report_pdf.py:29` (`_SECTION_CONFIDENCE = 1.0`)，使用于 line 75
- **输入场景**: parser 的 `locate_sections` 返回章节偏移（已知 BQ-5: `110011` 的 `§3` 存在定位问题）
- **实际分支**: 所有 `ReportSection` 对象均以 `confidence=1.0` 构建，无论 parser 匹配质量
- **预期行为**: confidence 应反映 parser 对每个章节定位的可信度；目前 parser 用纯正则匹配且已知 `§3` 有漏判，1.0 会误导下游依赖
- **实际行为**: 所有章节声明为完全可信，下游无法区分可靠定位与可能误判
- **直接证据**: adapter.py:29 定义常量，line 75 无条件传入每个 section
- **影响**: 下游模板分析（如 P1-S5 投资者收益率提取）可能基于错误定位的章节做数据提取，且无信号表明数据可能不可靠
- **建议改法和验证点**: P1-S1 不要求修复 parser，但至少应在 adapter 层区分 confidence：如 parser 匹配到章节用 0.8，fallback/推导用更低值。P1-S2 修复 `§3` 后可回升。当前最低成本修复：将常量降为 0.8 并加注释说明 P1-S2 之前为临时值
- **修复风险（低）**: 常量修改 + 注释
- **严重程度（中）**: 下游信任一个不准确的信号，但 P1-S2 即为章节定位修复，影响窗口限于 S1→S2 之间
