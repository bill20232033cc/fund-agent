**1. Type dispatch / value construction**

`_build_core_risk_value` (line 5061) 用 `isinstance` 检查区分 `_RiskCharacteristicValueCandidate` 和 `_CoreRiskRoleValueCandidate`。`_select_core_risk_values` (line 5035-5039) 确保 `risk_values` 只放入 `_RISK_CHARACTERISTIC_OUTPUT_PATH` 键下，`role_values` 只放入 role key 键下。类型一致性由构造路径保证，`isinstance` 是防御性检查。无 silent-drop 风险。

**2. Numeric body cell authorization**

`_collect_core_risk_role_cell_candidates` (line 4844-4855) 对含 ASCII 数字的 body cell 扩展 `candidate_texts` 以包含 `table.heading_text`、`table.table_caption_or_nearby_heading`、`cell.row_label_path`、`cell.column_header_path`。`_matches_guarded_core_risk_source` (line 8778-8782) 仍要求 strong token 或 generic token + guard context 命中。扩展后的 `candidate_texts` 不包含 cell 自身文本以外的 guard context 来源，guard context 仍通过独立的 `_core_risk_cell_guard_context` 传递。无 guard-context 自引用循环。

**3. Heading-only rejection for numeric values**

`_is_core_risk_role_heading_only` (line 4922-4924) 对短文本（<15 字符）且不含 ASCII 数字时返回 True。数值如 `180%`、`45%`、`5000万元` 含数字，不被拒绝。强 token 文本如 `换手率` 无数字且 <15 字符，被正确拒绝。无误放行或误拒绝。

**4. Ambiguity detection**

`_resolve_core_risk_role_candidate` (line 4976-4977) 用 `_normalize_match_text` 比较所有候选的规范化文本。完全相同的文本（如两个段落披露相同内容）取第一个；不同文本标记为 ambiguous。测试 `test_core_risk_source_truth_identical_role_text_resolves_first` 和 `test_core_risk_source_truth_role_ambiguity_omits_role` 覆盖了两种路径。

**5. Gap/status consistency**

- `_core_risk_status` (line 5174): value 为空 → `missing`；有值但不完整 → `partial`；全部五个 required subvalue 存在 → `accepted`。
- `_core_risk_source_truth_gaps` (line 5132-5157): value 为空且无 ambiguity → `field_family_missing`；有值但缺失 required top-level → `field_family_partial`；有 ambiguity → `ambiguous_table_or_locator`。
- `extraction_mode` (line 1241): `status == "missing"` 时设为 `"missing"`，否则 `"direct"`。
- 三者一致。status 为 `missing` 时 value 为空、gaps 包含 `field_family_missing`、extraction_mode 为 `"missing"`。

**6. Anchor collection**

`_extract_core_risk_source_truth` (line 1232-1235) 用 `_core_risk_emitted_output_paths` 获取实际进入 value 的 output paths，然后从 `selected_values` 中提取对应 anchor。`_core_risk_emitted_output_paths` (line 5092-5099) 同时检查 value 键和 selected_values 键存在。role subvalue 不内嵌 source_anchors；public anchors 只在 `FundFieldFamilyResult.anchors` 中。

**7. Proof-positive admission**

`_extract_core_risk_source_truth` 在 `_build_field_families` 中调用，后者要求 `_validate_source_truth_admission()` 返回 None。该验证函数要求 `FundDisclosureSourceTruthAdmissionProof`、`candidate_boundary is None`、`failure_class is None`、`source_provenance is not None`。Proof-missing/invalid 路径不进入 `_extract_core_risk_source_truth`。proof-positive direct path 的 `candidate_evidence=()` 正确抑制了 candidate evidence。

**8. Facade / bundle no-projection**

`data_extractor.py` 的 `_active_processor_result_to_bundle()` 仅在 `product_essence.v1` 缺少 `risk_characteristic_text` 时从 `core_risk.v1` fallback。不添加 `StructuredFundDataBundle.core_risk`，不投影 role keys 到 `tracking_error`、`turnover_rate`、`holdings_snapshot` 或 bundle 属性。测试 `test_explicit_disclosure_core_risk_role_keys_not_projected_to_bundle` 覆盖。

**9. Guard token specificity**

`_CORE_RISK_MATCH_GROUPS` 中每个 role 的 strong/generic/guard tokens 已检查：
- `liquidation_or_scale_risk`: strong tokens 如 `基金资产净值低于五千万元` 高度具体；generic `规模` 需 guard 如 `基金合同终止`。
- `tracking_error_or_deviation_risk`: strong `跟踪误差` 具体；generic `跟踪` 需 guard 如 `业绩比较基准`。
- `turnover_or_style_drift_risk`: strong `换手率` 具体；generic `换手` 需 guard 如 `投资策略`。
- `concentration_risk`: strong `持仓集中度` 具体；generic `持仓` 需 guard 如 `股票投资组合`。

无低特异性 strong token 导致误匹配风险。
