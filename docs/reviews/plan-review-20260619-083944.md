# Plan Review: FundDisclosureDocument S6 Field-family Extraction Plan

## Reviewed Target And Scope

- Target plan: `docs/reviews/funddisclosuredocument-s6-field-family-extraction-plan-20260619.md`
- Scope reviewed: S6 field-family extraction planning, processor contract changes, candidate-boundary semantics, public anchor strategy, facade leak guard, validation matrix.
- Source facts checked:
  - `FundFieldFamilyResult` requires anchors for `accepted`/`partial` families.
  - `FundProcessorResult` has no explicit invariant defining whether `contract_status="blocked"` may contain `partial` field families.
  - `admit_disclosure_intermediate()` currently maps non-`None` `candidate_boundary` to admitted but `contract_status="blocked"`.
  - `EvidenceAnchor.source_kind` is limited to `annual_report`, `external_api`, and `derived`.

## Assumptions Tested

- Candidate-only field-family evidence can safely be represented as public `FundFieldFamilyResult(partial)` under result-level `blocked`.
- Public `EvidenceAnchor(source_kind="annual_report")` plus note text is sufficient to prevent candidate-only evidence from being mistaken for proven annual-report facts.
- Extending `FundDisclosureDocumentIntermediate` with full content collections is the right contract boundary.
- A six-family keyword selector plan is specific enough for implementation without reopening design.
- Keeping `FundDataExtractor` fail-closed is enough to prevent candidate-only leakage.

### 001-未修复-高-blocked result with partial field families has no accepted contract semantics
- **位置**: plan §7 lines 105-108; `fund_agent/fund/processors/contracts.py` `FundFieldFamilyResult` / `FundProcessorResult`
- **问题类型**: 契约缺失 / 状态机漏洞 / 不可直接实施
- **当前写法**: plan allows direct processor results to contain `partial` field families while preserving result-level `contract_status="blocked"` for candidate-boundary input.
- **反例/失败场景**: Implementation emits `contract_status="blocked"` with useful `partial` families. One caller treats `blocked` as "do not consume"; another direct harness reads `field_families` because they are present. Later code may aggregate `result.anchors` or family values without checking result status, creating hidden candidate-only consumption.
- **为什么有问题**: Current `FundProcessorResult` validates only that result-level gaps are cross-family; it does not define whether blocked results may carry non-missing families, whether anchors should be aggregated, or whether blocked means "no usable output" versus "not consumable by facade but inspectable by harness".
- **直接证据**: plan lines 105-108 propose `partial` families under `contract_status="blocked"`; `contracts.py` lines 395-433 define result fields but no invariant for this mixed state; `fund_disclosure_dispatch.py` lines 119-125 maps candidate boundary to `contract_status="blocked"`.
- **影响**: Implementation agent must invent state semantics; review cannot mechanically verify consumer safety; future consumers may accidentally treat blocked candidate evidence as structured facts.
- **建议改法和验证点**: Before implementation, add an explicit contract decision. Safer options: keep blocked results fully missing; or introduce a separate internal candidate-evidence container not exposed as normal `FundFieldFamilyResult`; or amend `FundProcessorResult` docs/tests with a named invariant for "blocked but inspectable" and add tests proving no normal projection path consumes those families.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 002-未修复-高-public EvidenceAnchor is overloaded to carry candidate-only locator semantics
- **位置**: plan §7 lines 101-104; `fund_agent/fund/extractors/models.py` `EvidenceSourceKind`
- **问题类型**: 架构边界 / 契约缺失 / 最佳实践偏离
- **当前写法**: plan keeps `EvidenceAnchor.source_kind="annual_report"` and relies on `row_locator` / `note` text containing `candidate_only/not_proven`.
- **反例/失败场景**: A downstream audit or renderer sees `EvidenceAnchor(source_kind="annual_report")` and treats it like existing annual-report evidence. The fail-closed boundary is only encoded in free-text notes, which are not type-enforced and can be omitted or ignored.
- **为什么有问题**: The project is explicitly sensitive to not turning candidate artifacts into source truth. Current anchor type has no candidate-only structured boundary, and note text is not a stable contract.
- **直接证据**: plan lines 101-104 encode candidate boundary in anchor notes while prohibiting `EvidenceSourceKind` expansion; `extractors/models.py` line 11 limits `EvidenceSourceKind` to `annual_report`, `external_api`, and `derived`; `contracts.py` lines 389-390 requires anchors for non-missing families.
- **影响**: Candidate locator evidence can masquerade as annual-report evidence; the plan's safety depends on convention instead of type structure; implementation may pass tests while weakening source-truth boundaries.
- **建议改法和验证点**: Do not emit public `EvidenceAnchor` for candidate-only partial families until a typed candidate locator/anchor boundary is accepted. Alternative: add an internal `candidate_evidence_anchor` structure inside field-family value and keep `FundFieldFamilyResult.status="missing"` for public contract, or run a separate heavy schema gate to introduce a structured candidate boundary on anchors/results.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 003-未修复-中-six-family keyword extraction is underspecified and too broad for first implementation
- **位置**: plan §7 lines 90-100
- **问题类型**: 不可直接实施 / 切片过粗 / 测试缺口
- **当前写法**: plan asks for a module-level configuration table mapping all six field families to section/table keyword groups, then deterministic selectors over headings, captions, labels, headers, and paragraph text.
- **反例/失败场景**: Implementation agent invents Chinese keyword sets for "current_stage" or "core_risk"; tests use synthetic headings that match the invented keywords; real candidate shapes later miss or misclassify common annual-report sections. The code becomes deterministic but not semantically justified.
- **为什么有问题**: The plan does not provide the actual mapping table, canonical fixture examples, priority/tie-break rules, maximum evidence counts, duplicate handling, excerpt length rules, or ambiguity handling. That leaves core extraction behavior to implementation guesswork.
- **直接证据**: plan lines 90-100 specify selector inputs and output shape but no concrete keyword table or conflict rules; AGENTS requires fund analysis rules to be configurable and evidence-traceable, not implicit hardcoded guesses.
- **影响**: Code generation will create unreviewed extraction semantics; review can only inspect implementation after the fact; false positives or silent omissions become likely.
- **建议改法和验证点**: Reduce S6 implementation to one narrow family or one contract-only slice, or amend the plan with a concrete reviewed mapping table and at least two fixture scenarios per selected family: positive locator, ambiguous locator, and no-match fail-closed. Keep remaining families `missing`.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 中

### 004-未修复-中-extending the minimal intermediate protocol overcouples admission with content extraction
- **位置**: plan §6 lines 61-73; current `FundDisclosureDocumentIntermediate`
- **问题类型**: 过度耦合 / 架构边界 / 非最优方案
- **当前写法**: plan extends `FundDisclosureDocumentIntermediate` itself with `sections`, `paragraph_blocks`, and `table_blocks`.
- **反例/失败场景**: Existing or future test doubles and admission-only intermediates must implement full content collections even when the only operation is identity/provenance/failure-class validation. The minimal admission protocol stops being minimal and every processor guard path depends on content-shape attributes.
- **为什么有问题**: Admission and content extraction have different ownership and failure modes. The current protocol deliberately describes identity/provenance/failure boundary only; making content mandatory widens the base contract for all callers.
- **直接证据**: plan lines 61-73 modifies the base protocol; plan line 17 acknowledges the current protocol only exposes identity/provenance/boundary/failure fields; existing dispatch helper only needs those fields.
- **影响**: Unnecessary contract churn, brittle tests, and future processors forced to conform to a wider FDD content shape even when they only need admission semantics.
- **建议改法和验证点**: Preserve `FundDisclosureDocumentIntermediate` as the admission protocol. Add a second protocol such as `FundDisclosureDocumentContentIntermediate` and have extraction helpers require it after admission. Tests should prove admission-only stubs still work and content extraction fails closed when content protocol is absent.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## Open Questions

- Should candidate-only evidence ever be represented as public `FundFieldFamilyResult(partial)`, or should it remain in a separate internal candidate evidence result until source-truth qualification?
- If a blocked result may contain inspectable field families, what exact invariant prevents normal aggregation/projection paths from consuming those values?
- Is adding a typed candidate locator/anchor model in processor contracts preferable to extending public `EvidenceAnchor`?
- Which single field family, if any, has enough reviewed mapping evidence to be the first implementation slice?

## Residual Risks And Suggested Tracking

- Track candidate-boundary result semantics in a separate S6-A contract gate before implementation.
- Track public anchor/source-kind semantics in the same contract gate or a dedicated candidate locator model gate.
- Track selector mapping table acceptance separately from processor plumbing; do not let implementation invent the mapping.
- Track facade projection as a later gate only after consumer-visible candidate boundary exists.

## Final Plan Review Conclusion

`fail`

The plan identifies the right risks, but it still leaves core contract decisions unresolved. It is not yet code-generation-ready for implementation. Minimum amendments are: define blocked/partial result semantics, avoid overloading public `EvidenceAnchor` with candidate-only semantics, split admission protocol from content protocol, and narrow or fully specify the first extraction mapping slice.
