## Accepted Finding 1: Source-truth Admission Cannot Be Derived From candidate_boundary=None

### Original Finding

`candidate_boundary is None` does not prove source truth. The original plan treated non-candidate FDD inputs as sufficient for source-truth extraction, risking any stub with `candidate_boundary=None` and proper `source_provenance` being mistaken for proven source truth.

### Controller Requirement

> The plan must define a positive source-truth admission contract, or explicitly downgrade the work unit to a non-candidate direct-extraction prototype. Absence of `candidate_boundary` is not source-truth proof. The fixed plan must include negative and positive tests for admission.

### Fix Assessment: CLOSED

The fixed plan addresses this finding comprehensively:

1. **Positive proof contract added**: Slice A introduces `FundDisclosureSourceTruthAdmissionProof` frozen dataclass with explicit required fields: `proof_kind`, `source_boundary`, `fund_code`, `report_year`, `document_kind`, `intermediate_kind`, `source_kind`, `repository_identity_verified`, `source_provenance_verified`, `locator_identity_verified`, `parser_integrity_verified`, `producer`. All proof booleans must be `True`. Literal values are constrained.

2. **Protocol extension**: `FundDisclosureDocumentContentIntermediate` gains `source_truth_admission: FundDisclosureSourceTruthAdmissionProof | None`. Only content intermediates can carry proof because field extraction needs section/table/paragraph locator identity.

3. **Multi-condition admission rule**: The processor admission helper `_validate_source_truth_admission(intermediate, context)` requires ALL conditions:
   - `isinstance(intermediate, FundDisclosureDocumentContentIntermediate)`
   - `intermediate.candidate_boundary is None`
   - `intermediate.failure_class is None`
   - `intermediate.source_provenance is not None`
   - `intermediate.source_truth_admission is not None`
   - proof identity matches dispatch identity (fund_code, report_year, document_kind, intermediate_kind, source_kind)
   - all proof booleans validated by dataclass

   `candidate_boundary is None` is now a necessary but never sufficient condition.

4. **Fail-closed gap codes**: New `source_truth_admission_missing` and `source_truth_admission_invalid` gap codes with `source_boundary="source_truth_unverified"` ensure proof-missing inputs produce no public values or anchors.

5. **Required tests**:
   - `test_source_truth_admission_requires_positive_proof`: proof-missing fixture → no public value/anchor, `source_truth_admission_missing` gap
   - `test_source_truth_admission_rejects_identity_mismatch`: proof identity mismatch → no public value/anchor, `source_truth_admission_invalid` gap
   - `test_source_truth_admission_accepts_repository_loaded_identity_proof`: proof-positive → admission path recognizes proof state
   - `test_product_essence_source_truth_requires_proof_even_when_candidate_boundary_none`: same content as positive fixture but no proof → no public value/anchor, `source_truth_admission_missing` gap — directly closes the original gap

6. **Existing tests preserved**: `test_product_essence_selector_preserves_candidate_boundary_blocked_status` continues to assert candidate input remains blocked.

### Evidence

- Plan §"Slice A Contract - Positive Source-truth Admission Proof" defines exact dataclass, protocol extension, admission rule, validation, gap codes, and test specs.
- Plan §"Goal" explicitly states: "candidate_boundary is None is necessary but never sufficient."
- Plan §"Implementation Decisions" #1: "Preserve existing admission and identity checks before source-truth proof validation."

## Accepted Finding 2: product_essence.v1 Value Schema And Key/value Extraction Rules Must Be Code-generation-ready

### Original Finding

The plan lacked exact `product_essence.v1.value` shape, extraction rules for table/cell key-value pairs, ambiguity handling, and fixture specifications. An implementation agent would need to re-design extraction semantics.

### Controller Requirement

> The plan must define exact `product_essence.v1.value` keys, value types, required/optional fields, projection relationship, table/cell extraction rules, ambiguity behavior, and exact fixture/test assertions.

### Fix Assessment: CLOSED

The fixed plan addresses this finding comprehensively:

1. **Exact value shape defined** (`§Slice B Contract - product_essence.v1 Value Schema`):
   - Top-level keys: `basic_identity`, `product_profile`, `benchmark`, `risk_characteristic_text`
   - Unavailable top-level fields omitted (not `None`)
   - `basic_identity`: required `fund_name: str`, `fund_code: str`, `classified_fund_type: Literal["active_fund"]`, `classification_basis: tuple[str, ...]`; optional `fund_category`, `fund_scale`, `fund_manager`, `management_company`, `custodian`, `inception_date` as `str | None`
   - `product_profile`: `investment_objective`, `style_positioning`, `investment_scope`, `investment_strategy` as `str | None`; at least one of `investment_objective/investment_scope/investment_strategy` non-empty
   - `benchmark`: required `benchmark_text: str`
   - `risk_characteristic_text`: required `schema_version`, `fund_code`, `report_year`, `risk_characteristic_text`, `source_anchors`

2. **Label map defined** (`§Label Map`): 14 output-path-to-FDD-label mappings covering all four top-level values.

3. **Table/cell extraction rule** (`§Table/cell Rule`):
   - Both table and cell `locator_stability == "stable"`
   - Label match priority: `row_label_path` first, then `column_header_path`
   - Text selection: `cell_text_normalized` first, fallback to `cell_text` only when empty
   - Exclusion: selected text must not equal matched label or generic labels (`项目`, `指标`, `名称`, `内容`, `说明`)
   - `basic_identity.fund_code` must equal `context.fund_code`
   - Source path: `table_blocks[{table_index}].cells[{cell_index}]` using sorted `(row_index, column_index)` iteration
   - No sibling cell lookup (explicit non-goal)

4. **Priority and ambiguity rule** (`§Table/cell priority`):
   - Table/cell candidates before paragraph fallback
   - Single normalized value → use it
   - Multiple same normalized value → deduplicate, first stable locator as primary anchor
   - Multiple different normalized values → omit path, add `ambiguous_table_or_locator`

5. **Paragraph fallback rule** (`§Paragraph Fallback Rule`):
   - Only for `product_profile.*`, `benchmark.*`, `risk_characteristic_text.*`
   - Not for `basic_identity.*`
   - Label match via `heading_path` or `text_normalized` prefix
   - Same duplicate/ambiguity rule as table/cell
   - Ignored if table/cell already produced unambiguous value

6. **Anchor rule** (`§Anchor Rule`): Every emitted output path must have an `EvidenceAnchor` with `source_kind="annual_report"`, exact `section_id`/`table_id`/`row_locator` construction rules.

7. **Field-family status** (`§Field-family Status`): `accepted`/`partial`/`missing`/`blocked` definitions tied to top-level value presence.

8. **Missing and ambiguity behavior** (`§Missing And Ambiguity Behavior`): Missing optional subfields no gap; missing required creates `field_family_partial` or `field_family_missing`; ambiguous required path creates `ambiguous_table_or_locator`.

9. **Required tests** (Slice B):
   - `test_product_essence_source_truth_extracts_exact_value_shape`: full fixture with cells and paragraph, asserts exact top-level keys, nested required values, `classified_fund_type`, `candidate_evidence == ()`, `extraction_mode == "direct"`, `anchors` non-empty with `source_kind == "annual_report"`
   - `test_product_essence_source_truth_requires_proof_even_when_candidate_boundary_none`: content present but no proof → fail-closed
   - `test_product_essence_source_truth_ambiguous_duplicate_omits_conflicting_path`: two different `基金名称` cells → omit `basic_identity`, `ambiguous_table_or_locator` gap
   - `test_product_essence_source_truth_paragraph_fallback_for_descriptive_fields`: no table value for `投资范围` → paragraph fallback with anchor
   - `test_product_essence_source_truth_missing_keeps_family_missing`: proof-positive no labels → `missing`
   - `test_product_essence_source_truth_partial_when_required_groups_missing`: only fund name/code → `partial`

### Evidence

- Plan §"Slice B Contract - product_essence.v1 Value Schema" through §"Slice B Extraction Rules" define complete value shape, label map, extraction rules, priority/ambiguity handling, and anchor construction.
- Plan §"Tests / Validation Matrix" maps every requirement to specific test coverage.
- Plan §"Direct Code Evidence" correctly identifies `FundFieldFamilyResult.value` is `dict[str, object]` and the plan must define exact shape.

## New Blocker Check

### Protocol Extension Backward Compatibility

The new `source_truth_admission: FundDisclosureSourceTruthAdmissionProof | None` field on `FundDisclosureDocumentContentIntermediate` is additive and optional (defaults to `None`). The existing `_ContentIntermediateStub` in tests does not include this field, which means it defaults to `None` — correctly triggering `source_truth_admission_missing` in Slice A. This is backward-compatible.

### Gap Code And Source Boundary Expansion

New `FundExtractionGapCode` values (`source_truth_admission_missing`, `source_truth_admission_invalid`) and `FundExtractionSourceBoundary` value (`source_truth_unverified`) are additive to existing Literal types. They do not conflict with existing gap codes or boundaries. Existing tests that assert specific gap codes are not affected.

### Extraction Rule Consistency With FundDisclosureCellLike

The table/cell extraction rule uses only fields currently exposed by `FundDisclosureCellLike` (`cell_id`, `table_id`, `section_anchor`, `heading_path`, `row_index`, `column_index`, `row_label_path`, `column_header_path`, `cell_text`, `cell_text_normalized`, `locator_stability`). No sibling lookup API is required. The plan explicitly states: "Slice B must not read sibling cells."

### Candidate Evidence Isolation

The plan preserves existing candidate-only evidence behavior: candidate selectors remain unchanged, candidate evidence does not enter public value or anchors, and `candidate_evidence == ()` is asserted for promoted field families. No candidate-to-source-truth promotion risk.

### Implementation Ordering

Slice A (proof contract) before Slice B (extraction) is correct. Slice B depends on the proof validation guard from Slice A. No circular dependency.

### Deterministic And LLM-free Extraction

Plan §"Implementation Decisions" #7: "Keep extraction deterministic and lexical; no LLM, provider, PDF, Docling, repository, network, or filesystem reads inside the processor." This is preserved.

### No EvidenceSourceKind Expansion

The plan uses `source_kind="annual_report"` for public anchors, matching existing `EvidenceAnchor.source_kind` values. No new source kind is introduced. Gap codes are internal processor/fail-closed semantics, not `EvidenceSourceKind`.

### No Documentation Premature Update

Plan §"Docs Decision": "Docs must not be updated before code is accepted." Slice C (documentation sync) is explicitly after implementation acceptance.

### No New Blockers Found

The plan fix closes both accepted findings without introducing new blockers. The new proof contract is well-specified, the value schema is code-generation-ready, the extraction rules are deterministic and bounded, and the test matrix covers positive, negative, ambiguity, missing, and partial scenarios.
