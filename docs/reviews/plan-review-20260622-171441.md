## Findings

### 001-未修复-[高]-semantic aggregate cannot represent warn claim results

- **位置**: `State-machine and Data-flow Semantics` aggregate rules, lines 189-196 of the reviewed plan
- **问题类型**: 契约缺失 / 状态机漏洞 / 不可直接实施
- **当前写法**: Per-claim mapping allows `severity="warn"` for `insufficient`, and also requires an `entailed` result to be raised to warn when deterministic fact hard gate is warn. Aggregate rules then define only `contradicted / insufficient / entailed / not_applicable` and say `entailed` only when no block/warn remains.
- **反例/失败场景**: A claim is semantically `entailed`, but deterministic `anchor_precision` is warn. The per-claim result must have `severity="warn"` and `status="entailed"`. Aggregate cannot return `entailed` because a warn remains, cannot return `insufficient` because status is entailed and there is no block, and falls through to `not_applicable`, which is false.
- **为什么有问题**: The implementation agent would have to invent a third aggregation model or silently misclassify warning results. That violates gateflow's code-generation-ready requirement and can hide reviewable semantic/pathway warnings.
- **直接证据**: Reviewed plan lines 183-187 define warn severity; lines 191-194 define aggregate statuses without a warn status; line 196 says not to invent a third status model during implementation.
- **影响**: Implementation Agent 跑偏 / 状态不一致 / review 不可验收 / 后续返工
- **建议改法和验证点**: Amend the plan to define a separate aggregate status literal `EvidenceSemanticOverallStatus = Literal["pass", "warn", "fail", "not_applicable"]`. Map `contradicted` and block severity to `fail`, warn severity or semantic insufficient to `warn`, all entailed/info to `pass`, and empty/all not-applicable to `not_applicable`. Add tests for aggregate `warn` from anchor precision and from semantic insufficient.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 高
