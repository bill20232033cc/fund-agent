## 1. 目标与动机

### 1.1 目标

在 **仅覆盖 P1 数据层** 的前提下，建立一个可被后续分析层稳定消费的数据管道，满足以下三件事：

1. 基金文档访问必须收口到 **统一文档仓库接口**，上层不再直接拿 `Path`、不再直接操作文件系统。
2. 修复当前 `§3` 章节定位不稳定的问题，让 `110011` 的 2024 年年报能稳定定位到 `§3`，并为 P1 所需章节建立可测试的定位契约。
3. 基于稳定的文档访问和章节定位，落地 **12 项结构化提取**、两级缓存、样本基金验证矩阵和 P1 所需的净值数据适配器。

### 1.2 动机

- `docs/implementation-control.md:37-41` 已把 P1 标为 `in progress`，并明确“首个 implementation slice 以章节定位加固和数据提取契约定义为先”。
- `docs/implementation-control.md:115-119` 要求 P1 退出时具备下载、解析、12 项提取、两级缓存和样本核对能力。
- `docs/implementation-control.md:154-157` 明确当前只有 `fund_agent/fund/pdf/downloader.py`、`fund_agent/fund/pdf/parser.py` 原型，且 `locate_sections()` 在 `110011`/2024 年报上漏掉 `§3`。
- `AGENTS.md:44-50` 要求基金文档必须通过统一文档仓库接口访问、基金类型判断优先、证据必须可溯源。
- `AGENTS.md:84-107` 要求这类“理解基金类型、财报章节、证据锚点”的逻辑留在 `fund_agent/fund` Capability 层，不能越过边界放到上层。

## 2. 范围与非目标

### 2.1 本计划范围

- 年报 PDF 下载、解析、章节定位、缓存、结构化提取、净值数据适配器。
- 面向 P1 的数据契约、证据锚点结构、基金类型识别结果输出。
- 基于 3 只样本基金的准确率与缓存命中验证。

### 2.2 非目标

- 不实现 P2 的 `R=A+B-C` 计算、言行一致性判断、风险审计规则执行、模板渲染。
- 不实现 P3 的 CLI、端到端报告输出、温度计、scene/tool 注册。
- 不改 UI / Application / Runtime / Service / Engine 层。
- 不引入 OCR、图片型 PDF 处理或扫描件专用流程；若必须依赖 OCR，直接触发 stop condition。
- P1 仅落地 `annual_report` 文档类型的统一仓库接口；**招募说明书、季报、QDII 海外披露文件不在本 work unit**。

## 4. 关键决策

### 4.1 决策一：先建立统一文档仓库接口，再允许任何结构化提取落地

当前 `downloader.py` / `parser.py` 只能作为 **文档仓库内部的 PDF 适配器**，不能继续作为上层直接依赖的“公开 API”。  
P1 要新增一个 Capability 内部的稳定仓库接口，负责：

- 下载与刷新；
- 解析与章节索引；
- 文件缓存与 SQLite 缓存；
- 向提取器暴露 **已解析文档对象**，而不是裸 `Path`。

### 4.2 决策二：`§3` 章节定位在 P1 中单独成为冻结前置，不与 12 项提取混做

`§3` 同时支撑：

- 第 2 章的净值/基准表现；
- 第 4 章的投资者收益率；
- 份额估算 fallback 的输入。

因此 `§3` 章节定位必须在 extractor 开工前冻结，不能一边提取一边补 parser。

### 4.3 决策三：P1 虽然不渲染最终报告，但必须输出结构化证据锚点

P2/P3 才负责渲染 Markdown 报告与程序审计，但 **P1 就必须把每个结构化字段的来源锚点存进数据契约**，否则后续层无法满足 `AGENTS.md:50` 的可溯源约束。  
这里的落地边界是：

- P1 产出结构化 `EvidenceAnchor`；
- P2/P3 再把它渲染成 `> 📎 证据：...` 文本格式。

## 5. P1 数据契约与 Schema 变更

本节是 code-generation-ready 的公共契约，后续 slice 必须以此为准，除非触发 stop condition。

### 5.1 文档仓库契约

建议新增模块：

- `fund_agent/fund/documents/models.py`
- `fund_agent/fund/documents/repository.py`
- `fund_agent/fund/documents/cache.py`
- `fund_agent/fund/documents/adapters/annual_report_pdf.py`
- `fund_agent/fund/pdf/section_catalog.py`

建议核心类型：

```python
@dataclass(frozen=True)
class DocumentKey:
    fund_code: str
    year: int
    document_kind: Literal["annual_report"]


@dataclass(frozen=True)
class ReportSection:
    section_id: str
    title: str
    start_offset: int
    end_offset: int
    matched_rule: str
    confidence: float


@dataclass(frozen=True)
class ParsedTable:
    page_number: int
    table_index: int
    headers: tuple[str, ...]
    rows: tuple[tuple[str, ...], ...]


@dataclass(frozen=True)
class ParsedAnnualReport:
    key: DocumentKey
    raw_text: str
    sections: dict[str, ReportSection]
    tables: tuple[ParsedTable, ...]


class FundDocumentRepository(Protocol):
    async def load_annual_report(
        self,
        fund_code: str,
        year: int,
        *,
        force_refresh: bool = False,
    ) -> ParsedAnnualReport:
        ...
```

契约规则：

- `FundDocumentRepository` 是 **唯一允许对外暴露的文档读取入口**。
- `fund_agent/fund/documents/*` 承载公共契约、公共模型和默认仓库实现。
- `fund_agent/fund/pdf/*` 只作为 `documents` 仓库内部的 PDF 适配器，不承载对外公共契约。
- `downloader.py` / `parser.py` 只提供仓库内部 helper，不再作为 extractor 或上层的直接依赖点。
- 除 `documents/repository.py`、`documents/cache.py` 与其调用的 `pdf` 适配器外，其他模块不得直接 `mkdir`、`write_bytes`、`open(pdf_path)`。

### 5.2 结构化提取契约

建议新增模块：

- `fund_agent/fund/extractors/models.py`
- `fund_agent/fund/extractors/profile.py`
- `fund_agent/fund/extractors/performance.py`
- `fund_agent/fund/extractors/manager_ownership.py`
- `fund_agent/fund/extractors/holdings_share_change.py`
- `fund_agent/fund/fund_type.py`
- `fund_agent/fund/data_extractor.py`

建议核心类型：

```python
@dataclass(frozen=True)
class EvidenceAnchor:
    source_kind: Literal["annual_report", "external_api", "derived"]
    document_year: int | None
    section_id: str | None
    page_number: int | None
    table_id: str | None
    row_locator: str | None
    note: str | None


@dataclass(frozen=True)
class ExtractedField[T]:
    value: T | None
    anchors: tuple[EvidenceAnchor, ...]
    extraction_mode: Literal["direct", "derived", "estimated", "missing"]
    note: str | None


@dataclass(frozen=True)
class StructuredFundDataBundle:
    fund_code: str
    report_year: int
    basic_identity: ExtractedField[dict]
    product_profile: ExtractedField[dict]
    benchmark: ExtractedField[dict]
    fee_schedule: ExtractedField[dict]
    turnover_rate: ExtractedField[dict]
    nav_benchmark_performance: ExtractedField[dict]
    investor_return: ExtractedField[dict]
    share_change: ExtractedField[dict]
    manager_alignment: ExtractedField[dict]
    manager_strategy_text: ExtractedField[dict]
    holdings_snapshot: ExtractedField[dict]
    holder_structure: ExtractedField[dict]
```

约束：

- `basic_identity.value` 内必须包含 `classified_fund_type` 与 `classification_basis`。
- 所有数值型字段必须带 `anchors`；估算值必须使用 `extraction_mode="estimated"` 并写明 `note`。
- `data_extractor.py` 只作为 façade / orchestration，不承载全部章节解析细节。

### 5.3 缓存 Schema

P1 只需要满足当前阶段，不预设计 P2/P3 表。

最低需要：

- 文件缓存：原始 PDF。
- SQLite 缓存：
  - `documents`：文档元信息、下载时间、来源 URL、哈希。
  - `parsed_reports`：标准化文本、章节索引、表格序列化结果、schema_version。
  - `nav_cache`：净值数据响应、来源、时间范围、抓取时间。

冻结策略：

- `P1-S3` 只冻结 `documents` 与 `parsed_reports` 两类缓存 schema，并为 `nav_cache` 预留接入点。
- `structured_data` 缓存 **不在 P1-S3 冻结**；只在 `P1-S4 ~ P1-S7` 的 extractor schema 被 accepted 后，由 `P1-S8` 收口最终落地。
- `P1-S3` 期间若需要缓存 extractor 中间结果，只允许使用测试内或进程内临时对象，不得提前固化 SQLite `structured_data` 表。

## 11. Review Gates

### 11.1 Gate 顺序

1. `plan accepted`
2. `P1-S1 implementation + review`
3. `P1-S2 implementation + review`
4. `P1-S3 implementation + review`
5. `P1-S4 ~ P1-S7 implementation + review`
6. `P1-S8 implementation + aggregate review`
7. `P1 ready for controller acceptance`

### 11.2 每个 Gate 的审查重点

- `P1-S1`：是否真正收口了文件系统边界，并把公共契约放在 `documents` 而不是 `pdf` 命名空间。
- `P1-S2`：`§3` root cause 是否被直接修复，而非靠样本特判。
- `P1-S3`：raw PDF / parsed report 缓存是否被限制在仓库层内部，且未过早冻结 `structured_data`。
- `P1-S4`：基金类型识别是否先于通用抽取输出。
- `P1-S5`：投资者收益 fallback 是否带明确标记与证据来源。
- `P1-S6`：是否偷偷引入了分析判断而不是原始数据抽取。
- `P1-S7`：表格型 anchor 是否足够可追溯。
- `P1-S8`：36 格矩阵、缓存性能、README 同步是否全部完成。

## 12. Stop Conditions

出现以下任一情况，implementation worker 必须停下并回报，不得自行扩 scope：

1. 需要修改 UI / Application / Runtime / Service / Engine 才能推进 P1。
2. 需要引入 OCR、图像识别、人工标注流程才能定位关键章节。
3. 需要让上层直接读本地 PDF 路径或直接查询 SQLite。
4. 12 项定义与 `docs/design.md` / `docs/implementation-control.md` 出现不可调和冲突。
5. 为了通过样本测试，必须加入基金代码级的硬编码特判。
6. 投资者收益率 fallback 需要提前实现 P2 公式或结论判断。
7. 为了净值适配器稳定性，需要新增非 P1 数据源治理工作。
8. 若要把统一文档仓库公共契约继续放回 `fund_agent/fund/pdf/*`，立即停止并回报命名空间耦合问题。

## 13. 风险与开放问题

### 13.1 已识别风险

- 年报格式异构风险：
  - 缓解：章节规则配置化 + extractor 输出 `missing/estimated`，不在 P1 做人工例外表。
- `akshare` 稳定性风险：
  - 缓解：净值适配器做 provider 封装与缓存，不把源站细节泄露给 `data_extractor.py`。
- `§3` 章节标题变体风险：
  - 缓解：单独冻结 `section_catalog` 与样本断言，不在 extractor 中兜底。

### 13.2 开放问题

- `BQ-2`：2026 新规披露字段是否在 3 个样本上都存在。
  - 处理：P1 按 `direct / estimated / missing` 三态落地，不阻塞当前 phase。
- `BQ-4`：`akshare` 历史净值接口是否在全部样本上稳定。
  - 处理：作为 `nav_data.py` provider 的测试项，不阻塞仓库与 extractor 主线。

当前没有额外的 **blocking** 问题；以上问题按 P1 内部 fallback 与 provider 封装处理。
