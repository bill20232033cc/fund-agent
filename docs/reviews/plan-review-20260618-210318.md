## Findings

### 001-未修复-[高]-failure split precedence still conflicts on mixed redirect/render facts
- **位置**: Target plan §7.3 `redirect_unavailable` / `render_unavailable` Split Rules and §10.2 failure mapping tests.
- **问题类型**: 状态机漏洞 / 不可直接实施 / 测试缺口
- **当前写法**: The plan says `http_status` has highest precedence, then also says `redirect_domain` differing from the official EID domain triggers `identity_mismatch` regardless of HTTP status. It also mixes body/structure overrides with status-driven branches without specifying the exact ordering for mixed facts such as 5xx plus non-EID redirect, HTTP 200 plus non-HTML body, or HTML body plus consistently absent navigation/sections.
- **反例/失败场景**: A context for `redirect_unavailable` has `http_status=503` and `redirect_domain="evil.example"` or another non-EID domain. Following "HTTP status highest precedence" returns `unavailable`, enabling later fallback semantics. Following "domain mismatch regardless of HTTP status" returns `identity_mismatch`, fail-closing the source identity. Both are plausible readings of the same plan.
- **为什么有问题**: The accepted finding required typed context, precedence, and tests for every split sub-condition and ambiguity. The plan now has the context fields, but the precedence itself is not deterministic in mixed-fact cases. This affects canonical source failure classification, which controls whether fallback is allowed (`unavailable` / `not_found`) or blocked (`identity_mismatch` / `schema_drift` / `integrity_error`).
- **直接证据**:
  - Typed context is introduced at target plan lines 523-539.
  - Split rules list domain mismatch, 4xx known render URL, empty/non-HTML body, consistent structure absence, and 5xx/timeout branches at lines 498-507.
  - Precedence says `http_status` is highest at line 511, while line 514 says non-official `redirect_domain` triggers `identity_mismatch` regardless of HTTP status.
  - Required tests name broad classes and "multiple facts in context" at lines 666-677, but do not pin the mixed-fact precedence cases above.
- **影响**: 实施 Agent 需要自行裁决 source failure semantics / fallback eligibility 可能被错误打开或错误 fail-closed / review 不可验收 / 后续返工。
- **建议改法和验证点**: Rewrite §7.3 precedence as one total ordered decision table for `redirect_unavailable` and one for `render_unavailable`. Include explicit mixed-fact tests for at least: non-EID redirect plus 5xx, non-EID redirect plus 4xx, known render URL 404, HTTP 200 empty body, HTTP 200 non-HTML body, HTML body with consistent structure absence, and insufficient facts ambiguity.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 002-未修复-[中]-projection blockers are excluded in one section but remapped by the function contract
- **位置**: Target plan §7.1, §7.2, §7.4, §10.2.
- **问题类型**: 契约缺失 / 不可直接实施 / 测试缺口
- **当前写法**: §7.1 and §7.2 state that `value_unvalidated` and `raw_xml_not_proven` are projection-blocker statuses and must not map to `AnnualReportSourceFailureCategory`. §10.2 requires `TestProjectionBlockerDoesNotMap` to raise `ValueError`. But §7.4 defines the mapping function as accepting `FundDisclosureCandidateFailureCode` and says it must "Map every `FundDisclosureCandidateFailureCode` to exactly one canonical category."
- **反例/失败场景**: An implementation agent follows §7.4 literally and maps `value_unvalidated` / `raw_xml_not_proven` to a canonical source category to satisfy "every `FundDisclosureCandidateFailureCode`". The implementation then violates §7.1, §7.2, and the required `TestProjectionBlockerDoesNotMap`. Alternatively, the agent makes the test pass by rejecting those values and violates the function contract text.
- **为什么有问题**: This was an accepted low finding: projection blockers are not source-failure mappings. The fix closes it in the tables and tests, but the mapping contract reopens the ambiguity at the exact API surface an implementation worker will follow.
- **直接证据**:
  - `FundDisclosureProjectionBlocker` contains `value_unvalidated` and `raw_xml_not_proven` at lines 446-453.
  - Lines 455-459 state they must not be mapped to `AnnualReportSourceFailureCategory`.
  - Lines 461-465 define `FundDisclosureCandidateFailureCode` as source failures plus projection blockers.
  - Lines 553-559 require the mapping function to map every `FundDisclosureCandidateFailureCode`.
  - Line 678 requires projection blockers to raise `ValueError` if passed to the mapping function.
- **影响**: 实施 Agent 生成相互矛盾的 API 行为 / 测试与实现契约不可同时满足 / projection blocker 被错误升级为 source failure / review 不可验收。
- **建议改法和验证点**: Change the mapping function input type and bullet from `FundDisclosureCandidateFailureCode` to `FundDisclosureCandidateSourceFailureCode`. Keep an explicit fail-closed guard that raises `ValueError` if a projection blocker or unknown string reaches the mapping function. Tests should assert both the source-code complete mapping and projection-blocker rejection.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
