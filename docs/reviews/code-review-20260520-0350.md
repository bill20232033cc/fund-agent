## Findings

### P5S1-CR-1-已修复-高-`analyze` quality gate 复用 `run_extraction_score()` 会把单基金 CSV 误判为 fatal

- **入口 / 触发输入**: `FundAnalysisService.analyze()`，`quality_gate_policy != "off"`，且调用方传入只包含当前基金的合法 `quality_gate_source_csv`。
- **实际分支**: Service 调用 `run_quality_gate_for_bundle(...)`，adapter 生成单基金 `snapshot.jsonl` 后调用 `run_extraction_score(...)`。
- **期望行为**: P5-S1 plan 要求 source CSV 用于取得当前基金 App 类别；当前基金在 CSV 中时应运行 gate，或在 prerequisites unavailable 时返回 `not_run_reason`。
- **实际行为**: `run_extraction_score(...)` 无条件调用 `select_minimal_golden_set(source_csv)`，而该函数强制要求 CSV 包含 `004393` 和多类 minimal golden set。只含当前基金的合法 CSV 会抛 `ValueError: 精选基金池缺少必选基金代码：004393`，既不是 gate result，也不是 not-run reason。
- **直接证据**:
  - `run_quality_gate_for_bundle(...)` 原实现调用 `run_extraction_score(snapshot_path=..., source_csv=...)`。
  - `run_extraction_score(...)` 调用 `select_minimal_golden_set(source_csv)`。
  - `select_minimal_golden_set(...)` 固定 `_find_fund_by_code(..., MANDATORY_GOLDEN_CODE)`。
- **影响**: `analyze` quality gate 对合法单基金 CSV 不可用；错误路径变成普通异常，破坏 P5-S1 “运行/跳过/不可用显式可见”的契约。
- **修复**: 新增 `write_extraction_score_records(...)`，允许对已加载 snapshot records 写 score/gate 产物，并在未传 `golden_set` 时写入单基金 gate 专用空 `GoldenSetSelection`。`run_extraction_score(...)` 保持原 CLI 行为，仍选择 minimal golden set；`run_quality_gate_for_bundle(...)` 改为调用新函数，不再强制完整 golden set。
- **验证点**: `tests/fund/test_quality_gate_integration.py` 现在使用只含当前基金的 CSV 验证 adapter 可运行，并断言 `golden_set.records == []`。
- **严重程度**: 高
