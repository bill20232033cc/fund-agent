## Findings

### 1-未修复-中-holding_strategy_change 角色同时使用 strong token 作为 guard token 产生语义冗余

- **位置**: §7 Token / Guard Table, `holding_strategy_change` 行
- **问题类型**: 非最优方案
- **当前写法**: `holding_strategy_change` 的 strong tokens 和 guard tokens 同时包含 `报告期内基金投资策略和运作分析`。这意味着这个短语既能作为 strong token 直接命中，又作为 guard token 授权 generic token 通过。
- **反例/失败场景**: 年报中 `报告期内基金投资策略和运作分析` 是标准章节标题，它的每次出现都会直接命中 strong token，使同源 generic tokens（如 `策略`、`持仓`、`配置`）的 guard check 变为冗余——guard 永远会通过，因为 strong token 已经先匹配了。这不会导致功能错误，但 guard semantic 在此场景下是"dead code"，后续维护者可能误以为 guard 在生效而忽略了真正需要 guard 的边界情况。
- **为什么有问题**: 这不是 token-breadth 或 correctness 问题，而是 guard 设计的语义清晰度问题。如果 strong token 已经覆盖了该短语，guard tokens 中再列同一短语不会增加安全性，只会增加维护困惑。
- **直接证据**: plan §7 表格 holding_strategy_change 行：strong tokens 列和 guard tokens 列均含 `报告期内基金投资策略和运作分析`。
- **影响**: 轻微——implementation agent 会按表复制 token，guard function 行为仍是正确的（strong match 短路 generic check）。不影响正确性，但 future reader 可能误以为需要 guard 来 catch 该短语。
- **建议改法和验证点**: 从 guard tokens 中移除已被 strong tokens 覆盖的短语，或加注释说明冗余是故意复制以保持 source-level guard 自洽。不改也不影响正确性。

### 2-未修复-低-计划未显式描述 `_current_stage_cell_guard_context()` 的 role-invariant 语义是否确实与 S6-F 一致

- **位置**: §7 Guard rule / §9 Future Implementation Shape
- **问题类型**: 不可直接实施
- **当前写法**: §7 描述 cell guard context 应使用 `parent table heading_text, table_caption_or_nearby_heading, heading_path, cell row_label_path, cell column_header_path, cell heading_path`，排除 `cell_text` 和 `cell_text_normalized`。§9 列出了函数名 `_current_stage_cell_guard_context()` 但未给出签名或完整语义。
- **反例/失败场景**: implementation agent 不熟悉 S6-F 的 `_core_risk_cell_guard_context()` 模式（processor 代码 lines 2054-2078），可能自创 cell guard context shape，或错误引入 S6-E 的 role-dependent 分支（S6-E 的 `manager_alignment` 和 `subscription_redemption` 有例外路径）。
- **为什么有问题**: §7 guard rule 说"Do not copy S6-E same-cell default guard behavior"，但没有显式引用 S6-F 的 `_core_risk_cell_guard_context()` 作为参考实现。implementation agent 需要推断正确的排除集。
- **直接证据**: plan §7 guard rule 末尾："Cell generic guard must be role-invariant and fail-closed: do not include `cell_text` or `cell_text_normalized` in the guard context." 但 §9 函数列表中 `_current_stage_cell_guard_context()` 没有写"始终 role-invariant，与 `_core_risk_cell_guard_context()` 同形"。
- **影响**: 轻微——有经验的 agent 会参照 S6-F 模式，且 §7 的文字描述已足够明确。但 implementation agent 仍然需要主动查找 S6-F 代码确认语义。
- **建议改法和验证点**: §9 增加一句 `_current_stage_cell_guard_context()` 复用 S6-F 的 role-invariant pattern，排除 `cell_text` / `cell_text_normalized`。

### 3-未修复-低-测试 16 的 overlap regression fixture 构建方式未指定 baseline fixture 内容

- **位置**: §10 Required Tests, test 16 `test_current_stage_selector_preserves_overlap_family_semantics`
- **问题类型**: 不可直接实施
- **当前写法**: "Build a baseline fixture that produces S6-B/S6-C/S6-D/S6-E/S6-F candidate evidence. Build a second fixture by adding S6-G current-stage legal content. Compare each existing family signature: record count, exact path order, gap codes, public value, and public anchors."
- **反例/失败场景**: implementation agent 不知道 baseline fixture 应该包含哪些具体内容。如果 baseline fixture 太弱（只有 1-2 个 record 每族），regression 检测能力不足；如果太强（试图覆盖所有 role），test fixture 维护成本过高且引入不相关 token 依赖。
- **为什么有问题**: plan 未指定 baseline fixture 的最小覆盖范围。一个过弱的 baseline 可能在 S6-G 破坏现有语义时不触发 failure。
- **直接证据**: plan §10 test 16 描述只有需求，没有 fixture shape 指引。
- **影响**: 轻微——implementation agent 可以合理地从现有 S6-B/C/D/E/F 测试中提取 fixture pattern。但缺乏最小覆盖说明意味着 regression 保护的强度是不确定的。
- **建议改法和验证点**: 添加"baseline fixture 至少包含每个现有字段族的至少 2 条 candidate evidence（section + paragraph 或 table + cell），并验证 S6-G 添加后这些记录的数量、path 和 gap 不变"。
