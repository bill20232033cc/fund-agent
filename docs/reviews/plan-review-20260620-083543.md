## Material Findings

### F1 [MEDIUM] Direct-route-missing candidate_evidence suppression not explicitly tested

Plan §6: "Proof-positive direct route suppresses `manager_profile.v1.candidate_evidence` even when no direct value is found." This is a critical invariant: when the direct extractor runs but finds no allowed subvalues (`field_family_missing`), `candidate_evidence` must still be `()`, not fall back to S6-D candidate records.

Slice 1 test `test_manager_profile_source_truth_route_suppresses_candidate_evidence` likely covers the general case. Slice 2 test `test_manager_profile_source_truth_missing_when_no_allowed_labels` covers the missing case. But neither test name explicitly asserts the combination: "direct route + no values found + candidate_evidence == ()".

**Recommendation**: Implementation worker should ensure at least one test proves that proof-positive routing with zero matched subvalues still emits `candidate_evidence=()` and does NOT fall back to `_select_manager_profile_candidate_evidence()`. The existing named tests may already cover this if their assertions include candidate_evidence; the plan should make this explicit.

### F2 [LOW] holdings_snapshot source tables are §8 portfolio tables, not §4 manager tables

The plan sources `holdings_snapshot` from `前十名股票投资明细` and `报告期末按行业分类的股票投资组合` — these are Chapter 8 accounting/portfolio tables, not Chapter 4 manager profile tables. The existing `FIELD_FAMILY_MAPPINGS` already assigns `holdings_snapshot` to `manager_profile.v1` for Chapter 3 analysis, so this is not a plan error.

However, the naming tension is real: a field called `holdings_snapshot` in a family called `manager_profile` whose source data comes from portfolio accounting tables could confuse future readers. The plan defers all interpretation (concentration, style drift, core risk), which mitigates this.

**Recommendation**: Implementation evidence artifact should note this semantic tension as a residual. No plan change needed.

### F3 [LOW] Holdings snapshot value shape is narrower than parsed-annual-report extractor output

The plan's `holdings_snapshot` allows only `top_holdings`, `top_holdings_status`, `top_holdings_source`, `industry_distribution`, `industry_distribution_status`. Existing `HoldingsShareChangeExtractionResult.holdings_snapshot` is typed `ExtractedField[dict[str, object]]` — unconstrained.

Downstream consumers expecting keys beyond the plan's five (e.g., `all_stock_details`, `bond_holdings`) would get `None` from `_field_from_family` because the key is absent from value dict. This is fail-closed — consumers that check for None or extraction_mode == "missing" before using the value are safe.

**Recommendation**: Document this scope limitation in the implementation evidence. The plan already lists deferred items in §4, which is sufficient.

### F4 [LOW] manager_strategy_text concatenation may include section artifacts

The plan concatenates stable paragraph texts under strategy/outlook headings. Paragraphs may include section numbering, table references, or formatting artifacts. The plan doesn't specify filtering of section numbers (`1.1`, `(一)`) or cross-reference text.

**Recommendation**: Implementation should use `text_normalized` (not `text_raw`) for concatenation, which should already strip most artifacts. If artifacts persist, accept them as disclosed text; do not add text-cleaning logic in this gate.

### F5 [LOW] portfolio_managers source guard `主要人员情况` is broad

Section 5.1 allows stable table rows under headings containing `主要人员情况`. This heading could match non-manager personnel sections (board members, supervisors, key employees). The plan mitigates with the role guard: "A manager entry requires a non-empty `name` and a row/role context containing `基金经理`."

**Recommendation**: Implementation should ensure the row-level `基金经理` role check is applied strictly — if a row under `主要人员情况` has a name but no `基金经理` role context, it must be skipped.

### F6 [INFO] No cross-family anchor deduplication — correctly out of scope

The plan's anchor deduplication applies within `manager_profile.v1` only. Since no other family is being extracted for source-truth in this gate, cross-family deduplication is not needed. This is correctly out of scope per plan §2.

### F7 [INFO] Existing test regression coverage is adequate

Plan references existing test coverage at `tests/fund/processors/test_fund_disclosure_processor.py` (source-truth proof guard tests, product/return direct extraction tests, S6-D candidate-only tests) and `tests/fund/test_data_extractor.py` (facade projection tests). Plan requires existing S6-A through S6-D tests to remain passing. Plan §13 explicitly says no parser replacement or contract expansion is needed.

## Open Questions

1. **Q1**: Does `_select_manager_profile_candidate_evidence()` scan cells sorted by `(row_index, column_index)` today? The plan's source-truth selector uses the same ordering. If S6-D already sorts this way, the implementation can reuse the ordering logic. If not, the implementation must ensure consistency between candidate and source-truth selectors. *Not blocking for plan review — implementation detail.*

2. **Q2**: When `holdings_snapshot` `top_holdings` is emit but `industry_distribution` is missing (different tables under different headings), should `holdings_snapshot` contribute a `field_family_partial` gap? The plan's §4 status rules say `partial` is "at least one but fewer than five top-level subvalues are emitted." But `top_holdings` and `industry_distribution` are sub-subvalues of `holdings_snapshot`, not top-level subvalues. The plan's gap taxonomy in §6 says `field_family_partial` is "at least one subvalue is emitted and one or more of the five allowed top-level subvalues is missing." This suggests the partial gap is at the top-level (e.g., `holdings_snapshot` present but `turnover_rate` missing), not at the sub-subvalue level. *The plan is internally consistent on this point — `holdings_snapshot` as a whole is either present or missing, and missing sub-subvalues within it don't produce additional field_family_partial gaps. Implementation should follow this.*
