### R2-unfixed-low-capture-limitation-must-not-be-strengthened

- **位置**: Plan Sections 5, 6 A4, and 8 controller judgment requirements
- **问题类型**: observability gap / 证据边界
- **当前写法**: The plan correctly identifies `stdout/stderr capture limitation` and requires it be preserved.
- **反例/失败场景**: A control sync could accidentally phrase the run as having independently measured empty stdout or physically split stdout/stderr evidence.
- **为什么有问题**: The existing evidence explicitly says stdout byte count was `not_independently_measured`; strengthening that into strict stdout proof would violate same-source evidence discipline.
- **直接证据**: `docs/reviews/mvp-post-operator-provider-availability-evidence-gate-live-evidence-20260606.md` records that stdout/stderr were not split and strict stdout byte-count proof is a controller-review limitation.
- **影响**: The audit trail would overclaim what was captured, making the accepted checkpoint harder to defend.
- **建议改法和验证点**: Controller judgment and control docs should use limitation wording such as `stdout byte count not independently measured` and `stdout/stderr capture limitation preserved`; they should not claim stdout was independently empty. Verify via `rg -n "stdout/stderr|capture limitation"`.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低
