## Findings

### 001-未修复-高-blocking failure can still produce a high auditability score
- **位置**: `Score Semantics` lines 223-237; `Dimension Semantics` lines 169-203; `Plan Review Focus` line 333
- **问题类型**: 契约缺失 / 状态机漏洞 / 测试缺口
- **当前写法**: Plan says fact-level `auditability_score` is the average of numeric dimension scores, while `source_support`, `missing_evidence`, `proof_boundary`, and `value_match` each have independent numeric scores. It also says a proven same-anchor value mismatch can make `value_match` fail with score `40` while `source_support` can pass.
- **反例/失败场景**: A fact has a precise proven annual-report reference for its own anchor, but the value is absent from the excerpt. Under the plan, `anchor_precision=100`, `source_support=100`, `missing_evidence=100`, `proof_boundary=100`, and `value_match=40`; the fact hard gate fails, but the average score is `88`.
- **为什么有问题**: The work unit goal is to separate hard gate and dimension score, but the score is still expected to be a useful auditability summary. A failed E2 same-source value check with an 88 score is not a useful auditability signal; it can make downstream reviewers or later bridge work misread a hard failure as nearly clean. The plan itself names this risk in the review focus but does not resolve it.
- **直接证据**: Plan lines 198-202 define `value_match fail / score 40`; lines 225-235 define averaging; lines 209-221 define hard gate fail separately; current phase-1 code returns score `40` for E2 fail in `fund_agent/fund/evidence_confirm.py` lines 310-321, avoiding the proposed V2 averaging inflation.
- **影响**: Implementation agent can produce a V2 result where `overall_status="fail"` but `auditability_score` looks high. Review acceptance becomes ambiguous, later report-quality bridge can inherit misleading scoring, and future gates may need a semantic score rewrite.
- **建议改法和验证点**: Amend the plan before implementation with an explicit fail cap or status-aware score rule. Minimum acceptable rule: if fact hard gate is `fail`, fact `auditability_score` must be capped at the lowest failing blocking dimension score, or at a fixed fail-band such as `0-40`; aggregate score must either use capped fact scores or expose separate `pass_rate` / `blocking_failure_count`. Add required tests: value mismatch hard-gate fail cannot exceed `40`; candidate-only proof failure cannot exceed `0`; aggregate with one blocking fact cannot report a pass-like score.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 002-未修复-中-Slice 1 is not independently executable because its tests require V2 result construction before V2 functions exist
- **位置**: `Implementation Slices` lines 241-294
- **问题类型**: 切片过粗 / 不可直接实施 / 测试缺口
- **当前写法**: Slice 1 allows only V2 aliases/dataclasses and hard-gate helpers, and explicitly does not implement V2 public functions. Its required tests nevertheless include "V2 result uses `evidence_confirm.v2`" and "V2 fact result includes all five dimensions in deterministic order". Slice 2 later implements `confirm_chapter_evidence_v2()` and `confirm_projection_evidence_v2()`.
- **反例/失败场景**: A Slice 1 worker follows the allowed changes literally. There is no public V2 result-producing function yet, so the worker either has to test private helpers/dataclass literals only, weaken the required tests, or prematurely implement part of Slice 2. Any of those outcomes breaks the gate's slice ownership and makes review criteria unstable.
- **为什么有问题**: A Gateflow slice must be small and independently reviewable. This plan's first slice mixes "schema declaration only" with tests that require operational result assembly. That forces implementation-time redesign of the slice boundary.
- **直接证据**: Plan lines 249-253 restrict Slice 1 to aliases/dataclasses/private hard-gate helpers; lines 255-262 require V2 result and dimension-result behavior; lines 280-284 defer public V2 functions and dimension scoring implementation to Slice 2.
- **影响**: Implementation worker may overrun Slice 1, duplicate Slice 2 work, or produce tests that do not prove the public contract. Review cannot cleanly decide whether Slice 1 is accepted.
- **建议改法和验证点**: Amend the plan by either merging Slice 1 and Slice 2 into one contract/scorer implementation slice, or redefine Slice 1 as a pure schema/helper slice with tests limited to dataclass construction, hard-gate helper aggregation, and schema constant checks. If kept separate, add a named private builder contract that Slice 1 is allowed to implement and test; otherwise move "all five dimensions deterministic order" and `evidence_confirm.v2` public result tests to Slice 2.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## Architecture Boundary Review

No cross-layer implementation boundary violation was found in the plan. It keeps V2 in `fund_agent/fund/evidence_confirm.py`, keeps repository/PDF/source-helper/network access out of scope, and preserves no Service/UI/Host/renderer/quality-gate integration. The score-contract issues above are internal Fund-layer contract risks, not layering violations.

## Best-Practice Review

The plan follows local no-live typed-contract practice and preserves V1 compatibility. The main best-practice gap is that score semantics must be monotonic with hard failure severity; otherwise the score becomes hard to review and unsafe to bridge into later report-quality artifacts.

## Optimal-Solution Review

The overall approach, adding V2 result types beside V1 instead of replacing V1, is a practical path. The optimal fix is not a redesign of the whole feature; it is to add status-aware score caps and repair slice boundaries before implementation.

## Overengineering Review

The five dimensions are arguably more granular than phase-1 scoring, but the granularity is justified by the work unit goal. No standalone overengineering blocker found.

## Overcoupling Review

The plan avoids coupling Evidence Confirm V2 to report-quality `ScoreDimension` and avoids upper-layer consumption. The only overcoupling risk is semantic: multiple dimensions score the same proof/evidence failure, which causes finding 001's inflated average.

## Open Questions

- Should V2 expose both a numeric `auditability_score` and an explicit `blocking_failure_count` to prevent future consumers from relying on score alone?
- Should `proof_boundary` fail when any supplied same-anchor reference is invalid, or only when no valid proof reference exists? The plan currently leans toward the latter but should state the mixed-reference rule explicitly.

## Residual Risks And Tracking Destination

- Dayu upstream parity remains unverified locally. Track in implementation controller only if exact parity becomes a requirement; current plan should rely on project-local semantics.
- Live source/PDF Evidence Confirm, semantic entailment, report-quality bridge, and quality-gate adoption remain correctly assigned to future gates.

## Final Plan Review Conclusion

`fail`

The plan is directionally sound but not code-generation-ready. It needs a plan-fix gate for the score-fail cap and slice-boundary corrections before implementation.
