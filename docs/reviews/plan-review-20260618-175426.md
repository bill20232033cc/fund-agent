## Findings

### 1-未修复-[高]-`intermediate_kind` contract contradicts the S4 processor dispatch contract
- **位置**: Plan §6.1-§6.2, especially `FundDisclosureDocumentIntermediate` compliance and `FundDisclosureDocumentIdentity.intermediate_kind`; S4 processor identity validation.
- **问题类型**: 契约缺失 / 不可直接实施 / 架构边界
- **当前写法**: The plan says the concrete `FundDisclosureDocument` must satisfy `FundDisclosureDocumentIntermediate` and expose `intermediate_kind`, but the identity schema sets `intermediate_kind: FundIntermediateKind  # "eid_xbrl_html_render_candidate.v1"`.
- **反例/失败场景**: An implementation agent follows the identity sketch literally and builds a document whose `intermediate_kind` is `"eid_xbrl_html_render_candidate.v1"`. The S4 processor dispatch key for the accepted route is `"fund_disclosure_document.v1"`, so the processor rejects the object as `input_type_mismatch` before admission can exercise candidate-boundary behavior.
- **为什么有问题**: This gate is supposed to bridge S4 skeleton processor to a concrete candidate schema. The current plan conflates source artifact kind (`eid_xbrl_html_render_candidate.v1`) with processor intermediate kind (`fund_disclosure_document.v1`). That forces the implementation agent to invent either a second hidden field, a conversion layer, or a processor behavior change, all outside the allowed write set.
- **直接证据**:
  - Target plan states the concrete dataclass must expose `intermediate_kind` via `FundDisclosureDocumentIntermediate`: `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:198-201`.
  - Target plan assigns identity `intermediate_kind` to `"eid_xbrl_html_render_candidate.v1"`: `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:207-212`.
  - Current `FundIntermediateKind` includes both literals, but S4 processor supports only dispatch `fund_disclosure_document.v1`: `fund_agent/fund/processors/contracts.py:24-30`, `fund_agent/fund/processors/fund_disclosure_processor.py:76-80`.
  - S4 processor explicitly blocks when `intermediate.intermediate_kind != context.intermediate_kind`: `fund_agent/fund/processors/fund_disclosure_processor.py:206-218`.
  - S4 final closeout accepts default registry resolution for `active_fund + annual_report + fund_disclosure_document.v1` and says `FundDataExtractor.extract()` still does not consume this intermediate: `docs/reviews/s4-concrete-funddisclosuredocument-processor-final-closeout-controller-judgment-20260618.md:32-39`.
- **影响**: 实施 Agent 跑偏 / 生成不可接入代码 / review 不可验收 / 后续返工。
- **建议改法和验证点**: Make the top-level `FundDisclosureDocument.intermediate_kind` fixed to `"fund_disclosure_document.v1"`. If the source artifact identity must preserve EID render provenance, add a separate field such as `source_artifact_kind: Literal["eid_xbrl_html_render_candidate.v1"]` or `candidate_source_kind`, and specify that it is not the processor dispatch key. Add a no-live test that passes a concrete document to `FundDisclosureDocumentProcessor.extract()` with dispatch `fund_disclosure_document.v1` and proves the result reaches candidate-boundary/missing-field behavior, not `input_type_mismatch`.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 2-未修复-[高]-Failure split rules require facts the mapping contract cannot receive
- **位置**: Plan §7.3-§7.4 and §10.2 failure mapping tests.
- **问题类型**: 不可直接实施 / 测试缺口 / 状态机漏洞
- **当前写法**: The plan requires split rules for `redirect_unavailable` and `render_unavailable`, including non-EID domain detection, previously-known render URL behavior, and structure absence across retries. The mapping function signature accepts only `http_status`, `error_type`, `retry_count`, `body_empty`, and `body_is_html`.
- **反例/失败场景**: A redirect returns HTTP 302 to an unrecognized non-EID domain. The plan expects `identity_mismatch`, but `map_fund_disclosure_failure_to_source_category(...)` has no redirect target URL or domain status parameter. Likewise, "HTTP 4xx on previously-known render URL" and "navigation/sections consistently absent across retries" require historical URL knowledge and per-retry structure observations that are absent from the function contract.
- **为什么有问题**: The accepted schema controller judgment carried `redirect_unavailable` / `render_unavailable` as a binding residual specifically because the split had to become testable. The plan names the split cases, but the proposed API cannot encode several of them. Implementation would either weaken assertions, hard-code assumptions, or invent an unplanned context object.
- **直接证据**:
  - Controller residual requires testable split rules: `docs/reviews/funddisclosuredocument-candidate-source-schema-plan-controller-judgment-20260614.md:77-85`.
  - Target plan requires non-EID domain, previously-known render URL, and consistent structure absence branches: `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:432-447`.
  - Target mapping signature omits URL/domain, known-url state, navigation/section presence, and retry-observation details: `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:449-466`.
  - Required tests assert non-EID domain and render structure/body branches, but do not bind the missing input shape: `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:546-563`.
- **影响**: 实施 Agent 重新设计 mapping API / 测试只能覆盖 happy path / failure classification 误报为 `unavailable`、`not_found` 或 `schema_drift` / fallback semantics 后续返工。
- **建议改法和验证点**: Replace the loose keyword signature with a typed no-live context, for example `FundDisclosureFailureContext`, carrying `http_status`, `error_type`, `redirect_target_url`, `official_domain_status`, `render_url_known`, `body_empty`, `body_is_html`, `navigation_present`, `sections_present`, `retry_count`, and an explicit `structure_absence_observed_consistently` boolean or deterministic retry observation tuple. Define precedence when multiple facts are present. Tests must instantiate the exact context for every sub-condition and include negative ambiguity cases that fail closed instead of guessing.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 3-未修复-[中]-Non-consumption validation is weaker than the existing project-local guard pattern
- **位置**: Plan §9.1, §9.2, §10.1, and future validation commands.
- **问题类型**: 测试缺口 / 架构边界 / 最佳实践偏离
- **当前写法**: The plan says future tests should prove `FundDisclosureDocument` is not importable from Service/UI/Host/Agent/extractors via "Import check", plus a local module import analysis that only checks the new candidate module does not import `EvidenceAnchor` or `EvidenceSourceKind`.
- **反例/失败场景**: A future implementation accidentally imports `FundDisclosureDocument` from `fund_agent/services/fund_analysis_service.py` or `fund_agent/fund/template/renderer.py`. A package-root "not importable from `fund_agent/services/`" check can still pass because the symbol is not re-exported from the package root, while a deep module has already crossed the forbidden boundary. The plan also omits explicit scan targets for `fund_agent/fund/template`, `fund_agent/fund/audit`, and `fund_agent/fund/report_quality_validation.py` in §9.1 even though renderer/audit/quality gate leakage is forbidden.
- **为什么有问题**: The control truth forbids Service/UI/Host/renderer/quality gate direct consumption of Docling/pdfplumber/EID HTML render or candidate internals. The repo already has a stronger no-consumption pattern for Docling: AST scanning guarded directories and files for forbidden import prefixes. The new plan does not require reusing that pattern, so leakage can hide in a deep module while surface import tests pass.
- **直接证据**:
  - Target plan requires no imports by Service/UI/Host/Agent/renderer/quality gate, but the guard table specifies import checks rather than full guarded-tree static scans: `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:490-507`, `docs/reviews/funddisclosuredocument-candidate-source-no-live-implementation-plan-20260618.md:509-523`.
  - Current control truth forbids candidate conversion artifacts being consumed by Service/UI/Host/renderer/quality gate/LLM prompt/template: `docs/implementation-control.md:40-43`.
  - Existing project-local guard scans `fund_agent/services`, `fund_agent/ui`, `fund_agent/host`, `fund_agent/fund/template`, `fund_agent/fund/audit`, `fund_agent/fund/extractors`, and `fund_agent/fund/report_quality_validation.py` with AST import analysis: `tests/fund/documents/test_docling_no_consumption_guards.py:8-22`, `tests/fund/documents/test_docling_no_consumption_guards.py:25-47`.
- **影响**: 架构边界泄漏 / renderer 或 quality gate 可直接消费 candidate internals / review 难以发现 / 后续字段投影 gate 被当前 schema implementation 预先污染。
- **建议改法和验证点**: Require an AST-based guarded-tree test mirroring `test_docling_no_consumption_guards.py`, with forbidden prefixes including `fund_agent.fund.documents.candidates.fund_disclosure_document` and the new failure mapping module. Guard at least `fund_agent/services`, `fund_agent/ui`, `fund_agent/host`, `fund_agent/agent`, `fund_agent/fund/template`, `fund_agent/fund/audit`, `fund_agent/fund/extractors`, and `fund_agent/fund/report_quality_validation.py`. Keep public `__init__.py` non-export checks as an additional surface test, not the primary leakage proof.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
