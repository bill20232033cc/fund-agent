## Findings

### 1-未修复-高-source-truth admission 条件不足，non-candidate 可能被误当作 source truth

- **位置**: `Goal`, `Success Signal`, `Implementation Decisions`, `Tests / Validation Matrix`
- **问题类型**: 契约缺失 / 架构边界 / 测试缺口
- **当前写法**: plan 以 “non-candidate admitted FDD inputs” 作为 source-truth extraction 的主要前提，并要求输出 `EvidenceAnchor(source_kind="annual_report")`。
- **反例/失败场景**: 任意测试 stub 或未来内部 producer 只要把 `candidate_boundary=None`、`failure_class=None`、`source_provenance` 填好，就会被 implementation 视为可产出 source-truth 字段。该对象未必来自 `FundDocumentRepository` 加载的同一份年报，也未必是经过 source identity / section identity / parser integrity 证明的 FDD 中间态。
- **为什么有问题**: 当前代码契约只有 `CandidateBoundaryStatus` 用来阻止 candidate promotion；没有等价的 positive source-truth admission contract。`FundDisclosureDocumentIntermediate` 只暴露 `document_kind`、`fund_code`、`report_year`、`intermediate_kind`、`source_provenance`、`candidate_boundary`、`failure_class`。这些字段能排除一些坏输入，但不能证明 FDD 内容是 source truth。把 `candidate_boundary is None` 解释为 source truth，会把“不是 candidate”误当作“已证明为真源”。
- **直接证据**:
  - `contracts.py` 的 `FundDisclosureDocumentIntermediate` 没有 `source_truth_status`、`repository_loaded_identity`、`content_source_kind` 或 equivalent positive proof 字段。
  - `CandidateBoundaryStatus` 只校验 candidate 不得声明 proof/readiness。
  - `fund_disclosure_processor.py` 当前 admission 成功后只调用 `_field_families_for_intermediate()`，没有 positive source-truth guard。
  - `tests/fund/processors/test_fund_disclosure_processor.py` 的 `_ContentIntermediateStub` 默认 `candidate_boundary=None`，但它只是测试对象，不是 source-truth producer。
- **影响**: 实施 Agent 可能写出“只要不是 candidate 就产出 annual_report EvidenceAnchor”的路径；这会违反 candidate-only / not_proven 边界，导致 source truth 被过早提升，后续 review 无法区分真实 FDD producer 与测试 stub。
- **建议改法和验证点**:
  - 在 plan 中新增明确的 positive source-truth admission rule：本 slice 只接受 `candidate_boundary is None` 之外还满足的条件，例如 FDD 中间态必须声明稳定的 non-candidate source boundary / repository-mediated annual-report source identity / section identity provenance。若当前协议不具备该字段，则本 slice 不应命名为 source-truth promotion，而应降级为 “non-candidate FDD direct extraction contract prototype”。
  - 如果决定新增契约字段，plan 必须列出 exact field、allowed values、fail-closed behavior、affected tests。
  - 测试必须增加 negative case：`candidate_boundary=None` 但缺少 positive source-truth proof 时不得产出 public field values。
  - 测试必须增加 positive case：只有满足 positive source-truth proof 的 FDD fixture 才能产出 public `EvidenceAnchor`。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 2-未修复-高-product_essence 字段 schema 与 key/value 提取规则不够 code-generation-ready

- **位置**: `Product Essence Fields`, `Implementation Decisions`, `Slice A`, `Tests / Validation Matrix`
- **问题类型**: 不可直接实施 / 契约缺失 / 测试缺口 / 过度耦合
- **当前写法**: plan 要求提取 `basic_identity`、`product_profile`、`risk_characteristic_text`、`benchmark`，并说 “prefer table/cell values for identity-like key/value fields and paragraph/section text for descriptive fields”。
- **反例/失败场景**: FDD table cell protocol 只有单个 cell 的 row/column/header/text/locator 信息，没有 plan-level 规则说明如何从同一行的 key cell/value cell、header path、row label path 和 nearby heading 组合出 `fund_name`、`fund_code`、`benchmark` 等字段。实施者必须自行决定：命中 “基金名称” 的 cell_text 是 key 还是 value、相邻 cell 是否可读、多个候选冲突时怎么处理、`product_profile` 字段结构如何与当前 `StructuredFundDataBundle.product_profile.value` 对齐。
- **为什么有问题**: 这是 extractor contract 变更，不是 locator selector。Plan 必须给实现者 exact output schema、字段来源优先级、required key 列表、ambiguity handling 和 fixture shape。当前写法只给了字段名和方向，仍要求 implementation agent 重新设计 extraction semantics。
- **直接证据**:
  - `FundFieldFamilyResult.value` 是 `dict[str, object]`，没有强类型 schema。
  - `ActiveFundAnnualProcessor` 的 mapping table列出 source paths，但不定义 FDD product essence value shape。
  - `FundDisclosureCellLike` 暴露当前 cell，但不暴露 row object 或 sibling lookup API；plan 却要求 table/cell key-value extraction。
  - `tests/fund/processors/test_fund_disclosure_processor.py` 现有 `_CellStub` 只有一个 cell，`row_label_path=("基金名称",)` 且 `cell_text="测试基金"`，无法证明真实 key/value 表格提取策略。
- **影响**: 实施 Agent 会在 `fund_disclosure_processor.py` 内发明 ad hoc schema 和 table traversal；review 时难以判断输出是否正确，后续字段族可能被锁进不稳定的 value shape。
- **建议改法和验证点**:
  - Plan 必须为 `product_essence.v1.value` 写出 exact keys、required/optional、value types 和与 `StructuredFundDataBundle` 对应字段的投影关系。
  - Plan 必须定义 table/cell extraction rule：哪些 row/column/header/path 被视为 key，哪些被视为 value；是否允许 sibling lookup；没有 sibling API 时是否只允许 row_label_path 已含 key 且 cell_text 为 value 的 fixture。
  - Plan 必须定义 conflict handling：多个 fund_name / benchmark / risk_characteristic 命中时是 `ambiguous_table_or_locator`，还是按稳定 priority 选择。
  - Tests 必须包含 positive key/value fixture、ambiguous duplicate fixture、paragraph fallback fixture 和 missing fixture，并断言 exact output value shape。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

## Open Questions

- 当前项目是否已有或即将有一个 accepted non-candidate `FundDisclosureDocument` producer？如果没有，本 work unit 应先规划 source-truth admission contract，而不是直接规划 field extraction。
- `product_essence.v1` 的 public value 是否应该严格复用 `StructuredFundDataBundle.basic_identity/product_profile/risk_characteristic_text/benchmark` 的 shape，还是允许 FDD processor 输出新的 field-family-local dict shape？该问题必须在 implementation 前收敛。

## Final Plan Review Conclusion

`fail`

The plan is directionally right but not implementation-ready. It must be revised before implementation because it lacks a positive source-truth admission contract and exact `product_essence.v1` extraction/value semantics.
