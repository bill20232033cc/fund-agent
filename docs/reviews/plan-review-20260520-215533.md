## Findings

### P7S3-PR1-未修复-中-双 timeout 语义没有落到可实施接口
- **位置**: `Timeout, Retry, Session, User-Agent` / `PDF Download` / `EidAnnualReportSource` constructor
- **问题类型**: 契约缺失 / 不可直接实施 / 测试缺口
- **当前写法**: Plan 要求 `metadata_timeout_seconds` 用于 `validate_fund.do` 和 `advanced_search_report.do`，`pdf_timeout_seconds` 用于 PDF 下载；同时要求 “Use `httpx.AsyncClient(follow_redirects=True, headers=..., timeout=...)`” 和 “use one client per `fetch_annual_report_pdf(...)` call”。
- **反例/失败场景**: implementation agent 只在 `AsyncClient(timeout=metadata_timeout_seconds)` 或 `AsyncClient(timeout=pdf_timeout_seconds)` 设置一次 timeout。这样 metadata 和 PDF 会共享同一个 timeout，导致 metadata 等待过长或 PDF 下载过早超时，且测试可能只验证 happy path。
- **为什么有问题**: P7-S1/P7-S2 把 metadata timeout 和 PDF timeout 分成两个显式配置项，目的是让短元数据请求和长 PDF 下载的失败分类可控。P7-S3 如果不规定 per-request timeout 传递方式，就会让配置边界变成伪配置。
- **直接证据**: P7-S3 plan 同时要求两个 timeout 字段和单个 `AsyncClient(timeout=...)`，但没有说明 `client.post/get(..., timeout=config.metadata_timeout_seconds/pdf_timeout_seconds)` 或等价策略，也没有测试检查两者被分别使用。
- **影响**: implementation agent 可能实现出行为正确但配置错误的 EID source；真实 EID 慢 PDF 或慢 metadata 情况下会产生错误 unavailable 分类和不可预测 retry。
- **建议改法和验证点**: 修订 plan：`AsyncClient` 可使用一个安全默认 timeout，但 metadata endpoints 必须在 request 级传入 `metadata_timeout_seconds`，PDF request 必须传入 `pdf_timeout_seconds`；fake client/MockTransport 测试应验证 metadata 和 PDF request 的 timeout 参数被区分，若 MockTransport 不暴露 timeout，则通过 injectable request helper/client fake 记录 timeout。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
