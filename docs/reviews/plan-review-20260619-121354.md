## Findings

### 1-未修复-高-现有 regression test 预期需要更新但 plan 未明确指定

- **位置**: Section 7 "Existing regression tests that must keep passing, with expectation updates only where S6-E makes the old wording stale"
- **问题类型**: 不可直接实施
- **当前写法**: Plan 列出了 29 个 existing regression tests 并说"with expectation updates only where S6-E makes the old wording stale"，但未具体说明哪些 test 需要更新、更新什么。
- **反例/失败场景**: 当前 `_ContentIntermediateStub` 的默认内容（`heading_text_raw="基金简介"`, `table.heading_text="基金基本情况"`）匹配 S6-B `_PRODUCT_ESSENCE_MATCH_GROUPS` 的 `product_identity` role tokens。因此 `test_extract_satisfied_returns_fully_gapped_result` 当前应已因 S6-B 而失败（`product_essence.v1` 的 `gap_code` 应为 `candidate_only_not_source_truth` 而非 `field_family_missing`），或测试在 S6-B 之前通过且未被更新。S6-E 实现后，若 default stub 的 investor-related content 也命中 S6-E tokens，`investor_experience.v1` 的 `gap_code` 同样会从 `field_family_missing` 变为 `candidate_only_not_source_truth`。`test_extract_admits_candidate_boundary_but_returns_blocked` 断言所有 6 个 field family 的 `gap_code` 为 `field_family_missing`，也会受影响。
- **为什么有问题**: Implementation agent 不知道哪些 test 的哪些 assertion 需要更新。如果盲目更新所有 test，可能破坏 S6-B/C/D 的 contract invariant；如果不更新，test 会失败。
- **直接证据**:
  - `test_extract_satisfied_returns_fully_gapped_result` line 2015: `assert family.gaps[0].gap_code == "field_family_missing"` 对所有 6 个 family
  - `test_extract_admits_candidate_boundary_but_returns_blocked` line 1970: `assert family.gaps[0].gap_code == "field_family_missing"` 对所有 6 个 family
  - `_ContentIntermediateStub` 默认 section heading "基金简介" 匹配 S6-B tokens
- **影响**: Implementation agent 被迫重新设计 test 更新策略，增加返工风险。
- **建议改法和验证点**: Plan 应显式列出哪些 existing test 的哪些 assertion 需要从 `field_family_missing` 改为 `candidate_only_not_source_truth`（对 `product_essence.v1` 和 `investor_experience.v1`），哪些保持不变。或至少说明：默认 stub 的哪些 field family 会被 S6-B/S6-E 选择器命中并改变 gap_code。
- **修复风险（低/中/高）**: 中
- **严重程度（高）**: 高

### 2-未修复-中-shared helper 重构边界不明确

- **位置**: Section 5 "Required helper structure" 第 2-4 条
- **问题类型**: 过度耦合 / 不可直接实施
- **当前写法**: "The implementation may introduce small private shared helper functions for guarded source scanning, token matching, record construction, and limit handling if needed to avoid a fourth copy of traversal logic. Any shared helper must preserve existing S6-B/S6-C/S6-D order, paths, excerpts, limits, row locators, guard behavior, and tests."
- **反例/失败场景**: 如果 implementation agent 创建一个统一的 `_scan_candidate_evidence(match_groups, ...)` helper 并让 S6-B/C/D/E 都使用它，S6-B 的简单 token matching（无 guard）和 S6-D 的 guard context 差异化（`manager_alignment` vs 其它 role 使用不同 guard context）必须被精确保留。若 helper 统一了 guard context 语义，S6-D 的 `manager_alignment` paragraph guard（包含 `text_normalized`/`text_raw`）和其他 role 的 paragraph guard（仅 `heading_path`）可能被错误统一。
- **为什么有问题**: Plan 允许但不要求 helper 重构，且未明确 helper 的边界约束。Implementation agent 可能过度重构（统一 S6-B/C/D/E）或不足重构（第四份 copy），两种情况都有风险。
- **直接证据**:
  - S6-D `_manager_profile_paragraph_guard_context` line 1189-1206: `manager_alignment` role 使用 `(text_normalized, text_raw, *heading_path)`，其它 role 使用 `(*heading_path,)`
  - S6-E plan section 3 paragraph guard context: 所有 role 统一使用 `(text_normalized, text_raw, heading_path)`
  - S6-B 无 guard：`_PRODUCT_ESSENCE_MATCH_GROUPS` 只有 `(role, tokens)` tuple
  - S6-C 无 guard：`_RETURN_ATTRIBUTION_MATCH_GROUPS` 只有 `(role, tokens)` tuple
- **影响**: 重构可能引入 S6-B/C/D behavior drift；不重构则 S6-E 增加第四份 traversal copy。
- **建议改法和验证点**: Plan 应明确：(a) S6-E 的 paragraph guard context 是否对所有 role 统一使用 `(text_normalized, text_raw, heading_path)`，或是否像 S6-D 一样按 role 区分；(b) 共享 helper 的 scope 是仅限 S6-E 内部，还是可以跨 S6-B/C/D/E；(c) 若跨 family 共享，S6-B/C 的无 guard matching 如何与 S6-D/E 的 guarded matching 共存。
- **修复风险（低/中/高）**: 中
- **严重程度（中）**: 中

### 3-未修复-低-income_distribution strong tokens 与 guard tokens 高度重叠

- **位置**: Section 3 Match groups table, `income_distribution` row
- **问题类型**: 最佳实践偏离
- **当前写法**: Strong tokens: `基金收益分配`, `收益分配`, `利润分配`, `收益分配情况`, `基金利润分配`, `每10份基金份额分红数`。Guard tokens: `收益分配`, `利润分配`, `基金份额`, `分配`。
- **反例/失败场景**: 每个 strong token 都至少包含一个 guard token（`收益分配` 出现在所有 strong token 中，`利润分配` 也出现在多数中）。这意味着 `income_distribution` 的 generic guard 实际上永远不会阻断 strong token 命中后的 guard 检查——guard 对 strong tokens 是 no-op。Guard 只对 generic tokens `分红`/`红利` 有效。
- **为什么有问题**: 这不是功能错误，但 guard token 的选择使得 `income_distribution` role 的 guard 机制在 strong token 路径上完全冗余。Guard 的设计意图是防止 broad token 误命中，但当前 guard tokens 太接近 strong tokens 以至于不提供额外保护。
- **直接证据**: Plan section 3 match groups table: `income_distribution` strong tokens 全部包含 `收益分配` 或 `利润分配`，guard tokens 包含 `收益分配` 和 `利润分配`。
- **影响**: 低。Guard 对 generic tokens `分红`/`红利` 仍有效。Strong token 路径的 guard 冗余不影响正确性。
- **建议改法和验证点**: 可接受当前设计。如需改进，guard tokens 可考虑增加更 broad 的 context token（如 `基金份额持有人`）来为 `分红`/`红利` 提供更强的 context 要求。
- **修复风险（低/中/高）**: 低
- **严重程度（低）**: 低

## Open Questions

1. **S6-E paragraph guard context 是否对所有 5 个 role 统一？** Plan section 3 定义了统一的 paragraph guard context `(text_normalized, text_raw, heading_path)`，但 S6-D 的 `manager_alignment` role 使用了更宽的 guard context。S6-E 的 `investor_return` 和 `income_distribution` 在语义上接近 S6-C 的 performance data，而 `holder_structure` 接近 S6-D 的 manager_ownership data。是否需要按 role 区分 paragraph guard context？

2. **`income_distribution` 和 `investor_return` 共享同一 processor field surface (`performance.investor_return`)，当同一 source 同时命中两个 role 时，dedupe 策略是否符合预期？** Plan 的 role order 确保 `investor_return` 先扫描，dedupe 保留 first record。这是否是有意设计？

3. **S6-E 的 cell guard context 比 S6-D 更宽（所有 role 都包含 parent table context + cell-level fields），这是有意设计还是需要与 S6-D 对齐？** S6-D 的 `manager_alignment` 使用宽 cell guard，其它 role 只用 `table_context + cell.heading_path`。S6-E plan 对所有 role 统一使用宽 cell guard。

## Conclusion

**pass-with-risks**

Plan 的设计方向正确、token 定义具体、guard 规则清晰、boundary 字段完整、non-goals 明确、与 S6-B/C/D ownership 无冲突。主要风险是现有 regression test 的预期更新未明确指定（Finding 1），implementation agent 需要自行判断哪些 test 的哪些 assertion 需要变更。Shared helper 重构边界（Finding 2）也需要 implementation agent 做架构判断。这两个风险不阻断 plan 接受，但应在 implementation gate 的 acceptance criteria 中显式覆盖。
