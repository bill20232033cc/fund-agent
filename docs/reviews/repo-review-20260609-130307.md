## Findings

### [P1] Untracked `fund_agent.tools` helper would place a secret-writing workstation tool inside the distributable package namespace

Evidence:

- `pyproject.toml` packages `include = ["fund_agent*"]` and excludes `scripts*`, but not `fund_agent/tools` (`pyproject.toml:72-75`).
- Current package discovery confirms `fund_agent.tools` is discoverable as a namespace package under the configured `fund_agent*` pattern.
- `fund_agent/tools/claude_mimo.py` is untracked and sits under that namespace. It writes `~/.claude/settings.json` and `~/.claude.json` (`fund_agent/tools/claude_mimo.py:66-92`), putting `ANTHROPIC_AUTH_TOKEN` into the JSON payload (`fund_agent/tools/claude_mimo.py:70-78`).
- The same helper shape also exists as untracked `scripts/claude_mimo_simple.py`, which writes the same home-level files and token payload (`scripts/claude_mimo_simple.py:46-73`).

What can go wrong:

If this workspace residue is promoted, a local Claude/MiMo credential configuration helper becomes part of the project namespace/distribution surface. That violates the repository's product boundary: this repo is a fund analysis agent, while this helper mutates user-home Claude config and handles authentication tokens. The risk is not that normal `fund-analysis` imports it today; the risk is package-surface contamination and accidental publication of a secret-handling workstation tool under `fund_agent.*`.

Root cause:

The helper is placed under `fund_agent/` instead of a local ignored tooling location, while the packaging pattern is broad enough to discover `fund_agent.tools`.

Concrete fix:

Classify the helper as local workstation residue: remove it from `fund_agent/`, keep it outside the package if still needed, or explicitly exclude/ignore that path. Add a repo hygiene guard that fails if unexpected untracked Python files appear under `fund_agent/` or if `find_namespace_packages(... include=["fund_agent*"])` discovers non-product namespaces.
