### 001 - 未修复 - 高 - Repository cannot persist current tracking-error extractor output

- Location: `fund_agent/fund/extractor_output_repository.py:288`
- Related current model: `fund_agent/fund/extractors/models.py:619`
- Related current extractor path: `fund_agent/fund/extractors/performance.py:426`

`ExtractorOutputRepository.save()` serializes every `StructuredFundDataBundle` field through `_jsonable(...)`. `_jsonable(...)` accepts JSON primitives, mappings, dataclasses, and sequences, but rejects `Decimal`. Current `TrackingErrorValue.value` is typed as `Decimal`, and the current performance extractor constructs `TrackingErrorValue(value=value, ...)` for directly disclosed tracking error.

Impact: a real bundle with non-missing `tracking_error` can fail during `extractor-output-save`, even though that value is part of the current accepted extractor model. Existing repository tests only save a fixture whose `tracking_error.value` is `None`, so this regression is not covered.

Required fix: preserve current exact numeric semantics by explicitly converting supported `Decimal` values to a deterministic JSON representation, and add a regression that saves a bundle with non-missing `TrackingErrorValue`.
