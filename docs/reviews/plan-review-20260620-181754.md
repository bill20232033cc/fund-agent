## New Blocker Check

The fixed plan did not introduce any new implementation blockers. Specific checks:

1. **Union type consistency**: `_CoreRiskRoleValueCandidate` has `output_path`, `role`, `value`, `anchor`, `source_field_path`. `_RiskCharacteristicValueCandidate` has `output_path`, `value`, `anchor`, `source_field_path`. The union is structurally compatible. Both types have `output_path` and `source_field_path`, so `_core_risk_emitted_output_paths()` and anchor generation can operate on either type without unsafe narrowing.

2. **Dispatch correctness**: `_build_core_risk_value()` dispatches on `_RiskCharacteristicValueCandidate` → `_build_risk_characteristic_text_value()` and `_CoreRiskRoleValueCandidate` → `_build_core_risk_role_disclosure_value()`. The plan specifies that unexpected candidate types are programming errors (line 266).

3. **Gap semantics consistency**: `field_family_missing` for no subvalues, `field_family_partial` for partial required top-level, `ambiguous_table_or_locator` for conflicting candidates. All use valid `FundExtractionGapCode` values. `deferred_role` removal is clean (not in `FundExtractionGapCode`).

4. **Status semantics consistency**: accepted = all five, partial = at least one but not all, missing = none. Contract-valid: `FundFieldFamilyResult.__post_init__` allows `"partial"` status with anchors.

5. **Test coverage**: Positive test (five-role shape), negative tests (risk-text-only → partial, heading-only → missing/partial, ambiguity → `ambiguous_table_or_locator`), structural boundary tests (header cell rejection, placeholder rejection, guard requirement), facade tests (no bundle projection). All prior tests that assert `accepted` with risk-text-only are explicitly enumerated for update.
