## Findings

### F1-未修复-高-cell级generic token自守卫导致guard失效

- **位置**: Plan §3 Match groups—`subscription_redemption` role cell guard context definition
- **问题类型**: 架构边界 / 过度耦合
- **当前写法**: Plan定义cell guard context包含`cell.cell_text_normalized`和`cell.cell_text`（§3 Guard rule: "For cell records, guard context is parent table heading_text, table_caption_or_nearby_heading, parent heading_path, cell row_label_path, cell column_header_path, cell cell_text_normalized, cell cell_text, and cell heading_path"）。同时`subscription_redemption`的generic token包含`申购`和`赎回`，guard token包含`份额`。
- **反例/失败场景**: 一个table cell内容为`"本期总申购份额 50亿份"`——该cell自身文本同时命中generic token `申购`和guard token `份额`，形成自守卫（self-guard），guard未提供任何额外过滤能力。同理`"总赎回份额"`、`"申购份额"`等常见cell内容均可自守卫通过。基金年报中大量表格cell（费率表、份额变动表、持有人结构表）都包含`申购`/`赎回`与`份额`的共现，这些cell与subscription/redemption volume披露无关，但会全部被标记为candidate evidence。
- **为什么有问题**: 与S6-D `manager_profile`的实践不一致——S6-D `_manager_profile_cell_guard_context()` 对non-`manager_alignment` roles排除`cell.cell_text`自守卫，只使用table-level和heading_path guard context（`fund_disclosure_processor.py:1209-1242`）。Plan未定义任何role-specific cell guard context narrowing，implementation agent需自行设计，超出plan授权范围。16条limit面对大量自守卫cell可能很快被低质量记录填满，挤掉真正有价值的section/table级candidate。
- **直接证据**: `fund_disclosure_processor.py:1209-1242` S6-D cell guard context按role分支：`manager_alignment`包含cell自文本，其他role（`portfolio_managers`, `holdings_snapshot`等）排除cell_text。Plan §3 guard rule未区分role，统一允许cell自守卫。Plan §7 test 7只测试generic token无guard不通过，未测试自守卫cell的过多false positive场景。
- **影响**: Implementation agent需要独立设计cell guard context narrowing规则（超出plan授权）；cell级false positive可能填满16-record limit导致更相关的section/paragraph记录被截断；后续extraction gate需付出额外过滤成本。但candidate-only / NOT_READY姿态限制了级联影响——不会传播到public value或判断。
- **建议改法和验证点**:
  1. 为`subscription_redemption`和`income_distribution` role定义role-specific cell guard context，仿照S6-D `_manager_profile_cell_guard_context` pattern，排除`cell.cell_text` / `cell.cell_text_normalized`自守卫。
  2. 或显式在plan中声明cell guard context使用table-only级别（`heading_text`, `table_caption_or_nearby_heading`, `heading_path`），替代当前包含cell自文本的定义。
  3. Test 7增加自守卫cell场景：构造cell内容为`"申购份额"`、table heading为无关内容（如`"费率表"`），assert不产生`subscription_redemption` candidate。
- **修复风险（中）**: 修改guard context定义需同步更新plan中5个role的cell guard说明，且需确保不误杀真正的subscription/redemption cell记录。
- **严重程度（高）**: 不修复会导致cell级candidate质量显著下降，16条limit被无效记录占据，implementation agent需自行填补设计空白。

### F2-未修复-中-`subscription_redemption` guard token `份额` 粒度过粗

- **位置**: Plan §3 Match groups—`subscription_redemption` guard tokens列
- **问题类型**: 契约缺失 / 最佳实践偏离
- **当前写法**: `subscription_redemption` guard tokens = `份额`, `基金份额`, `份额变动`, `基金总申购`, `基金总赎回`
- **反例/失败场景**: 年报段落："本基金申购份额的计算方式为：净申购金额=申购金额/(1+申购费率)，申购份额=净申购金额/T日基金份额净值"——此段落在基金招募说明书/年报的"基金份额的申购与赎回"章节中极其常见，它讨论的是申购份额的**计算方式**（费率机制），而非**实际申购赎回发生额**。该段落同时命中generic token `申购`和guard token `份额`（且`基金份额`也因为"基金份额净值"被substring匹配命中），因此被误标记为`subscription_redemption` candidate。
- **为什么有问题**: `份额`在年报中是最高频术语之一（出现在净值、费率、持有人结构、份额变动等几乎所有章节）。将其作为guard token无法有效区分"申购赎回发生额披露"和"申购计算方式/费率说明/申购限制条款"。Plan §3 Do not add list显式禁止`上涨`、`下跌`、`追涨`等broad tokens，但未将`份额`纳入guard过宽审视。
- **直接证据**: Plan §3 guard tokens for `subscription_redemption`包含`份额`作为guard。`_matches_any_token`（`fund_disclosure_processor.py:1330-1349`）使用substring matching，因此`"基金份额净值"`会命中guard token `"基金份额"`。Plan §7 test 7对`subscription_redemption`仅要求generic token无对应guard不产生记录，未测试`申购`+`份额`在非实际成交量上下文中的false positive。
- **影响**: 中等——paragraph级false positive会增加candidate noise，但16条limit和candidate-only状态限制damage不会传播到public value。
- **建议改法和验证点**:
  1. 将`subscription_redemption` guard从`份额`升级为更窄的`总申购份额`和`总赎回份额`（与strong token对齐），或增加`申购份额`→需同时命中`总申购`/`总赎回`/`申购赎回`上下文。
  2. Test 7增加反例：构造包含"申购份额的计算方式"的paragraph，table heading稳定为费率相关信息，assert不产生`subscription_redemption` candidate。
- **修复风险（低）**: guard token调整只影响匹配召回率，不改变其他role行为。
- **严重程度（中）**: guard过宽导致candidate quality下降但已被plan自行列为residual risk，且candidate-only边界限制下游影响。

### F3-未修复-中-`income_distribution` role缺少独立字段映射锚点

- **位置**: Plan §3 Selector Contract—`income_distribution` role mapping
- **问题类型**: 范围漂移 / 架构边界
- **当前写法**: `income_distribution` mapping: `performance.investor_return` (same as `investor_return` role)。Plan §10 Residual Risks明示："`income_distribution` is not a distinct current `ActiveFundAnnualProcessor` public field."
- **反例/失败场景**: Implementation agent需要在`investor_return`和`income_distribution`两个role间分配同一个source field的candidate记录。两个role在同一field family内不dedupe（plan §4: "Deduplicate within investor_experience.v1 by exact source_field_path"），但两个role使用不同的strong tokens匹配同一source可能产生重复路径——按照dedupe rule，先扫描的role（`investor_return`排在`income_distribution`之前）会保留记录，后者被去重丢弃。这导致income_distribution的strong token命中的记录可能被investor_return的弱匹配记录先行占据source_field_path。
- **为什么有问题**: 两个role共享一个source field的mapping但tokens完全不同（`investor_return`关注投资者实际收益/获得感，`income_distribution`关注分红/利润分配），scan顺序差异导致一个role的匹配可能被另一个role的先行记录压制。去重规则说"keep the first record for a path according to the ordering rule"，但未定义当同一path被不同role以不同token命中时如何处理。
- **直接证据**: Plan §3 mapping table: `investor_return` → `performance.investor_return`, `income_distribution` → `performance.investor_return`。Plan §4 dedupe rule: "Deduplicate within investor_experience.v1 by exact source_field_path." Plan §4 ordering rule: `investor_return` → `holder_structure` → `share_change` → `subscription_redemption` → `income_distribution`（income_distribution最后扫描）。
- **影响**: 中等——income_distribution candidate可能被investor_return的记录覆盖，导致分红相关信息丢失。但两个role都在同一个field_family内，合并后的candidate set仍然覆盖了相关disclosure，只是role label可能不准确。
- **建议改法和验证点**:
  1. 考虑将dedupe key从纯`source_field_path`扩展为`(source_field_path, role)`，允许同一path被不同role保留，各自携带不同role label和excerpt。
  2. 或在plan中显式声明income_distribution在当前gate中不生成独立candidate（仅作为investor_return的token补充），理由是不存在独立字段锚点。
  3. Test 6增加场景：同一source被两个role的strong tokens同时命中，assert两个role的candidate都得到保留。
- **修复风险（低）**: dedupe key变更只影响investor_experience.v1内部去重逻辑，不跨family。
- **严重程度（中）**: 影响income_distribution role的candidate召回完整性，但不改变candidate-only/NOT_READY边界。

### F4-未修复-低-implementation helper抽取未指定回归保护细节

- **位置**: Plan §5 Required helper structure
- **问题类型**: 切片过粗 / 不可直接实施
- **当前写法**: "The implementation may introduce small private shared helper functions for guarded source scanning, token matching, record construction, and limit handling if needed to avoid a fourth copy of traversal logic."
- **反例/失败场景**: Implementation agent因token数量多（5个role共约70+ token）和guard规则复杂（5种不同的strong/generic/guard组合），自然倾向于将S6-B/S6-C/S6-D的遍历逻辑也重构为共享helper。重构过程中参数化错误（如S6-D的4-tuple vs S6-E的3-tuple guard groups，S6-D的role-specific cell guard context vs S6-E的统一cell guard）可能导致现有selector回归。
- **为什么有问题**: Plan只说"Any shared helper must preserve existing S6-B/S6-C/S6-D order, paths, excerpts, limits, row locators, guard behavior, and tests"，但没有具体定义新helper的signature或参数化契约。S6-B/S6-C/S6-D selector使用不同的match group tuple结构（S6-B: 2-tuple，S6-C: 2-tuple，S6-D: 4-tuple），S6-E plan使用5-column table（role/strong/generic/guard/evidence_role），实际数据结构可能是3-tuple（strong only roles: holder_structure, share_change）、4-tuple（guarded roles: investor_return, subscription_redemption, income_distribution）或统一5-tuple。这些差异使通用helper设计不trivial。
- **直接证据**: Plan §5: "may introduce small private shared helper functions...if needed to avoid a fourth copy of traversal logic"——将设计权委托给implementation agent。Plan §7 regression list含28个现有测试，但均测试behavior而非helper structure。
- **影响**: 低——现有S6-B/S6-C/S6-D regression tests应在implementation gate捕获回归，但修复shared helper回归可能比独立实现S6-E遍历逻辑耗费更多时间。
- **建议改法和验证点**:
  1. Plan中显式声明：S6-E implementation优先独立实现遍历逻辑（fourth copy），shared helper refactor留到后续独立cleanup gate。
  2. 或显式定义shared helper的最小signature和参数化契约（match group tuple格式、guard context callback签名）。
- **修复风险（低）**: 只是implementation strategy选择，不改变contract。
- **严重程度（低）**: 现有regression tests提供安全网，implementation agent可选择保守策略（独立实现）。

## Open Questions

1. **OQ1**: `income_distribution` role的`每10份基金份额分红数` strong token匹配到分红表中的具体数值行时，cell文本为纯数字（如`"0.50"`），cell自身不含任何token——此类cell如何被匹配？是否需要row_label_path或column_header_path提供token上下文？Plan §3允许cell匹配时检查`row_label_path`和`column_header_path`，但strong token表格未说明数字型cell的特殊处理。→ **建议**: Implementation gate前澄清或接受cell级分红数值不产生独立candidate。

2. **OQ2**: `investor_return` role的`行为损益` strong token——"行为损益"是一个高度分析性/结论性的术语，通常不出现在年报原文中（年报原文使用"投资者实际收益"或"加权平均投资者收益率"等）。该token在locator-only阶段的实际命中率可能为0。→ **建议**: 不影响correctness，但应记录预期命中率以指导后续field-extraction gate的candidate充足性判断。

3. **OQ3**: S6-E implementation后，`_FAMILY_ORDER`中`current_stage.v1`和`core_risk.v1`在`_field_families_for_intermediate()`中保持`_missing_field_family()`调用不变，而`investor_experience.v1`的ternary表达式将新增第4层嵌套（当前已有3层：product_essence → return_attribution → manager_profile）。嵌套深度是否在implementation gate需要重构为mapping table？→ **建议**: 不在plan gate解决，属于implementation style decision。

## Final Conclusion: **pass-with-risks**

**Rationale**: The plan correctly maintains architecture boundaries (candidate locator stays inside `FundDisclosureDocumentProcessor`, no leak to Service/UI/Host/renderer/quality gate/LLM), preserves candidate-only/not_proven/NOT_READY source truth boundary, explicitly excludes S6-C/S6-D-owned tokens, and provides a comprehensive test matrix (9 new + 28 regression). The write set is properly bounded and forbidden files are explicitly listed.

Two material risks prevent a clean `pass`:

1. **F1 (High)**: Cell-level self-guard for `subscription_redemption` — the generic tokens `申购`/`赎回` with guard token `份额` plus cell self-text in guard context creates trivial guard satisfaction. Unlike S6-D which has role-specific cell guard context narrowing, the plan provides no narrowing, leaving the implementation agent to fill a design gap. Risk: 16-record limit filled by low-quality cell records.

2. **F3 (Medium)**: `income_distribution` shares `performance.investor_return` source field mapping with `investor_return` role, and scan-order-based dedupe may suppress income_distribution records in favor of earlier-scanned investor_return records.

These risks are contained by the `candidate_only`/`NOT_READY` posture — no risk propagates to public value, EvidenceAnchor, field correctness, or release readiness. The plan can proceed to controller judgment with these findings recorded as acceptance conditions for the implementation gate.
