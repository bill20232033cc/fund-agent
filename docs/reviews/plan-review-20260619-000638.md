## Findings

### F1. Failure/Blocked Matrix Inconsistency: "Raise RuntimeError" vs "No Bundle" — Severity: Low

The failure matrix (lines 221-239) uses two distinct phrasings for blocked/unsupported conditions:

- "Raise `RuntimeError`" (explicit raise)
- "Facade raises `RuntimeError`; no bundle" (same pattern, more explicit)

These are consistent. However, **Contract Decision #11** (line 200-201) introduces a different pattern:

> If `contract_status in ("missing", "partial", "satisfied")`, project through existing `_active_processor_result_to_bundle()` and preserve field-family missing semantics.

This means a `contract_status="missing"` result (which `FundDisclosureDocumentProcessor` returns for admitted non-candidate intermediates) **does** produce a bundle — all fields will be missing but a `StructuredFundDataBundle` is returned.

The failure matrix does not explicitly list the `contract_status="missing"` case. Adding a row would prevent ambiguity for the implementation worker:

| Condition | Required S5 behavior |
|---|---|
| `contract_status="missing"` (admitted non-candidate intermediate, S4 skeleton) | Return bundle with all six field families as `ExtractedField(value=None, anchors=(), extraction_mode="missing", ...)`. No `RuntimeError`. |

**Risk:** Implementation worker might incorrectly raise `RuntimeError` for `missing` status, treating it like `blocked`/`unsupported`.

### F2. `source_kind` Dispatch Key Discrepancy — Severity: Medium

The plan's Contract Decision #2 (line 173-184) constructs the dispatch key with `source_kind="annual_report"`:

```python
FundProcessorDispatchKey(
    fund_type="active_fund",
    report_type="annual_report",
    intermediate_kind="fund_disclosure_document.v1",
    source_kind="annual_report",   # <--
    ...
)
```

However, the concrete `FundDisclosureDocument` carries source provenance from `eid_xbrl_html_render_candidate` (its `source_kind` literal is `"eid_xbrl_html_render_candidate"` per `fund_disclosure_document.py:22`).

The plan does **not explain why** the dispatch key uses `"annual_report"` instead of `"eid_xbrl_html_render_candidate"`. Both `ActiveFundAnnualProcessor` and `FundDisclosureDocumentProcessor` ignore `source_kind` in their `supports()` method (they only check `fund_type`, `report_type`, `intermediate_kind`, `processor_goal`), so this is not a functional bug. But the rationale should be explicit: the dispatch key `source_kind` represents the **report category** (annual report), not the **source provenance** (EID XBRL HTML render).

**Risk:** If a future processor's `supports()` checks `source_kind`, the dispatch key's `"annual_report"` may conflict with the disclosure's actual provenance. The plan should either (a) explain the rationale, or (b) use `"eid_xbrl_html_render_candidate"` and verify processor `supports()` compatibility.

### F3. Test Matrix Missing `contract_status="missing"` Positive Test — Severity: Medium

The plan lists 17 test cases. However, none explicitly tests the **success path** where a non-candidate intermediate with no failure_class and no candidate_boundary produces a bundle with all missing fields (the S4 skeleton's admitted output).

The closest test is `test_explicit_disclosure_missing_result_projects_gapped_bundle()`, which tests a "Local or default non-blocked missing result." But this test's description says "Local or default non-blocked missing result produces only missing `ExtractedField`s with no anchors and no candidate value" — it is ambiguous about whether it uses a concrete `FundDisclosureDocument` (which has `candidate_boundary` set) or a Protocol stub without `candidate_boundary`.

**Recommendation:** Add an explicit test:

| Test | Acceptance expectation |
|---|---|
| `test_explicit_disclosure_non_candidate_admitted_produces_missing_bundle()` | Protocol stub with `candidate_boundary=None`, `failure_class=None`, `source_provenance=non-None` produces bundle with `contract_status="missing"` and all six field families as missing `ExtractedField`s. |

### F4. Test Matrix Missing Registry Resolution Failure Test — Severity: Low

The plan does not include a test for the case where `FundProcessorRegistry.resolve()` raises `UnsupportedFundProcessorError` (e.g., wrong `fund_type` or `intermediate_kind` in the dispatch key). The failure matrix row "Registry cannot resolve `fund_disclosure_document.v1`" lists the expected behavior (`UnsupportedFundProcessorError`; no fallback), but no corresponding test exists.

**Recommendation:** Add:

| Test | Acceptance expectation |
|---|---|
| `test_explicit_disclosure_registry_resolution_failure_raises()` | Dispatch key with unsupported `intermediate_kind` (after mocking registry) raises `UnsupportedFundProcessorError`; no fallback. |

### F5. Non-Active Fund Disclosure Route Test Ambiguity — Severity: Low

The test `test_explicit_disclosure_non_active_fund_does_not_use_legacy_path()` (line 326) has a description that conflates two scenarios:

> "Index/bond/QDII/FOF parsed classification plus disclosure parameter raises; default no-disclosure legacy path still passes separately."

This is a single test asserting two things: (a) non-active + disclosure raises, and (b) non-active without disclosure still uses legacy path. Splitting into two tests would be clearer and more robust.

### F6. `_active_processor_result_to_bundle` Reuse for Disclosure Route — No Finding

The plan reuses the existing `_active_processor_result_to_bundle()` function for the disclosure route. This is correct and prevents duplication. The function already handles `missing` field families by calling `_field_from_family()`, which returns `ExtractedField(value=None, anchors=(), extraction_mode="missing", ...)` for missing families/fields.

However, `_active_processor_result_to_bundle()` takes `profile_result` as input for `index_profile` (line 576: `index_profile=profile_result.index_profile`). The disclosure route also loads `ParsedAnnualReport` (Contract Decision #2, step 1), so `profile_result` is available. This is consistent.

### F7. `bond_risk_evidence` and `drawdown_metric` for Disclosure Route — Severity: Low

The existing `_extract_active_fund_via_processor()` calls `_load_drawdown_metric_for_bond_fund()` and `extract_bond_risk_evidence()`. The plan's Contract Decision #2 does not mention whether the disclosure route should also compute `bond_risk_evidence` and `drawdown_metric`.

Looking at the plan's output semantics (lines 244-276), it says `StructuredFundDataBundle` may contain fields "projected from processor field families" and "existing NAV degraded-unavailable semantics." But `bond_risk_evidence` is **not** from processor field families — it's from the existing `extract_bond_risk_evidence()` function called after processor extraction.

**Risk:** The implementation worker may omit `bond_risk_evidence`, `portfolio_managers`, and `risk_characteristic_text` (which are also not from processor field families but from the `_extract_active_fund_via_processor` path). The plan should clarify whether these fields are included in the disclosure route's bundle or left as default missing.

### F8. `fund_agent/fund/README.md` Change Scope — No Finding

The plan correctly identifies that `README.md` must be updated if `data_extractor.py` changes (per AGENTS.md trigger rule). The change description is bounded: "Update only current Fund package boundary wording." This is appropriate.

### F9. Forbidden File List Completeness — No Finding

The forbidden file list (lines 113-124) covers all files that should not be touched:
- Document repository internals
- Processor contracts (not to be changed)
- Concrete processor implementations (not to be changed)
- Candidate modules (not to be imported)
- Service/UI/Host/renderer/quality-gate files
- Root README, design/control/startup docs

This is complete and correct.

### F10. Import Boundary Enforcement — No Finding

The plan's import boundary (lines 280-304) correctly:
- Allows only `from fund_agent.fund.processors.contracts import FundDisclosureDocumentIntermediate` in `data_extractor.py`
- Forbids `from fund_agent.fund.documents.candidates...`
- Forbids direct consumer imports from Service/UI/Host/Agent/template/audit/extractors

This matches the current codebase's import patterns.

## Conclusion

**Pass with risks.**

The plan is code-generation-ready for the S5 implementation gate. It correctly identifies the current code state, preserves the production default path, maintains candidate-only boundaries, and defines explicit fail-closed behavior. The import boundaries, forbidden file list, and non-goals are well-defined.

The material findings are:
- **F2** (medium): `source_kind="annual_report"` in dispatch key lacks rationale — could cause future `supports()` conflicts.
- **F3** (medium): Missing explicit test for the `contract_status="missing"` success path with a non-candidate admitted intermediate.
- **F7** (low): Disclosure route's handling of `bond_risk_evidence`, `portfolio_managers`, and `risk_characteristic_text` (non-processor-field-family fields) is unspecified.

These risks are addressable by the implementation worker with clarifying comments or minor plan amendments; they do not block the planning gate from passing.
