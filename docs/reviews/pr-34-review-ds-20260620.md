## Findings

### 1-未修复-[低]-`_PRODUCT_ESSENCE_LABELS[_RISK_CHARACTERISTIC_OUTPUT_PATH]` 共享 label 配置无显式解耦保护

- **入口/函数**: `_collect_risk_characteristic_table_candidates` / `_collect_risk_characteristic_paragraph_candidates` → `_PRODUCT_ESSENCE_LABELS`
- **文件(行号)**: `fund_disclosure_processor.py:3832, 3916`
- **输入场景**: 未来 gate 需要为 product_essence 和 core_risk 使用不同的风险收益特征 label 集合时。
- **实际分支**: 两个中性 helper 都通过 `_PRODUCT_ESSENCE_LABELS[_RISK_CHARACTERISTIC_OUTPUT_PATH]` 获取 label = `("风险收益特征", "基金风险收益特征")`。
- **预期行为**: 当前两个家族对 risk_characteristic_text 有同一语义定义，共享 label 配置是正确的。
- **实际行为**: 对共享 label 集合的任何修改会同时影响 product_essence 和 core_risk 两个家族的提取行为，但没有编译时/测试时的解耦强制。
- **直接证据**: `fund_disclosure_processor.py:3832` (`labels = _PRODUCT_ESSENCE_LABELS[_RISK_CHARACTERISTIC_OUTPUT_PATH]`) 与 `:3916` 均直接引用 product_essence 命名空间下的配置常量，而非独立的 risk_characteristic label 常量。
- **影响**: 未来若两族需要分化的 label 语义，当前结构缺少显式边界。当前 25/25 product_essence focused tests 已确认无回归。
- **建议改法和验证点**: 后续 gate 若需分拆 label，应将 `_PRODUCT_ESSENCE_LABELS[_RISK_CHARACTERISTIC_OUTPUT_PATH]` 抽取为独立常量（如 `_RISK_CHARACTERISTIC_LABELS`）并分别验证两族行为。
- **修复风险（低）**: 当前无行为 bug，纯结构性考虑。
- **严重程度（低）**: 当前 scope 内无功能影响；仅在后续 label 分化时成为 blocker。

### 2-未修复-[低]-`_core_risk_status()` 二元模型对 deferred_role gaps 消费者可能造成疑惑

- **入口/函数**: `_core_risk_status` / `_core_risk_source_truth_gaps`
- **文件(行号)**: `fund_disclosure_processor.py:4791-4806, 4735-4788`
- **输入场景**: 消费者读取 `core_risk.v1.status == "accepted"` 并假定该族已完整覆盖。
- **实际分支**: `_core_risk_status` (line 4804) 仅检查 `"risk_characteristic_text" in value` 即返回 `"accepted"`。
- **预期行为**: `accepted` 表示该 gate scope 下的唯一 required subvalue 已获得。四个 `required=False` `deferred_role` gaps 供消费者区分"有意不完整"与"完整族"。

- **直接证据**: `_core_risk_status:4804` 单条件判断；`_core_risk_source_truth_gaps:4777-4787` 追加 `required=False` deferred_role gaps。plan 第 62 行注明这是刻意单 subvalue 设计且后续多 subvalue gate 需重写。

- **建议改法和验证点**: 后续多 subvalue gate 重写 `_core_risk_status()` 时同步更新 consumer 文档。当前无需修改。
