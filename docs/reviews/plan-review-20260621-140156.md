## Findings

### 001-未修复-中-New helper may diverge from existing row-field oracle builder

- **位置**: `Design Decisions` / `Slice A - Tests-only oracle helper`
- **问题类型**: 测试缺口 / 过度耦合 / review 不可验收
- **当前写法**: The plan allows the new helper to “move or copy” the currently needed oracle/report-building mechanics, and says not to import private functions from `test_small_golden_set_extractor_correctness.py`.
- **反例/失败场景**: Implementation copies a subset of the report builder into `small_golden_oracle_helpers.py`, but leaves the existing row-field tests on their old private builder. The two builders then diverge: row-field tests prove one synthetic `ParsedAnnualReport` shape, while integration tests prove another. The evidence chain no longer proves that the same accepted oracle input travels from row-field extraction into ProcessorRegistry/output persistence.
- **为什么有问题**: The stated goal is to connect existing retained-excerpt row-field correctness to the registry/output path. That connection is only reviewable if both test layers share the same oracle loading and report construction code, or if the new test explicitly proves equivalence.
- **直接证据**:
  - Existing row-field correctness currently uses private helpers in `tests/fund/test_small_golden_set_extractor_correctness.py`.
  - The plan permits copying helper logic instead of sharing it.
- **影响**: False confidence; implementation can pass while proving a parallel synthetic input rather than the accepted row-field input.
- **建议改法和验证点**:
  - Require extracting the shared oracle loader/report builder into `tests/fund/small_golden_oracle_helpers.py`.
  - Update `tests/fund/test_small_golden_set_extractor_correctness.py` to use the shared helper for oracle loading/report construction.
  - Keep old row-field assertions unchanged.
  - Validation must include the full small golden row-field test file plus the new integration test.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 中
