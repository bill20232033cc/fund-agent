## Findings

### 001-未修复-中-`_fact_hard_gate` fail 判定依赖 dimension status 与 issue severity 之间的涌现不变量

- **入口/函数**: `_fact_hard_gate()` → `_confirm_fact_v2()` → `confirm_chapter_evidence_v2()`
- **文件(行号)**: `fund_agent/fund/evidence_confirm.py:1165`
- **输入场景**: 新增一个维度函数返回 `status="fail"` 但只产生 `severity="reviewable"`（而非 `severity="blocking"`）的 issue
- **实际分支**: L1165 `if failed > 0 and blocking_issue_ids:` — 该条件要求同时满足 `failed > 0` 和 `blocking_issue_ids` 非空。若 fail 维度的 issues 全是 reviewable，`blocking_issue_ids` 为空，条件为 False，落入 L1167 `elif warned > 0 and not blocking_issue_ids:`；若 `warned==0`，一路 fall through 到 L1172 `else: status = "pass"`，硬门控错误返回 pass
- **预期行为**: 维度 `status="fail"` 应独立驱动 hard gate fail，不应依赖 issue severity 作为代理信号
- **实际行为**: 当前所有 fail 维度都产生 blocking issues，未触发该路径。但门控逻辑将 `failed > 0` 与 `blocking_issue_ids` 耦合，没有结构性强约束
- **直接证据**: L1156 `blocking_issue_ids` 来自全局 issues 而非 fail 维度专属 issues；L1165 的 `and` 条件使得单独 `failed>0` 不足以驱动 fail
- **影响**: 若未来新增维度（如语义性 audit 维度）以 fail+reviewable 组合返回，hard gate 会静默降级为 pass/warn。当前行为正确，但属于维护性脆弱点
- **建议改法和验证点**: 将 hard gate fail 判定改为 `if failed > 0: status = "fail"`（直接以维度失败驱动），或增加运行时断言 `assert not (failed > 0 and not blocking_issue_ids)`。若保持当前逻辑，至少在 `_fact_hard_gate` docstring 中记录该不变量
- **修复风险（低）**: 仅涉及门控判定逻辑，不影响维度计算
- **严重程度（中）**:

### 002-未修复-中-`_fact_score_v2` 的 blocking_fail_scores 过滤依赖 issue severity 耦合

- **入口/函数**: `_fact_score_v2()` → `_confirm_fact_v2()`
- **文件(行号)**: `fund_agent/fund/evidence_confirm.py:1293-1302`
- **输入场景**: 同 finding 001——新增维度 `status="fail"` 但 issue 为 reviewable
- **实际分支**: L1294-1300 的 `blocking_fail_scores` 推导式要求 `d.status == "fail"` 且 `d.issue_ids` 与 `hard_gate.blocking_issue_ids` 有交集。若 fail 维度的 issue 非 blocking，其 score 被排除在 `blocking_fail_scores` 之外，L1301 的 `if blocking_fail_scores:` 为 False，L1304 返回 `raw_average`，绕过 score cap
- **预期行为**: fail 维度 score 应无条件进入 score cap 计算
- **实际行为**: 当前所有 fail 维度均产生 blocking issues，条件始终满足。但 cap 逻辑依赖 issue severity 作为维度 score 的代理筛选条件，而非直接使用 `d.status == "fail"`
- **直接证据**: L1296-1299 的 `any(issue_id in hard_gate.blocking_issue_ids for issue_id in d.issue_ids)` 交叉检查
- **影响**: 同 finding 001——未来维度变更可能绕过 score cap。当前所有路径正确
- **建议改法和验证点**: 将 `blocking_fail_scores` 计算改为直接筛选 `d.status == "fail"` 的维度 score：`blocking_fail_scores = [d.score for d in dimension_results if d.status == "fail" and d.score is not None]`
- **修复风险（低）**: 行为等价（当前所有 fail 维度的 issues 均为 blocking），无测试回归
- **严重程度（中）**:
