## Findings

### CR-S1-001 — 新增 provenance 字段作为必填 dataclass 参数，破坏既有 quality gate 集成入口

- Severity: blocking
- Status: accepted
- Entry point: `tests/fund/test_quality_gate_integration.py::_summary()` 构造 `EvidenceConfirmProductionSummary`
- Triggering input: 既有 quality gate 集成测试传入 v1 形状 summary fixture，不包含 S1 新增 provenance 字段
- Actual branch: Python dataclass 构造器直接要求所有字段必填
- Expected behavior: S1 在引入 summary schema v2 字段时，不应让既有 quality gate 集成入口在进入 ECQ 投影逻辑前因构造器参数缺失崩溃；如果 S2 才接 quality gate projection，S1 至少要保持旧 fixture/调用方可构造或同步更新该测试入口
- Actual behavior: `EvidenceConfirmProductionSummary.__init__()` 抛出 `TypeError`，导致 `tests/fund/test_quality_gate_integration.py` 11 个用例失败
- Direct evidence:
  - `fund_agent/fund/evidence_confirm_production.py:95-99` 新增 `provenance_status`、`minimum_provenance_tier`、`provenance_missing_fact_count`、`strict_precision_residual_count`、`strict_precision_issue_ids`，且没有默认值
  - `tests/fund/test_quality_gate_integration.py:796-815` 既有 `_summary()` helper 仍按旧字段构造 summary
  - 本地复现命令：`uv run pytest tests/fund/test_quality_gate_integration.py -q`
  - 结果：`11 failed, 9 passed`，失败根因为 `TypeError: EvidenceConfirmProductionSummary.__init__() missing 5 required positional arguments: 'provenance_status', 'minimum_provenance_tier', 'provenance_missing_fact_count', 'strict_precision_residual_count', and 'strict_precision_issue_ids'`
- Impact: S1 虽然 focused tests 通过，但仓库内既有 quality gate integration suite 当前不可运行；这会把 schema v2 的落地变成跨 slice 同步修改要求，S1 不能作为可接受 slice checkpoint
- Fix direction: 在 S1 fix gate 内选择其一：
  - 给新增 provenance dataclass 字段提供安全默认值：`provenance_status="not_run"`、`minimum_provenance_tier="none"`、计数为 `0`、issue tuple 为 `()`，保持既有构造入口可用，同时由 `summary_from_repository_result()` 继续显式填充真实 v2 provenance
  - 或把 `tests/fund/test_quality_gate_integration.py` helper 更新纳入 S1，并把该测试命令加入 S1 validation
- Fix risk: 默认值方案需要确认不会把生产 summary 的真实 provenance 误默认为 pass；应只使用保守 `not_run/none/0/()`，并补跑 S1 focused tests 加 quality gate integration suite
