- **直接证据**: `processor.py:1321-1348` (S6-D)、`processor.py:1680-1707` (S6-E)、`processor.py:2024-2051` (S6-F)，三处实现逐行比对 structually identical。

## Residual Risk

1. **Token false positive 风险（已知，plan §10 已记录）**: Generic token `风险` 在 table heading 包含 "基金简介"（guard token for risk_characteristic）时可通过。这可能导致在该 table 内任意 cell 出现 `风险` 字即产生 candidate record。plan 对此已接受为 locator-only 边界内的已知风险；后续 field-extraction gate 负责。

2. **重叠 family 回归测试覆盖范围**: `test_core_risk_selector_preserves_overlap_family_semantics` 使用单一 fixture 配置（四个 section 分别命中 S6-B/C/D/E + 一个 S6-F section）。该测试覆盖了 record count、path order、gap semantics、value 和 anchors 的一致性比较，但仅覆盖 section 级匹配。paragraph 级和 table/cell 级的 S6-B/C/D/E 回归未在 S6-F overlap test 中覆盖。S6-B/C/D/E 各自的独立测试套件不受影响；此处仅指 overlap test 的覆盖深度有限。

3. **`_cell_guard_context` role-invariant 设计对 future role 扩展的适配风险**: `_core_risk_cell_guard_context()` 不接受 `role` 参数，与 S6-D/S6-E 的 `_manager_profile_cell_guard_context(role, ...)` 和 `_investor_experience_cell_guard_context(role, ...)` 不一致。若未来 core_risk 某 role 需要更宽松的 cell guard context（类似 S6-D `manager_alignment` 的 cell-text-inclusive guard），将需要签名变更。plan 明确要求 role-invariant，当前无行为错误。

4. **Docs sync 审查**: `docs/design.md` v2.27 和 `fund_agent/fund/README.md` 的 S6-F 措辞均保持 candidate-only、not_proven、NOT_READY 边界，未声称 source truth、parser replacement、field correctness、readiness 或 release。未发现越权表述。

5. **未覆盖的 adversarial 场景**: 
   - 超大输入（数万 sections/paragraphs/cells）下的性能：`_matches_any_token` 使用 nested loop O(n*m)，其中 n 为 texts 数量，m 为 tokens 数量；每个 candidate 匹配固定开销可接受。
