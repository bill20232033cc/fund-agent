### 逐项证据

**1. disclosure_intermediate route dispatch 基于 ParsedAnnualReport 的 classified_fund_type**

- `data_extractor.py:310-311`：始终先通过 `FundDocumentRepository.load_annual_report()` 加载 ParsedAnnualReport
- `data_extractor.py:332`：`classified_fund_type = _classified_fund_type(profile_result.basic_identity)` —— 分类来自 ParsedAnnualReport 的 profile 结果，非 FDD intermediate
- `data_extractor.py:334-342`：`disclosure_intermediate is not None` 时才进入 FDD 路径，且传入 `classified_fund_type` 参数
- `data_extractor.py:463-466`：若 `classified_fund_type is None`，在 dispatch 前 fail-closed，不会由 FDD intermediate 决定基金类型
- `data_extractor.py:468-475`：`FundProcessorDispatchKey(fund_type=classified_fund_type, ...)` —— dispatch key 的 fund_type 完全来自 ParsedAnnualReport 分类

**2. shared base + 六类 concrete FDD processor 无串类、无 unsupported gap**

- `fund_disclosure_processor.py:772-804`：`_FundDisclosureDocumentFundProcessor` 基类，`supports()` 三重匹配 `fund_type == supported_fund_type AND report_type == "annual_report" AND intermediate_kind == "fund_disclosure_document.v1"`
- 六类 concrete processor（`fund_disclosure_processor.py:933-978`）：
  - `FundDisclosureDocumentProcessor` → active_fund
  - `IndexFundDisclosureDocumentProcessor` → index_fund
  - `EnhancedIndexDisclosureDocumentProcessor` → enhanced_index
  - `BondFundDisclosureDocumentProcessor` → bond_fund
  - `QdiiFundDisclosureDocumentProcessor` → qdii_fund
  - `FofFundDisclosureDocumentProcessor` → fof_fund
- 五类 non-active（index/enhanced_index/bond/qdii/fof）全部覆盖，覆盖全部六种 `FundType`
- identity validation：
  - `data_extractor.py:321-325`：`_validate_disclosure_intermediate_identity` —— facade 入口身份校验
  - `fund_disclosure_processor.py:981-1053`：`_check_identity` —— processor 内部 intermediate_kind / document_kind / fund_code / report_year 四维校验
  - `data_extractor.py:619-658`：`_validate_processor_result_identity` —— processor 结果与 dispatch key 五字段交叉校验
- source-truth proof：
  - `fund_disclosure_processor.py:1181-1227`：`_validate_source_truth_admission` —— proof 存在性、类型、身份三向交叉校验（proof ↔ context ↔ content_intermediate）
  - 无 proof 或 proof 非法时全部字段族附加 `source_truth_admission_missing` / `source_truth_admission_invalid` gap
- candidate-only boundary：
  - `fund_disclosure_dispatch.py:119-125`：`candidate_boundary is not None` 时 `admitted=True, contract_status="blocked"`
  - `fund_disclosure_processor.py:895-906`：`contract_status == "blocked"` 时跳过 source-truth admission 校验，`source_truth_extraction_allowed=False`
  - `fund_disclosure_processor.py:1208-1209`：即使 content intermediate 存在，`candidate_boundary is not None` 也会使 `_validate_source_truth_admission` 返回 invalid
  - 字段族仅附加 candidate evidence (`FundCandidateEvidenceRecord`)，不会误提升为 source truth

**3. registry 与 __init__.py 注册/导出正确，不抢 parsed_annual_report.v1 route**

- `registry.py:143-173`：`create_default()` 注册 6 个 parsed_annual_report processor（priority 90-100）+ 6 个 FDD processor（priority 50）
- `supports()` 的 `intermediate_kind` 维度保证：
  - `active_annual.py:231`：parsed annual processor 仅匹配 `intermediate_kind == "parsed_annual_report.v1"`
  - `fund_disclosure_processor.py:802`：FDD processor 仅匹配 `intermediate_kind == "fund_disclosure_document.v1"`
  - 两套 processor 互不交叉命中
- priority 排序只在同 `supports()` 返回 True 的 processor 间生效，不会跨 intermediate_kind 匹配
- `__init__.py:18-25,31`：六个 FDD processor 全部导出（FundDisclosureDocumentProcessor + 五个 non-active 子类）

**4. 未违反 AGENTS.md 边界约束**

- FundDocumentRepository 边界：
  - `fund_disclosure_processor.py` 不 import `FundDocumentRepository`、PDF/cache/source helper、Docling
  - `fund_disclosure_dispatch.py` 同样不触碰文档仓库
  - FDD 中间态通过 `FundDisclosureDocumentIntermediate` Protocol 接入 processor，不直接依赖具体文档仓库实现
- EvidenceSourceKind / EvidenceAnchor：
  - `EvidenceSourceKind = Literal["annual_report", "external_api", "derived"]` 定义在 `extractors/models.py:11`，未被扩展
  - `EvidenceAnchor` 在 `extractors/models.py:87-107` 定义，字段集合未被修改
  - `chapter_facts.py` 中的 `ChapterEvidenceSourceKind` 是独立类型（加了 `"unknown"`），不是对 `EvidenceSourceKind` 的扩展

  - 未在 Service/UI/Host/renderer/quality gate 层出现

## Residual Risk

- FDD processor 的字段族 body extraction（`_extract_product_essence_source_truth` 等六个 source-truth extractor）依赖 `FundDisclosureDocumentContentIntermediate` 的 section/paragraph/table 结构，当前这些 extractor 在非 proof-positive 路径下全部返回 candidate-only evidence。若后续 content intermediate 的 locator stability 或 cell/paragraph text 规范化规则变更，可能导致字段族 extraction 行为变化——这属于 FDD schema gate 的待验证范围，不在本次 review scope 内。
- `fund_disclosure_processor.py` 文件体积较大（~9170 行），主要来自 6 个 source-truth extractor 的字段级匹配逻辑。当前文件结构在 processor gate 内可接受，但后续若新增字段族或 fund type，应考虑按 field family 或 fund type 拆分 extractor 模块以降低单文件维护负担。这属于 maintainability 观察，不构成 correctness/stability finding。
