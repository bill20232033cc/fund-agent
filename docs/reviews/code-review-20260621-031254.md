## Findings

### 01-未修复-中-_blocked_result 双份实现签名不一致，维护期有漂移风险

- **入口/函数**: `_blocked_result` 在 `active_annual.py:641` 与 `fund_disclosure_processor.py:9122`
- **文件(行号)**: `fund_agent/fund/processors/active_annual.py:641–688`、`fund_agent/fund/processors/fund_disclosure_processor.py:9122–9170`
- **输入场景**: 任一 processor 在 `extract()` 中调用 `_blocked_result` 构造 fail-closed 结果时
- **实际分支**: 两个模块各自调用自己模块内的 `_blocked_result`
- **预期行为**: 同名同语义的工具函数应保持签名与行为一致；如需差异化能力，应通过参数显式表达
- **实际行为**: `active_annual.py` 的 `_blocked_result` 不接受 `contract_status` 参数，直接按 `gap_code.startswith("unsupported_")` 推导；`fund_disclosure_processor.py` 的版本接受可选的 `contract_status: FundProcessorContractStatus | None = None` 参数，缺省时使用相同推导逻辑
- **直接证据**:
  - `active_annual.py:648` 签名：`def _blocked_result(processor_id, context, *, gap_code, message, source_boundary)` — 无 `contract_status`
  - `fund_disclosure_processor.py:9122–9129` 签名：`def _blocked_result(processor_id, context, *, gap_code, message, source_boundary, contract_status=None)` — 有 `contract_status`
- **影响**: 当前两个模块各自调用各自的版本，行为正确。但若未来任一 `_blocked_result` 需要新增参数或修改契约状态推导规则，另一份可能被遗漏，导致两个 processor 族对同一种 fail-closed 场景返回不同 `contract_status`，破坏 extractor 层的统一 fail-closed 语义
- **建议改法和验证点**:
  1. 将 `_blocked_result` 抽取到 `contracts.py` 或新建 `_processor_helpers.py` 作为单一定义
  2. 或至少在两个模块的 `_blocked_result` 上方加注释标注对端位置，降低漂移概率
  3. 验证点：确认所有 `_blocked_result` 调用方在提取公共版本后 contract_status 推导结果不变
- **修复风险（低）**: 纯内部重构，不改变对外行为；两个版本的缺省推导逻辑相同
- **严重程度（中）**: 非即时缺陷，但属于结构性维护风险；两份实现若漂移会影响整个 fail-closed 语义一致性

### 02-未修复-低-FDD 路由的 source_provenance 空值检查晚于 processor 执行

- **入口/函数**: `FundDataExtractor._extract_fund_disclosure_via_processor`
- **文件(行号)**: `fund_agent/fund/data_extractor.py:476`（读取 provenance）、`:491–492`（空值检查）
- **输入场景**: `disclosure_intermediate.source_provenance` 为 `None` 时
- **实际分支**: `source_provenance = disclosure_intermediate.source_provenance` → `processor = self._processor_registry.resolve(dispatch_key)` → `result = processor.extract(processor_input)` → `_validate_processor_result_identity(result, dispatch_key)` → 最后才 `if source_provenance is None: raise RuntimeError(...)`
- **预期行为**: provenance 缺失应在 processor resolution 前 fail-fast，避免无 provenance 的输入进入 processor 执行路径
- **实际行为**: processor 已完整执行（包括 admission 判定、identity 校验、字段族构造），之后才因 provenance 缺失抛出 RuntimeError；浪费了 processor 计算且让 fail-closed 语义的执行点后移
- **直接证据**: `data_extractor.py:476` 赋值、`:484` processor.extract 调用（早于 `:491` 的空值检查）、`:491–492` 空值抛出
- **影响**: 仅在 provenance 缺失的 pathological 场景下浪费 processor 计算；不产生错误状态或错误返回值（仍然 fail-closed）
- **建议改法和验证点**: 将 `if source_provenance is None` 检查移到 `dispatch_key` 构造之后、`registry.resolve()` 之前（大约 `data_extractor.py:476` 行附近）
- **修复风险（低）**: 纯执行顺序调整，不改变 fail-closed 语义
- **严重程度（低）**: 仅在退化输入下浪费计算，不影响正确性

## Open Questions

1. **非主动基金类型的 ParsedAnnualReport processor 字段族完整性**：当前六个 `_ParsedAnnualReportFundProcessor` 子类共享完全相同的窄 extractor 编排（`_collect_existing_extractor_fields`）。指数基金的 `tracking_error` 字段和债券基金的 `drawdown_metric` 字段在 processor 内部均按通用逻辑处理，再由 `_processor_result_to_bundle` 中的 `_tracking_error_for_fund_type` 和外部 `_load_drawdown_metric_for_bond_fund` 做基金类型裁剪。是否需要在 processor 层为不同基金类型提供差异化字段族 mapping（例如债券基金可能需要额外的信用风险字段）？当前设计预留了子类覆盖能力，但尚未使用。

2. **FDD processor 字段族全部 missing 的语义**：六个 `_FundDisclosureDocumentFundProcessor` 子类当前的 `extract()` 实现中，admission 通过后调用 `_field_families_for_intermediate`，但该函数在 `source_truth_extraction_allowed=False` 时返回全 missing 字段族。这是否意味着当前 branch 的 FDD 路由在非主动基金上仍是 "admission gate only, 字段抽取 deferred"？如是，是否需要在 control doc 中记录此状态？

## Residual Risk

- **六个 ParsedAnnualReport processor 子类的字段族行为完全一致**：当前每个子类仅通过 `processor_id` 和 `supported_fund_type` 区分，`extract()` 逻辑完全相同。如果未来某类基金需要不同的 extractor 编排（例如 QDII 需要额外读取境外投资章节），当前架构支持子类覆盖，但未被测试证明。建议在后续 gate 中为至少一种非 active_fund 类型编写差异化 extraction 测试。
- **legacy path 仍可被归类为 active_fund 的基金触发**：`_classified_fund_type()` 的最终 fallback 将无法匹配任何规则的基金归类为 `active_fund`（`fund_type.py:394–400`），这意味着所有基金最终都能被分类。legacy path (`_extract_bundle_direct_legacy_path`) 只在 `classified_fund_type is None` 时触发，而 `_classified_fund_type` 从不返回 None（最后总会返回 `active_fund`）。因此 legacy path 在当前实现中实际不可达。如果这是预期行为，建议在 docstring 中说明；如果 legacy path 仍需保留作为安全网，需重新审视 `_classified_fund_type` 的 fallback 策略。
- **FDD 路由的 `admit_disclosure_intermediate` 和 `_field_families_for_intermediate` 未在本次 scope 内逐行走读**：这两个函数属于 `fund_disclosure_processor.py` 的内部实现，本次 review 仅验证了它们的调用路径和 fail-closed 语义，未对内部字段抽取逻辑做逐行审查。
