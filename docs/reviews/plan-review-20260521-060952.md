# Plan Review — P8-S3 Source Fallback Policy Design

## Reviewed Target

- Plan: `docs/reviews/p8-s3-source-fallback-policy-plan-20260521.md`
- Scope: adversarial review before implementation of annual-report source fallback taxonomy and fail-closed provenance.

## Assumptions Tested

- The plan preserves the design truth that EID is the official annual-report primary source and Eastmoney is fallback only.
- Source policy remains inside Fund Capability document source code.
- The proposed exception contract is specific enough for implementation and tests.
- The plan does not weaken fail-closed behavior for official-source schema drift, identity mismatch, or PDF integrity errors.
- The plan does not require Service/UI/Engine changes or direct fund-document filesystem access.

## Findings

No blocking findings.

## Review Notes

- Architecture boundary is acceptable: implementation ownership stays in `fund_agent/fund/documents/sources.py` and document-source tests; production access remains through `FundDocumentRepository`.
- The plan correctly keeps `not_found` and `unavailable` fallback-eligible while making `schema_drift`, `identity_mismatch`, and `integrity_error` blocking categories.
- Adding `AnnualReportSourceFallbackBlockedError` is justified because existing `AnnualReportSourceMismatchError` and `AnnualReportSourceSchemaError` do not carry source/category/provenance, and `AnnualReportSourceAggregateError` means source exhaustion rather than intentional fallback block.
- The plan's provenance model is minimal: successful fallback keeps `AnnualReportSourceMetadata.fallback_used=True`; blocked fallback carries structured failures on the exception rather than inventing success metadata.
- Repository and adapter code do not currently depend on catching mismatch/schema specially; they allow lower-level exceptions to propagate. Therefore changing orchestrator mismatch/schema paths to a structured blocking exception is a controlled internal contract change.

## Open Questions

无。

## Residual Risks

- Existing orchestrator tests currently assert direct `AnnualReportSourceMismatchError` / `AnnualReportSourceSchemaError` from the orchestrator. Implementation must deliberately update those assertions to `AnnualReportSourceFallbackBlockedError` while preserving direct `EidAnnualReportSource` tests for source-level mismatch/schema behavior.
- `AnnualReportSourceIntegrityError` changes direct EID invalid-PDF tests from schema-style failure to integrity-style failure. Implementation should update source docstrings and tests together so the new category is not only exercised through orchestrator fakes.

## Conclusion

`pass`

The plan is code-generation-ready for P8-S3 implementation.
