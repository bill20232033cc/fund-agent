## Findings

### 01-未修复-高-identity mismatch plan references a non-existent gap code while forbidding contract edits
- **位置**: Target plan section 3.1 write set, 3.2 forbidden files, 3.4 extract logic, 3.6 identity validation, and 4.1 tests.
- **问题类型**: 不可直接实施 / 契约缺失
- **当前写法**: The plan forbids `fund_agent/fund/processors/contracts.py`, but requires identity mismatches to return an `identity_mismatch` gap and requires tests to assert `identity_mismatch`.
- **反例/失败场景**: Implementation follows the plan literally and constructs `FundExtractionGap(gap_code="identity_mismatch", ...)`. That value is not part of `FundExtractionGapCode`, so the implementation either violates the typed contract, weakens tests by ignoring the gap code, or edits `contracts.py` despite the plan forbidding that file.
- **为什么有问题**: A code-generation-ready plan cannot require a value outside the current public contract while simultaneously banning the file that defines the contract. This pushes a design decision into implementation and makes review criteria unstable.
- **直接证据**: Target plan lines 51-54 allow only processor/registry/tests; lines 60-64 forbid `contracts.py`; line 116 says identity mismatch returns `identity_mismatch`; line 168 repeats a single `identity_mismatch` gap; lines 196-197 require tests asserting `identity_mismatch`. Current `FundExtractionGapCode` literals in `fund_agent/fund/processors/contracts.py` lines 43-60 do not include `identity_mismatch`.
- **影响**: Implementation agent will either generate invalid contract usage, expand contracts without authorization, or silently remap identity mismatch to a different existing code without plan backing. Any of those outcomes blocks deterministic review and risks source-failure taxonomy confusion.
- **建议改法和验证点**: Choose one path before implementation. Preferred narrow fix: keep `contracts.py` forbidden and map dispatch/intermediate identity mismatch to an existing gap code such as `candidate_boundary_blocked` with `source_boundary="candidate_only"` for candidate/source identity violations, or `unsupported_intermediate` / `input_type_mismatch` where the mismatch is type-boundary rather than source identity. The fixed plan must name the exact existing gap code per mismatch category and update tests to assert only existing `FundExtractionGapCode` values. If a new `identity_mismatch` processor gap is desired, make `contracts.py` part of the write set and justify it as a public contract change with its own tests and review scope.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 02-未修复-中-implementation write set omits required documentation sync despite requiring code and tests changes
- **位置**: Target plan section 3.1 write set and section 7 README / Control Doc Sync Required.
- **问题类型**: 范围漂移 / 不可直接实施
- **当前写法**: The plan's implementation write set includes only processor source, registry, and tests. Later it says `fund_agent/fund/README.md`, `docs/implementation-control.md`, and `docs/current-startup-packet.md` must be updated after implementation acceptance.
- **反例/失败场景**: Implementation worker follows the exact write set and omits README/control sync; then acceptance requires documents that were not authorized. Or the worker updates docs anyway and violates the write set.
- **为什么有问题**: The repository rules require documentation synchronization after code/test changes, and phaseflow requires control-doc state tracking at key gates. The plan must say which docs belong to implementation evidence, accepted-slice bookkeeping, or controller-only follow-up.
- **直接证据**: Target plan lines 51-54 list only four implementation files. Lines 281-283 require README/control/startup updates after implementation acceptance. `AGENTS.md` requires README sync when `fund_agent/fund/` and `tests/` change, and phaseflow requires control-doc updates at gate checkpoints.
- **影响**: Review cannot tell whether missing docs are a planned deferred controller task or an implementation omission. This creates avoidable fix-loop churn at implementation acceptance.
- **建议改法和验证点**: Split write ownership explicitly. Add `fund_agent/fund/README.md` to implementation write set if the implementation worker should sync Fund package docs. State whether `tests/README.md` is unchanged because no test convention changes, or include it if conventions change. Keep `docs/implementation-control.md` and `docs/current-startup-packet.md` as controller bookkeeping unless the implementation gate explicitly delegates them, and state that they are not part of implementation worker write set.
