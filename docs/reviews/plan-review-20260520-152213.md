## Findings

### PR-01-未修复-中-explicit facets 与 fund_type 冲突时会静默触发错误规则
- **位置**: `Deterministic Evaluation Rules`
- **问题类型**: 契约缺失 / 不可直接实施 / 测试缺口
- **当前写法**: plan 写明规则触发条件是 `fund_type in rule.fund_types_any`，如果显式 `facets` 命中 `rule.facets_any`，也触发；空或不支持的 facet 不触发、不抛错。
- **反例/失败场景**: 调用方传入 `fund_type="bond_fund"`，同时错误传入 facet `指数增强基金`。当前计划会触发指数增强相关 ITEM_RULE，导致债券基金报告被判定应该渲染指数增强条件段落。
- **为什么有问题**: 项目硬约束要求“基金类型判断优先于通用分析”。显式 facet 可以作为更细粒度信息，但不能静默覆盖已识别的标准基金类型。否则 P6-S4 会引入一个新的不一致输入通道，后续 audit/renderer 会把错误 facet 当作同源事实。
- **直接证据**:
  - `fund_agent/fund/fund_type.py` 定义标准 `FundType` 为 `index_fund/active_fund/bond_fund/enhanced_index/qdii_fund/fof_fund`。
  - plan 的 `Deterministic Evaluation Rules` 明确“explicit facets are also triggering if any explicit facet is in rule.facets_any”，但没有要求 facet 与 `fund_type` 的支持关系一致。
  - 当前模板 facets 具有明显 fund type 归属，如 `指数增强基金` 属于 `enhanced_index`，不应与 `bond_fund` 并存后仍触发。
- **影响**: 实施 Agent 可能把任意用户/App facet 当作强触发条件；后续报告可出现跨基金类型条件段落；测试即使覆盖 facet trigger，也不会覆盖冲突输入。
- **建议改法和验证点**:
  - plan 增加 deterministic facet-to-fund-type map。
  - evaluator 规则改为：先校验所有显式 facets 是否属于当前 `fund_type` 可接受集合；若存在冲突 facet，fail closed with `ValueError`，不要触发。
  - 只有 facet 与当前 `fund_type` 一致时，facet 才能触发更细粒度规则。
  - 测试增加：`bond_fund + ("指数增强基金",)` raises `ValueError`；`enhanced_index + ("指数增强基金",)` triggers expected rules。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

### PR-02-未修复-中-segment_markers_any 使用普通短语会误判条件段落存在
- **位置**: `Built-in Manifest Content`、`Segment Presence Helper`
- **问题类型**: 测试缺口 / 不可直接实施 / false positive 风险
- **当前写法**: `segment_markers_any` 允许任一 literal substring 出现即判定 segment present。示例 markers 包含 `跟踪指数`、`投资哲学`、`超额收益稳定性`、`日均偏离度` 等普通短语。
- **反例/失败场景**: 第 1 章的普通产品说明可能写“跟踪指数为沪深300”，这会让 `chapter_1_index_constituents` 的 `rendered_segment_present(...)` 返回 true，即使并没有渲染“指数编制规则与成分股”这个条件段落。后续如果用该 helper 证明“conditional inactive + segment present -> violation”，会产生误报。
- **为什么有问题**: P6-S4 的 acceptance signal 是证明条件段落删除/保留策略，而不是普通词出现。segment marker 必须定位到段落边界或稳定 label，不能用段内可能自然出现的短语。
- **直接证据**:
  - plan 的 `Segment Presence Helper` 使用 literal substring，无 regex、无 heading parsing。
  - plan 的 manifest 示例将 `chapter_1_index_constituents.segment_markers_any` 设为 `指数编制规则与成分股, 跟踪指数, 前十大成分股`，其中 `跟踪指数` 是普通产品描述词。
  - 当前 renderer 第 1 章已有“业绩基准”“产品本质”等普通段落，未来很容易自然出现 `跟踪指数`。
- **影响**: 后续 audit 或测试会把普通文本误判为条件段落，导致 false positive；或者为了避免误报，implementation agent 会弱化 helper 断言，削弱 P6-S4 目标。
- **建议改法和验证点**:
  - plan 要求 `segment_markers_any` 使用稳定段落标题或唯一 label，首选 Markdown heading marker，例如 `#### 指数编制规则与成分股`。
  - 删除普通短语 markers：`跟踪指数`、`投资哲学`、`选股标准`、`超额收益稳定性`、`日均偏离度` 等。
  - `rendered_segment_present(...)` 仍可做 literal substring，但测试必须覆盖普通句子包含短词不应误判。例如文本含“本基金跟踪指数为...”但不含 `#### 指数编制规则与成分股` 时返回 false。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## Open Questions

1. ITEM_RULE manifest 是否需要引用 `rule_source_span` 或 `source_line_hint` 来增强源文档 drift 检查？当前不是 blocker，但可作为后续文档同步增强。
2. P6-S4 是否需要 design doc 同步？当前 plan 只加 README，不改 design；由于 `docs/design.md` 已定义 ITEM_RULE 概念，首版 manifest 不一定需要同步，但 implementation 若改变设计术语则必须更新。

## Residual Risks

- P6-S4 只做 manifest/evaluator/helper，不接 renderer/audit。它能证明 policy，不证明报告已经实际执行 ITEM_RULE。该风险应在后续 renderer/audit slice 跟踪。
- Built-in manifest 手写自模板草稿，仍可能与模板 drift。计划中的 source-fidelity tests 能降低但不能完全消除该风险。

## Conclusion

`fail` for immediate implementation.

方向正确，但实施前必须收紧 explicit facet 与 fund_type 的一致性规则，并把 `segment_markers_any` 改成唯一段落 marker，避免 P6-S4 在最核心的“是否应渲染/删除条件段落”判断上产生静默误触发。
