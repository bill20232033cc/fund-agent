### 次要观察（非阻断）

1. **`_active_processor_result_to_bundle` 的 `classified_fund_type` 类型标注**：参数标注为 `FundType`（L713），但 `_classified_fund_type()` 返回 `FundType | None`。两处调用点（`_extract_active_fund_via_processor` 和 `_extract_active_fund_disclosure_via_processor`）均已提前验证基金类型为 `"active_fund"`，运行时安全。`# type: ignore[arg-type]` 已标注在 L759 等位置。这是静态类型系统的已知 gap，不影响正确性。

2. **`source_provenance is None` 检查顺序**：L494-495 的 None provenance 显式检查位于 processor extraction（L487）和 identity validation（L488）之后。默认 `FundDisclosureDocumentProcessor` 在 admission 阶段即返回 blocked，因此 L494-495 在当前路径下为防御纵深。该顺序不会导致安全缺口，因为 processor 接收 None provenance 输入后会先行阻断。

3. **`_StubDisclosureIntermediate.test` 的 `failure_class` 字段**：测试 stub 实现的 `failure_class` 属性与 `FundDisclosureDocumentIntermediate` Protocol 一致（protocol L279）。facade code 不直接读取 `failure_class`，而是委托给 processor admission helper，符合 plan 设计。
