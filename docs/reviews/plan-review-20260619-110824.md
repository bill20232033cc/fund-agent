### Focus 2: Generic Guard 可实现性

**Verdict: Pass with material gap — guard specification is ambiguous for implementation**

The plan defines two context-dependent guards:

1. `portfolio_managers` guard (plan §3 table row 1): Generic header tokens require "same source heading path, table heading/caption, or section heading to include `基金经理` or `管理人`"

2. `manager_alignment` guard (plan §3 table row 4): `持有本基金` requires "same source text or heading path to include `基金经理`, `从业人员`, or `基金管理人`"

The problem is not the concept — the concept is sound. The problem is that the current `_matches_any_token()` implementation (fund_disclosure_processor.py:899-918) is a flat substring check with no context evaluation mechanism:

```python
def _matches_any_token(texts, tokens):
    normalized_tokens = tuple(_normalize_match_text(token) for token in tokens)
    return any(token and token in _normalize_match_text(text) for text in texts for token in normalized_tokens)
```

The implementation worker must invent guard mechanics for three unresolved questions:

- **Q1: What is the guard scope for a cell match?** When a cell's `cell_text` matches `姓名`, does the guard check the cell's own `heading_path`, the parent table's `heading_text`/`table_caption_or_nearby_heading`, the containing section's heading, or all of these? The plan says "same source heading path, table heading/caption, or section heading" — this is a disjunction of three different locator fields at three different structural levels. The implementation must decide which field to check and whether ALL matched cells in a table benefit from the table-level heading match.

- **Q2: Guard application order.** Is the guard applied as a PRE-filter (only scan generic tokens if the structural context already has `基金经理`) or a POST-filter (scan all tokens, then filter out records whose context fails the guard)? The S6-B/S6-C pattern does per-source-type per-role traversal, which suggests POST-filter is the natural fit, but POST-filter means the implementation must associate each candidate record with its context after matching.

- **Q3: "Same source text" for manager_alignment paragraph match.** When a paragraph's `text_normalized` contains `持有本基金`, does the guard check the same paragraph's text for `基金经理`, the paragraph's `heading_path`, or the containing section's heading? The plan says "same source text or heading path" — for paragraph records, both are available but have different semantics.

These are resolvable implementation decisions, not fundamental design flaws. An implementation worker who reads the S6-B/S6-C code will see the structural context fields available on each record type (`heading_path` on all types, `table_caption_or_nearby_heading` on tables, section heading via section lookup). The guard can be implemented as:
- For cell records: check `heading_path`, parent table's `heading_text`, and parent table's `table_caption_or_nearby_heading`
- For paragraph records: check the paragraph's own `text_normalized` and `text_raw` (for `manager_alignment` "same source text") AND `heading_path`
- For section/table records: check their own heading fields

This is a material gap because the plan claims "code-generation-ready" (plan §9) but the guard specification is at the conceptual level rather than the field-mapping level. The MiMo review classified this as LOW severity ("Edge cases in non-基金 contexts possible") and did not flag the specification ambiguity. I classify it as MEDIUM because the implementation worker must resolve three non-trivial design decisions that affect correctness.

**Recommendation**: The controller judgment should add a binding amendment specifying guard implementation mechanics before the implementation gate, OR the implementation gate should accept that the worker will propose a specific guard design and validate it through positive+negative tests (see Focus 4).

### Focus 3: holdings_snapshot 和 turnover_rate Scope Leakage

**Verdict: Pass — leakage risk is adequately contained**

The mapping table proves `holdings_snapshot` maps to `manager_profile.v1` (Ch3), `current_stage.v1` (Ch5), AND `core_risk.v1` (Ch6). `turnover_rate` maps to `manager_profile.v1` (Ch3) AND `core_risk.v1` (Ch6).

The plan's containment is three-layered and adequate:

1. **Token selection** (plan §3): Match tokens are disclosure/behavioral locators (`报告期末按行业分类的股票投资组合`, `换手率`), not interpretive tokens. The plan explicitly forbids `持仓`, `配置`, `风险`, `收益` as standalone tokens.

2. **Candidate-only boundary** (plan §4): All records are `candidate_only=True`, `field_correctness_status="not_proven"`, `source_truth_status="not_proven"`. Public `value={}`, `anchors=()`, `status="missing"`. Future selectors scanning the same source paths with their own tokens cannot be corrupted by S6-D's locator decisions because S6-D produces no public facts.

3. **Family-local dedupe** (plan §4): "Do not deduplicate across `product_essence.v1`, `return_attribution.v1`, and `manager_profile.v1`; each field family owns its own candidate evidence set." Future selectors independently scan.

One observation (not a finding): the plan includes `持仓集中度` as a match token for `holdings_snapshot` (plan §3 table row 5). "Concentration" is inherently evaluative — it implies a judgment about the degree of position concentration. The plan says "Do not infer concentration or style; locator only." This is a tension between matching an evaluative disclosure heading and claiming pure locator semantics. Acceptable for candidate-only because the match is on the disclosure heading text, not on a computed concentration metric, but the token name itself implies an interpretation. LOW severity, no action required beyond awareness.

### Focus 4: Required Tests Adequacy

**Verdict: Pass with one gap — missing positive guard test**

The plan specifies 10 tests. Tests 1-6 and 8-10 are well-specified and cover the critical paths:

| Test | Coverage | Adequate? |
|---|---|---|
| 1: candidate_evidence_only | Basic selector: status=missing, value={}, gap code | Yes |
| 2: preserve_candidate_boundary_fields | All records have correct boundary literals | Yes |
| 3: keep_other_families_without_candidate_evidence | Non-target families stay field_family_missing, S6-B/S6-C unchanged | Yes |
| 4: no_match_keeps_field_family_missing | Negative: neutral content → no evidence | Yes |
| 5: preserves_candidate_boundary_blocked_status | candidate_boundary input → blocked | Yes |
| 6: orders_dedupes_limits_and_truncates | Role order, source order, dedup, 16-record limit, excerpt <= 160 | Yes |
| 7: requires_context_for_generic_tokens | Generic tokens without context → no evidence | Partial |
| 8-10: S6-A/S6-B/S6-C/import regression | Existing tests pass unchanged | Yes |

**Gap: Test 7 has no positive counterpart.** Test 7 proves that `姓名`/`职务`/`任职日期` without `基金经理`/`管理人` context DOES NOT produce evidence, and that `持有本基金` without `基金经理`/`从业人员`/`基金管理人` context DOES NOT produce evidence. This is necessary but not sufficient.

Without a positive test proving that `姓名` WITH `基金经理` context DOES produce evidence, the guard implementation could be over-aggressive. For example: a section with `heading_text="基金管理人及基金经理情况"` containing a table with `姓名` column header — the guard must recognize that the section heading provides the required context. A naive implementation that only checks cell-level `heading_path` would miss this and produce a false negative.

This gap combines with the guard specification ambiguity (Focus 2) to create a compounded risk: the implementation worker must both invent guard mechanics AND has no positive test to validate them against.

**Recommendation**: Add test 7b: `test_manager_profile_selector_allows_generic_tokens_with_context` — a section with heading containing `基金经理` that has a table with `姓名`/`职务` column headers should produce portfolio_managers evidence. Similarly for `manager_alignment`: a section with `基金管理人` heading containing `持有本基金` text should produce manager_alignment evidence.

## Material Findings

### MF1 [MEDIUM] Generic guard specification is not implementation-ready

- **Location**: Plan §3 match group table, rows 1 and 4; cross-checked against `fund_disclosure_processor.py:899-918` `_matches_any_token()` implementation
- **Problem**: The plan defines context-dependent guards ("same source heading path, table heading/caption, or section heading") but the current matching infrastructure (`_matches_any_token`) is a flat substring check with no context evaluation. The implementation worker must resolve three design questions: guard scope for cell matches, guard application order (pre-filter vs post-filter), and "same source text" semantics for paragraph matches.
- **Evidence**: `fund_disclosure_processor.py` lines 899-918 show `_matches_any_token` takes flat `(texts, tokens)` with no structural context parameter. S6-B/S6-C selectors use only this function for matching.
- **Impact**: Implementation could produce false positives (failing to enforce guard) or false negatives (over-aggressive guard blocking legitimate matches). The plan's "code-generation-ready" claim (plan §9) is weakened.
- **Mitigation in plan**: None explicit. Test 7 covers negative guard cases but does not constrain implementation mechanics.
- **Suggested resolution**: Controller judgment should add a binding amendment specifying field-level guard mechanics, or accept that the implementation gate will propose and validate a specific guard design.

### MF2 [MEDIUM] Missing positive test for context-dependent generic token guards

- **Location**: Plan §7 test 7; cross-checked against test taxonomy in `test_fund_disclosure_processor.py`
- **Problem**: Test 7 proves only the negative case (generic tokens without context → no evidence). There is no positive counterpart proving that generic tokens WITH proper context DO produce evidence. Combined with MF1, this creates a compounded risk of over-aggressive guarding.
- **Evidence**: Plan §7 lists 10 tests. Test 7 description: "Generic `姓名` / `职务` / `任职日期` content without `基金经理` or `管理人` context must not produce `portfolio_managers` evidence." No test for "Generic `姓名` / `职务` content WITH `基金经理` context must produce evidence."
- **Impact**: An implementation that requires the context token to appear in every matched field (rather than in the structural parent) would silently fail to match valid manager roster tables, producing zero portfolio_managers evidence for valid input.
- **Mitigation in plan**: None. The plan relies on test 7's negative assertion plus implementation worker judgment.
- **Suggested resolution**: Add a positive guard test (test 7b) in the plan or as a controller amendment before implementation.

## Observations (Non-material)

### OBS1 [LOW] `持仓集中度` token is evaluative despite locator-only claim

Plan §3 match group for `holdings_snapshot` includes `持仓集中度` as a match token. "Concentration" implies a quantitative or qualitative assessment of position concentration. The plan says "Do not infer concentration or style; locator only." Matching on a disclosure heading that itself implies concentration is a semantic choice at the boundary of "pure locator." This is acceptable for candidate-only evidence but the tension should be acknowledged.

### OBS2 [LOW] turnover_rate guard asymmetry

`turnover_rate` has only negative token exclusions (`交易`, `买入`, `卖出`, `股票`) while `portfolio_managers` and `manager_alignment` require positive context (`基金经理`, `管理人`). `换手率` appearing in a risk disclosure section (e.g., "流动性风险" discussing turnover) would pass the negative guard and produce a false positive locator. The MiMo review noted this. The candidate-only boundary contains the blast radius.
