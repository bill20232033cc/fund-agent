## Findings

### 001. Generic repository `FileNotFoundError` is misclassified as ambiguous

- Severity: medium
- Entry point: `run_repository_bounded_evidence_confirm()`
- Triggering input: an injected or default repository whose `load_annual_report()` raises a plain `FileNotFoundError`
- Actual branch: `_classify_repository_failure()` checks `category`, `blocking_failure`, aggregate `failures`, then exact class names, but has no generic `isinstance(exc, FileNotFoundError)` branch.
- Expected behavior: accepted EC-P2 plan requires `FileNotFoundError -> not_found`, and `FundDocumentRepository.load_annual_report()` documents `FileNotFoundError` as the not-found contract.
- Actual behavior: plain `FileNotFoundError` falls through to `ambiguous_repository_failure`.
- Direct evidence:
  - `fund_agent/fund/evidence_confirm_sources.py:490-527` lacks a generic `FileNotFoundError` branch.
  - `fund_agent/fund/documents/repository.py:318-338` documents `FileNotFoundError` from `load_annual_report()`.
  - `docs/reviews/evidence-confirm-productionization-ec-p2-plan-20260622.md` requires `FileNotFoundError -> not_found`.
  - `tests/fund/test_evidence_confirm_sources.py:594-607` covers source class names and fake category attributes, but not plain `FileNotFoundError`.
- Impact: a valid repository not-found result can be reported as ambiguous, which weakens EC-P2 failure taxonomy and contradicts the accepted plan. It does not create a false pass, but it does break the promised source/PDF failure classification.
- Fix direction: add a generic `FileNotFoundError` branch after explicit category extraction and before class-name fallthrough, and add a no-live regression case for plain `FileNotFoundError`.
- Fix risk: low; `AnnualReportSourceNotFoundError` is already a `FileNotFoundError`, and both should map to the same `not_found` category.
