# Plan Review: RR-09 A4 Row-material Precision Plan

`docs/reviews/evidence-confirm-productionization-rr-09-a4-row-material-precision-plan-20260624.md`

Scope:

Adversarial review of the A4 planning artifact before no-live implementation. Review focused on Evidence Confirm materializer safety, row locator protocol contract, fail-closed behavior, architecture boundary, testability and residual routing.

## Assumptions Tested

- Processor-style locator parsing is the minimal useful A4 path.

## Findings

### PR-001-未修复-高-Bad Processor locator validation can still degrade into proof-bearing table references

- **位置**: `Slice A4-S1: Processor Row Locator Protocol Materialization` / `Exact allowed changes`
- **问题类型**: 契约缺失 / 测试缺口 / 最佳实践偏离
- **当前写法**: Plan says parsing or validation failures should "keep existing semantic downgrade behavior unless the failure is already covered by an existing blocking table/row preflight", while also listing `processor_row_locator_table_mismatch` and `processor_row_locator_invalid_row` issue reasons.
- **反例/失败场景**: An anchor has a compatible `anchor.table_id="page-3-table-0"` and a Processor locator `field=fee_schedule.management_fee; table_id=page-99-table-0; row=1`. The current plan could be implemented as "parse failed or validation failed, so degrade to table excerpt for `page-3-table-0` with an informational issue". That would preserve a proof-bearing annual-report reference even though the explicit Processor locator contradicts the anchor's table identity. Similar ambiguity exists for non-integer, negative or missing `row`.
- **为什么有问题**: A4's stated motivation is to improve proof precision without weakening Evidence Confirm. A contradictory Processor locator is not merely an unsupported semantic string; it is a source/projection consistency defect. Degrading it into a table-level proof can hide the defect and reproduce the same `coarse_reference_insufficient` class while making the pathway appear acceptable.
- **直接证据**: The plan's success signals require malformed/inconsistent locators not to create false row-level proof, but the exact changes do not specify whether mismatched embedded `table_id`, invalid row, missing row or missing embedded `table_id` are blocking no-reference outcomes or informational table downgrades. Current materializer behavior in `fund_agent/fund/evidence_confirm_sources.py` degrades unsupported semantic row locators to table excerpts when a compatible `anchor.table_id` exists.
- **影响**: Implementation agent could preserve table-level references for known-bad Processor locators. Review would then be unable to distinguish safe unsupported locator fallback from a projection consistency defect, and A4 live re-evidence could still fail with coarse references while the new diagnostic reason is non-actionable.
- **建议改法和验证点**:
  - Split unsupported arbitrary semantic locators from recognized Processor protocol failures.
  - For a recognized Processor locator prefix/shape with mismatched embedded `table_id`, missing embedded `table_id`, missing `row`, non-integer row, negative row or out-of-range row, return no reference with blocking issue.
  - Keep table/section downgrade only for non-Processor semantic locators or Processor locators that cannot be recognized as the bounded protocol.
  - Add tests asserting these recognized Processor protocol failures produce `status="fail"`, `references == ()`, blocking issue severity and no table excerpt fallback.
  - Keep the existing unsupported arbitrary semantic locator downgrade test as an informational pathway-pass regression.
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

## Special Lens Results

- **Architecture boundary review**: Pass. The plan keeps work inside `fund_agent/fund` and does not cross Service/Host/UI boundaries.
- **Best-practice review**: Fail on PR-001. Validation failures in a source-proof protocol need explicit terminal semantics, not ambiguous fallback.
- **Optimal-solution review**: Pass with PR-001 fixed. Processor row protocol parsing is more direct than broad token-matching expansion or schema migration.
- **Overengineering review**: Pass. The plan avoids cell-level proof and sub-anchor migration.
- **Overcoupling review**: Pass. The plan avoids coupling to product CLI, provider, Docling or external source helpers.

## Conclusion

`fail`

The plan is directionally sound, but PR-001 must be fixed before implementation because it controls whether A4 preserves fail-closed behavior for recognized bad Processor locators.
