## Findings

### 1-未修复-中-`_manager_profile_status` 在有歧义缺口时返回 accepted，违反已接受计划规则

- **入口/函数**: `_manager_profile_status` → `_extract_manager_profile_source_truth` → `_select_manager_profile_values`
- **文件(行号)**: `fund_disclosure_processor.py:2494-2511`
- **输入场景**: proof-positive FDD 内容包含全部五个 top-level subvalues，但其中某个子表（例如 `holdings_snapshot.top_holdings` 或 `portfolio_managers`）内部存在冲突行——部分行被省略并追加了 `ambiguous_table_or_locator` gap。
- **实际分支**: `_manager_profile_status` 仅检查 `all(top_level in value for top_level in _MANAGER_PROFILE_REQUIRED_TOP_LEVEL)` → 全部五个 top-level key 都在 value 中 → 返回 `"accepted"`。未检查 `ambiguous_paths` 是否非空。
- **预期行为**: 已接受计划 Section 6 明确规定："if any ambiguity gap is present, status must be `partial`, not `accepted`"。`_manager_profile_status` 应在返回 `"accepted"` 前确认无歧义缺口。
- **实际行为**: 当 holdings_snapshot 的内部行存在冲突（例如同一股票代码在不同 locator 出现不同公允价值），冲突行被省略但仍有非冲突行保留，`top_holdings` 非空 → `holdings_snapshot` top-level key 进入 value → `_manager_profile_status` 返回 `"accepted"`，同时 `_manager_profile_source_truth_gaps` 追加了 `ambiguous_table_or_locator` gap。下游消费者看到 `status="accepted"` 但实际存在内部歧义。
- **直接证据**:
  - `_select_manager_profile_holdings_rows` (行 2034-2069)：冲突行被省略，`ambiguous_paths.add(output_path)`，但 `accepted_rows` 仍可非空
  - `_select_manager_profile_portfolio_managers` (行 1172-1205+)：同样模式——冲突 entry 被省略但 portfolio_managers 仍可非空
  - `_manager_profile_status` (行 2494-2511)：仅检查 top-level key 存在性，不检查 `ambiguous_paths`
  - `_extract_manager_profile_source_truth` (行 1072-1109)：`ambiguous_paths` 仅在构造 gaps 时使用，不传入 `_manager_profile_status`
- **影响**: 内部存在歧义的 manager_profile 被标记为 `accepted` 输出，下游可能将含歧义的数据当作可信结果消费。违反计划 fail-closed 规则。
- **建议改法和验证点**:
  1. 修改 `_manager_profile_status` 签名以接收 `ambiguous_paths: set[str]`
  2. 在返回 `"accepted"` 前增加 `and not ambiguous_paths` 条件
  3. 修改 `_extract_manager_profile_source_truth` 中调用处传入 `ambiguous_paths`
  4. 验证：新增测试证明 5 个 top-level 全部存在但有内部歧义时 status=`"partial"`，且有 `ambiguous_table_or_locator` gap
- **修复风险（低）**: 改动范围极小（仅 `_manager_profile_status` 签名字段和一处调用点），不影响其他语义。
- **严重程度（中）**: 违反已接受计划的 fail-closed 歧义规则，可能导致下游消费错误标记的数据，但仅在同时存在歧义和全部 top-level key 的边缘场景触发。

### 2-未修复-低-`_manager_profile_cell_original_index` 找不到 cell 时静默返回 0

- **入口/函数**: `_manager_profile_cell_original_index`
- **文件(行号)**: `fund_disclosure_processor.py:2175-2195`
- **输入场景**: 当传入的 `target_cell` 对象与 `table.cells` 中的所有对象均不满足 `is` 同一性时（在当前正常代码路径中不会发生，因为 cell 源自同一 table 的 cells 迭代）。
- **实际分支**: `for` 循环遍历全部 cells 后未匹配 → 返回 `0`。
- **预期行为**: 不应返回误导的有效索引；应抛出异常或返回 sentinel 值由调用方处理。
- **实际行为**: 返回 `0`，`source_field_path` 变为 `table_blocks[{table_index}].cells[0]`，指向错误的 cell，使 anchor 的 source_field_path 不可信。
- **直接证据**:
  - 行 2192-2195：`for cell_index, cell in enumerate(table.cells): if cell is target_cell: return cell_index` → 未找到 → `return 0`
  - 调用方（行 1751-1764、2041-2050）：直接使用返回值构造 `source_field_path`
- **影响**: 在当前的正常代码路径中不会触发（cell 确来自同一 table 的 `_manager_profile_rows_by_index` 迭代，`is` 同一性成立）。但若未来代码重构改变了 cell 传递方式而未更新此函数，可能静默产生错误的 source locator。影响为静默数据损坏（错误 anchor），不会导致崩溃。
- **建议改法和验证点**: 将 `return 0` 改为 `return -1` 或抛出 `ValueError`，由调用方决定如何处理。
- **修复风险（低）**: 需要调整所有调用方处理不可能的返回值。
- **严重程度（低）**: 当前代码路径不会触发；属防御性编程问题。

## Open Questions

1. **Alignment 表格 guard context 复用 source_texts**：`_manager_profile_alignment_source_allowed(source_texts, source_texts)`（行 1746）中，表格 cell 上下文同时作为被检查文本和 guard context。当前所有 alignment test 均使用 paragraph-only 路径，无 table-only alignment 测试覆盖。是否需要新增表格版 alignment 测试以覆盖 `_manager_profile_alignment_row_value` 的 label/value 分离逻辑和 cell 级 guard 行为？

2. **Holdings row identity 回退可靠性**：`_manager_profile_holdings_row_identity`（行 2147-2172）在无 identity label 匹配时回退到 `next(iter(row))`（按 column_index 排序后第一个非 generic header 的列值）。若所有列均为 generic header，row 会返回 `None`（在 `_manager_profile_holdings_row_dict` 中），因此不会触发此回退。但若第一个非 generic 列恰好不是身份列（例如行业分布表中行业名称列在第二列、公允价值在第一列），回退身份可能不具区分性。计划中 identity_labels 已覆盖已知表头，实际触发概率低，但值得在 Slice 4 文档化。

## Residual Risk

- **测试未覆盖表格 alignment 路径**：6 个新增 Slice 3 测试中，alignment 相关测试（generic guard rejection、multi-locator first-anchor、all-groups-accepted）均使用 paragraph stub。`_manager_profile_alignment_row_context`、`_manager_profile_alignment_row_value` 和 table-path alignment 的 `_manager_profile_alignment_source_allowed(source_texts, source_texts)` 路径无直接测试覆盖。需在后续补充。
- **Holdings snapshot 只有第一行 anchor**：`_select_manager_profile_holdings_rows` 只返回 `anchors[0]` 作为公共 anchor，多行持仓的快照仅锚定首行。若首行 later 被修改或移除，剩余行的溯源链断裂。当前与 roster 的 anchor 策略一致（仅首行），但多行数据丢失溯源粒度的风险存在。
- **Holdings 身份回退依赖 dict 插入顺序**：Python 3.7+ dict 保证插入顺序，当前实现依赖 `sorted(cells, key=lambda item: item.column_index)` 的迭代顺序确定 `row` dict 的首个 key，作为无 identity label 时的回退身份。若未来 row dict 构造逻辑变更（如改用 comprehension 或并行构造），身份可能漂移。当前代码路径可证明正确但结构脆弱。
