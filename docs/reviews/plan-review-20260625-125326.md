## Assumptions Tested

- The plan is code-generation-ready and does not require implementation workers to invent a new contract.
- Atomic source facts have a clear output location in existing Processor/Extractor contracts.
- Default parsed annual migration can preserve child facts without fabricating provenance from old composite dicts.
- Derived composite views can remain compatibility-only without changing current CLI/renderer/quality-gate policy.
- The slice order is small enough to review and prevents cross-layer drift.
- Tests prove atomic extraction truth separately from composite compatibility views.

## Findings

### F1-未修复-高-atomic facts 的承载位置仍是“or equivalent”，Processor/Bundle 契约不可直接实施

- **位置**: Target plan sections `Target Contract`, `Public Contract Impact`, `S1 - Atomic contract and compatibility view primitives`
- **问题类型**: 契约缺失 / 不可直接实施 / public contract ambiguity
- **当前写法**: Plan 定义 `AtomicSourceFact`、`AtomicSourceFactStore` 和 `CompositeAnalysisView`，然后写“Add a keyword-default `source_facts: AtomicSourceFactStore` or equivalent compatibility accessor to the bundle/result surface”。S1 只说“Add bundle/result wiring so a `StructuredFundDataBundle` can carry atomic facts”。
- **反例/失败场景**: Implementation worker 可以选择把 atomic facts 放在 `StructuredFundDataBundle.source_facts`、`FundProcessorResult.source_facts`、`FundFieldFamilyResult` metadata、或仅放在 `data_extractor.py` 的临时 assembler 中。每一种都能声称满足 “or equivalent”，但它们对 FDD route、parsed annual route、chapter facts 和 Evidence Confirm 的数据流完全不同。后续 S2/S3/S4/S5 会因为读取来源不同而出现重复 adapter 或互相绕过。
- **为什么有问题**: 当前代码中 `FundProcessorResult` 只有 `field_families`、`gaps`、`anchors`、`source_provenance` 等字段；`FundFieldFamilyResult` 的 public value 是 dict；`StructuredFundDataBundle` 是 frozen/slots dataclass 且当前不含 source facts。`data_extractor.py` 当前通过 `_field_from_family()` 从 `FundFieldFamilyResult.value` 投影顶层 `ExtractedField`。如果计划不明确 atomic facts 首先属于哪个 contract，implementation worker 必须重新设计公共数据流，违反 plan gate “code-generation-ready” 要求。
- **直接证据**:
  - Plan: `Public Contract Impact` 写 “`source_facts: AtomicSourceFactStore` or equivalent compatibility accessor”。
  - Code: `fund_agent/fund/processors/contracts.py` 中 `FundProcessorResult` 当前只包含 `field_families: tuple[FundFieldFamilyResult, ...]`，没有 atomic fact surface。
  - Code: `fund_agent/fund/processors/contracts.py` 中 `FundFieldFamilyResult.value: dict[str, object]` 是现有字段族 value。
  - Code: `fund_agent/fund/data_extractor.py` 中 `StructuredFundDataBundle` 是 frozen/slots dataclass，当前不含 source facts。
  - Code: `fund_agent/fund/data_extractor.py` 中 `_field_from_family()` 只从 `family_result.value.get(field_name)` 和 `_anchors_for_family_field()` 投影 bundle 字段。
- **影响**: 实施 Agent 跑偏；不同 slice 可能把 atomic facts 接在不同层；review 无法判断 public contract impact；后续 Evidence Confirm 和 ChapterFactProvider 可能消费不同真源，导致“atomic source fact store”成为旁路 metadata 而不是真源。
- **建议改法和验证点**:
  - 在 plan-fix 中选择唯一承载路径，例如：`FundProcessorResult.source_facts: AtomicSourceFactStore = empty_store()` 是 Processor 输出真源；`StructuredFundDataBundle.source_facts` 只是 facade 携带同一 store；`FundFieldFamilyResult.value` 与 legacy `ExtractedField` 只由 store 派生。
  - 明确是否需要 `FundFieldFamilyResult.source_fact_ids` 或 `CompositeAnalysisView` registry；如果不需要，写明不得新增。
  - 明确 dataclass 兼容策略：frozen/slots 下只能新增 defaulted trailing field，禁止 positional construction break；所有 existing constructor tests 必须覆盖。
  - 验证点：processor result contract test 断言 parsed annual 与 FDD route 都在同一 `FundProcessorResult.source_facts` surface 输出 canonical ids；bundle projection test 断言 bundle 携带同一 store 且 composite fields 从 store 派生。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### F2-未修复-高-S2 只允许改 processor wrapper，无法从默认 parsed annual composite dict 安全产生独立子字段事实

- **位置**: Target plan section `S2 - Default parsed annual processor atomic emission`
- **问题类型**: 架构边界 / 不可直接实施 / 测试缺口 / no-fabrication risk
- **当前写法**: S2 允许模块主要是 `fund_agent/fund/processors/active_annual.py`、`fund_agent/fund/data_extractor.py` 和 tests；变化描述为“Preserve child paths already known by parsed annual matchers as atomic fact ids”并“Emit atomic facts for direct child facts in `fee_schedule`, `nav_benchmark_performance`, `manager_strategy_text`, and `manager_alignment`”。
- **反例/失败场景**: 当前 default parsed annual path 在 `ActiveFundAnnualProcessor` 中拿到的已经是 legacy extractor 输出的 `ExtractedField[dict]`，mapping 只有 `output_field_name` 顶层字段。若 S2 不允许改 `fund_agent/fund/extractors/profile.py`、`performance.py`、`manager_ownership.py` 或相应 result model，implementation worker 只能从 dict key 如 `management_fee`、`nav_growth_rate` 反推 atomic fact。这正是 plan 自己禁止的“从 composite dict shape 推断 child provenance”。
- **为什么有问题**: 用户目标是“每个字段独立提取”，不是在 processor 层把旧 composite dict 拆开。当前 code fact 显示 `active_annual.py` 的 `_field_scoped_anchors()` 明确把 anchors 附加为顶层 `source_field_path={mapping.output_field_name}`；`FieldFamilyMapping` 只记录顶层 `output_field_name`；默认 path 并没有 child fact carrier。FDD route 的 child output paths 不能证明 default parsed annual legacy extractors 也有等价 child provenance。
- **直接证据**:
  - Plan S2 allowed modules 不包含 `fund_agent/fund/extractors/profile.py`、`fund_agent/fund/extractors/performance.py`、`fund_agent/fund/extractors/manager_ownership.py`。
  - Code: `fund_agent/fund/processors/active_annual.py` 中 `_field_scoped_anchors()` docstring 是“顶层字段范围定位”，实现调用 `_with_source_field_path(anchor, mapping.output_field_name)`。
  - Code: `fund_agent/fund/processors/active_annual.py` mapping rows show `profile.fee_schedule -> fee_schedule`、`performance.nav_benchmark_performance -> nav_benchmark_performance`、`manager_ownership.manager_strategy_text -> manager_strategy_text`、`manager_ownership.manager_alignment -> manager_alignment`。
  - Code: `fund_agent/fund/data_extractor.py` 当前 `_field_from_family()` 从 `family_result.value.get(field_name)` 投影顶层字段。
- **影响**: S2 实施大概率会生成伪 atomic facts，或者卡住要求扩大 write set。前者会破坏 provenance truth；后者说明 plan 不可直接实施。最坏情况下 strict V2 看似改善，但只是把 composite dict keys 包装成 child source facts，仍没有真正独立提取。
- **建议改法和验证点**:
  - 将 S2 拆成两个明确 slice：
    - S2a: legacy parsed annual path 只输出 compatibility composite view，不从 composite dict 伪造 child facts；测试断言 absence/gap。
    - S2b: 修改实际 legacy extractors/result models，使 `profile`、`performance`、`manager_ownership` 直接产出 child-level `ExtractedField` 或 `AtomicSourceFact`，再由 processor 汇总。
  - 或者明确 default parsed annual atomic migration 在第一轮只覆盖已有直接 child provenance 的 FDD route，default parsed annual child facts 延后到 extractor split slice。
  - 验证点：测试必须证明每个 emitted atomic fact 的 anchor 来自 child-level extractor output，而不是从 dict key 或 top-level anchor 派生。
- **修复风险（低/中/高）**: 中到高
- **严重程度（低/中/高/严重）**: 高

### F3-未修复-中-S4/S5 让 ChapterFactProvider 与 Evidence Confirm 同时改源模型，缺少最小兼容 adapter 验收点

- **位置**: Target plan sections `S4 - ChapterFactProvider projection from atomic facts and derived views`, `S5 - Evidence Confirm atomic audit behavior`
- **问题类型**: 过度耦合 / 切片过粗 / 测试缺口
- **当前写法**: S4 要 `ChapterFactProvider` 从 `AtomicSourceFactStore` 投影，S5 要 Evidence Confirm materialization prefer atomic fact value and anchors。计划说保留 legacy projection，但没有指定一个最小 adapter contract，用于保证未迁移字段、无 atomic store 的旧 bundle、以及 derived view fallback 在两个模块中行为一致。
- **反例/失败场景**: Chapter facts 可能把 atomic facts 映射成 `field_path=AtomicSourceFact.<fact_id>`，但 Evidence Confirm 仍从旧 `ChapterFactEntry.value` flatten；或者 Evidence Confirm 直接读取 `AtomicSourceFactStore`，绕过 ChapterFactProvider 的 derived view dependency ids。两者都能局部测试通过，但 audit unit 和 writer fact unit 不一致。
- **为什么有问题**: 当前 `chapter_facts.py` 用 `_SOURCE_FIELD_IDS` 顶层 `structured.<field>` 和 `_fact_entry()` 生成 `StructuredFundDataBundle.<field_name>`；Evidence Confirm 的 `_material_tokens(fact.value)` 和 `_material_tokens_for_anchor_scope()` 当前基于 fact value 与 semantic locator。计划没有定义迁移期间 `ChapterFactEntry` 是否新增 `source_fact_ids`、`view_id`、或 dependency metadata，也没有规定 Evidence Confirm 以 ChapterFactEntry 为唯一输入还是可直接访问 store。
- **直接证据**:
  - Code: `fund_agent/fund/chapter_facts.py` `_SOURCE_FIELD_IDS` 仅定义 top-level `structured.fee_schedule`、`structured.nav_benchmark_performance`、`structured.manager_alignment`、`structured.manager_strategy_text`。
  - Code: `fund_agent/fund/chapter_facts.py` `_fact_entry()` 当前 `field_path=f"StructuredFundDataBundle.{spec.field_name}"`。
  - Code: `fund_agent/fund/evidence_confirm.py` 和 `fund_agent/fund/evidence_confirm_sources.py` 当前 materialization 入口围绕 `fact.value` 与 `row_locator`。
  - Plan S4/S5 没有明确 `ChapterFactEntry` 的新增字段或 Evidence Confirm 的 single source of truth。
- **影响**: Writer/auditor 可能继续不一致；atomic migration 变成两个平行接入点；review 难以证明“写作层自由组合独立字段”和“审计层审核同一独立字段”引用的是同一事实。
- **建议改法和验证点**:
  - 在 plan-fix 中添加一个最小 bridge contract，例如 `ChapterFactEntry.source_fact_ids: tuple[str, ...]` 与 `derived_view_id: str | None`，并规定 Evidence Confirm 只通过这些 ids 回查 atomic store / derived view dependency，不直接重新推断。
  - S4 只实现 chapter projection contract 和 compatibility fallback；S5 只消费 S4 暴露的 stable ids，不另开独立 mapping。
  - 验证点：同一 `fee_schedule.management_fee` 在 chapter fact、Evidence Confirm diagnostic 和 source fact store 中 fact id 相同；无 atomic store 时 legacy path 仍按旧 summary 行为退化。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 中

## Open Questions

- Atomic source facts 是否应该只进入 `FundProcessorResult.source_facts`，还是也进入 `FundFieldFamilyResult`？当前 plan 必须二选一或明确禁止某一层承载。
- Default parsed annual 第一轮是否必须覆盖 child-level atomic facts？如果必须，write set 需要包括 legacy extractor/result model；如果不必须，计划应承认 default parsed annual composite fields 在首轮只保留 compatibility/gap。
- `ChapterFactEntry` 是否新增 source fact dependency fields？如果不新增，Evidence Confirm 如何稳定知道一个 chapter fact 对应哪个 atomic source fact？

## Residual Risks And Suggested Tracking Destination

- `fund_disclosure_processor.py` 文件过大、职责集中。当前 plan 可以先以 adapter 迁移，但后续仍需要单独 refactor work unit；本 plan 不应把它作为当前 blocker。
- Live/PDF strict V2 evidence、B1 `017641 / 2024` runtime product CLI residual、release/readiness 均应继续留在后续 exact authorization gates；当前 review 未改变这些 residual owner。
- Existing unrelated dirty/untracked workspace residue remains outside this review.

## Final Plan Review Conclusion

`fail`

The plan direction is consistent with the accepted design truth, but it is not yet code-generation-ready. It leaves the core output surface for atomic facts ambiguous and asks default parsed annual migration to emit independent child facts without authorizing or specifying the legacy extractor changes needed to prove child provenance. Fix the plan before implementation.
