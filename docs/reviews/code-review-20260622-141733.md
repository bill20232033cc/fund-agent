## Findings

### EC-P1A-R1-未修复-中-negative-max-section-excerpt-chars-bypasses-bound

- **入口/函数**: `_section_excerpt` → `build_annual_report_evidence_confirm_references`
- **文件(行号)**: `fund_agent/fund/evidence_confirm_sources.py:437`
- **输入场景**: 调用方传入 `max_section_excerpt_chars < 0`（例如 `-1`、`-100`）的 `EvidenceConfirmReferenceBuildRequest`
- **实际分支**: 条件 `request.max_section_excerpt_chars >= 0` 为 `False`，不执行切片，`excerpt` 保持为 `_normalize_whitespace` 的完整结果
- **预期行为**: 按 plan 要求 section excerpt "bounded to request.max_section_excerpt_chars from the current section offsets only"，负数应被视为非法输入并 fail-closed（例如产生 `empty_section_excerpt` 或新增 `invalid_max_section_excerpt_chars` issue），或至少在 request 构造时被拒绝
- **实际行为**: 负数值静默地禁用了截断上限，返回完整 section text，无论多长。这使调用方可以通过传递 `-1` 绕过 plan 要求的上限约束
- **直接证据**: 第 437 行 `if request.max_section_excerpt_chars >= 0:` 仅对非负数执行 `excerpt[: request.max_section_excerpt_chars]`；`EvidenceConfirmReferenceBuildRequest` dataclass（第 55 行）的 `max_section_excerpt_chars: int` 类型注解不对运行时传入的负数值做任何校验（Python dataclass 不强制类型）
- **影响**: 静默绕过 section excerpt 上限。当前 EC-P1A 是 no-live 的，但未来 live gate（EC-P2）中若调用方误传负数，会导致超长 section text 进入 reference excerpt，进而影响 V2 value_match 行为和 auditability 判定
- **建议改法和验证点**: 在 `_section_excerpt` 中将负值视为等同于 0（fail-closed 为空 excerpt）或在 request 初始化后增加 invariant check。验证点：新增 `max_section_excerpt_chars=-1` 和 `max_section_excerpt_chars=0` 两项参数化测试
- **修复风险（低）**: 修改仅影响边界输入，正常路径（default=1200、显式正数）不受影响
- **严重程度（中）**: 非正常输入静默绕过安全约束，虽然当前 no-live 阶段调用方都是测试代码，但违反了 plan 明确要求的 bounded 语义

### EC-P1A-R2-未修复-低-dead-code-in-anchor-excerpt-dispatch

- **入口/函数**: `_anchor_excerpt`
- **文件(行号)**: `fund_agent/fund/evidence_confirm_sources.py:300-305`
- **输入场景**: 任何调用路径
- **实际分支**: 第 300-305 行的 `if anchor.row_locator:` 分支在逻辑上永远不可达。当 `anchor.row_locator` 为 truthy 时，要么 `not anchor.table_id` 为 True（被第 292-297 行捕获），要么 `anchor.table_id` 也为 truthy（被第 298-299 行的 `if anchor.table_id:` 捕获）
- **预期行为**: 无功能影响，但 dead code 增加了维护负担，后续修改者可能误以为存在第三条可达路径
- **实际行为**: 该分支永远不会被执行
- **直接证据**: 第 292 行 `if anchor.row_locator and not anchor.table_id:` 与第 300 行 `if anchor.row_locator:` 形成全覆盖：row_locator truthy 时 table_id 或为 falsy（命中 292）或为 truthy（命中 298），不存在第三种情况
- **影响**: 仅 maintainability — 无运行时行为错误
- **建议改法和验证点**: 删除第 300-305 行的不可达分支，或将第 300 行的注释改为 `# unreachable: captured above` 以明确意图
- **修复风险（低）**: 纯删除
- **严重程度（低）**: dead code，不影响 correctness

### EC-P1A-R3-未修复-低-no-test-for-zero-max-section-excerpt-chars

- **入口/函数**: `_section_excerpt`
- **文件(行号)**: `fund_agent/fund/evidence_confirm_sources.py:437-438`, `tests/fund/test_evidence_confirm_sources.py`
- **输入场景**: `max_section_excerpt_chars=0` 且 section 有非空文本
- **实际分支**: 第 437 行 `0 >= 0` 为 True，`excerpt[:0]` 得 `""`，第 439 行 `if not excerpt:` 为 True，返回 `empty_section_excerpt`
- **预期行为**: 行为正确（空字符串 → blocking issue），但无显式测试覆盖此边界
- **实际行为**: 无测试验证 `max_section_excerpt_chars=0` 产生 blocking issue 而非静默通过
- **直接证据**: `test_locator_and_section_failures_are_explicit` 只覆盖了 `blank_section` 导致空 excerpt，不覆盖 `max_section_excerpt_chars=0` 截断非空文本；`test_section_only_reference_uses_get_section_text_and_max_bound` 只覆盖正数截断。`test_default_request_is_not_proven_and_section_bound_is_1200` 只覆盖默认值。0 值边界无覆盖
- **影响**: 边界行为未经测试证明，未来修改可能引入回归
- **建议改法和验证点**: 在 `test_locator_and_section_failures_are_explicit` 参数化中增加一个 case：`("default", {}, "empty_section_excerpt")` 搭配 `max_section_excerpt_chars=0` 的 request。或新增独立测试
- **修复风险（低）**: 仅新增测试
- **严重程度（低）**: 当前实现行为正确，仅缺测试覆盖
