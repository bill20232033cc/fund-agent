### 1-未修复-低-_flatten_material_values 对非预期类型静默返回空元组

- **入口/函数**: `_flatten_material_values()` → `_material_tokens()` → `_confirm_fact()`
- **文件(行号)**: `evidence_confirm.py:668`
- **输入场景**: `ChapterFactEntry.value` 包含 `set`、`bytes`、自定义类实例等未在 `isinstance` 链中覆盖的类型
- **实际分支**: `evidence_confirm.py:668` — `return ()` 对所有无法识别的类型静默返回空元组
- **预期行为**: 对无法提取 material token 的类型，至少应产生可观测信号而非静默跳过
- **实际行为**: `_flatten_material_values(set_value)` → `()` → `_material_tokens` 返回空 → `_confirm_fact` line 303 `not tokens` → 若 fact 非 required → `not_applicable`, score=None，无 issue
- **直接证据**: `evidence_confirm.py:648-668`：isinstance 链覆盖 `None, bool, str, int, float, Decimal, dataclass, dict, tuple, list`，最后 `return ()` 捕获其余类型；`evidence_confirm.py:303-308` 中 `not tokens` 对非 required fact 走 `not_applicable`
- **影响**: 若未来 `ChapterFactEntry.value` 引入新类型（如 `set` 作为去重集合），该 fact 的 value 会被静默跳过而不触发任何 issue。当前 `ChapterFactEntry.value` 的类型限于 `dict/list/scalar/None/dataclass`，实际风险极低
- **建议改法和验证点**: 可考虑在 `_flatten_material_values` 末分支记录一个 debug-level 日志或返回 sentinel；或在 `_confirm_fact` 的 `not tokens` 分支对未知类型产生 informational issue
- **修复风险（低）**: 只影响当前不可达的代码路径
- **严重程度（低）**: 当前生产类型闭集内不可触发；属于防御性编程缺口
