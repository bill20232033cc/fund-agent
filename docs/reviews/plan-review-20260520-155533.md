## Assumptions Tested

- FQ5 can use manifest-derived facts in `extraction_score.py` while `quality_gate.py` continues consuming only `score.json`.

- App category conflict, unsupported fund type, and manifest resolution failure must remain blocking FQ5 mismatches.

## Findings

### P6S5-PLAN-001-未修复-高-classified_fund_type 多值冲突被计划归为 not_applicable
- **位置**: `Not-applicable Rules`、`Tests` 第 4 条。
- **问题类型**: 契约缺失 / 状态机漏洞 / 测试缺口。
- **当前写法**: plan 写明 `classified_fund_type` 缺失或跨行冲突时使用 `not_applicable`，并在测试中要求 “Missing or multi-value `classified_fund_type` yields `preferred_lens_status="not_applicable"`”。
- **反例/失败场景**: 同一基金 snapshot 同时出现 `active_fund` 与 `bond_fund`，且字段 `value_present=True`。这种情况下 FQ2/FQ4 不会因为缺失而阻断，App 类别也可能是未知；如果 FQ5 输出 `not_applicable`，`quality_gate.json` 不生成 block issue，报告会在基金类型矛盾的状态下继续进入后续流程。
- **为什么有问题**: 基金类型判断优先于通用分析，且 preferred_lens 路径依赖单一稳定基金类型。多值冲突是系统已经观测到的确定性矛盾，不是模板不适用；把它降级成 `not_applicable` 会隐藏 root cause。当前 `_unique_optional_text(...)` 已返回冲突 reason，但会把 value 置为 `None`，plan 若只看 `classified_fund_type is None` 就会丢失“缺失 vs 冲突”的区别。
- **直接证据**:
  - `docs/reviews/p6-s5-quality-gate-fq5-upgrade-plan-20260520.md` 的 `Not-applicable Rules` 写到 “`classified_fund_type` has conflicting values across rows” 使用 `not_applicable`。
  - `fund_agent/fund/extraction_score.py` 中 `_unique_optional_text(...)` 对多值返回 `(None, f"{key} 存在冲突值：...")`。
  - `fund_agent/fund/extraction_score.py` 当前 `_build_fund_quality_row(...)` 只把该 reason 拼入文本，后续 `_preferred_lens_status(...)` 不能区分缺失与冲突。
- **影响**: 实施 Agent 会按计划实现非阻断状态，导致同一基金类型矛盾无法被 FQ5/block 捕获；review 也难以从测试发现，因为 plan 明确要求错误断言。
- **建议改法和验证点**:
  - 把 fund type 多值冲突从 `not_applicable` 改为 `mismatch`。
  - 在 `_derive_contract_applicability(...)` 的输入中显式传入或解析 `fund_type_reason`，当 reason 表示 `classified_fund_type 存在冲突值` 时直接返回 `mismatch`，不进入 manifest evaluation。
  - 测试拆成两类：缺失 `classified_fund_type` -> `not_applicable`；多值 `classified_fund_type` -> `mismatch` 且 quality gate 生成 FQ5/block。
  - reason/message 必须包含冲突值，便于追溯。
- **修复风险（低/中/高）**: 低。
- **严重程度（低/中/高/严重）**: 高。

## Open Questions

- `app_category="货币基金类"` 且 `classified_fund_type` 为当前不支持的非货币类型时，应优先表达模板不适用还是类型冲突？建议实现顺序先判 fund type 多值冲突和 App 类别冲突，再判货币类不适用，避免已知矛盾被 `not_applicable` 掩盖。

## Residual Risks And Tracking

- P6-S5 仍只证明 contract / item-rule manifest applicability，不证明 renderer compliance；该风险已被 plan non-goal 和 docs guardrail 覆盖，应在后续 renderer/audit slice 继续跟踪。
- `quality_gate.json.rule_results` 是新增输出面，旧消费者若使用严格 schema 可能需要确认；当前仓库内 gate 消费者主要读 status/issues，实施时应保留顶层 status/issues 不变。

## Conclusion

`fail`

该 plan 在总体依赖方向上可取，但 `classified_fund_type` 多值冲突语义必须先修正。否则 P6-S5 会把最关键的基金类型矛盾降级为非阻断信息，直接违背基金类型优先和 FQ5 升级目标。
