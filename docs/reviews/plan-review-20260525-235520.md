## Findings

### F1-未修复-中-输入来源未锁定会迫使 implementation agent 重新设计或误触禁区

- **位置**: §4 Recommended Path, "Allowed input choices, in priority order"; §5 Concrete Consumer
- **问题类型**: 不可直接实施 / 契约缺失 / 架构边界
- **当前写法**: plan 允许三类输入：既有 reviewed/quasi-reviewed bundle JSON-like content、从已有 `StructuredFundDataBundle` 投影、或手工组装 quasi-real bundle；但没有指定本仓库当前应使用哪一个具体入口、哪一个测试 helper、哪一个 accepted artifact，或哪一组最小字段。
- **反例/失败场景**: implementation agent 为了拿到 "production-shaped" bundle，可能去调用 extractor、`FundDocumentRepository`、PDF/cache/source helper，或重新设计一个 bundle builder；这会违反当前 gate 禁止 fresh annual-report fetch/parse、禁止直接 repository/PDF/cache/source helper、禁止改 source/test/product flow 的约束。
- **为什么有问题**: 当前 gate 的价值是低风险 evidence loop。输入选择不收敛会把计划从 evidence command 变成探索性实现，使下一步不可验收，也难以证明没有越过 Service/Capability/文档仓库边界。
- **直接证据**:
  - plan §4 同时列出 3 个 input choices，但未指定本次执行采用哪一个。
  - plan §8 stop condition 禁止 fetching/parsing new annual report 和 repository/PDF/cache/source helper。
  - 代码事实：`validate_report_quality_bundle()` / `validate_report_quality_jsonl()` 已存在于 `fund_agent/fund/report_quality_validation.py`；测试里已有 `_valid_bundle_dict()` 和 `ReportEvidenceBundle` 构造样例，但 plan 没有引用这些可控输入形态。
- **影响**: 实施 Agent 可能跑偏、触碰禁止边界、产出无法 review 的 scratch 数据，或者把准真实证据误称为 repository-verified / scoring-ready。
- **建议改法和验证点**:
  - 在交给 implementation 前补一个 plan patch，明确本轮只允许使用 "quasi-real manually assembled bundle from accepted S0/S1/S2 review evidence plus current test serialization shape"。
  - 明确禁止从生产 extractor 构造 `StructuredFundDataBundle`，除非另开 gate。
  - 写出最小字段清单：`schema_version`、`corpus_id`、`review_status`、`source_documents`、`facts`、`data_gaps`、`score_issues`、`evidence_anchors`，以及至少一个能触发当前 validator issue localization 的 fact/gap/issue relation。
  - 验证点：evidence artifact 必须声明 input label 为 `quasi_real_review_evidence`，不得声明 `repository_verified`、`scoring_ready` 或 `accepted_baseline`。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
