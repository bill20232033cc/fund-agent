## Findings

### 001-unresolved-low-S6-G share-change token overlap with S6-E

- **位置**: Section 7 (Token / Guard Table), `share_scale_change` role; Section 8 (Overlap Handling), S6-E overlap
- **问题类型**: 过度耦合
- **当前写法**: S6-G `share_scale_change` strong tokens include `基金份额变动`, `份额变动`, `基金份额总额变动`, `报告期期初基金份额总额`, `报告期期末基金份额总额`, `期初基金份额总额`, `期末基金份额总额`. These are identical to S6-E `share_change` strong tokens. Both selectors independently capture the same source records from `FundDisclosureDocumentContentIntermediate`.
- **反例/失败场景**: If a future change modifies S6-G token behavior or guard rules for `share_scale_change`, it could semantically "shadow" S6-E share-change evidence. The plan's test 16 baseline fixture must include share-change content to catch this; but the plan does not explicitly require the baseline fixture to contain share-change content that would trigger both S6-E and S6-G simultaneously.
- **为什么有问题**: Cross-family token identity is intentional per the S6-A through S6-F pattern (each family independently collects candidate evidence). However, the plan does not explicitly document that test 16's baseline fixture must include S6-E share-change content to verify the overlap does not alter S6-E signatures when S6-G share_scale_change tokens also fire.
- **直接证据**: Plan Section 7 `share_scale_change` strong tokens are verbatim copies of `_INVESTOR_EXPERIENCE_MATCH_GROUPS[2]` (`share_change`) strong tokens in `fund_disclosure_processor.py` lines 196-208.
- **影响**: Low. Implementation agent may build test 16 baseline without share-change content, weakening the overlap regression. No code correctness issue because families independently collect.
- **建议改法和验证点**: In test 16 specification, explicitly require the baseline fixture to contain content matching S6-E `share_change` strong tokens. Verify S6-E record count, path order, and gap codes are unchanged after adding S6-G content.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低

### 002-unresolved-low-Cell guard context defined by exclusion not inclusion

- **位置**: Section 7 (Token / Guard Table), guard rule for cells
- **问题类型**: 不可直接实施
- **当前写法**: Plan states "Cell generic guard must be role-invariant and fail-closed: do not include `cell_text` or `cell_text_normalized` in the guard context." It also states "Do not copy S6-E same-cell default guard behavior." The guard context fields for cells are: parent table `heading_text`, `table_caption_or_nearby_heading`, `heading_path`, cell `row_label_path`, `column_header_path`, `heading_path`.
- **反例/失败场景**: An implementation agent reading "do not copy S6-E" might not know what to do instead. The allowed guard fields are stated in the guard context definition for cells but scattered across two paragraphs (the guard context paragraph and the cell self-guard paragraph).
- **为什么有问题**: The plan specifies the correct cell guard context fields, but the "do not copy S6-E" instruction combined with the exclusion-based guard definition ("do not include cell_text") may cause an implementation agent to second-guess the explicit field list. The S6-F `_core_risk_cell_guard_context()` implementation at `fund_disclosure_processor.py` lines 2054-2078 provides the exact pattern to follow.
- **直接证据**: Plan Section 7 guard rule paragraph and cell guard context paragraph. Code `fund_disclosure_processor.py` `_core_risk_cell_guard_context()` lines 2054-2078.
- **影响**: Low. Implementation agent can resolve by reading existing S6-F code. Minor implementation ambiguity.
- **建议改法和验证点**: In Section 9 (Future Implementation Shape), explicitly list the cell guard context fields as a positive definition (what TO include) rather than only exclusion. Add: "Follow `_core_risk_cell_guard_context()` pattern: `(table.heading_text, table.table_caption_or_nearby_heading, *table.heading_path, *cell.row_label_path, *cell.column_header_path, *cell.heading_path)`."
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/严重）**: 低

### 003-unresolved-low-Fee-adjustment current-stage under-coverage as residual

- **位置**: Section 12 (Residual Risks)
- **问题类型**: 范围漂移
- **当前写法**: Plan explicitly lists "Fee-adjustment current-stage evidence is under-covered because current parsed-route `current_stage.v1` mapping does not include `profile.fee_schedule`, and fee tokens are S6-C-owned" as a residual risk with owner "later Chapter 5 field-family mapping gate" and destination "separate plan if fee-adjustment locator becomes required."
- **反例/失败场景**: Chapter 5's `must_answer.item_02` asks for "过去一年最关键的 1-3 个变化（基金经理、规模、策略、费率、仓位或大额申赎）". Fee changes are explicitly listed as a key change type. The plan's residual correctly identifies this gap but does not give the implementation agent a way to partially address it.
- **为什么有问题**: This is a known design gap, not a plan defect. The plan correctly scopes fee tokens to S6-C and documents the residual. No action required in this gate.
- **直接证据**: `docs/fund-analysis-template-draft.md` Chapter 5 `ch5.must_answer.item_02` includes `费率`. Plan Section 12 residual risk #2.
- **影响**: Low. Fee-adjustment locator evidence will be absent from `current_stage.v1` candidate evidence. This does not block the current gate.
- **建议改法和验证点**: No change required. Controller should accept this as a documented residual with clear owner and destination.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低
