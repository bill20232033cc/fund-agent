## Findings

### F1 High: golden v1 blocker exists in score/control artifacts, but no executable readiness gate consumes it

`docs/implementation-control.md:60-66` states QDII coverage is blocked, `110020` remains `not_promoted`, FOF remains `data_gap / taxonomy_pending`, `006597` remains blocked by `bond_risk_evidence_missing.baseline_blocking=true`, and golden answer corpus v1 remains blocked until coverage/source/quality/fund-type/fixture-promotion blockers are resolved or explicitly deferred. `docs/implementation-control.md:100-105` repeats these residual owners.

The bond evidence artifact confirms this is not a pure data gap. `docs/reviews/release-maintenance-bond-positive-risk-evidence-20260527.md:63-70` records that public CLI artifacts still emit the blocker, while repository inspection found candidate evidence. `docs/reviews/release-maintenance-bond-positive-risk-evidence-20260527.md:120-136` concludes final state is extractor/evidence anchor work, not disclosure absence, and says the blocker must remain. The controller judgment repeats that golden corpus v1 remains blocked in `docs/reviews/release-maintenance-bond-positive-risk-evidence-controller-judgment-20260527.md:26-32` and `:55-62`.

In code, `fund_agent/fund/extraction_score.py:172-254` defines all seven bond evidence groups with `baseline_blocking=True`; `fund_agent/fund/extraction_score.py:884-925` emits the issue for exact `bond_fund`; `fund_agent/fund/extraction_score.py:1733-1780` constructs `bond_risk_evidence_missing` with `baseline_blocking=True`. But `fund_agent/fund/quality_gate.py:887-941` projects that issue as a normal `FQ2F/warn` and does not carry `baseline_blocking`. Repository search found no production consumer of `baseline_blocking` outside `extraction_score.py`.

Impact: someone looking only at `quality_gate.status=warn` can incorrectly believe the row is promotable, while control truth says baseline/golden remains blocked. This explains the “迟迟排查不出原因” symptom: the blocking fact is real, but it is split across score JSON, markdown/control docs, and review artifacts rather than surfaced by one golden-readiness command.

Recommended next step: do not weaken FQ0-FQ6. First run the accepted next gate, `bond risk evidence extractor / anchor hardening design gate`, to define positive `bond_risk_evidence.v1` records and anchors. Then add a golden-readiness/preflight command or artifact contract that explicitly fails when `baseline_blocking=true` or accepted coverage dispositions remain unresolved.

### F2 High: Eastmoney fallback still risks treating PDF integrity/schema failures as fallback-eligible unavailable

`AGENTS.md` requires fallback only for `not_found` and `unavailable`; `schema_drift`, `identity_mismatch`, and `integrity_error` must fail closed. The orchestrator honors this category split in `fund_agent/fund/documents/sources.py:40-42` and blocks non-eligible categories. However `EastmoneyAnnualReportSource.fetch_annual_report_pdf()` catches `(httpx.HTTPError, OSError, ValueError)` and rethrows all as `AnnualReportSourceUnavailableError` in `fund_agent/fund/documents/sources.py:522-553`.

The underlying downloader uses `ValueError` for PDF magic-byte failure in `fund_agent/fund/pdf/downloader.py:96-124`, which is an integrity failure, not availability. `_find_annual_report_id()` also directly indexes Akshare dataframe columns like `公告标题` and `报告ID` in `fund_agent/fund/pdf/downloader.py:39-45`; schema drift there can surface as raw dataframe/key errors and then be mapped as unavailable by the Eastmoney wrapper.

The real network smoke increased this finding’s relevance: `004393/2024` succeeded through Eastmoney fallback with `fallback_used=True` and primary failure `unavailable`. That success path is useful, but it also means correctness of failure taxonomy now materially affects whether fallback is trustworthy.

Impact: an upstream non-PDF response or schema drift from the fallback source can be mislabeled as availability, allowing a path that should fail closed to look retry/fallback-eligible.

Recommended next step: split downloader exceptions into typed source errors. PDF magic-byte/content-type failures should become `AnnualReportSourceIntegrityError`; dataframe shape drift should become `AnnualReportSourceSchemaError`; only transport/timeouts should remain `AnnualReportSourceUnavailableError`.

### F3 Medium: source metadata bool parsing can corrupt fallback provenance when reading cached metadata

`fund_agent/fund/documents/models.py:96-128` deserializes `fallback_used` with `bool(payload.get("fallback_used", False))`. A JSON/string value like `"false"` therefore becomes `True`. `docs/implementation-control.md:106` already records this as an open residual.

Impact: replaying or migrating cached metadata can produce false fallback provenance. That matters because current debugging and golden readiness depend heavily on distinguishing primary-source success from eligible fallback.

Recommended next step: make metadata parsing strict: accept booleans as booleans, tolerate missing as `False`, and reject or explicitly normalize legacy string forms with tests.

## Golden Corpus V1 Root Cause

`golden-build` is not the blocker. It produced strict JSON successfully: 11 funds, 150 records, 29 skipped.

The active blockers are:

- `006597` bond: annual-report evidence exists, but current extractor/score contract cannot encode positive `bond_risk_evidence.v1`; `baseline_blocking=true` remains active.
- QDII coverage: automatic probing stopped; multiple candidates are provenance-eligible but quality `block` and `not_promoted`.
- FOF coverage: no accepted pure `fof_fund`; current state is `data_gap / taxonomy_pending`.
- `110020`: accepted only as reviewed coverage candidate input, still `not_promoted`.
- Fixture/promotion process: no single executable readiness gate aggregates the above into pass/block with owners and artifact links.

So the shortest path is not “rerun golden-build again.” The next useful step is the accepted bond extractor/anchor design gate, followed by an explicit golden-readiness preflight that consumes score applicability, coverage disposition, source provenance, quality gate and fixture promotion state in one place.

## Uncertainties

- I did not use sub-agents in this run. The task was performed as a single-agent repository-wide audit with local validation, because current tool permissions require explicit user authorization for spawning/delegating to other agents.
- The real smoke used live external sources and can vary by network/source availability. The observed successful fallback path should be treated as one run’s evidence, not as a permanent source guarantee.
- I did not modify or delete the pre-existing untracked `--help` file or other untracked review artifacts.

## Final Assessment

The project is operational and well tested for deterministic paths, but release/golden work is now blocked by representation and gate-design gaps rather than basic test failures. The highest-leverage fix is to make “baseline/golden readiness” executable and to give bond risk evidence a first-class positive record/anchor contract. Until then, `quality_gate=warn` and `golden-build=pass` can coexist with “golden corpus v1 blocked,” which is exactly the confusing state observed today.
