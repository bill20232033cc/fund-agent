## Findings

### 001-未修复-[高]-计划要求 `FundDataExtractor` 接入不存在的中间态入口
- **位置**: plan §3 Allowed New/Modified Files, §5.2 S3 Dispatch Extension, §8.1 Test Plan, §11 Stop Condition
- **问题类型**: 不可直接实施 / 架构边界 / 切片过粗
- **当前写法**: plan 要修改 `fund_agent/fund/data_extractor.py`，让 `_extract_active_fund_via_processor()` 支持 `FundDisclosureDocument` intermediate kind，并要求 `FundDataExtractor.extract()` 在 active fund + `fund_disclosure_document.v1` 时抛 `UnsupportedFundProcessorError`。
- **反例/失败场景**: 当前 `FundDataExtractor.extract(fund_code, report_year, force_refresh=False)` 没有任何参数可传入 `FundDisclosureDocument` 或 intermediate kind；它固定调用 `FundDocumentRepository.load_annual_report()`，拿到 `ParsedAnnualReport` 后才分类，并在 active fund 路径固定构造 `intermediate_kind="parsed_annual_report.v1"`。实施者若为了满足测试而给 repository 增加 candidate loader，会触碰当前禁止的 repository behavior/candidate route；若只测私有方法，则计划宣称的 facade admission 没有被证明。
- **为什么有问题**: 当前 gate 是 planning-only，且控制真源明确候选 Docling/FundDisclosureDocument/pdfplumber/EID HTML render 仍未进入生产 facade。plan 同时说“不实现 concrete FundDisclosureDocumentProcessor”和“让 data_extractor 支持新 intermediate”，但没有定义受控中间态从哪里进入 Fund 层。implementation agent 必须自行发明入口、测试缝或 repository 行为，无法 code-generation-ready。
- **直接证据**: plan §3 行 79-87 要改 `data_extractor.py` 并加 facade negative tests；plan §5.2 行 196-200 要把 non-ParsedAnnualReport intermediate 传入 `FundProcessorInput`；plan §8.1 行 288-290 要测试 `FundDataExtractor.extract()` 的 `fund_disclosure_document.v1` 路径。当前代码 `data_extractor.py` 行 280-307 固定 repository 加载年报，行 321-330 仅 active fund 进入 processor，行 362-376 固定 dispatch/input 为 `parsed_annual_report.v1` + `ParsedAnnualReport`。`tests/fund/documents/test_repository.py` 行 1254-1273 明确验证生产 repository 没有 candidate/document-disclosure route。
- **影响**: 实施 Agent 跑偏；可能新增隐藏 API、改 repository 行为、或只能写无法证明目标的私有方法测试；review 无法验收“facade admission”。
- **建议改法和验证点**: 将 S3 收缩为纯 Fund processor contract/admission helper，不修改 `FundDataExtractor.extract()`；或者先补一个明确的 Fund documents 内部 loader/adapter contract gate，定义 `FundDisclosureDocument` 如何由 repository 内部受控输出并如何传入 extractor。若仍保留 `data_extractor.py`，plan 必须给出显式 public/internal method signature、调用方、禁止 repository 行为变化的测试、以及不影响 legacy facade 的证明。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 002-未修复-[高]-`candidate_only` 进入 `EvidenceAnchor.source_kind` 违反当前锚点边界
- **位置**: plan §6.2 EvidenceAnchor / Provenance / Gap State, §9 Residual Risks, Appendix A
- **问题类型**: 架构边界 / 契约缺失 / 过度耦合
- **当前写法**: plan 要把 `candidate_only` 加入 `EvidenceSourceKind`，并允许 candidate paths 使用 `candidate_only` 作为 `EvidenceAnchor.source_kind`。
- **反例/失败场景**: 当前 candidate 层已有约束：candidate route/source kind 保存在 candidate note 或 processor/candidate metadata 中，公共 `EvidenceAnchor.source_kind` 仍保持 `annual_report`。如果 S3 把 `candidate_only` 写入公共 anchor source kind，下游 closure/anchor guard 会把非 annual_report source kind 视为 candidate metadata violation；同时会扩大所有消费 `EvidenceAnchor` 的 renderer/audit/quality/golden surface，而 plan 又禁止这些层直接消费 candidate/internal representation。
- **为什么有问题**: `EvidenceAnchor.source_kind` 是公共证据语义来源，不是 parser/candidate route 身份。把 candidate 状态塞进 source kind 会混同 repository source、processor route、candidate status 和 evidence semantic source，正好违反当前设计里“candidate 只能作为中间态，事实仍需经 EvidenceAnchor/fail-closed 分类”的边界。
- **直接证据**: plan §6.2 行 223-227 明确要求 candidate paths 可用 `candidate_only` source kind；plan §9 行 332 把扩展 `EvidenceSourceKind` 作为本 slice 必做。当前 `extractors/models.py` 行 10-11 的 `EvidenceSourceKind` 只有 `annual_report/external_api/derived`。`tests/fund/documents/test_docling_locators.py` 行 225-258 验证 candidate source kind 必须留在 note，`EvidenceAnchor.source_kind` 保持 `annual_report`。`tests/fund/documents/test_docling_source_truth_residual_closure.py` 行 2255-2273 验证 repository source、processor source kind、anchor source kind 分离；行 2309-2331 验证非 annual_report anchor source kind fail-closed。`tests/fund/documents/test_docling_candidate_models.py` 行 184-199 也明确 candidate internals 不进入 public documents exports 且 Docling source kind 不进入 `EvidenceSourceKind`。
- **影响**: 生成错误公共契约；candidate metadata 可能泄漏到公共 EvidenceAnchor；既有候选 guard 测试和 source-truth residual closure 语义被破坏；后续 renderer/audit/quality/golden 可能误把 candidate status 当证据来源类型。
- **建议改法和验证点**: 不扩展 `EvidenceSourceKind` 为 `candidate_only`。保留 `EvidenceAnchor.source_kind="annual_report"`，把 candidate route 放在 candidate anchor note、`CandidateBoundaryStatus`、`FundProcessorResult.candidate_boundary` 或显式 `source_boundary="candidate_only"` gap 中。验证点应保留现有 no-leak tests，并新增 candidate processor/admission 测试证明 candidate route 不进入 public `EvidenceAnchor.source_kind`。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 003-未修复-[中]-failure taxonomy 和 gap/status 输出未落到当前 processor contract
- **位置**: plan §4.3 Parser Failure Taxonomy, §6.3 Fail-closed Categories, §7 Failure Handling, §8 Test Plan
- **问题类型**: 契约缺失 / 不可直接实施 / 测试缺口
- **当前写法**: plan 要 `fund_disclosure_dispatch.py` enforce canonical five failure mapping，非 `None` failure class 要 emit `FundExtractionGap`，`schema_drift/identity_mismatch/integrity_error` blocked，`not_found/unavailable` partial or missing；测试要覆盖 all five failure-class-to-gap-code mapping。
- **反例/失败场景**: 当前 `FundExtractionGapCode` 没有 `not_found`、`unavailable`、`schema_drift`、`identity_mismatch`、`integrity_error` 这些 gap code；plan 又没有指定映射到现有 `candidate_boundary_blocked`、`source_provenance_unsafe`、`unsupported_intermediate` 等哪个稳定 code，也没有定义 helper 返回 `FundProcessorResult`、`FundExtractionGap`、异常还是中间 admission decision。因为本 slice 不实现 concrete processor，身份 mismatch 和 failure_class 的 blocked/partial/missing 输出只能靠新 helper 模拟，不能证明 future processor 的字段族、anchors、gaps 和 result contract。
- **为什么有问题**: AGENTS 和 documents layer 的 failure taxonomy 是来源 fallback 决策边界；processor contract 的 `FundExtractionGapCode` 是另一个闭集。plan 把二者直接等同，但没有给出可类型检查、可 review 的投影表。implementation agent 必须自行发明 gap code、result shape 和 blocking semantics。
- **直接证据**: plan §4.3 行 160-174 要从 failure class emit gap 并设置 status；plan §6.3 行 228-241 把 five canonical failures 写成 result behavior；plan §8.1 行 286-290 要测试 failure-class-to-gap-code mapping。当前 `processors/contracts.py` 行 39-56 的 `FundExtractionGapCode` 不包含 five canonical source failure names；行 208-231 定义 `FundExtractionGap` 需要闭集 gap code 和 source boundary；行 284-337 定义 result-level gaps 只能承载跨字段族缺口。candidate failure 到 source category 的现有映射位于 `documents/candidates/failures.py` 行 9-24、45-60，但它只返回 `AnnualReportSourceFailureCategory`，不是 processor gap/result contract。
- **影响**: 实现不可直接生成；测试容易只验证自造 helper 字典而非真实 processor output；后续 processor 可能产生不一致的 gap/status，导致 fail-closed 语义不可审计。
- **建议改法和验证点**: 在 plan 中先定义一个精确投影表：`AnnualReportSourceFailureCategory -> FundExtractionGapCode -> FundExtractionSourceBoundary -> FundProcessorContractStatus -> field-family/result-level location`，并明确哪些 code 需要加入 `contracts.py`、哪些复用现有 code。若无 concrete processor，则 S3 只能测试 admission decision object；blocked/partial/missing 的 `FundProcessorResult` 行为应下放到 S4 concrete processor gate。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

### 004-未修复-[中]-`FundDisclosureDocumentStub` 放入 public documents models 的边界未收敛
- **位置**: plan §3 Allowed New/Modified Files, §4 Input Contract, Appendix A
- **问题类型**: 架构边界 / 范围漂移 / 过度设计
- **当前写法**: plan 要在 `fund_agent/fund/documents/models.py` 添加 `FundDisclosureDocumentStub` dataclass，作为 minimal identity/provenance envelope；同时又声明 full schema、concrete processor 和 candidate source implementation 都 deferred。
- **反例/失败场景**: `documents/models.py` 当前承载生产年报公共数据模型，且现有测试明确 production repository 没有 candidate route、documents public init 不导出 candidate internals。若把 “Stub” 放入公共 models，implementation agent 很容易把它当已接受 public document surface 或 future schema seed；若不导出、不被 repository 返回、不被 processor 消费，它只是为测试构造的 object，应该留在 test/helper 或 candidate/internal module，不应进入 production public models。
- **为什么有问题**: 设计真源说 `FundDisclosureDocument` 是未来 Fund documents 内部中间态，且后续 schema planning gate 必须核对 EvidenceAnchor/source outcome/failure mapping，不得让 implementation worker 自行发明 source ownership/source kind/failure outcome。当前 plan 跳过 schema gate，先把 stub 放进 production models，会把 future schema 名称提前固化为当前代码事实。
- **直接证据**: plan §3 行 81-83、Appendix A 行 384-385 要在 documents public models 添加 `FundDisclosureDocumentStub`。`docs/design.md` 行 686-688 说明该设计不是当前代码事实，后续 schema planning gate 必须显式规划 source kind/failure outcome。`tests/fund/documents/test_repository.py` 行 1254-1273 验证 repository 无 candidate route；`tests/fund/documents/test_docling_no_consumption_guards.py` 行 50-65 验证 documents public init 不导出 candidate internals。
- **影响**: 生成当前事实与未来设计混杂；公共 surface 可能提前承诺未设计 schema；后续 schema gate 需要返工或迁移。
- **建议改法和验证点**: 删除 production `FundDisclosureDocumentStub` 写项；如本 slice 只需要 protocol conformance，用 test-local fake 或 `fund_disclosure_dispatch.py` 内部 fixture protocol test。若确需 production minimal envelope，plan 必须先定义稳定字段、导出策略、repository ownership、candidate/non-candidate boundary和为何不与未来 schema gate冲突。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## Open Questions

- S3 的真实目标是“纯 admission helper/contract”还是“把 document representation 接入 `FundDataExtractor` facade”？两者当前需要不同入口和测试策略。
- `FundDisclosureDocument` 是否应先经过独立 schema gate，还是可以在 processor contract gate 中只用 protocol/test fake 表达？
- Candidate route identity 应放在 candidate note、processor input、source provenance extension，还是单独的 processor route metadata？当前 plan 的 `EvidenceSourceKind` 方案不安全。
- `not_found/unavailable` 在 processor 层是否允许产生 `partial`，还是应仅由 documents/source layer 决定 fallback 并把 processor 输入限定为已加载/可处理中间态？

## Final Plan Review Conclusion

fail

The plan is not code-generation-ready.
