## Findings

### 1. Medium - Public source provenance 仍输出 `primary_then_fallback`，与当前 EID single-source 真源冲突

**Evidence**

- 当前控制真源把 annual-report source policy 定义为 EID single-source，且 fallback invocation / Eastmoney / CNINFO source expansion 未授权：`docs/implementation-control.md:30-35`、`docs/current-startup-packet.md:37-42`。
- 运行时代码默认只装配 EID source，并拒绝多 source：`fund_agent/fund/documents/sources.py:594-603`。
- repository cache 复用也要求 current EID single-source metadata：`fund_agent/fund/documents/repository.py:267-291`。
- 但 public provenance 仍把稳定策略类型和值固定为 `primary_then_fallback`：`fund_agent/fund/source_provenance.py:14-31`，并在 default / non-fallback EID 成功路径中继续输出该值：`fund_agent/fund/source_provenance.py:93-101`、`fund_agent/fund/source_provenance.py:126-137`。
- 该 provenance 进入真实 extraction bundle：`fund_agent/fund/data_extractor.py:320-342`，并进入 snapshot record public fields：`fund_agent/fund/extraction_snapshot.py:197-204`。
- 测试也固化旧值：`tests/fund/test_source_provenance.py:33`、`tests/fund/test_extraction_snapshot.py:198`、`tests/fund/test_extraction_score.py:59`。

**What can go wrong**

运行时 fallback 已被禁用，但对外 snapshot / provenance 仍表达“主源后 fallback”策略。后续 quality/golden/readiness 或 reviewer 可能把旧 source strategy 当当前 operational fact，重新打开 fallback 设计或误判 release evidence。

**Recommendation**

开独立 `EID source provenance truth alignment gate`：把 public provenance schema/fields 对齐 current source policy，例如显式输出 `selected_source=eid`、`source_mode=single_source_only`、`fallback_enabled=false`；若必须保留 `source_strategy` 兼容字段，必须重命名/标记为 legacy/deferred，并更新 extractor/snapshot/score/readiness tests。

### 2. Medium - `--use-llm` 先读取 provider env / 构造 client，再校验业务请求，错误归因会被环境配置遮蔽

**Evidence**

- CLI `--use-llm` 路径先调用 `_build_llm_execution_request_or_fail()`：`fund_agent/ui/cli.py:781-784`。
- `_build_llm_execution_request_or_fail()` 委托 `build_fund_llm_execution_request()`：`fund_agent/ui/cli.py:1347-1367`。
- `build_fund_llm_execution_request()` 当前顺序是先读取 env / 构造 provider client，再解析业务 contract：`fund_agent/services/fund_analysis_service.py:939-942`。
- 业务请求校验在更后面才发生：`fund_agent/services/execution_contract.py:419-434`、`fund_agent/services/execution_contract.py:468-487`、`fund_agent/services/fund_analysis_service.py:1123-1162`。
- provider env 是 fail-closed 必填：`fund_agent/config/llm.py:79-100`；CLI 把 config/construction error 映射为 exit 1：`fund_agent/ui/cli.py:814-819`。
- 现有测试覆盖“provider config error before host run”：`tests/services/test_fund_analysis_service_llm.py:272-294`，但没有覆盖“非法业务请求 + 缺失 provider env”时应先报告哪类错误。

**What can go wrong**

非法 fund code、非法 report year、product mode 携带 developer override 等请求事实，可能先被“缺少 LLM provider 配置”遮蔽。用户看到的是环境问题，root cause 实际是请求契约问题；错误归因依赖本机 env 状态，不利于 fail-closed debugging。

**Recommendation**

开 `LLM execution request validation ordering gate`：先完成 `normalize_fund_llm_analysis_input()`、ExecutionContract / quality policy / request identity 校验，再读取 env 和构造 provider client。补测试覆盖非法 request 与缺失 provider env 同时存在时的错误优先级。

### 3. Medium - UI 层直接持有 Host runner，和 AGENTS/UI 边界存在控制冲突

**Evidence**

- `AGENTS.md` 要求 UI 只依赖 Service，不直接调用 Host 或 Agent 内部模块。
- `docs/design.md` 的架构表也写 UI 不得直接调用 Host；但同一设计文档后续 current implementation 裁决又描述 CLI 把 operation closure 与 Host timeout 交给本地 Host runner。
- 当前代码中 `fund_agent/ui/cli.py` 直接 import `fund_agent.host` 类型与 `HostRuntimeRunner`：`fund_agent/ui/cli.py:29-36`。
- `--use-llm` 路径在 CLI 内构造 operation closure 并调用 `HostRuntimeRunner().run_sync()`：`fund_agent/ui/cli.py:1374-1431`。
- UI tests 已固化这个跨层形态，见 `tests/ui/test_cli.py` 中对 `build_fund_llm_execution_request` 与 Host path 的 monkeypatch 覆盖。

**What can go wrong**

Host lifecycle governance 入口散落到 CLI。后续非 CLI UI、durable Host session/resume/memory/outbox 或多入口 runtime 接入时，容易复制 UI -> Host 穿透，而不是由 Service 暴露统一用例 façade。

**Controller note**

该 finding 不是简单实现 bug：当前 `docs/design.md` 同时包含“UI 不直连 Host”的边界规则和“当前 CLI 调 Host runner”的 current fact。需要 controller 先裁决是修实现到 UI -> Service，还是把真源文档明确改成当前过渡例外；以 `AGENTS.md` 为最高规则时，推荐修实现。

**Recommendation**

开 `UI-Service-Host boundary reconciliation gate`：优先将 Host-runner 托管迁入 Service-owned façade，UI 只构造显式 request 和 progress sink；若保留过渡例外，必须先做 design/control truth-sync 并明确退出条件。

### 4. Low - EID source orchestrator docstring 仍保留 Eastmoney fallback 旧口径

**Evidence**

- `fund_agent/fund/documents/sources.py:583-585` 的参数说明仍写“未提供时使用 EID 主源与 Eastmoney fallback”。
- 实际代码默认只创建 `EidAnnualReportSource`：`fund_agent/fund/documents/sources.py:594-599`。
- 同一构造函数拒绝多 source：`fund_agent/fund/documents/sources.py:600-603`。

**What can go wrong**

这是 source policy 边界文件中的旧口径，短期不是 runtime defect，但会误导后续 fallback/source expansion gate。

**Recommendation**

在 source provenance / EID wording alignment gate 中顺手修正文档字符串：当前默认仅 EID single-source；Eastmoney 仅为 deferred / historical candidate，不是 production fallback。
