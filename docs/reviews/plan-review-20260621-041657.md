## Findings

### 001-未修复-高-宽松文本 fallback 会把 schema drift 写成合法 JSON

- **位置**: Slice 1 / Changes / “Convert `Path` or non-JSON primitives with stable text fallback only where current bundle payloads require it.”
- **问题类型**: 契约缺失 / schema drift / 最佳实践偏离
- **当前写法**: Plan 允许对非 JSON primitives 使用 stable text fallback。
- **反例/失败场景**: 如果未来某个 extractor value 引入 `Decimal`、`date`、自定义对象或异常对象，repository 会把它悄悄转成字符串并写入 `structured_fund_data.json`。多年分析或 LLM route 后续读取时无法判断这是原始语义、格式化文本还是序列化降级。
- **为什么有问题**: 该 work unit 的目标是提供“稳定输入层”。宽松 fallback 会把 schema drift 和数据损坏伪装成成功写盘，违反当前项目 fail-closed 和证据可溯源原则。
- **直接证据**: Plan 的 Slice 1 明确授权 fallback；`StructuredFundDataBundle` 中包含 `ExtractedField`、`EvidenceAnchor`、`NavDataResult`、`PublicSourceProvenance`、`BondRiskEvidenceValue` 等嵌套结构，当前 serialization contract 是本 gate 新增的唯一边界。
- **影响**: 写出不可审计 JSON；后续多年分析/LLM route 消费错误值；schema drift 难以在测试中暴露。
- **建议改法和验证点**: Plan 应改为定义严格 `_jsonable(...)`：只接受 `None`、`str`、`int`、`float`、`bool`、dataclass、mapping、sequence；`Path` 只有在明确字段允许时转字符串；未知类型直接 `TypeError` fail-closed。新增测试覆盖 unsupported object rejected。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 高

### 002-未修复-中-Service 计划要求 fake 测试但没有注入点

- **位置**: Public Contract / Service API；Slice 2 / Tests
- **问题类型**: 不可直接实施 / 测试缺口
- **当前写法**: `ExtractorOutputService.save()` 描述为构造默认 `FundDataExtractor()` 和 `ExtractorOutputRepository(root)`；Slice 2 测试又要求 “Service calls extractor once and repository save once using fakes”。
- **反例/失败场景**: implementation agent 按计划写死默认构造后，Service 单元测试无法注入 fake extractor/repository，只能走真实 `FundDocumentRepository` 或 monkeypatch 模块全局，导致测试变成脆弱集成测试或触发真实文档路径。
- **为什么有问题**: 计划的 API 与测试策略互相矛盾。当前项目已有 `ExtractionSnapshotService` 薄编排模式，但此处需要显式验证 Service 不越界、只编排；没有注入点就不能用直接证据证明。
- **直接证据**: Plan 的 Service behavior 写 “Construct `FundDataExtractor()` and `ExtractorOutputRepository(root)`”；同一计划的 Tests 写 “using fakes”。
- **影响**: implementation 阶段需要重新设计 Service 构造器；测试可能触发真实仓库或网络；review 难以验收 Service 层边界。
- **建议改法和验证点**: Plan 应增加 `ExtractorOutputService.__init__(extractor=None, repository_factory=None)` 或等价 Protocol 注入点。生产默认仍使用真实依赖；测试通过 fake extractor 和 fake repository factory 断言调用参数。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## Open Questions

## Residual Risks and Suggested Tracking

- Typed hydration from repository JSON remains deferred to annual-period/LLM consumer gate.

- Report type expansion beyond `annual_report` remains deferred to schema extension gate.

## Special Lens Results

- **Architecture boundary review**: Direction is sound: Fund repository + Service wrapper + UI thin command. Finding 002 must be fixed to preserve testable Service boundary.
- **Best-practice review**: Finding 001 violates storage contract best practice by allowing silent lossy conversion.
- **Optimal-solution review**: Dedicated bundle-level repository is better than reusing snapshot or adding extractor side effects.
- **Overengineering review**: No database/index/locking overreach found.
- **Overcoupling review**: No cross-layer consumption leak found after Service injection fix.

## Conclusion

`fail`

The plan is directionally correct but not code-generation-ready until findings 001 and 002 are fixed and re-reviewed.
