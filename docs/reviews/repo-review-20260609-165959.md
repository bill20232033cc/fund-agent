## Findings

### [P1] Eastmoney PDF 完整性失败被误归类为 fallback-eligible `unavailable`

证据：

- `fund_agent/fund/pdf/downloader.py:99-100` 下载后调用 `_validate_pdf_bytes(resp.content, context=url)`；`fund_agent/fund/pdf/downloader.py:123-124` 对非 PDF body 抛出 `ValueError("PDF 响应缺少 %PDF- 文件头...")`。
- `fund_agent/fund/documents/sources.py:544-553` 的 `EastmoneyAnnualReportSource.fetch_annual_report_pdf()` 捕获 `(httpx.HTTPError, OSError, ValueError)` 并统一抛出 `AnnualReportSourceUnavailableError`。
- `fund_agent/fund/documents/sources.py:636-641` 将 `AnnualReportSourceUnavailableError` 构造成 `unavailable` 并允许继续 fallback；同文件 `650-653` 才把 `AnnualReportSourceIntegrityError` 作为 `integrity_error` fail-closed。
- 对照实现：EID 路径在 `fund_agent/fund/documents/sources.py:1157-1174` 对 Content-Type / PDF magic 失败抛出 `AnnualReportSourceIntegrityError`，同类失败没有被降级。
- 测试缺口：`tests/fund/pdf/test_downloader.py:236-284` 已证明 Eastmoney 底层 helper 对 HTML body 抛 `ValueError` 且不写缓存；`tests/fund/documents/test_annual_report_sources.py:1284-1365` 只覆盖 Eastmoney HTTP/OSError -> unavailable，没有覆盖 ValueError -> integrity_error 或 orchestrator stop。

影响：

- 默认两源配置下，Eastmoney 作为最后 fallback 源时，最终错误类别会把“内容不是 PDF”的完整性失败报告成 `unavailable`，破坏来源失败类别与审计语义。
- 若未来或测试注入更多来源，Eastmoney 非 PDF 响应会被视为 fallback-eligible，可能继续尝试后续来源，违反 `integrity_error` 必须 fail-closed 的规则。
- 该问题直接落在生产年报 PDF 来源编排路径，不能用全量 pytest 通过抵消。

修复要求：

- 让 Eastmoney adapter 区分底层完整性失败与临时不可用；至少将 `_download_pdf()` 的 PDF magic failure 映射为 `AnnualReportSourceIntegrityError`。
- 增加 Eastmoney wrapper 测试：downloader 抛出 PDF 完整性语义错误时，wrapper 抛 `AnnualReportSourceIntegrityError`。
- 增加 orchestrator 测试：Eastmoney integrity failure 不进入后续 fallback，并保留 `category="integrity_error"`。
