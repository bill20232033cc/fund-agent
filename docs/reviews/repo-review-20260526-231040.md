## Findings

### [P2] NAV 外部数据失败会阻断并不依赖 NAV 的年报报告主链路

**位置**

- `fund_agent/fund/data_extractor.py:161-167`
- `fund_agent/fund/data/nav_data.py:184-222`
- `fund_agent/fund/data/nav_data.py:224-240`
- `fund_agent/fund/template/renderer.py:1164-1178`
- `fund_agent/fund/README.md:91-96`

**从第一性原理看**

`fund-analysis analyze` 的用户目标是“基于年报生成基金体检报告”。年报抽取出的 `nav_benchmark_performance.nav_growth_rate` 已经能支撑 R=A+B-C 的核心收益率输入；`nav_data` 当前不进入 `ReportEvidenceBundle` 事实投影，README 也明确 `nav_data` 当前不投影为报告事实。renderer 对净值序列的当前使用只是在第 5 章显示记录条数和提示“继续补充同口径净值序列”。

实际代码中，`FundDataExtractor.extract()` 在完成年报加载后、任何章节 extractor 执行前，直接等待 `self._nav_provider.load_nav_data(...)`。`FundNavDataAdapter.load_nav_data()` 在缓存未命中时调用 akshare fetcher，文档和代码都允许抓取函数异常向上抛出，没有转换为显式 `unavailable` 或空结果。于是只要 akshare、网络、SQLite 缓存、接口字段或本地依赖异常，即使年报已经可用，`analyze` / `checklist` / snapshot 主链路也会整体失败。

**可触发场景**

1. 年报 PDF 与抽取链路可用。
2. NAV 缓存未命中或 `--force-refresh`。
3. akshare 接口超时、返回结构漂移、网络失败、本地环境缺依赖或 SQLite 缓存错误。
4. `FundDataExtractor.extract()` 抛出异常，后续年报 extractor、quality gate、renderer 都无法运行。

**影响**

这会把一个当前报告质量上“辅助/可缺省”的外部时间序列，提升成单基金报告的可用性前置条件。对夜间小样本 baseline、用户快速运行 `fund-analysis analyze 004393 --report-year 2024`、以及后续自动化 agent 来说，都容易把“数据源暂不可用”误表现为“基金分析不可用”。

**建议**

在单独 implementation gate 中把 NAV 读取改为显式降级契约，而不是在本次审核中直接改代码：

- 给 `NavDataResult` 增加稳定的 unavailable/error 表达，或约定 `source="nav_unavailable"`、`records=[]`、`cached=False` 加原因字段。
- `FundDataExtractor.extract()` 对 NAV provider 的外部失败做局部捕获，只在基金代码为空、缓存完整性不可恢复等真正输入/本地状态错误时 fail-closed。
- 补测试：NAV provider 抛异常时，年报字段仍被抽取，renderer 第 5 章显示“数据不足/未可用”，quality gate 行为不被错误削弱。
- 保持 `FundDocumentRepository` 年报失败分类不被混入 NAV 降级逻辑；这两个数据源的 fail-open/fail-closed 规则不能串线。

### [P2] renderer 设计 gate 已在本地收口，但尚未形成耐久 checkpoint，后续 Agent 仍可能读到分叉状态

**位置**

- `docs/implementation-control.md:24-35`
- `docs/implementation-control.md:225-238`
- 本地 `git status --short --branch`
- 本地 `docs/design.md` diff

**从第一性原理看**

本仓库用 `docs/implementation-control.md` 的 Startup Packet 作为总控 Agent 恢复任务的入口。一个长时间多 agent 项目最怕的不是“某个文档没写漂亮”，而是“几个 agent 对当前 gate 的理解不同”。

审核过程中观察到本地工作区已从“renderer minimal integration design gate 待进入”推进为“renderer minimal integration design accepted locally”，并把下一入口写成 `renderer minimal integration implementation gate`。这与新增的 renderer design plan、review、rereview、controller judgment artifacts 方向一致，说明状态正在被收口。

风险在于：这些 control/design 修改和多份 renderer design artifacts 仍处于未提交/未跟踪状态；分支也显示相对 remote ahead。也就是说，工作区文本已经表达一个较新的 gate 状态，但它还不是 durable accepted checkpoint。若后续 agent 基于 `git log`、remote branch、另一个 tmux pane 的旧缓存、或只读已提交 artifact 恢复任务，仍可能把当前状态理解为旧 gate，或者把未提交的 design accepted 状态当成已经发布到远端。

**影响**

这不是代码运行时 bug，但会直接影响你当前的“总控 + 4 个 tmux agent”工作方式：同一轮开发可能出现三种状态认知：

- 仍等待进入 renderer minimal integration design gate；
- renderer design plan 已完成但未 accepted；
- renderer design 已被当作 accepted future design。

这会导致重复 plan、越权 implementation、或把未裁决设计写进当前代码事实。

**建议**

先做一次 artifact disposition / control reconciliation，只处理状态，不改产品代码：

- 若 renderer design 已 accepted：把 `docs/design.md`、`docs/implementation-control.md` 和对应 `docs/reviews/` artifacts 作为同一个 checkpoint 提交，确保 `git log`、Startup Packet 和 artifact ledger 一致。
- 若还需复核：在控制文档中明确写成 pending，不要让“accepted locally”和“requires explicit implementation authorization”同时被不同 agent 解读为可实现授权。
- 无论选择哪种状态，都应先收口未跟踪 artifacts；不要在脏工作区上开启 renderer implementation gate。

### [P3] dev-only 写作审计的交易建议禁词覆盖窄于生产 renderer

**位置**

- `fund_agent/fund/report_writing_audit.py:66-76`
- `fund_agent/fund/report_writing_audit.py:269-303`
- `fund_agent/fund/report_writing_audit.py:1090-1103`
- `fund_agent/fund/template/renderer.py:60-72`
- `tests/fund/test_report_writing_audit.py:148`

**从第一性原理看**

AGENTS.md 的基金分析硬约束是“禁止直接输出买入/卖出建议”。这类规则应当尽量形成同一套语义，至少不应在未来 renderer minimal integration gate 中出现“审计认为通过，但 renderer 会拒绝”或“审计漏过直接交易建议”的错觉。

当前 dev-only `report_writing_audit.py` 用固定短语列表判断交易建议，只覆盖 `建议买入`、`可以买入`、`立即买入`、`建议卖出`、`应该卖出`、`清仓`、`加仓`、`减仓`、`仓位比例`。生产 renderer 另有更宽的 `_DIRECT_TRADING_ADVICE_PATTERN`，覆盖 `建议|推荐|应当|应该|直接|立即|马上` + `买入|卖出|加仓|减仓`。因此 `推荐买入`、`应当买入`、`马上卖出` 等文本会被 renderer 识别为直接交易建议，但 dev-only 写作审计不会命中。

**影响**

当前影响主要在 dev-only report-writing audit 和小样本 baseline 解释层，产品 renderer 本身仍有更宽校验。风险在于下一 gate 正在讨论 renderer/report-writing minimal integration，如果直接复用 dev-only audit 的判断而不统一规则，会把“章节草稿安全性”评估做松。

**建议**

后续实现 gate 中不要复制两套禁词逻辑：

- 抽出共享的交易建议检测函数，或让 dev-only audit 复用 renderer 的直接交易建议 pattern。
- 增加参数化测试：`推荐买入`、`应当买入`、`直接卖出`、`马上加仓`、`长期持有不等于买入建议`。
- 保留 renderer 当前允许普通语境“买入并持有”等非指令性文本的行为，避免回到早期全局字符串误杀。

### [P3] report-quality 离线 wrapper 对坏 bundle 输入硬崩，不能产出可聚合失败摘要

**位置**

- `scripts/report_quality_eval.py:26-49`
- `scripts/report_quality_eval.py:105-138`
- `scripts/report_quality_eval.py:141-160`
- `tests/scripts/test_report_quality_eval.py:109-133`

**从第一性原理看**

`scripts/report_quality_eval.py` 是 maintainer-only / dev-only 工具，用于夜间或批量评估 report-quality artifact。对这种工具，最重要的不是用户体验，而是“失败也能被稳定统计”。当前 JSONL validator 可以把很多内容错误转换为 issue summary；但 wrapper 在读取单个 bundle JSON 时，如果 JSON 顶层不是 object 或 JSON 解析失败，会在 `_load_bundle_json()` 直接抛 `TypeError` / `JSONDecodeError`，`main()` 不写出 summary JSON。

测试目前明确断言非 object bundle 会抛 `TypeError`，说明这是当前接受行为，不是未测试偶然行为。但从自动化 baseline 的角度看，硬崩会让一轮多输入评估丢失统一失败分类，只留下进程失败。

**影响**

当夜间 agent 或总控把多个 bundle 交给该 wrapper 时，一个坏文件会阻断整批 summary 输出；后续 controller 需要读 traceback 而不是读稳定 JSON issue taxonomy。这会降低“可观测失败循环”的价值。

**建议**

这不必阻塞当前产品链路，但适合进入 future audit-output ergonomics gate：

- 给 wrapper 增加 `--fail-fast/--collect-errors` 策略，默认是否保持现状由 gate 裁决。
- 在 collect 模式下，把坏 JSON / 非 object bundle 转成 `failed_closed=true` 的单输入结果，写入 summary。
- 保留 validator 自身的 fail-closed 语义，不把坏 artifact 误标为 scoring-ready。
