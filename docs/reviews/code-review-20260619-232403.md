- **输入场景**: 多个 plan 规定必须 fail-closed 的场景缺少独立测试：

  - cell 的 `column_header_path` 匹配（仅测试了 `row_label_path` 优先路径）
  - `_PRODUCT_ESSENCE_GENERIC_CELL_TEXTS` 过滤（`项目`/`指标`/`名称`/`内容`/`说明` 被拒绝）
  - table 或 cell 的 `locator_stability != "stable"` 被静默跳过

  - paragraph heading_path 匹配 fallback（当前 paragraph 测试仅覆盖 text prefix 模式）
- **实际分支**: 这些代码路径存在且逻辑正确，但缺少独立 assertion
- **预期行为**: plan 要求的 fail-closed 行为应被独立测试覆盖
- **实际行为**: 这些路径的覆盖依赖正例（如 Slice A 测试的 side effect）或未被显式 assertion

### 2-未修复-低-`_normalize_match_text` 的去空白行为可能在长文本值去重时过度合并

- **入口/函数**: `_resolve_product_essence_candidate`
- **文件(行号)**: `fund_disclosure_processor.py:1158-1190`（调用 `_normalize_match_text` 在 1181 行和 1183 行）；
  `_normalize_match_text` 定义在 3643–3658 行
- **输入场景**: 两个不同 cell 对同一 label 给出长文本值（如 risk_characteristic_text），区别仅为 token 间空白位置不同（如 `"本基金属于 主动权益基金"` vs `"本基金属于主动 权益基金"`）
- **实际分支**: `_normalize_match_text` 执行 `"".join(str(text).split())` 去掉全部空白，两条值归一化为相同字符串 → `len(normalized_values) == 1` → 被当作重复值保留第一个
- **预期行为**: 两个不同值应触发 `ambiguous_table_or_locator` 省略该路径
- **实际行为**: 两个语义可能不同的值被当作同值去重，返回第一个候选
- **直接证据**:
  - 3643–3658 行：`_normalize_match_text` 调用 `"".join(str(text).split()).replace("　", "")`
  - 1181 行：`normalized_values = {_normalize_match_text(candidate.value) for candidate in candidates}`
  - 1188 行：`if len(normalized_values) > 1` — 仅当去空白后仍不同才触发歧义
- **影响**: 在真实年报中，同一表格出现格式变体时可能静默合并而丢失歧义信号。但中文金融文本中 token 间空白极少，实际触发概率低
- **建议改法和验证点**: 可在去重前对值做 `" ".join(text.split())`（保留单空格分隔）而非完全去空白，或对长文本（>20 字符）使用原始文本比较
- **修复风险（低）**: 仅影响重复/歧义判断逻辑，不改变核心提取行为
- **严重程度（低）**: 实际触发可能性低，且最多导致重复去重而非数据错误
