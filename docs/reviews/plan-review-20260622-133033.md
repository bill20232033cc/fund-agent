## Findings

### 1-CRITICAL-`ParsedTable`缺少`table_id`字段导致锚点主匹配路径不可实施

- **位置**: EC-P1 Work Unit, Anchor matching order (plan line 130–136); `ParsedTable` definition `fund_agent/fund/documents/models.py:676–688`
- **问题类型**: 不可直接实施
- **当前写法**: Plan specifies anchor matching order as "prefer exact `section_id + table_id + row_locator`" and reference construction copies "locator fields copy the matched anchor where possible."
- **反例/失败场景**: `ParsedTable` has fields `page_number: int`, `table_index: int`, `headers: tuple[str, ...]`, `rows: tuple[tuple[str, ...], ...]` — there is no `table_id` field. The plan says to match `anchor.table_id` against a table identifier, but `ParsedTable` has no identifier to match against. The implementation agent would need to invent a convention (e.g., deriving `table_id` from `page_number` + `table_index`), which is a design decision the plan does not make and does not authorize.
- **为什么有问题**: The primary matching path is structurally impossible against the current parsed report model. The implementation agent at EC-P1 would face an immediate blocker: either stop and return to planning, or invent a mapping convention that the plan doesn't specify (and which later live gates would need to honor consistently).
- **直接证据**: `fund_agent/fund/documents/models.py:676–688` — `ParsedTable` dataclass has exactly four fields: `page_number`, `table_index`, `headers`, `rows`. No `table_id`. `fund_agent/fund/chapter_facts.py:98–119` — `ChapterEvidenceAnchor` has `table_id: str | None`. Plan line 130–136 specifies matching on `table_id`.
- **影响**: 实施 Agent 跑偏 / 生成错误代码 / 后续返工 — EC-P1 cannot proceed as specified. If the implementation agent invents a convention, it may be wrong for live data and require EC-P2 rework.
- **建议改法和验证点**: Plan must specify: (a) how `ChapterEvidenceAnchor.table_id` maps to `ParsedTable` (e.g., via `page_number + table_index` convention, or a `f"{section_id}_table_{table_index}"` derived key, or explicit table numbering from section context); (b) whether the convention is Fund-documents-internal or a public contract; (c) test that the mapping is round-trip deterministic. If no clean mapping exists, EC-P1 should explicitly constrain to `page_number`-based matching only.
- **修复风险**: 中
- **严重程度**: 严重

### 2-HIGH-`ReportSection`没有段落级结构导致摘录提取边界不明确

- **位置**: EC-P1 Work Unit, Anchor matching fallback (plan line 135–136); `ReportSection` definition `fund_agent/fund/documents/models.py:609–626`
- **问题类型**: 不可直接实施
- **当前写法**: Plan says "fall back to exact `section_id + page_number` paragraph/section excerpt if table row is absent" and "`excerpt_text` is bounded and normalized but not summarized."
- **反例/失败场景**: `ReportSection` has `section_id`, `title`, `start_offset`, `end_offset`, `matched_rule`, `confidence`. There is no paragraph segmentation, no page-to-offset mapping, and no paragraph-level sub-structure. The implementation agent trying to extract a "paragraph/section excerpt" from a section offset range must decide: extract the entire section raw text? Extract a bounded window around a keyword? How is the excerpt bounded without paragraph boundaries?
- **为什么有问题**: "Bounded excerpt" extraction from a section identified only by character offsets is underspecified. The plan says "bounded and normalized but not summarized" — but doesn't say bounded by what (character count? paragraph count? sentence boundary?). The implementation agent must invent paragraph/page segmentation policy, which is a material design decision.
- **直接证据**: `fund_agent/fund/documents/models.py:609–626` — `ReportSection` fields are only section-level index metadata. Plan line 135–136 and line 143 specify excerpt extraction but no bounding rules. `ParsedAnnualReport.raw_text` is the only text source — the materializer must extract excerpts from raw text by offset, with no paragraph structure from the model.
- **影响**: 实施 Agent 跑偏 / 生成错误代码 / 后续返工 — implementation agent would invent arbitrary excerpt bounding (e.g., "first 500 chars after section start"), which may be wrong for live data and would need rework at EC-P2.
- **建议改法和验证点**: Plan must specify: (a) excerpt bounding rules (character limit? paragraph boundary heuristic using newlines?); (b) whether the full section text is acceptable as "bounded excerpt" or must be trimmed; (c) how `page_number` in the anchor maps to a position within `raw_text` when `ReportSection` has no page-level indexing. Alternatively, EC-P1 could explicitly defer paragraph-level excerpt extraction to EC-P2 (live) and use section-level raw text in EC-P1.
- **修复风险**: 低
- **严重程度**: 高

### 3-HIGH-`row_locator`到`ParsedTable.rows`的映射规则缺失

- **位置**: EC-P1 Work Unit, Anchor matching order (plan line 132); `ParsedTable` definition `fund_agent/fund/documents/models.py:676–688`
- **问题类型**: 契约缺失
- **当前写法**: Plan specifies matching on `row_locator` but does not define how `ChapterEvidenceAnchor.row_locator` maps to a specific row in `ParsedTable.rows`.
- **反例/失败场景**: `ParsedTable.rows` is `tuple[tuple[str, ...], ...]` — rows are indexed by position only, with no row-level identifier. `ChapterEvidenceAnchor.row_locator` is an arbitrary string (e.g., "row 3", "管理费", "2024年度"). The implementation agent must match a string locator against table rows, but: (a) is `row_locator` a row number? (b) a header match? (c) a cell substring match? (d) None of the above — and anchoring fails? The plan doesn't say.
- **为什么有问题**: Without a `row_locator` mapping contract, the implementation agent at EC-P1 could choose any of several incompatible strategies. If the strategy is wrong for live tables (which have multi-line headers, merged cells, and Chinese column names), EC-P2 live evidence will fail in ways that are a design problem, not an implementation bug.
- **直接证据**: `fund_agent/fund/documents/models.py:689` — `ParsedTable.rows: tuple[tuple[str, ...], ...]`, no row-level metadata. `fund_agent/fund/chapter_facts.py:118` — `row_locator: str | None` with no format specification. Plan line 132 references `row_locator` matching without mapping definition.
- **影响**: 实施 Agent 跑偏 / 生成错误代码 — materializer may silently match wrong rows or fail to match valid rows, producing false-positive or false-negative references.
- **建议改法和验证点**: Plan must specify: (a) `row_locator` format contract (numeric index? header cell match? cell substring?); (b) matching algorithm (exact header match, substring in any cell, first-column match); (c) fail-closed behavior when `row_locator` is ambiguous. Alternatively, explicitly scope row_locator matching as "exact row index if `row_locator` is a parseable integer, fail otherwise" for EC-P1.
- **修复风险**: 中
- **严重程度**: 高

### 4-MEDIUM-EC-P8质量门控FQ7规则码的schema扩展点未充分规格化

- **位置**: EC-P8 Work Unit (plan line 309–327)
- **问题类型**: 契约缺失
- **当前写法**: Plan says "Extend score payload or quality-gate integration payload with an optional `evidence_confirm` summary." The two options ("score payload or quality-gate integration payload") are materially different — one changes `score.json` schema (affecting `extraction_score.py`), the other changes the integration adapter (affecting `quality_gate_integration.py`).
- **反例/失败场景**: If the `evidence_confirm` summary is added to `score.json`, it becomes the responsibility of `extraction_score.py` — which is a scoring module that currently knows nothing about Evidence Confirm. If added to the integration adapter, it means `run_quality_gate_for_bundle()` must accept a new parameter, which changes the Service→Fund integration contract. The plan doesn't choose, leaving the implementation agent to make a public-contract decision.
- **为什么有问题**: The choice of extension point determines which modules change, which tests must be updated, and what the backward-compatibility surface looks like. This is a schema/contract decision that belongs in planning, not implementation.
- **直接证据**: Plan line 313–314: "Extend score payload or quality-gate integration payload." `fund_agent/fund/quality_gate.py:166` — `run_quality_gate()` takes `score_path: Path` and reads score JSON internally. `fund_agent/fund/quality_gate_integration.py:48` — `run_quality_gate_for_bundle()` takes `bundle: StructuredFundDataBundle`.
- **影响**: 实施 Agent 跑偏 — the wrong choice could require rework at EC-P9 or EC-P10 when the extension point proves incompatible with live quality gate evidence.
- **建议改法和验证点**: Plan must commit to one extension point: either (a) extend `score.json` schema with optional `evidence_confirm` key and update `extraction_score.py`; or (b) add `evidence_confirm_summary` parameter to `run_quality_gate_for_bundle()` and pass through to `run_quality_gate()`. The choice should be made based on whether `evidence_confirm` is a property of the score (option a) or a separate quality signal (option b).
- **修复风险**: 低
- **严重程度**: 中

### 5-MEDIUM-EC-P1测试未覆盖`ParsedAnnualReport`结构性边界情况

- **位置**: EC-P1 Required no-live tests (plan line 154–162)
- **问题类型**: 测试缺口
- **当前写法**: Required tests cover happy path (exact anchor locator match), wrong year, dangling table row, non-annual_report anchors, candidate-only metadata, value round-trip, and import isolation. No test covers: (a) empty `sections` dict, (b) empty `tables` tuple, (c) section exists but has no relevant tables, (d) `raw_text` at section offsets doesn't contain expected content, (e) multi-table sections where the "correct" table is ambiguous.
- **反例/失败场景**: In live data, annual reports routinely have sections with no tables (narrative-only sections), tables without clear section association, or sections where the raw text doesn't cleanly map to the expected content. If EC-P1 doesn't test against these structural edge cases, the materializer may crash or produce wrong references at EC-P2.
- **为什么有问题**: The materializer is the foundation primitive for the entire 11-gate program. If its behavior on realistic-but-unexpected parsed report structures is unspecified and untested, EC-P2 live evidence will discover design gaps that should have been caught at EC-P1.
- **直接证据**: Plan line 154–162 — test list covers six scenarios, none of which exercise structural edge cases. `ParsedAnnualReport.sections: dict[str, ReportSection]` can be empty. `ParsedAnnualReport.tables: tuple[ParsedTable, ...]` can be empty.
- **影响**: 风险后移 — structural edge case failures discovered at EC-P2 would require going back to EC-P1 for redesign.
- **建议改法和验证点**: Add test cases: (a) empty sections → all section_id-anchored facts produce blocking issues; (b) empty tables → all table_id-anchored facts produce blocking issues; (c) section has text but no matching table → materializer falls back to section excerpt; (d) raw_text offsets produce empty or whitespace-only excerpt → issue, not reference.
- **修复风险**: 低
- **严重程度**: 中

### 6-MEDIUM-Service请求契约变更点未定位到具体dataclass

- **位置**: EC-P6 Work Unit, Public contract changes (plan line 261–266)
- **问题类型**: 契约缺失
- **当前写法**: Plan says "Add explicit request field, for example `evidence_confirm_policy: Literal["off", "warn", "block"] = "off"` or a nested policy object." The plan uses "for example" language, which means the implementation agent must choose the exact field location and type.
- **反例/失败场景**: `FundAnalysisRequest` is a frozen dataclass in `fund_agent/services/`. Adding a field there changes every caller (CLI, tests, multi-year analysis) and requires all existing request constructions to be updated. A "nested policy object" would be less invasive but adds nesting complexity. The plan doesn't commit, leaving the implementation agent to make a public-contract decision that affects the entire Service surface.
- **为什么有问题**: The Service request contract is the primary public API. Changes to it propagate to CLI, tests, and documentation. The plan's hesitation between two materially different approaches means the implementation agent must design the contract at implementation time, which is a planning responsibility.
- **直接证据**: Plan line 263–264: "Add explicit request field, for example... or a nested policy object." `fund_agent/services/fund_analysis_service.py` — `FundAnalysisRequest` is accessed at `_run_analysis_core()` (line 1089). No current `evidence_confirm_policy` field exists in the request.
- **影响**: 实施 Agent 跑偏 — choosing a nested object when a flat literal is needed (or vice versa) would require rework across Service, CLI, and tests.
- **建议改法和验证点**: Plan must commit to one approach. Given the project convention of typed frozen dataclasses with explicit fields (not nested policy blobs), a flat `evidence_confirm_policy: Literal["off", "warn", "block"] = "off"` on `FundAnalysisRequest` is the simpler, auditable choice. The plan should also specify: (a) whether `FundChecklistRequest` gets the same field; (b) whether `MultiYearAnnualAnalysisRequest` inherits or independently declares it.
- **修复风险**: 低
- **严重程度**: 中

### 7-LOW-EC-P4语义entailment Protocol的位置定义未明确

- **位置**: EC-P4 Work Unit, Design (plan line 222–225)
- **问题类型**: 架构边界
- **当前写法**: "Define `EvidenceEntailmentClient` Protocol in Fund or Service boundary with fake implementation in tests." The plan says "Fund or Service," leaving the Protocol's home module unspecified.
- **反例/失败场景**: If the Protocol is defined in Service, Fund cannot import it without creating an illegal upward dependency (Fund → Service). If defined in Fund, Service implements it — which is clean. But the plan's ambiguity means the implementation agent could place it in Service and create a dependency violation that only later review catches.
- **为什么有问题**: Architecture boundary rules (`AGENTS.md` line 96–97) prohibit upper-layer dependencies. Fund → Service import is illegal. The Protocol must be in Fund (or a shared types module) for Fund to accept it.
- **直接证据**: Plan line 223: "Define `EvidenceEntailmentClient` Protocol in Fund or Service boundary." `AGENTS.md:96` — UI → Service → Host → Agent, no upward dependencies. `fund_agent/fund/evidence_confirm.py` is in Agent/Fund layer.
- **影响**: 后续返工 — if placed incorrectly, must be moved and all imports updated.
- **建议改法和验证点**: Specify: "Define `EvidenceEntailmentClient` Protocol in `fund_agent/fund/` (Fund layer). Service constructs the real provider client and injects it; Fund accepts only the Protocol." Add import isolation test.
- **修复风险**: 低
- **严重程度**: 低

## Open Questions

1. Does the current extractor/fact-projection pipeline populate `ChapterEvidenceAnchor.table_id` with values that can be mapped to `ParsedTable` in the current codebase? (The plan assumes it does, but this was not verified — it may be a closed-loop assumption.)
2. What is the actual format of `row_locator` values produced by the current `project_chapter_facts()` pipeline? Are they numeric indices, header cell strings, or something else?
3. Is `EvidenceConfirmReferenceBuildRequest.source_truth_status` defaulting to `"proven"` (plan line 126) intentional, or should it default to `"not_proven"` to force explicit caller admission? The current `EvidenceConfirmReference.source_truth_status` already defaults to `"proven"` (line 104), so the plan's default matches — but this means materializer-built references are proven-by-default, which weakens the fail-closed invariant.
4. EC-P9 Production Default Decision Gate says "Set deterministic Evidence Confirm default to `block`" as an option. Has the product impact been assessed? If EC is default-block and the materializer has structural gaps against real parsed reports, every analysis would block.

## Conclusion

**FAIL**. The EC-P1 implementation gate is not code-generation-ready. Findings 1, 2, and 3 together establish that the anchor-to-parsed-report locator mapping is structurally underspecified against the current `ParsedAnnualReport` / `ParsedTable` / `ReportSection` model. The implementation agent cannot proceed without inventing three design decisions that the plan should make: (1) `table_id` mapping to `ParsedTable`, (2) excerpt bounding rules from offset-only `ReportSection`, and (3) `row_locator` to `ParsedTable.rows` mapping. These are not implementation details — they are the core contract of the reference materializer, and getting them wrong would cascade failures through all subsequent gates.

Findings 4–7 are individually fixable through plan amendments and would not independently block, but together they indicate the plan is not sufficiently precise for a heavy-gate program plan.

The architecture boundary alignment is soundly reasoned (F1-HIGH quality), and the gate ordering correctly separates source access, deterministic proof, semantic judgment, and integration concerns. The program structure is the right one. But EC-P1 cannot begin until the `ParsedAnnualReport` locator contract is reconciled with the proposed anchor matching rules.

**Recommended next action**: Return to planning. Reconcile `ChapterEvidenceAnchor` locator fields against `ParsedAnnualReport` / `ParsedTable` / `ReportSection` actual fields. Either: (a) specify the derived mapping convention in the plan, or (b) accept that EC-P1 scope should be narrowed to `page_number`-based section excerpt matching only (deferring `table_id`+`row_locator` matching to EC-P2 after live table structure evidence), or (c) propose a `ParsedTable` model extension (adding `table_id`) as a prerequisite documents-model gate.
