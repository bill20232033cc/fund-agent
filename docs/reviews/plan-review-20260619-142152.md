## Reviewed Target And Scope

Plan proposes a single `core_risk.v1` candidate locator selector for `FundDisclosureDocumentProcessor`, adding S6-F as the sixth field family with candidate-only evidence. The selector maps five locator roles (`risk_characteristic`, `liquidation_or_scale_risk`, `tracking_error_or_deviation_risk`, `turnover_or_style_drift_risk`, `concentration_risk`) to section/paragraph/table/cell sources with strong-token-first matching and generic-token-with-guard rules. Candidate evidence stays `candidate_only`/`not_proven`/`NOT_READY`; public `value` and `anchors` remain empty; `current_stage.v1` stays untouched.

## Assumptions Tested

1. Plan is code-generation-ready: match groups, tokens, guards, source paths, ordering, dedupe, limit, excerpt, result semantics, and test specs are all explicit.
2. `core_risk.v1` locator-only: no veto, causality, risk level, pressure test, or final judgment inference.
3. Broad token guards: generic tokens require same-source role-specific guard context; cell self-guard prevented.
4. No boundary violations: `current_stage.v1`, `FundDataExtractor`, `EvidenceAnchor`/`EvidenceSourceKind`, repository/source/cache/live/LLM/renderer/quality gate/Service/UI/Host untouched.

## Findings

### F1-未修复-中-`_field_families_for_intermediate()` 六层嵌套三元式未处理

- **位置**: Plan §5 Implementation Design, current processor lines 476–491
- **问题类型**: 不可直接实施 / 过度耦合
- **当前写法**: Plan 描述 core_risk evidence selection 语义，但未指定 `_field_families_for_intermediate()` 的具体代码结构。当前函数使用五层嵌套三元式（product_essence → return_attribution → manager_profile → investor_experience → missing），添加 core_risk 后将成为六层。
- **反例/失败场景**: Implementation agent 直接追加第六层三元式，使函数可读性进一步劣化到超过合理阈值；或自作主张重构为 dict/mapping 模式导致 S6-B/C/D/E regression。
- **为什么有问题**: 当前嵌套三元式已到可维护性边界。Plan 不在 implementation design 中处理此问题，把代码结构决策完全推给 implementation agent。对于 `heavy planning gate`，这种决策应该由 planner 做出。
- **直接证据**:
  - Processor lines 476–491：五层嵌套三元式
  - Plan §5: "In `_field_families_for_intermediate()`, select: existing product_essence... existing return_attribution... existing manager_profile... existing investor_experience... new core_risk... `current_stage.v1` stays fully missing"
- **影响**: 实施 Agent 跑偏 / 后续返工
- **建议改法和验证点**: Plan 应在 implementation design 中明确推荐结构：是继续追加三元式，还是用 dict 映射 `family_id → evidence` 重构。若选择重构，必须将 S6-B/C/D/E regression 加入必须保留的测试列表。
- **修复风险**: 低 — 只需在 plan 中补充两行 code sketch 或明确决策。
- **严重程度**: 中

### F2-未修复-中-cell guard context 设计未显式对照 S6-E 的 role-variant 模式

- **位置**: Plan §3 Guard Rule (cell guard context), 对照 S6-E `_investor_experience_cell_guard_context()` lines 1635–1673
- **问题类型**: 不可直接实施
- **当前写法**: Plan 指定 cell guard context 为 "parent table heading_text, parent table_caption_or_nearby_heading, parent heading_path, cell row_label_path, cell column_header_path, and cell heading_path"，对所有五个 role 统一适用，不包含 cell_text/cell_text_normalized。
- **反例/失败场景**: Implementation agent 以 S6-E `_investor_experience_cell_guard_context()` 为模板复制修改，但该函数对 `subscription_redemption` 以外的 role **包含** cell_text 在 guard context 中（lines 1666–1673）。若 agent 逐字模仿此模式，会为 `risk_characteristic`、`concentration_risk` 等 role 意外包含 cell_text，导致 `风险`/`持仓` 等 broad token 在 same-cell 内自守卫。
- **为什么有问题**: Plan 的 guard context 设计是正确的（role-invariant、不含 cell_text），但未显式标注"不要照搬 S6-E 的默认分支逻辑"。S6-E 的默认分支恰好在 guard context 中包含 cell_text。这是一个容易踩的坑。
- **直接证据**:
  - Processor lines 1666–1673：非 `subscription_redemption` role 的 cell guard context 包含 `cell.cell_text_normalized, cell.cell_text`
  - Plan §3 Guard Rule: guard context 列表不含 cell_text
- **影响**: 生成错误代码 — cell self-guard 失效，broad token 在不相关 cell 中自授权
- **建议改法和验证点**: Plan 应在 cell guard context 描述后加上显式禁止："实现不得在 guard context 中包含 cell_text 或 cell_text_normalized，即使 S6-E 的 `_investor_experience_cell_guard_context` 对某些 role 这样做。" Test 9 (cell self-guard) 已设计覆盖此场景，plan 本身再加一行显式禁止即可消除风险。
- **修复风险**: 低 — plan 补充一行显式说明即可。
- **严重程度**: 中

### F3-未修复-低-shared helper 重构决策未收敛

- **位置**: Plan §5 Required helper structure
- **问题类型**: 不可直接实施
- **当前写法**: "Prefer a small private shared helper for guarded source scanning only if it preserves existing S6-B/S6-C/S6-D/S6-E order, paths, excerpts, limits, row locators, guard behavior, and tests."
- **反例/失败场景**: Implementation agent 创建 shared helper 并重构 S6-D/S6-E 使用它，引入 regression；或保守地复制 `_matches_guarded_core_risk_source()` 增加技术债。两个方向都有成本，plan 未做出选择。
- **为什么有问题**: S6-D 和 S6-E 已有结构相同但命名不同的 `_matches_guarded_manager_profile_source()` 和 `_matches_guarded_investor_experience_source()`。Plan 的 "prefer" 不是决策，把架构选择推给 implementation gate。
- **直接证据**: Processor — 两个 `_matches_guarded_*_source()` 函数（lines 1246, 1605）实现逻辑完全相同
- **影响**: 实施 Agent 跑偏 / 后续返工
- **建议改法和验证点**: Plan 应明确选择：(A) 新增 `_matches_guarded_core_risk_source()` 不复用现有函数（安全但重复），或 (B) 先重构为单一 `_matches_guarded_source()` 再用于 S6-F（干净但有重构风险）。若选 B，重构必须是独立 slice，先验证 S6-D/S6-E regression 再进入 S6-F。
- **修复风险**: 低
- **严重程度**: 低

### F4-未修复-低-测试未覆盖 mixed scenario：Chapter 6 输出词与合法 locator token 在同一 source 中共存

- **位置**: Plan §7 Test 10
- **问题类型**: 测试缺口
- **当前写法**: Test 10 "Content containing only Chapter 6 reasoning/output words" —— 只测纯输出词场景。
- **反例/失败场景**: 正文中 "根据压力测试结果，本基金风险收益特征为R3等级" —— 同时包含禁止的输出 token `压力测试` 和合法 strong token `风险收益特征`。Test 10 不覆盖此混合场景。Selector 应正确产生 `risk_characteristic` 记录（因 strong token 命中），但不因 `压力测试` 产生记录。当前 test description 不足以证明 selector 在混合场景下行为正确。
- **为什么有问题**: Pure-negative tests 不足以证明 selector 不因 forbidden token 产生误报。需要 mixed-positive-negative tests。
- **直接证据**: Plan §7 Test 10 description: "Content containing only Chapter 6 reasoning/output words"
- **影响**: 风险后移 — 混合场景行为未验证
- **建议改法和验证点**: Test 10 应扩展为两个 sub-case：(a) 纯 forbidden 词不产生记录（现有），(b) forbidden 词与合法 strong token 共存时，forbidden 部分不单独产生记录，strong token 正常命中。
- **修复风险**: 低
- **严重程度**: 低
