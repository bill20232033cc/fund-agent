### 1-未修复-高-Host cancel/deadline 被投影成 LLM runtime failure

- **入口/函数**: `run_agent_chapter_orchestration_bridge()` -> `_service_stop_reason_from_task()`.
- **文件(行号)**: `fund_agent/agent/runner.py:774`; `fund_agent/services/agent_bridge.py:502`; `fund_agent/services/chapter_orchestrator.py:100`; `tests/agent/test_service_bridge.py:107`.
- **输入场景**: Host cancel 或 deadline 在章节执行边界触发。
- **实际分支**: Agent 构造 `terminal_state="blocked_scheduler_interrupted"` 的 task，bridge 把该 terminal state 映射成 Service stop reason `llm_exception`。
- **预期行为**: scheduler interruption 应保留为调度/Host 生命周期终态，不应伪装成 LLM/provider runtime failure。
- **实际行为**: Service artifact / first-failed 分类会看到 `llm_exception`；测试也把 cancel 后的 stop reason 固化为 `llm_exception`。
- **直接证据**: `fund_agent/agent/runner.py:774-783` 写入 `stop_reason="scheduler_interrupted"` 和 `failure_category="scheduler_interrupted"`；`fund_agent/services/agent_bridge.py:502-503` 将 `blocked_scheduler_interrupted` 返回为 `llm_exception`；当前 `ChapterFailureCategory` 不包含 scheduler 分类。
- **影响**: 取消/超时会污染 incomplete-run diagnostics 和 provider/root-cause triage，使生命周期中断被误判为 LLM 异常。
- **建议改法和验证点**: 为 Service stop reason / failure category 增加显式 scheduler interruption 分类，至少区分 `scheduler_cancelled` 与 `scheduler_deadline_exceeded`；bridge 从 `AgentSchedulerInterruption.status/reason` 投影；更新 cancel/deadline 测试和 artifact serialization 测试。
- **修复风险（低/中/高）**: 中。涉及 public typed literals 和 tests，但应局限于 no-live scheduler path。
- **严重程度（低/中/高/严重）**: 高。

### 3-未修复-高-Host safe diagnostics 只检查 key 不检查敏感 string value

- **入口/函数**: `build_safe_diagnostics()` / `HostRunContext.record_diagnostic()`.
- **文件(行号)**: `fund_agent/host/runtime.py:141`; `fund_agent/host/runtime.py:347`.
- **输入场景**: 任意 Host operation 调用 `record_diagnostic(message="Bearer sk-test-secret system_prompt raw_response")`，或把 secret/prompt/raw response 放进普通诊断 value。
- **实际分支**: key 通过 forbidden-fragment 检查后，string value 只被截断，不做 value redaction 或 fail-closed。
- **预期行为**: Host safe diagnostics 是公共 runtime 边界，应对 key 和 string value 都执行安全策略。
- **实际行为**: `HostRunEvent.diagnostics`、`HostRunResult.safe_diagnostics` 和 event sink 可能保留敏感 value。
- **直接证据**: `runtime.py:141-155` 构造 safe diagnostics；`runtime.py:134-137` 对 string 只按长度返回/截断；`record_diagnostic()` 在 `runtime.py:347` 统一接受调用方 diagnostics。
- **影响**: 后续 artifact/event/outbox 或 progress 复用 Host diagnostics 时可能泄漏 API key、Authorization、prompt、draft 或 raw response。
- **建议改法和验证点**: 增加 forbidden value 检查或 redaction helper。当前 key 违规是 fail-closed，建议 string value 命中 forbidden fragment 时也 fail-closed，或者显式替换为 `[redacted]` 并测试 event sink 结果。覆盖 `session_id`、`record_diagnostic()` value 和 direct `build_safe_diagnostics()`。
- **修复风险（低/中/高）**: 中。fail-closed 可能暴露已有调用方违规，需要测试校准。
- **严重程度（低/中/高/严重）**: 高。
