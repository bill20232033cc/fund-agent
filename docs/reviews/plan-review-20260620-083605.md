## Material Findings

### 001-unresolved-medium-holdings_snapshot-cross-family-ownership

- **位置**: Section 4 (Exact Public Value Shape), Section 5.5 (Holdings Snapshot), Section 12 (Residual Risks)
- **问题类型**: 过度耦合 / 契约缺失
- **当前写法**: Plan includes `holdings_snapshot` as one of five allowed `manager_profile.v1` top-level keys. Section 5.5 says "Allowed source only if owned by `manager_profile.v1` in this gate". Section 12 defers "Holdings snapshot shape breadth" to a future refinement gate.
- **反例/失败场景**: `ActiveFundAnnualProcessor.FIELD_FAMILY_MAPPINGS` maps `holdings_share_change.holdings_snapshot` to three families: `manager_profile.v1` (chapter 3, priority `(3,)`), `current_stage.v1` (chapter 5, priority `(5,)`), and `core_risk.v1` (chapter 6, priority `(6,)`). When FDD processor emits `holdings_snapshot` only in `manager_profile.v1`, downstream consumers of `current_stage.v1` or `core_risk.v1` that expect `holdings_snapshot` from those families will get empty values. If the `ActiveFundAnnualProcessor` (used for non-FDD paths) provides `holdings_snapshot` to all three families but the FDD processor provides it only to `manager_profile.v1`, there is a behavioral divergence between processor routes.
- **为什么有问题**: The plan treats `holdings_snapshot` as semantically belonging to `manager_profile.v1` (manager behavior), but the existing `ActiveFundAnnualProcessor` mapping treats it as shared data also relevant to `current_stage.v1` (stage analysis) and `core_risk.v1` (risk analysis). The plan's non-goal "Do not route `current_stage.v1` or `core_risk.v1` through these values in this gate" is acceptable for scoping, but the plan does not document this cross-family divergence as a residual risk or specify how a future gate should reconcile it.
- **直接证据**: `fund_agent/fund/processors/active_annual.py:23` maps `holdings_share_change.holdings_snapshot` → `manager_profile.v1`; `active_annual.py:29` maps same to `current_stage.v1`; `active_annual.py:34` maps same to `core_risk.v1`. Plan Section 12 lists "Holdings snapshot shape breadth" as residual but does not mention cross-family ownership divergence.
- **影响**: Implementation worker may not realize that `holdings_snapshot` in `manager_profile.v1` creates a behavioral gap for `current_stage.v1` / `core_risk.v1` consumers. Future gates for those families must either re-extract `holdings_snapshot` independently or route through `manager_profile.v1`.
- **建议改法和验证点**: Add to Section 12 residual risks: "`holdings_snapshot` is mapped to `manager_profile.v1`, `current_stage.v1`, and `core_risk.v1` in `ActiveFundAnnualProcessor`. FDD direct extraction emits it only in `manager_profile.v1`. Future `current_stage.v1` / `core_risk.v1` gates must either independently extract `holdings_snapshot` from FDD content or explicitly route through `manager_profile.v1` value. Owner: future field-family gates." Add a test in Slice 3 or 4 asserting that FDD direct `manager_profile.v1` does not leak `holdings_snapshot` into `current_stage.v1` or `core_risk.v1` field-family results.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

### 002-unresolved-low-facade-divergence-between-fdd-and-active-processor-routes

- **位置**: Section 8 (Facade Projection Decision)
- **问题类型**: 契约缺失
- **当前写法**: "No `FundDataExtractor` production code change is planned." The plan asserts `data_extractor.py:708-793` already maps `manager_profile.v1` to the five bundle fields.
- **反例/失败场景**: `data_extractor.py:761-765` maps `manager_profile.v1` value keys to `portfolio_managers`, `turnover_rate`, `manager_alignment`, `manager_strategy_text`, and `holdings_snapshot` via `_field_from_family()`. This works for the FDD route. But the `ActiveFundAnnualProcessor` route (used for non-FDD `ParsedAnnualReport` input) may produce different value shapes for the same keys because it uses different extractors (`ManagerOwnershipExtractionResult`, `HoldingsShareChangeExtractionResult`). If FDD direct extraction produces values with different subkey structures than what `_field_from_family()` expects, the facade may silently project `None` or wrong values.
- **为什么有问题**: The plan assumes existing shapes are compatible but does not verify that FDD direct extraction output subkeys match what `_field_from_family()` extracts. For example, `_field_from_family(manager_profile, "portfolio_managers")` expects `value["portfolio_managers"]` to exist; if FDD direct extraction puts the manager list under a different key, the projection fails silently.
- **直接证据**: `data_extractor.py:761`: `portfolio_managers = _field_from_family(manager_profile, "portfolio_managers")`. Plan Section 4 defines `portfolio_managers` shape with `schema_version`, `fund_code`, `report_year`, `portfolio_managers` keys. The `_field_from_family` helper extracts by top-level key from `value` dict. If FDD emits `value = {"schema_version": "manager_profile.v1", "portfolio_managers": {...}, ...}`, then `_field_from_family(manager_profile, "portfolio_managers")` would get the inner dict, which is correct. But the plan's Slice 4 facade regression test must explicitly prove this.
- **影响**: Silent facade projection failure if FDD value shape diverges from active processor shape.
- **建议改法和验证点**: Slice 4 facade regression test should explicitly assert that FDD direct `manager_profile.v1` values project correctly through `_field_from_family()` to the expected bundle fields. The test should use the exact value shapes defined in Section 4.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低

### 003-unresolved-low-paragraph-block-section-id-availability

- **位置**: Section 7 (Anchor Construction And Gap Semantics)
- **问题类型**: 契约缺失
- **当前写法**: "section_id: from section/paragraph/table/cell section field"
- **反例/失败场景**: The S6-D implementation evidence shows paragraph blocks have `heading_path` but the plan does not verify that `FundDisclosureDocumentContentIntermediate.paragraph_blocks[*]` exposes a `section_id` field. If paragraph blocks lack `section_id`, the anchor construction would need to derive it from `heading_path` or set it to `None`, which the plan does not specify.
- **为什么有问题**: Implementation worker may be uncertain about anchor construction for paragraph-backed values (strategy text, turnover basis text, alignment text) if `section_id` is not directly available on paragraph blocks.
- **直接证据**: S6-D plan Section 4 lists paragraph fields as `text_normalized`, `text_raw`, `heading_path`; no `section_id` is listed. Plan Section 7 says `section_id` comes from "section/paragraph/table/cell section field".
- **影响**: Minor implementation ambiguity; worker may set `section_id=None` for paragraph anchors, which is acceptable but should be explicit.
- **建议改法和验证点**: Add to Section 7: "For paragraph-backed anchors, if `section_id` is not directly available on the paragraph block, use `None` or derive from `heading_path` first element; document the choice in implementation evidence."
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 低

## Conclusion

**pass-with-risks**

The plan is code-generation-ready. It correctly reuses the existing `FundDisclosureSourceTruthAdmissionProof` guard, follows the accepted `return_attribution.v1` pattern, defines explicit public value shapes compatible with existing `StructuredFundDataBundle` projection, and implements well-scoped fail-closed rules. The four implementation slices are small, progressive, and bounded to the correct file set.

The one material finding (001: `holdings_snapshot` cross-family ownership) is explicitly deferred by the plan's scope boundary and does not block implementation. The two low-severity findings (002, 003) are implementation clarifications that the worker should address in evidence.

The plan does not authorize source-truth extraction from candidate-only locators or weak proof. Docs sync is properly bounded. `NOT_READY` and `not_proven` are preserved.
