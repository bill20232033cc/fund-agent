### 1-未修复-中-token 子串匹配存在 false positive 和 false negative

- **入口/函数**: `_matched_anchor_ids` (L525-547) 和 `_material_tokens` (L550-564)
- **文件(行号)**: `evidence_confirm.py:545`
- **输入场景**: fact value 的数值 token 出现在 excerpt 中更大数值的子串中，或 fact value 与 excerpt 中相同数值但格式不同
- **实际分支**: `any(token in normalized_excerpt for token in tokens)` 执行纯子串包含检查
- **预期行为**: 数值 token 匹配应具备数值语义等价性，或至少避免跨数值边界误匹配
- **实际行为**:
  - False positive 示例: fact value `12` 归一化为 token `"12"`，excerpt 为 `"管理费率1.20%"`，归一化后 `"管理费率1.20%"`，`"12" in "管理费率1.20%"` → **True**（误匹配，12 出现在 1.20 的子串中）
  - False negative 示例: fact value `1.20` 归一化为 `"1.20"`，excerpt 为 `"管理费率1.2%"`，归一化后 `"管理费率1.2%"`，`"1.20" in "管理费率1.2%"` → **False**（漏匹配，1.20 与 1.2 数值等价但字符串不等）
- **直接证据**: `evidence_confirm.py:545` — `if any(token in normalized_excerpt for token in tokens)` 纯子串匹配，无数值边界保护
- **影响**: 真实年报数据中数值格式差异（省略尾零、千分位、百分比格式）可导致 E2 误判。当前测试用例刻意使用精确匹配文本（如 `"年报披露换手率为 120%。"` 对应 value `120`）规避了此问题
- **建议改法和验证点**:
  1. 短期: 对纯数值 token 增加边界保护，匹配时要求 token 前后字符不是数字/小数点（如 regex `(?<!\d){token}(?!\d|%)`），防止 "12" 匹配 "1.20"
  2. 中期: 对数值 token 做归一化对齐（去除尾零、统一小数位），支持 `"1.20"` 匹配 `"1.2"`
  3. 验证: 新增测试用例覆盖 `value=12` 不匹配 `excerpt="1.20%"`，以及 `value=1.20` 匹配 `excerpt="1.2%"`
- **修复风险（低/中/高）**: 中 — 需要平衡匹配精度与召回率，可能需要数值解析逻辑
- **严重程度（中）**: 当前测试数据刻意规避，但生产年报数据格式多样，此问题可能在真实场景触发

### 2-未修复-低-"reviewed_note" source_kind 在 phase 1 为死代码

- **入口/函数**: `_PROOF_SOURCE_KINDS` (L37) 和 `_reference_is_proof` (L421-446)
- **文件(行号)**: `evidence_confirm.py:37`
- **输入场景**: 调用方传入 `source_kind="reviewed_note"` 的 reference
- **实际分支**: `_reference_is_proof` 中 `reference.source_kind in _PROOF_SOURCE_KINDS` 通过（"reviewed_note" 在 frozenset 中），但 `_reference_matches_anchor` 中 `anchor.source_kind != reference.source_kind` 必然失败
- **预期行为**: phase 1 的 `_PROOF_SOURCE_KINDS` 应只包含当前可用的 anchor source kinds
- **实际行为**: `ChapterEvidenceSourceKind = Literal["annual_report", "external_api", "derived", "unknown"]`（`chapter_facts.py:61`），不含 `"reviewed_note"`。`ChapterEvidenceAnchor.source_kind` 的类型约束意味着运行时 anchor 的 source_kind 永远不可能是 `"reviewed_note"`，因此 `_PROOF_SOURCE_KINDS` 中的 `"reviewed_note"` 和 `_REFERENCE_SOURCE_KIND_PAIRS` 中的 `"reviewed_note": "reviewed_note"` 永远不会生效
- **直接证据**: `chapter_facts.py:61` — `ChapterEvidenceSourceKind` 闭集不含 `"reviewed_note"`；`evidence_confirm.py:37` — `_PROOF_SOURCE_KINDS` 含 `"reviewed_note"`
- **影响**: 不影响正确性（该路径不可达），但造成代码意图不清 — 阅读者无法判断是 phase 2 forward-looking 设计还是笔误
- **建议改法和验证点**:
  1. 若为 phase 2 设计，在 `_PROOF_SOURCE_KINDS` 旁加注释说明 `"reviewed_note"` 为 phase 2 预留
  2. 若非有意设计，从 `_PROOF_SOURCE_KINDS` 和 `_REFERENCE_SOURCE_KIND_PAIRS` 中移除 `"reviewed_note"`
  3. 验证: 无行为变更
- **修复风险（低/中/高）**: 低
- **严重程度（低）**: 死代码不影响运行时行为

### 3-未修复-低-_precision_issue 对 derived_calculation 无定位校验

- **入口/函数**: `_precision_issue` (L493-522)
- **文件(行号)**: `evidence_confirm.py:510`
- **输入场景**: reference_kind 为 "derived_calculation"，且 section_id/page_number/table_id/row_locator 全部为 None
- **实际分支**: `if reference.reference_kind == "derived_calculation": return None` 直接跳过
- **预期行为**: derived_calculation 作为 "derived" 来源，应有计算来源标识（如 section_id 标注计算依据）；至少应评估是否需要最低精度要求
- **实际行为**: derived_calculation reference 不检查任何定位字段，直接返回 None（无 precision issue）
- **直接证据**: `evidence_confirm.py:510-511` — derived_calculation 提前 return None
- **影响**: 一个无任何定位信息的 derived_calculation reference 可通过 precision 检查。若 `_reference_is_proof` 的 identity 校验也通过（anchor source_kind 为 "derived"），该 reference 可关闭 E2/E3。当前 phase 1 中 derived fact 会被 `_fact_is_derived` 拦截为 not_applicable，因此此路径实际不可达；但若 phase 2 扩展 derived fact 的可审计范围，此处会成为审计漏洞
- **建议改法和验证点**: 在 derived_calculation 分支增加最低定位要求（至少 section_id 或 note 非空），或显式注释说明 phase 1 不检查 derived 的原因
- **修复风险（低/中/高）**: 低
- **严重程度（低）**: 当前 phase 1 不可达，仅 phase 2 扩展时有风险

### 5-未修复-低-report_year=0 静默跳过年份校验

- **入口/函数**: `_report_year_from_facts` (L824-841) 和 `_reference_matches_anchor` (L470)
- **文件(行号)**: `evidence_confirm.py:841` 和 `evidence_confirm.py:470`
- **输入场景**: fact_id 格式不含年份（如 `"fund:110011:unknown:turnover_rate"`），`_report_year_from_facts` 返回 0
- **实际分支**: `_reference_matches_anchor` 中 `if report_year and reference.document_year is not None` — `report_year=0` 为 falsy，跳过整个年份校验
- **预期行为**: 无法提取报告年份时应产生 warning 或显式标记年份未验证
- **实际行为**: `report_year=0` 使年份校验静默失效，任何 `document_year` 的 reference 都能通过年份检查
- **直接证据**: `evidence_confirm.py:470` — `if report_year and ...`，0 为 falsy
- **影响**: 若 fact_id 格式异常导致年份提取失败，不会有任何告警，2023 年的 reference 可关闭 2024 年的 fact
- **建议改法和验证点**: 在 `_confirm_fact` 或 `confirm_chapter_evidence` 中，若 `report_year == 0` 则生成一个 E1 reviewable issue 提示年份不可验证；或在 `_reference_matches_anchor` 中对 `report_year == 0` 做显式处理
- **修复风险（低/中/高）**: 低
- **严重程度（低）**: 当前测试 fact_id 格式固定含年份，生产环境中 fact_id 由 chapter_facts 投影生成，格式受控

- `_matched_anchor_ids` 的 `dict.fromkeys(matched)` 去重语义正确但依赖 Python 3.7+ dict 有序保证，若未来支持 Python 版本变更需注意。
