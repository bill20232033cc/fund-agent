## Contract Decision Stress Test

**Decision**: Four deferred roles → required `core_risk.v1` field-family subvalues using `core_risk_role_disclosure.v1` schema. No `StructuredFundDataBundle.core_risk`. No `EvidenceSourceKind` expansion.

**Stress-test result**: PASS. The decision is first-principles-sound.

- The four roles are Chapter 6 risk disclosure surfaces (`AGENTS.md` R=A+B-C framework, chapter 6 = core risk). Their natural public owner is `core_risk.v1`.
- Existing bundle fields (`tracking_error` → `return_attribution.v1`, `turnover_rate` → `manager_profile.v1`, `holdings_snapshot` → `manager_profile.v1`) already have separate owners. The plan explicitly forbids overwriting them (lines 37-40).
- Keeping roles as field-family subvalues (not bundle fields) is consistent with the current architecture: `core_risk.v1` value stays within the processor boundary; the facade (`data_extractor.py:724`) only projects `risk_characteristic_text` fallback.
- No `EvidenceSourceKind` expansion is required because `annual_report` already covers these disclosure surfaces.

## Blocker Analysis

### Blocker 1: `_resolve_core_risk_role_candidate(...)` algorithm unspecified

**Severity**: Code-generation-blocking.

The plan names `_resolve_core_risk_role_candidate(...)` (line 205) as a new helper but provides zero specification:

- What defines a conflict? Two paragraphs matching the same role with different text? Multiple table cells? Cross-source (paragraph vs table) matches?
- Resolution strategy: first-match-wins, longest-text-wins, most-specific-token-match-wins?
- Relationship to `ambiguous_paths` return from `_select_core_risk_role_values()`: does the resolver produce ambiguous paths, or does the selector produce them and the resolver only picks?

Every other new helper (`_collect_core_risk_role_paragraph_candidates`, `_collect_core_risk_role_cell_candidates`, `_core_risk_role_anchor`, `_core_risk_role_text_value`, `_build_core_risk_role_disclosure_value`) has clear input/output semantics derivable from the plan. This one doesn't.

**Required fix**: Specify the conflict detection condition and resolution strategy for this function, or remove it from the plan and inline the logic into `_select_core_risk_role_values()` where the ambiguity set is already computed.

### Blocker 2: Source discrimination rules are aspirational, not code-mappable

**Severity**: Implementation-drift risk → code-generation-blocking.

Lines 149-157 define "allowed" vs "disallowed" direct sources in natural language:
- Allowed: "paragraph text with role-specific strong token or generic token plus source-level guard context"
- Disallowed: "section-only heading matches", "table-only heading/caption matches with no substantive row/cell disclosure"

The plan reuses `_CORE_RISK_MATCH_GROUPS` tokens (strong/generic/guard tuples). The candidate evidence selector already matches paragraphs and cells against these tokens. The critical discrimination the plan requires — distinguishing "substantive disclosure paragraph" from "paragraph that happens to contain a token in a heading/structural context" — has no code-level specification.

The structural information exists in the intermediate representation (sections → paragraph_blocks → table_blocks → cells hierarchy), but the plan doesn't specify:
- Is a `paragraph_block` that is only a section heading distinguishable from body paragraphs? (Check: `block_kind` or `heading_level` field?)
- Is a `table_block` header row distinguishable from data rows?
- When a paragraph both matches a heading token AND contains body-level disclosure text, which rule applies?

The stop condition (line 312) says "A role cannot be extracted from direct paragraph/cell disclosure without relying on candidate evidence or heading-only locators" → `BLOCKED`. Without structural discrimination rules, the implementer cannot programmatically distinguish these cases.

**Required fix**: Add a structural discrimination specification: (a) heading-only blocks are excluded when the matching block has structural heading properties (specify field); (b) table header-only matches are excluded when the cell is in a header/heading row with no data-row context; (c) a paragraph that matches both heading and body tokens must contain at least one non-heading sentence with role-specific strong tokens to qualify.

### Blocker 3: Anchor duplication inside role subvalues

**Severity**: Design inconsistency → should be resolved before implementation.

The value shape (lines 93-131) embeds `source_anchors` as a list of dicts within each role subvalue, e.g.:

```python
"source_anchors": [
    {
        "section_id": anchor.section_id,
        "page_number": anchor.page_number,
        "table_id": anchor.table_id,
        "row_locator": anchor.row_locator,
    }
]
```

Every role subvalue duplicates `EvidenceAnchor` fields that already live in `FundFieldFamilyResult.anchors` (line 1216, populated via `_dedupe_anchors` from emitted output paths). No other field family (`product_essence.v1`, `return_attribution.v1`, `manager_profile.v1`, `investor_experience.v1`, `current_stage.v1`) embeds anchors inside values.

The duplication creates:
- Synchronization risk: value-internal anchors and `FundFieldFamilyResult.anchors` can diverge
- No serialization contract specified: `EvidenceAnchor` → dict mapping is ad-hoc
- Unclear selection: if a role subvalue is present but its anchor was deduped out of `result.anchors`, is the value-internal anchor still valid?

**Required fix**: Either (a) remove `source_anchors` from the role subvalue shape and rely solely on `FundFieldFamilyResult.anchors` (consistent with other field families), or (b) specify the anchor serialization contract, selection rules, and keep-in-sync guarantee.

### Blocker 4: `_select_core_risk_values()` return type change underspecified

**Severity**: Implementation confusion risk.

Current signature (line 4665):
```python
def _select_core_risk_values(...) -> tuple[dict[str, _RiskCharacteristicValueCandidate], set[str]]
```

The plan adds `_CoreRiskRoleValueCandidate` (lines 190-196) and says to "merge existing `_select_risk_characteristic_value()`; new role text candidates." But the merged return type is not specified:
- Does the dict contain both `_RiskCharacteristicValueCandidate` and `_CoreRiskRoleValueCandidate` values? If so, the type must become a union or a shared base class.
- Does `_build_core_risk_value()` (line 4685) accept this merged dict? Its current parameter type is `dict[str, _RiskCharacteristicValueCandidate]`.
- Does `_core_risk_emitted_output_paths()` need to distinguish between candidate types?

All callers of these functions must change. The plan should specify the unified value-candidate type hierarchy.

**Required fix**: Specify whether `_CoreRiskRoleValueCandidate` shares a base class with `_RiskCharacteristicValueCandidate`, or define a `_CoreRiskValueCandidate` union type, and update the type signatures of all affected functions.

3. **`deferred_role` removal**: Verified NOT in `FundExtractionGapCode` (contracts.py:43-62). The plan correctly identifies and removes this non-standard gap code. Current usage at fund_disclosure_processor.py:4779 uses `# type: ignore[arg-type]` to bypass the type checker — this must be cleaned up.

4. **`_CORE_RISK_REQUIRED_TOP_LEVEL` references**: The plan says to rename/replace `_CORE_RISK_DEFERRED_ROLES`. Current reference at line 4786 (gap emission) must be updated. No other references found.

5. **`_core_risk_emitted_output_paths()` output path naming**: The plan doesn't specify output path naming for role subvalues. Presumably the role name (`"liquidation_or_scale_risk"`, etc.), but this should be explicit.

6. **`risk_disclosure_text` normalization**: The plan says "normalized direct annual-report disclosure" but doesn't define normalization (whitespace collapse? newline removal? CJK fullwidth conversion?).

7. **`_CORE_RISK_MATCH_GROUPS` `risk_characteristic` entry**: The plan says "skip `risk_characteristic`" when reusing match groups for role extraction (line 207). This is correct — `risk_characteristic` is already handled by `_select_risk_characteristic_value()`.

## Conclusion

The plan's contract decision is correct. The four roles belong in `core_risk.v1` as field-family subvalues. The status semantics change (accepted→partial when roles missing) is contract-valid. The gap semantics change (remove `deferred_role`, use `field_family_partial`/`ambiguous_table_or_locator`) uses valid gap codes.

However, 4 blockers must be resolved before code generation:
1. `_resolve_core_risk_role_candidate` algorithm specification
2. Source discrimination rules from aspirational to code-mappable
3. Anchor duplication design decision
4. `_select_core_risk_values` return type specification

With all 4 resolved, the plan is code-generation-ready.
