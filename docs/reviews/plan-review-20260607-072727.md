## Findings

### F1-未修复-中-typed deleted required-output 规则没有可实施契约入口

- **位置**: Plan §5 Slice 1A, exact implementation decision 1 and 7
- **问题类型**: 契约缺失 / 不可直接实施 / 过度耦合
- **当前写法**: Plan 要求 `_audit_contract_markers()` 在 typed path 下“require markers for typed item ids whose action is not deleted by the writer path”，并要求“deleted typed items must not be required”。
- **反例/失败场景**: Implementation agent 按计划实现时，`ChapterAuditInput` 当前只有 `writer_input` 和 `draft`，没有 writer 已计算出的 `RequiredOutputEvidencePlan` 或 `action` 列表。writer 的删除裁决由 `chapter_writer.py::_required_output_evidence_plan()` / `_required_output_action()` 私有函数根据 `EvidenceAvailability` 动态计算；auditor 没有稳定公共 API 读取“哪些 typed required output 被 delete”。
- **为什么有问题**: 为了满足 plan，implementation agent 只能在 auditor 中复制 writer 的私有 required-output action 推导、扩大 `ChapterAuditInput` 公共契约、或跨模块导入 writer 私有函数。三种路径都会扩大 Slice 1A，增加 writer/auditor 双实现漂移或 public contract 变更风险。当前真实 Ch1 证据并不需要处理 `delete_if_not_applicable`，因为 Ch1 required-output entries 的 `when_evidence_missing` 均为 null；计划把未来 delete 行为绑定进当前 Ch1 修复，超出最小闭环。
- **直接证据**:
  - `chapter_auditor.py::_audit_contract_markers()` 当前只读取 `input_data.writer_input.chapter.contract.required_output_items` 和 markdown。
  - `chapter_writer.py::_prompt_required_output_marker_items()` 用 `required_output_plan` 过滤 `plan.action != "delete"`。
  - `chapter_writer.py::_required_output_evidence_plan()` / `_required_output_action()` 是 writer 内部逻辑，不是 auditor 公共契约。
  - `docs/fund-analysis-template-draft.md` 当前 Ch1 required-output entries 的 `when_evidence_missing` 为 null，当前 Slice 1A 不需要 solve delete semantics。
- **影响**: 实施 Agent 可能复制私有逻辑或修改过宽契约，导致 writer/auditor required-output action 规则未来漂移；也可能为了实现 deleted-filter 改动 Service/Fund API，扩大 gate blast radius，review 难以确认 Ch1 root cause 是否真正闭环。
- **建议改法和验证点**:
  - 把 Slice 1A 收窄为当前事实：typed path 下 programmatic auditor 使用 `writer_input.typed_required_output_items[*].item_id` 作为 required marker set；legacy path 继续使用 `chapter.contract.required_output_items`。
  - 明确 `delete_if_not_applicable` filtering 不在 Slice 1A 内实现，因为当前 canonical JSON 没有该行为的 active Ch1 evidence；若未来需要，单独开 “writer/auditor required-output plan sharing contract” gate。
  - 或者，如果 controller 坚持支持 delete filtering，本 plan 必须先设计一个稳定契约，例如 `ChapterAuditInput` 接收 writer-produced marker obligation tuple，并补 tests 覆盖 render/delete/gap/block 四类 action；这应成为单独 slice，而不是 Ch1 narrow fix。
  - 验证点：typed Ch1 positive/negative tests 和 legacy marker tests 足够；不要要求 implementation agent处理 deleted typed items。
- **修复风险（低/中/高）**: 低，如果收窄 Slice 1A；中，如果设计共享 marker-obligation 契约。
- **严重程度（低/中/高/严重）**: 中
