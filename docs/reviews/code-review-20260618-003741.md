## Findings

### 1-MEDIUM — `_blocked_result` 对所有阻断原因使用固定 `source_boundary="unsupported_intermediate"`

- **入口/函数**: `_blocked_result()` → `ActiveFundAnnualProcessor.extract()`
- **文件(行号)**: `active_annual.py:548-593` (定义), `active_annual.py:244-257` (调用点)
- **输入场景**: 当 dispatch key 的 `fund_type != "active_fund"`、`report_type != "annual_report"` 或 `processor_goal` 不匹配时，`supports()` 返回 False，触发 `gap_code="unsupported_processor"`
- **实际分支**: `_blocked_result` 始终构造 `FundExtractionGap(source_boundary="unsupported_intermediate", ...)`（line 585），不区分阻断原因
- **预期行为**: `source_boundary` 应反映真正的阻断原因：fund_type 不匹配应使用 `"unsupported_fund_type"`，report_type 不匹配应使用对应的 boundary（当前 Literal 无该值时可保留为 `"unsupported_intermediate"` 作为跨切面 fallback），intermediate_kind 不匹配使用 `"unsupported_intermediate"`
- **实际行为**: 所有 `supports()` 返回 False 的场景（含 fund_type、report_type、processor_goal 不匹配）都被标记为 `source_boundary="unsupported_intermediate"`，误导下游消费者认为中间态格式是问题根因而非基金类型或报告类型不匹配
- **直接证据**: `active_annual.py:585` 硬编码 `source_boundary="unsupported_intermediate"`，两个调用点（line 245-249 与 line 251-256）均不覆盖 `source_boundary` 参数
- **影响**: 错误 gap metadata；若未来下游系统根据 `source_boundary` 做路由或诊断决策，fund_type/report_type 阻断会被误归类为中间态格式问题
- **建议改法和验证点**: `_blocked_result` 接受可选 `source_boundary` 参数，调用点根据实际阻断原因传入对应值；`unsupported_processor` 场景下可按阻断维度选择 `FundExtractionSourceBoundary` 中最匹配的枚举值；验证不同 dispatch key 阻断场景的 gap 归因正确性
- **修复风险（低）**: 只改 metadata 字段值，不改变 fail-closed 行为或 contract_status
- **严重程度（中）**: 错误归因 metadata，当前 no-live 无生产影响，但会误导未来集成阶段的问题诊断

### 2-LOW — `_blocked_result` 的 `gap_code` 参数类型为 `str` 而非 `FundExtractionGapCode`

- **入口/函数**: `_blocked_result()`
- **文件(行号)**: `active_annual.py:548-553`
- **输入场景**: 任意调用 `_blocked_result` 的场景
- **实际分支**: 函数签名 `gap_code: str`，传入 `FundExtractionGap` 时使用 `# type: ignore[arg-type]` 绕过类型检查（line 581）
- **预期行为**: `gap_code` 参数应声明为 `FundExtractionGapCode`，消除 `type: ignore` 注释
- **实际行为**: 类型检查器无法验证调用方传入的 gap_code 是否在 Literal 内；当前两个调用点传入的值（`"unsupported_processor"`, `"input_type_mismatch"`）恰好在 Literal 中，但类型系统未提供保护
- **直接证据**: `active_annual.py:552` 声明 `gap_code: str`，`active_annual.py:581` 的 `# type: ignore[arg-type]` 注释
- **影响**: 类型安全缺口；若未来新增调用点传入不在 Literal 中的值，运行时不会报错但会污染 gap 数据
- **建议改法和验证点**: 将参数类型改为 `FundExtractionGapCode`，移除 `type: ignore` 注释
- **修复风险（低）**: 纯类型注解修改
- **严重程度（低）**: 当前运行时行为正确，仅类型安全性受影响

### 3-LOW — `field_family_missing` gap code 已定义但从未被发出

- **入口/函数**: `_field_gap()` → `_build_field_family_result()`
- **文件(行号)**: `active_annual.py:398-423`（`_field_gap` 始终发出 `field_family_partial`）, `contracts.py:39-50`（Literal 定义含 `field_family_missing`）
- **输入场景**: 字段族内所有 required mapping 均缺失
- **实际分支**: `_derive_family_status` 返回 `"missing"` 状态（line 447-448），但每个缺失 mapping 的 gap 仍为 `gap_code="field_family_partial"`（line 417）
- **预期行为**: 计划要求区分 `field_family_missing` 与 `field_family_partial`。当整个字段族完全缺失时，gap 应使用 `field_family_missing` 而非 `field_family_partial`；或至少在字段族级产生一个 `field_family_missing` gap
- **实际行为**: 所有字段级 gap 统一使用 `field_family_partial`，`field_family_missing` 只存在于 Literal 定义中。下游消费者无法从 gap_code 区分"字段族部分缺失"与"字段族完全缺失"，需要通过 `FundFieldFamilyResult.status` 间接判断
- **直接证据**: `active_annual.py:417` 固定 `gap_code="field_family_partial"`；`active_annual.py:443-449` 的 `_derive_family_status` 可返回 `"missing"` 但不改变 gap_code
- **影响**: gap code 粒度不够精细；`field_family_missing` 与 `field_family_partial` 的语义差异无法通过 gap code 表达
- **建议改法和验证点**: 当 `_derive_family_status` 返回 `"missing"` 时，为该字段族产生一个 `gap_code="field_family_missing"` 的顶层 gap；或将完全缺失字段族的每个字段 gap 改为 `field_family_missing`
- **修复风险（低）**: 只改变 gap code 字符串，不改变 status 行为。需更新测试中的 gap code 断言
- **严重程度（低）**: 两种 gap 语义可通过 `FundFieldFamilyResult.status` 间接区分，当前 no-live 无功能影响

## Residual Risk

- **Registry 未测试 `report_type`/`intermediate_kind` 无匹配场景**：`test_registry_unsupported_context_fail_closed` 只验证 `_NeverProcessor` 不匹配导致 `UnsupportedFundProcessorError`，未验证具体 dispatch key 字段（如 `report_type="semi_annual_report"` 或 `intermediate_kind="docling_pdf_candidate.v1"`）无 processor 支持的场景。ActiveFundAnnualProcessor 的 `supports()` 对这些值的返回正确（False），但 registry 集成路径未被直接测试。
- **Field-level mapping table 硬编码耦合**：`_collect_existing_extractor_fields` 中 extractor 输出字段名（15 个）与 `FIELD_FAMILY_MAPPINGS` 中 source_path（24 个引用）通过字符串硬编码耦合。若窄 extractor 新增输出字段，必须同步更新两处，否则新字段不会进入 field family。当前无测试覆盖"extractor 有输出但 mapping 未引用则不出现在 value 中"的防御语义。
- **`_derive_contract_status` 不支持 `not_applicable`/`blocked` 状态**：当未来 field family 出现 `not_applicable` 或 `blocked` 状态时，`_derive_contract_status` 会将它们归入 `"missing"`，这可能是错误的整体契约状态。S1 不触发此场景，但函数缺乏 defensive 处理。
- **`_dedupe_anchors` 使用 `set[EvidenceAnchor]`**：`EvidenceAnchor` 是 frozen dataclass，hash 基于所有字段。若两个 anchor 逻辑上等价但 `note` 字段微有差异，会被视为不同 anchor 导致去重不彻底。当前影响较低（S1 只透传 extractor 产生的 anchor），但未来 anchor 数量增长时可能产生大量语义重复 anchor。
