## 3. Findings

### F1-未修复-中-source_kind 字段重载会误导后续实现写错锚点语义

- **位置**: Sections 4.1, 4.2, 5; especially source-layer field `source_kind`, processed-layer references to parser source, and processor dispatch `source_kind`.
- **问题类型**: 契约缺失 / 架构边界 / 不可直接实施
- **当前写法**: Design uses `source_kind` for repository/source identity and says current production annual reports remain `eid` source metadata. The same artifact also discusses processed parser identities such as `docling_pdf_candidate` and current `EvidenceAnchor` semantics.
- **反例/失败场景**: An implementation worker adds proof rows and maps `source_kind="eid"` or `source_kind="docling_pdf_candidate"` into an `EvidenceAnchor`-shaped candidate, or uses `source_kind` ambiguously in matrix generation. Current production `EvidenceAnchor.source_kind` only accepts semantic evidence source classes such as `annual_report`, `external_api`, and `derived`; candidate model separately has `CandidateRepresentationSourceKind`.
- **为什么有问题**: The design's goal is to avoid one-off scripts by becoming an implementation handoff. If the contract overloads `source_kind`, the next worker must infer whether it means repository source, parser route, or EvidenceAnchor source. That creates exactly the kind of implicit design choice this gate is supposed to remove.
- **直接证据**:
  - `docs/design.md` defines `EvidenceAnchor` fields including `source_kind`, with annual-report anchors using `annual_report`.
  - `fund_agent/fund/documents/candidates/representation_models.py` already defines `CandidateRepresentationSourceKind` values such as `docling_pdf_candidate`, `pdfplumber_pdf_candidate`, and `eid_xbrl_html_render_candidate`.
  - The reviewed design Section 4.1 uses `source_kind` to mean repository/source identity and Section 5 uses `source_kind` as a dispatch key.
- **影响**: Implementation agent may generate ambiguous schemas, write wrong source-kind values into anchor candidates, or create evidence matrices that appear to prove source truth while mixing repository source identity with parser route identity.
- **建议改法和验证点**:
  - Rename source-layer `source_kind` to `disclosure_source_name` or `repository_source_name`.
  - Rename processed parser identity to `processor_source_kind` or `processor_profile`.
  - Add a boundary rule: `EvidenceAnchor.source_kind` remains semantic evidence kind and must stay `annual_report` for annual-report anchor candidates; repository source and processor route must only be preserved in metadata/note/proof rows.
  - Update dispatch wording so the key is `document_kind + repository_source_name + media_type + processor_profile`.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中
