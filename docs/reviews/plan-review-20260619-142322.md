### 001-unfixed-low-overlap-regression-test-gap

- **位置**: Section 7 Required Tests, test 12 `test_core_risk_selector_preserves_overlap_family_semantics`
- **问题类型**: 测试缺口
- **当前写法**: Test 12 asserts S6-B/C/D/E candidate records "if produced by their own tokens, keep their own `field_family_id`, row locator, limit, gap semantics, and `value == {}` / `anchors == ()`." The "if produced" qualifier means the test only checks overlap semantics when the test fixture happens to produce S6-B/C/D/E records; it does not assert that S6-B/C/D/E record counts, ordering, limits, or gap semantics are unchanged compared to a no-S6-F-content baseline.
- **反例/失败场景**: An implementation bug could subtly alter S6-D's guard context or ordering when S6-F content is present, and test 12 would pass as long as S6-D records still exist with correct field_family_id. The existing S6-D regression tests (test `test_manager_profile_selector_orders_dedupes_limits_and_truncates` etc.) would catch this only if S6-F's `_field_families_for_intermediate()` call path is exercised in those tests, which currently it is not since those tests use intermediates with S6-D-relevant content only.
- **为什么有问题**: The existing S6-B/C/D/E regression tests do not include S6-F-relevant content in their fixtures, so they cannot detect cross-selector interference from the S6-F code path. Test 12 is the designated overlap regression test but its assertions are conditional ("if produced") rather than comparative.
- **直接证据**: Plan Section 7 test 12 wording: "if produced by their own tokens, keep their own..."
- **影响**: Low. The existing S6-B/C/D/E tests validate each selector in isolation. If S6-F's implementation introduces a shared-helper regression, it would likely manifest as a change in those tests too (since `_field_families_for_intermediate()` calls all selectors). The risk is that a subtle ordering/limit change in S6-D when S6-F content co-exists goes undetected.
- **建议改法和验证点**: Strengthen test 12 to explicitly assert: (a) construct a fixture with S6-D-relevant AND S6-F-relevant content; (b) run the processor; (c) assert S6-D record count, path ordering, limit, and gap semantics are identical to running with S6-D-relevant content only. Same for S6-B, S6-C, S6-E. Alternatively, add a dedicated test `test_core_risk_selector_does_not_change_s6d_record_count_or_order` that uses the S6-D ordering/dedupe test fixture augmented with S6-F risk tokens.
- **修复风险（低/中/高）**: Low
- **严重程度（低/中/高/严重）**: Low

## Conclusion

**pass-with-risks**

The plan is code-generation-ready. It correctly restricts `core_risk.v1` to locator-only candidate evidence, prevents Chapter 6 reasoning/output token capture, guards broad tokens with source-level context, prevents same-cell self-guarding, handles overlap with S6-B/C/D/E without changing their semantics, and specifies 12 focused tests covering positive matching, negative broad-token cases, guard behavior, cell self-guard blocking, ordering/dedupe/limit, and overlap regression.

One low-severity finding: the overlap regression test (test 12) uses conditional assertions ("if produced") rather than comparative assertions against a no-S6-F-content baseline. This is mitigated by existing S6-B/C/D/E regression tests running in isolation, but a dedicated cross-selector interference test would strengthen the suite. The implementation worker should consider adding it.

All residual risks are owned by later gates and do not block this planning artifact. Release/readiness remains `NOT_READY`.
