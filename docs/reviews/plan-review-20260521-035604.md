# Plan Review — P8-S1 Must Answer Audit Contract Design

## Reviewed Target

- Plan: `docs/reviews/p8-s1-must-answer-audit-contract-design-plan-20260521.md`
- Scope: design and implementation plan for routing `CHAPTER_CONTRACT.must_answer` into auditable coverage categories.
- Review mode: adversarial plan review using code facts from `fund_agent/fund/audit/*`, `fund_agent/fund/template/contracts.py`, and existing audit/template tests.

## Assumptions Tested

- The plan is code-generation-ready for an implementation agent.
- Deterministic C2 remains honest and does not pretend to prove semantic answers.
- Audit/template ownership stays inside Capability.
- Existing C2 behavior and `ProgrammaticAuditResult` public shape remain stable.
- Tests can prove both coverage completeness and runtime behavior.

## Findings

### PR-001-未修复-高-缺少逐条 must_answer 覆盖映射，implementation agent 仍需重新设计核心决策
- **位置**: `S3. Seed Coverage Manifest Conservatively`
- **问题类型**: 不可直接实施 / open question 未收敛 / 测试缺口
- **当前写法**: plan 只给出分类策略：“Most `must_answer` items that already correspond...” 和 “Questions that require semantic quality judgment...”，但没有列出每条 `must_answer` 应归入哪个 coverage kind、对应哪个 required item、marker 或 rationale。
- **反例/失败场景**: implementation agent 在实现 `ContractMustAnswerCoverageRule` 时必须逐条阅读 8 章 `must_answer`，自行判断“当前最大的风险是什么”是 `covered_by_required_item`、`llm_semantic_audit` 还是 `narrative_guidance`。另一个 agent 可能给出不同分类，但两者都能通过“每条出现一次”的机械校验。
- **为什么有问题**: 本 slice 的核心目标正是把 hidden gap 变成显式 contract。如果 plan 不给初始覆盖表，最关键的设计决策被后移到代码实现阶段，review 无法判断分类是否符合审计边界，也无法验证未来实现是否偏离 controller 裁决。
- **直接证据**:
  - `contracts.py` 中 `ChapterContract.must_answer` 是自然语言问题列表。
  - `contract_rules.py` 当前 required item 规则是逐条显式枚举的 marker manifest，而不是由实现时临场推断。
  - plan 的 S3 只有策略性描述，没有每章/每题 mapping table。
- **影响**: 实施 Agent 跑偏 / review 不可验收 / 后续返工 / 可能把语义问题误放进 deterministic C2 或把可程序化问题降级为 narrative guidance。
- **建议改法和验证点**:
  - 在 plan 中新增 “Initial Must Answer Coverage Map” 附录。
  - 每行至少包含 `chapter_id`、`question_text`、`coverage_kind`、`required_item_text` 或 `markers_any`、`rationale`。
  - 对所有当前 `must_answer` 做 exact-text 覆盖。
  - implementation 只允许按该表编码；任何分类调整必须先修订 plan/review artifact。
- **修复风险（中）**: 需要一次性审阅 8 章所有 `must_answer`，但这是本 slice 的核心设计工作，不能下放给 implementation。
- **严重程度（高）**:

### PR-002-未修复-中-把 LLM/evidence/narrative routing 放进 ProgrammaticContractRules 容易模糊 C2 的证明边界
- **位置**: `S1. Extend Audit Contract Rule Model`
- **问题类型**: 架构边界 / 过度耦合 / 最佳实践偏离
- **当前写法**: plan 要给 `ProgrammaticContractRules` 增加 `must_answer_coverages`，其中包含 `llm_semantic_audit`、`evidence_confirm`、`narrative_guidance` 等非程序化 coverage kind。
- **反例/失败场景**: 后续代码或文档看到 `load_programmatic_contract_rules()` 返回了所有 `must_answer` coverage，误以为 C2 已经“消费”了这些语义问题；未来 LLM/evidence confirm slice 也可能直接耦合到 `ProgrammaticContractRules`，把本应独立的语义审计 manifest 绑进程序化 C2 规则模块。
- **为什么有问题**: 现有 `contract_rules.py` 模块 docstring 明确说它维护“确定性 C2 审计”和 marker 映射；`run_programmatic_audit()` 只执行 deterministic checks。把非程序化 routing 放进 `ProgrammaticContractRules` 名下，会削弱该边界。
- **直接证据**:
  - `fund_agent/fund/audit/contract_rules.py` docstring: “只维护 manifest 到程序审计 marker 的显式映射，不做 LLM 判断或语义推断。”
  - plan S2 又规定 non-programmatic coverage kinds “do not emit C2 issues in P8-S1”。
- **影响**: C2 语义被文档/调用方误读；未来 LLM/evidence slice 与程序化审计过度耦合；review 难以判断 `checked_rules=("C2")` 到底证明了什么。
- **建议改法和验证点**:
  - 将 coverage routing 模型命名为 `ContractMustAnswerCoverageRule` / `ContractAuditCoverageManifest`，不要把全部 coverage 直接命名为 `ProgrammaticContractRules` 的一部分。
  - `ProgrammaticContractRules` 可以只保留 deterministic runtime rules，或从 coverage manifest 派生 `programmatic_marker` 子集。
  - README/design 明确：non-programmatic coverage 是 routing/observability，不是 C2 pass 证明。

## Open Questions

- `structured_data_availability` 是应该由程序审计消费，还是由 quality gate 消费？plan 当前把它 defer，但没有指定 tracking artifact。
- 如果所有 `must_answer` 首版都不使用 `programmatic_marker`，是否仍应实现 runtime branch，还是只先实现 coverage manifest validation？

## Residual Risks

- 即使 coverage manifest 完整，`covered_by_required_item` 也只证明报告存在某个输出 marker，不证明该 marker 下的回答质量。
- Future LLM/evidence audit 仍需要单独 prompt/schema 设计，不能从本 slice 自动得到语义审计能力。

## Conclusion

`fail`

当前 plan 方向合理，但还不是 code-generation-ready。必须先补齐逐条 `must_answer` coverage map，并澄清 non-programmatic coverage 与 `ProgrammaticContractRules` 的边界，再进入 implementation。
