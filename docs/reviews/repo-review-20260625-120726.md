### F1-未修复-高-提取真源契约仍以复合字段为核心，阻断“独立字段事实库”路线

- **入口/函数**: `FundDataExtractor.extract()` default parsed annual path and explicit FDD path, through `StructuredFundDataBundle`
- **文件(行号)**:
  - `fund_agent/fund/data_extractor.py:193`
  - `fund_agent/fund/data_extractor.py:227`
  - `fund_agent/fund/data_extractor.py:229`
  - `fund_agent/fund/data_extractor.py:233`
  - `fund_agent/fund/data_extractor.py:234`
  - `fund_agent/fund/extractors/profile.py:982`
  - `fund_agent/fund/extractors/performance.py:1052`
  - `fund_agent/fund/extractors/manager_ownership.py:795`
  - `fund_agent/fund/extractors/manager_ownership.py:878`
- **输入场景**: 年报中披露了任一独立 source fact，例如 `management_fee`、`custody_fee`、`nav_growth_rate`、`benchmark_return_rate`、`strategy_summary`、`market_outlook`、`manager_holding` 或 `employee_holding`。
- **实际分支**: extractor 先抽取多个子值，再把它们组装为一个 `ExtractedField[dict[str, object]]`，随后 `StructuredFundDataBundle` 把 `fee_schedule`、`nav_benchmark_performance`、`manager_alignment`、`manager_strategy_text` 作为顶层 dict 字段公开。
- **预期行为**: 按当前产品目标，数据提取层应产出独立 source facts；模板分析、章节写作和审计层再自由组合这些 facts。提取层不应把组合视为事实真源。
- **实际行为**:
  - `fee_schedule` 返回 `{"management_fee": ..., "custody_fee": ...}`。
  - `nav_benchmark_performance` 返回 `{"nav_growth_rate": ..., "benchmark_return_rate": ...}`。
  - `manager_strategy_text` 返回 `{"strategy_summary": ..., "market_outlook": ...}`。
  - `manager_alignment` 返回 `{"manager_holding": ..., "employee_holding": ..., "judgment": None}`。
- **直接证据**:
  - `StructuredFundDataBundle` 字段类型把上述四项定义为 `ExtractedField[dict[str, object]]`。
  - `_build_fee_schedule()` 在一个 `ExtractedField` 中同时写入 `management_fee` 和 `custody_fee`。
  - `_build_nav_benchmark_performance()` 在一个 `ExtractedField` 中同时写入 `nav_growth_rate` 和 `benchmark_return_rate`。
  - `_build_manager_strategy_text()` 通过 `_build_field_from_matches()` 合并 strategy/outlook。
  - `_build_manager_alignment()` 通过 `_build_field_from_matches()` 合并 manager/employee holding。
- **影响**: source truth、缺口、anchor、value match、章节 fact 都被迫围绕 composite dict 推理。后续若要证明单个数值或单段文字，系统需要在 audit/materializer 层反向拆 dict，而不是直接消费原子事实，导致 RR-09 strict V2 类问题成为结构性问题。
- **建议改法和验证点**:
  - 新增 Fund 层 atomic source fact contract，例如 `AtomicSourceFact(field_id, value, status, extraction_mode, anchors, provenance, gaps)`。
  - extractor 输出 `fee_schedule.management_fee`、`fee_schedule.custody_fee`、`nav_benchmark_performance.nav_growth_rate` 等独立 facts。
  - `StructuredFundDataBundle` 或新的 analysis view 只作为 derived/composite view，由 assembler 从 atomic facts 构造，不作为 extraction truth。
  - 验证点：默认 parsed annual path 和 FDD path 都能输出相同 atomic fact ids；章节写作可以按模板组合 atomic facts；Evidence Confirm 针对单 fact 做 provenance/value-match。
- **修复风险（低/中/高）**: 高。会触及 public data shape、chapter facts、tests、quality/evidence projection；需要分阶段兼容 adapter。
- **严重程度（低/中/高/严重）**: 高

### F2-未修复-高-默认 parsed annual processor 明确丢弃子字段 provenance，只保留顶层 source scope

- **入口/函数**: `ActiveFundAnnualProcessor` and shared parsed annual processor projection into `FundFieldFamilyResult`
- **文件(行号)**:
  - `fund_agent/fund/processors/active_annual.py:439`
  - `fund_agent/fund/processors/active_annual.py:456`
  - `fund_agent/fund/processors/active_annual.py:475`
  - `fund_agent/fund/processors/active_annual.py:477`
  - `fund_agent/fund/evidence_confirm_sources.py:1355`
  - `fund_agent/fund/evidence_confirm_sources.py:1361`
  - `tests/fund/test_data_extractor.py:1651`
  - `tests/fund/test_data_extractor.py:1673`
- **输入场景**: 默认 `FundDataExtractor.extract("110011", 2024)` 走 classified fund + parsed annual processor，不传 `disclosure_intermediate`。
- **实际分支**: `_field_scoped_anchors()` 对每个 field 的 anchors 调用 `_with_source_field_path(anchor, mapping.output_field_name)`，即写入 `source_field_path=<top_level>`。
- **预期行为**: 若提取层已经识别了独立 source fact，应把 provenance 保留到独立 fact 或至少子字段路径，例如 `fee_schedule.management_fee`。上游不应在默认产品路径中把子字段归属降级为顶层。
- **实际行为**:
  - row locator 变成 `source_field_path=fee_schedule; locator=management_fee`。
  - 测试明确断言 `source_field_path` 不包含 `.`，即不输出 `fee_schedule.management_fee`。
  - Evidence Confirm materializer 遇到 `len(path_parts) == 1` 时仍对整个 fact value 取 tokens，因此顶层 scope 无法缩到单个 subvalue。
- **直接证据**:
  - `_field_scoped_anchors()` 的 docstring 写明“顶层字段范围定位”，实现传入 `mapping.output_field_name`。
  - `_with_source_field_path()` 写入 `source_field_path={source_field_path}; locator=...`。
  - `test_default_parsed_annual_processor_projects_top_level_source_field_path()` 断言 `("fee_schedule", "fee_schedule")` 且所有 path 不含 `.`。
  - `_material_tokens_for_anchor_scope()` 只有 `source_field_path` 含子路径时才 `_resolve_fact_value_path()`；顶层路径直接返回整个 dict 的 material tokens。
- **影响**: 即便底层 matcher 已经知道 `management_fee` 或 `nav_growth_rate`，默认产品路径仍把 audit scope 降回 composite top-level。Evidence Confirm 只能在章节/表格摘录内匹配多个 tokens，不能稳定证明单一值来源。这是 strict V2 全 fail 的直接结构原因之一。
- **建议改法和验证点**:
  - parsed annual extractor 的 anchor 应保留 atomic `source_field_path`，或 atomic fact store 应直接保存独立 anchor，不再通过 top-level composite field 包装。
  - 删除或替换“默认不推断复合子字段”的测试口径；改为验证 `management_fee` 等 facts 的 source path 是 extraction truth，而 composite view 是 derived。
  - 验证点：R1-R4 的 materializer 可以对 `fee_schedule.management_fee` 等单 fact 生成匹配 token，不需要从 composite dict 反解。
- **修复风险（低/中/高）**: 中。需要避免把不可靠推断伪装成子字段 provenance；但对已由 matcher 直接命名的字段，保留原子 path 风险可控。
- **严重程度（低/中/高/严重）**: 高

### F3-未修复-中-FDD source-truth 路线已经有原子候选信息，但又在同一 processor 内重组回复合 public value

- **入口/函数**: `FundDisclosureDocumentProcessor` source-truth extraction path
- **文件(行号)**:
  - `fund_agent/fund/processors/fund_disclosure_processor.py:1064`
  - `fund_agent/fund/processors/fund_disclosure_processor.py:1082`
  - `fund_agent/fund/processors/fund_disclosure_processor.py:1311`

  - `fund_agent/fund/processors/fund_disclosure_processor.py:3626`
  - `fund_agent/fund/processors/fund_disclosure_processor.py:3697`
- **输入场景**: `FundDataExtractor.extract(..., disclosure_intermediate=...)` receives proof-positive FDD input.
- **实际分支**: FDD route selects candidates by atomic output paths such as `nav_benchmark_performance.nav_growth_rate` and `fee_schedule.management_fee`, then `_build_return_attribution_value()` and `_build_manager_profile_value()` recompose them into top-level family values before projection back to `StructuredFundDataBundle`.
- **预期行为**: FDD route should be the natural source of atomic source facts. It should preserve selected atomic output paths as first-class output records and let report views compose them later.
- **实际行为**:
  - `_build_return_attribution_nav_benchmark_value()` requires both nav and benchmark and returns one dict under `nav_benchmark_performance`.
  - `_build_return_attribution_fee_schedule_value()` returns one dict under `fee_schedule`.
  - `_return_attribution_emitted_output_paths()` has to reconstruct which subpaths entered the composite public value so anchors can be attached.
  - The processor has six parallel source-truth family variables and a 9,174-line file combining admission, candidate selectors, source-truth selectors, gap logic, anchor construction, and composite family assembly.
- **直接证据**:
  - `_build_field_family_results()` initializes one variable per field family and calls all source-truth extractors in sequence.
  - `_extract_return_attribution_source_truth()` selects atomic values, builds a composite value, derives gaps/status, then emits `FundFieldFamilyResult`.
  - `_return_attribution_emitted_output_paths()` exists because the public `value` shape no longer directly records the atomic records that were selected.
  - `wc -l` shows `fund_disclosure_processor.py` has 9,174 lines, while the path includes many `_select_*`, `_build_*value`, `_candidate`, `_source_truth`, `_gap`, and `_anchor` helpers in one module.
- **影响**: The project pays the complexity cost of atomic selection and composite output at the same time. Adding one new independent fact requires editing selector rules, composite builder, emitted output path derivation, gaps/status, facade projection, chapter facts, and tests. This is overcoupled and makes source-truth evolution expensive.
- **建议改法和验证点**:
  - Split FDD processing into: admission, atomic candidate/source-fact selection, family/view assembly, and candidate-only evidence.
  - Make selected atomic output paths the primary output; family result can become a derived view over atomic fact ids.
  - Validation should assert atomic fact records and separately assert derived view assembly, rather than using one composite `FundFieldFamilyResult.value` as both source truth and analysis shape.
- **修复风险（低/中/高）**: 中到高. File is large and behavior is heavily tested; refactor should start with adapter and narrow field family slices.
- **严重程度（低/中/高/严重）**: 中

### F4-未修复-中-章节事实和 Evidence Confirm 以 composite top-level field 为审计单位，导致写作层无法自然按独立字段组合

- **入口/函数**: `ChapterFactProvider.project()` -> `project_chapter_facts()` -> Evidence Confirm material token extraction
- **文件(行号)**:
  - `fund_agent/fund/chapter_facts.py:75`
  - `fund_agent/fund/chapter_facts.py:80`
  - `fund_agent/fund/chapter_facts.py:82`
  - `fund_agent/fund/chapter_facts.py:86`
  - `fund_agent/fund/chapter_facts.py:87`
  - `fund_agent/fund/chapter_facts.py:513`
  - `fund_agent/fund/chapter_facts.py:1422`
  - `fund_agent/fund/chapter_facts.py:1425`
  - `fund_agent/fund/evidence_confirm.py:1793`
  - `fund_agent/fund/evidence_confirm.py:1810`
  - `fund_agent/fund/evidence_confirm_value_diagnostics.py:235`
- **输入场景**: Report writing or Evidence Confirm consumes projected chapter facts for any chapter that uses fee, NAV/benchmark, manager alignment, or manager strategy fields.
- **实际分支**: `_SOURCE_FIELD_IDS` maps top-level bundle field names to `structured.<field>`. `_build_chapter_fact()` writes one fact with `field_path=StructuredFundDataBundle.<field>` and `value=<entire dict>`. Evidence Confirm recursively flattens the dict into material tokens.
- **预期行为**: Chapter writing should be able to request independent facts, for example `management_fee` in cost analysis, `nav_growth_rate` in return attribution, or `strategy_summary` in manager profile, without inheriting unrelated sibling values and audit requirements.
- **实际行为**:
  - `structured.fee_schedule` is one source field id, not separate `structured.fee_schedule.management_fee` and `structured.fee_schedule.custody_fee` facts.
  - `structured.nav_benchmark_performance` is one source field id, not separate NAV and benchmark facts.
  - Evidence Confirm value diagnostics and V2 material tokens use `_material_tokens(fact.value)`, which recursively flattens all scalar values inside the composite dict.
- **直接证据**:
  - `_SOURCE_FIELD_IDS` defines only top-level composite source ids for the four fields.
  - `project_chapter_facts()` takes `StructuredFundDataBundle`, not an atomic fact store.
  - `_build_chapter_fact()` records `field_path=f"StructuredFundDataBundle.{spec.field_name}"`.
  - `_flatten_material_values()` recursively walks dict values and returns all scalar tokens.
  - `summarize_value_match_diagnostics()` uses `_material_tokens(fact.value)`, so diagnostics also inherit composite token behavior.
- **影响**: Writer/auditor cannot cleanly choose or audit one independent fact. Any chapter using one sibling value carries the entire composite value into fact identity and token matching. This creates false coupling between analysis needs and extraction shape.
- **建议改法和验证点**:
  - `ChapterFactProvider` should project from atomic facts into chapter-required fact views.
  - A chapter fact should be allowed to reference one atomic fact id or an explicit derived analysis view id with dependency fact ids.
  - Evidence Confirm should evaluate atomic source facts directly; derived composite views should pass provenance only if all required child facts have sufficient provenance.
- **修复风险（低/中/高）**: 中. This affects LLM prompts, deterministic renderer inputs, Evidence Confirm summaries, and tests, but can be introduced additively with old view compatibility.
- **严重程度（低/中/高/严重）**: 中
