## Findings Re-review

### F1-已修复-高-atomic facts 的承载位置仍是“or equivalent”，Processor/Bundle 契约不可直接实施

- **Prior issue**: The plan left the atomic fact output surface ambiguous and allowed `source_facts ... or equivalent`.
- **Fix evidence**:
  - The fixed plan now states `FundProcessorResult.source_facts` is the only primary production surface for atomic source facts.
  - It explicitly allows only `StructuredFundDataBundle.source_facts` and `ChapterFactProjection.source_facts` as mirrors.
  - It explicitly prohibits atomic fact payloads inside `FundFieldFamilyResult`.
  - S1 requires tests proving Processor result exposes source facts only through `FundProcessorResult.source_facts`.
- **Re-review judgment**: Fixed. Implementation workers no longer need to choose among competing contract surfaces.
- **Residual risk**: The actual dataclass compatibility will still need tests in implementation because current relevant models are frozen/slots and widely constructed.

### F2-已修复-高-S2 只允许改 processor wrapper，无法从默认 parsed annual composite dict 安全产生独立子字段事实

- **Prior issue**: The plan asked default parsed annual processing to emit child facts while S2 only authorized processor-wrapper files, creating a risk of deriving child facts from composite dict keys.
- **Fix evidence**:
  - The fixed plan splits the work into `S2A - Default parsed annual extractor child fact split` and `S2B - Default parsed annual processor atomic emission`.
  - S2A adds the legacy extractor modules `profile.py`, `performance.py`, and `manager_ownership.py` to the allowed scope.
  - S2A requires child-level extractor output before Processor assembly.
  - S2B may emit Processor atomic facts only when S2A supplied child-level evidence.
  - Stop conditions forbid child paths inferred from the old composite dict.
- **Re-review judgment**: Fixed. The plan now matches the user's independent-field extraction direction and closes the no-fabrication failure mode.
- **Residual risk**: S2A may be the largest implementation slice because it touches legacy extractor result shapes; code review should watch for broad rewrites and keep compatibility outputs intact.

### F3-已修复-中-ChapterFactProvider 与 Evidence Confirm 缺少最小兼容 adapter 验收点

- **Prior issue**: The plan did not define a stable bridge between writing facts and Evidence Confirm facts.
- **Fix evidence**:
  - The fixed plan adds `ChapterFactEntry.source_fact_ids`, `ChapterFactEntry.derived_view_id`, `ChapterFactProjection.source_facts`, and `ChapterFactProjection.derived_views`.
  - S4 carries the projection store and derived views forward.
  - S5 resolves Evidence Confirm values and anchors through those bridge fields.
  - The plan prohibits Evidence Confirm from rediscovering atomic mappings from dict keys, `field_path`, top-level `source_field_id`, row locator text, or free-form locators.
- **Re-review judgment**: Fixed. Writer and auditor can now share a single fact id bridge.
- **Residual risk**: Existing legacy facts without bridge ids still need regression coverage to ensure old summary behavior remains unchanged.
