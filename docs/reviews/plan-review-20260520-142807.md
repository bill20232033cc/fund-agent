## Assumptions Tested

1. P6-S3 can add contract-aware audit without importing `renderer.py` from audit.
2. Required output item checks can be deterministic rather than semantic NLP.
3. `must_not_cover` can be checked as a small exact-marker subset.
4. Adding a C-family audit code is compatible with `docs/design.md` and existing tests.
5. Renderer changes can remain narrow if required output markers are missing.
6. The plan is specific enough for an implementation agent to execute without re-designing the rule layer.

## Findings

### PR-01-未修复-中-required_output_items 规则仍未到可直接实施粒度
- **位置**: `Deterministic Interpretation Rules`、`File-level Implementation Plan`、`Required test updates/additions`
- **问题类型**: 不可直接实施 / 测试缺口 / 范围漂移
- **当前写法**: plan 要求 `ContractRequiredItemRule` 覆盖每个 manifest `required_output_items`，并要求“如果 current renderer 缺 label，就添加 minimal explicit bullet labels / 数据不足 placeholders”，但没有列出 0-7 章每个 required item 对应的 deterministic marker，也没有列出哪些 renderer chapter 需要补 label。
- **反例/失败场景**: implementation agent 只能在实现时临场决定 marker 文本。若它选择复用自然语言片段，audit 可能对同义表达误报；若它选择把所有 required item label 原样写进 renderer，P6-S3 会从 audit slice 漂移成 8 章报告结构重写，改变用户可见报告。
- **为什么有问题**: P6-S3 的核心目标是 deterministic contract audit。required item 规则是本 slice 的核心公共行为，不是实现细节。没有完整 marker matrix，reviewer 无法判断规则是同源、最小还是过宽，也无法在实现后区分“必要 renderer label”与“报告改写”。
- **直接证据**:
  - plan 写明“manifest required output item has no rule”必须 fail closed。
  - plan 又写明“Current renderer output ... may not expose every required item label literally”，允许实施阶段补 renderer bullets。
  - 当前 `renderer.py` 仅返回 Markdown 和 `chapter_blocks`，没有 item-level marker 元数据；`audit_programmatic.py` 当前没有任何 contract rule 层。
- **影响**: 实施 Agent 重新设计规则；review 无法验收 deterministic coverage；renderer 用户可见输出可能被一次性大改。
- **建议改法和验证点**:
  - 在 plan artifact 内新增 `Required Item Marker Matrix`，按 chapter_id 列出每个 `required_output_items` 的 `markers_any`。
  - 同一节标注每个 marker 的来源：`already-rendered` 或 `renderer-label-needed`。
  - 规定 P6-S3 implementation 只允许添加 matrix 标成 `renderer-label-needed` 的标签或数据不足占位，禁止扩大章节文案。
  - 测试必须断言 `load_programmatic_contract_rules()` 覆盖 manifest 全量 required items，并至少抽查一个 `renderer-label-needed` 项在 renderer 输出中出现。
- **修复风险（低/中/高）**: 中
- **严重程度（低/中/高/严重）**: 中

### PR-02-未修复-中-C2 阻断语义与设计真源存在状态冲突
- **位置**: `Audit Rule Code Choice`、`ProgrammaticAuditResult.checked_rules`、`docs/design.md` 第 5.2 节
- **问题类型**: 设计真源冲突 / open question 未收敛
- **当前写法**: plan 建议新增 `C2`，并把 deterministic C2 failures 设为 `blocker`，加入 `checked_rules`。但 `docs/design.md` 当前把 C2 定义为“章节越界（展开禁止话题）”，阻断级别为“低优先级”，MVP 阶段为 v2。
- **反例/失败场景**: implementation 合入后，代码和 README 会声明 C2 已进入程序审计并阻断，但 design 仍显示 C2 是 v2 低优先级。后续 controller/reviewer 按 design 审查时会判定代码越界，或者反过来按代码改 README 造成三套口径并存。
- **为什么有问题**: 项目硬约束要求 `docs/design.md` 是设计真源，且 README、设计、实现不能长期双重口径。P6-S3 若要把 C2 的 deterministic subset 前移到 MVP，必须同步规定设计文档如何改写：不是把完整 C2 v2 提前实现，而是新增“C2-programmatic subset”或更新规则表状态。
- **直接证据**:
  - `docs/design.md` 第 5.2 节：`C2 | 章节越界（展开禁止话题） | 低优先级 | ⬜ v2`。
  - plan：`Add C2 to checked rules` 且 `P6-S3 should emit severity="blocker" for deterministic C2 failures`。
- **影响**: 代码、README、design 不一致；review 可能误判 P6-S3 范围；后续质量 gate 和修复策略无法判断 C2 是 patch 还是 blocker。
- **建议改法和验证点**:
  - plan 必须明确二选一：
    - 使用现有 MVP rule code，如 P1/P3/CHECK contract structure 子项，不新增 C2；或
    - 接受新增 `C2` 作为 deterministic `CHAPTER_CONTRACT` 子集，同时把 `docs/design.md` 第 5.2 节更新为“C2 deterministic subset implemented in MVP; semantic C2 remains v2/patch”，并说明当前 result model 无 warning-only pass，因此程序子集暂为 blocker。
  - 如果选择 C2，implementation acceptance commands 必须包含 design doc diff review，并更新 `fund_agent/fund/README.md` 与 `tests/README.md` 同口径。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

### PR-03-未修复-中-计划把共享章节块抽取和规则审计绑在同一实现 slice，缺少失败隔离
- **位置**: `Architecture And Dependency Direction`、`File-level Implementation Plan`
- **问题类型**: 切片过粗 / 过度耦合 / 回滚风险
- **当前写法**: plan 同时要求移动 `RenderedChapterBlock` / splitter 到 `template/chapter_blocks.py`、修改 renderer imports、添加 rule mapping、修改 audit、可能修改 renderer 文案、更新多组集成测试。
- **反例/失败场景**: 如果 rule mapping 或 renderer label 改动导致大面积测试失败，implementation 需要同时判断是抽取 shared module 造成 import/export 问题，还是新 audit 规则造成行为变化。回滚计划虽然说可保留 `chapter_blocks.py`，但没有把“纯移动、行为不变”的验收放在规则审计之前。
- **为什么有问题**: 当前 P6-S2 已经通过 `split_rendered_chapter_blocks(...)` 测试。为了避免 audit/renderer 循环，抽 shared module 是必要的依赖方向 cleanup，但它本身应是 behavior-preserving。把它和 C2/P3 行为变更一次交给 implementation，会增加 review 面积并降低故障定位能力。
- **直接证据**:
  - `renderer.py` 当前定义 `RenderedChapterBlock` 和 splitter，同时导入 `ProgrammaticAuditInput`。
  - plan 已识别直接 `audit_programmatic.py -> renderer.py` 会循环，因此要求新建 `template/chapter_blocks.py`。
  - plan 同时允许 renderer 为 required items 添加 bullets。
- **影响**: 实施返工成本增加；reviewer 难以证明“移动无行为变化”；如果测试失败，容易出现为让 audit 通过而顺手改 renderer 的范围漂移。
- **建议改法和验证点**:
  - 将 implementation sequencing 明确拆成两个 commit-like steps，但仍可在 P6-S3 一个 slice 内完成：
    1. `chapter_blocks.py` extraction only，运行 `tests/fund/template/test_renderer.py -q`，确认无行为变化。
    2. 添加 `ProgrammaticAuditInput.chapter_blocks`、contract rules、C2/P3 audit 和必要 renderer labels。
  - plan 要求 implementation summary 分别报告这两个 step 的 diff 与测试结果。
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: 中

## Open Questions

1. `C2` 是否作为 deterministic subset 前移到 MVP？需要 controller 在 plan rereview 前明确裁决，并同步设计真源。
2. required item marker matrix 是否允许包含全部原始 item label？如果允许，必须明确这是“机器标签”，不是报告自然叙事重写。
3. `must_not_cover` forbidden markers 是否只覆盖少数高风险词，还是每个 `must_not_cover` 都需要一条 rule？当前更安全的选择是只覆盖可确定的高风险词，并在 docs 标注 semantic subset deferred。

## Residual Risks And Tracking

- 即使 P6-S3 完成，required item marker 也只能证明“报告暴露了对应栏目或占位”，不能证明回答质量。该风险应继续留在 LLM audit / Evidence Confirm v2。
- Per-chapter evidence line 只能证明格式存在，不能证明证据支持正文断言。该风险应继续留在 E1/E2/E3。
- Renderer 标签增加可能改变 golden report 文风。若 P6-S3 实现中出现大段文案调整，应单独触发 renderer output review。

## Conclusion

`fail` for immediate implementation.

计划方向基本正确，尤其是把 shared chapter block 抽到 `template/chapter_blocks.py` 来避免 audit/renderer 循环。但在实施前必须补齐 required item marker matrix，并裁决 C2 与 design truth 的状态冲突；否则 implementation agent 会在核心规则上重新设计，且代码、README、design 会形成双重口径。
