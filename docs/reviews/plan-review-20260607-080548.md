## Findings

### F1-未修复-中-计划没有收集 LLM blocking issues，却允许裁决 Ch3/Ch5 marker residual covered

- **位置**: `Direct Evidence To Collect` / `Completion Criteria` / `Validation Commands`
- **问题类型**: 测试缺口 / review 不可验收 / 范围漂移
- **当前写法**: Plan E1 只要求解析 `programmatic_issues`，验证脚本也只累计 `attempt.programmatic_issues` 中的 C2 marker issue；但 Completion Criteria 又要求如果 Ch3/Ch5 有 additional non-marker blocking evidence 就 separate route。
- **反例/失败场景**: retained artifact 的 Ch3 auditor feedback 同时包含 required-output marker C2 和 LLM blocking feedback。当前脚本不会读取 `llm_issues`、`audit_result.llm_issues` 或等价字段，因此 evidence artifact 可能只看到 marker C2，并把 Ch3 residual 误判为“被 Slice 1A locally covered”。
- **为什么有问题**: 本 slice 目标是 routing evidence，不是只证明 marker C2 存在。若不枚举 non-programmatic blocker，controller 无法安全区分“marker-protocol root covered”与“章节 residual covered”。
- **直接证据**: retained file `reports/llm-runs/006597-2024-20260606T231450Z-host_run_435c8c7c2b8d4e2/chapters/chapter-03-attempt-00-auditor-feedback.md` includes `llm:0:blocking` through `llm:4:blocking` in issue ids alongside programmatic C2 marker issues.
- **影响**: controller 可能错误关闭 Ch3 residual，后续 live rerun 或 slice selection 仍会被 LLM semantic blocker 卡住，导致 gate 状态不一致。
- **建议改法和验证点**: Plan 必须要求解析并记录每章 final/terminal attempt 的 programmatic issues 和 LLM issues，输出 `marker_issue_count`、`llm_blocking_issue_count`、`parse_failure` or equivalent。Completion Criteria 应拆成两层：marker-protocol residual locally covered；non-marker blockers remain separately routed and must not be closed。
