# Plan Review: FundDisclosureDocument core_risk.v1 Deferred Risk Roles Source-truth Direct Extraction

## Reviewed Target

`docs/reviews/funddisclosuredocument-core-risk-deferred-roles-source-truth-extraction-plan-20260620.md`

## Scope

Adversarial review of the implementation plan for converting four deferred `core_risk.v1` roles (`liquidation_or_scale_risk`, `tracking_error_or_deviation_risk`, `turnover_or_style_drift_risk`, `concentration_risk`) from candidate-only/deferred gaps into proof-positive public `core_risk.v1` subvalues.

## Assumptions Tested

1. Four deferred roles should become required `core_risk.v1` public subvalues.
2. `deferred_role` gap code is invalid and should be removed.
3. Status semantics change from "risk text only => accepted" to "risk text only => partial" is safe.
4. Source-truth extraction boundary is precise enough for implementation.
5. `_CoreRiskRoleValueCandidate` can be cleanly merged with existing `_RiskCharacteristicValueCandidate`.
6. `core_risk_role_disclosure.v1` subvalue schema is well-defined.
7. Facade no-projection guard is sufficient.
8. Test plan covers all affected test assertions.

## Findings

### 001-未修复-高-source-truth 角色提取边界规格不足
- **位置**: Public Value Shape, Anchor / Gap Semantics - Allowed direct sources
- **问题类型**: 不可直接实施
- **当前写法**: Plan 声明 "paragraph text with role-specific strong token or generic token plus source-level guard context" 和 "table cell text with role-specific strong token or generic token plus table/row/column/heading guard context" 以及 "table row or cell value where the selected disclosure text is substantive, not only a generic heading"。
- **反例/失败场景**: Implementation agent 面对一个包含 "基金资产净值低于五千万元" token 的段落时，无法确定：(a) 该段落是否需要额外的 guard context 才算 source truth；(b) guard context 的具体组成是什么（section heading？table caption？row label？）；(c) 什么程度的文本算 "substantive" 而非 "heading-only"。例如，一个表格的 row_label_path 包含 "基金合同终止事由" 但 cell_text 为空或只有 "——"，这是 heading-only 还是 missing？
- **为什么有问题**: 当前 `_CORE_RISK_MATCH_GROUPS` (processor.py:677-746) 定义了 strong_tokens、generic_tokens 和 guard_tokens，但这些是为 candidate evidence 设计的 locator tokens。Plan 要求 source-truth helpers "reuse the same tokens and scan order" 但 "build public values from direct paragraph/cell sources"。两者之间的判定逻辑差异未被规格化。现有 `_select_risk_characteristic_value()` 的实现（processor.py:3759-4110）包含 paragraph candidate collection、table candidate collection、label matching、ambiguity resolution 等复杂逻辑，但 plan 对四个新 role 的对应逻辑只列出了函数签名，没有描述核心判定规则。
- **直接证据**: Plan "Anchor / Gap Semantics" 章节的 "Allowed direct sources" 和 "Disallowed direct sources" 列表；processor.py:677-746 `_CORE_RISK_MATCH_GROUPS` 定义；processor.py:3759-4110 `_select_risk_characteristic_value()` 实现复杂度。
- **影响**: Implementation agent 必须重新设计提取判定逻辑，可能导致：(a) 过度宽松，把 heading-only locator 提升为 source truth；(b) 过度严格，把有效 disclosure 误判为 missing；(c) 不同 role 之间的判定标准不一致。
- **建议改法和验证点**: 为每个 role 定义具体的 source-truth 提取规则：(1) 明确 paragraph source 的 guard context 组成（例如：section heading path 必须包含 role-specific heading token，或 paragraph heading_path 必须匹配 guard_tokens）；(2) 明确 cell source 的 guard context 组成（例如：row_label_path 或 column_header_path 必须包含 role-specific token）；(3) 定义 "substantive" 的具体标准（例如：cell_text 长度 > N 且不全是标点/破折号/空白）；(4) 为每个 role 给出一个 happy-path 提取示例和一个 heading-only 拒绝示例。
- **修复风险（低/中/高）**: 中
- **严重程度（高）**: 高 - 直接影响 implementation agent 能否安全编码

### 002-未修复-中-status 语义变更影响范围未完整枚举
- **位置**: Anchor / Gap Semantics - Status semantics
- **问题类型**: 测试缺口
- **当前写法**: Plan 声明 "This intentionally changes the current 'risk text only => accepted' behavior to 'risk text only => partial' once the four roles are promoted from deferred to required"，并在 Expected tests 中列出 "Replace `test_core_risk_source_truth_extracts_risk_characteristic_text_only`"。
- **反例/失败场景**: 至少有两处未被 plan 枚举的测试也断言 `core_risk.status == "accepted"` 且 value 仅含 `schema_version` + `risk_characteristic_text`：(1) `test_product_essence_source_truth_extracts_minimal_shape` (test line 972-975) 断言 `core_risk.status == "accepted"` 和 `set(core_risk.value) == {"schema_version", "risk_characteristic_text"}`；(2) `test_investor_experience_source_truth_does_not_pollute_stage_or_core_risk` (test line 6200-6210) 断言 `core_risk.value` 仅含 risk_characteristic_text。这些测试在 status 语义变更后会失败，但 plan 只提到了一个测试的替换。
- **为什么有问题**: Implementation agent 可能只更新 plan 明确提到的测试，导致其他测试在运行时失败，需要额外的调试和修复周期。
- **直接证据**: test line 972-975 `assert core_risk.status == "accepted"` + `assert set(core_risk.value) == {"schema_version", "risk_characteristic_text"}`；test line 6200-6210 同样断言 core_risk accepted with risk text only。
- **影响**: Implementation agent 测试通过率下降，需要额外排查和修复。
- **建议改法和验证点**: 在 plan 的 Expected tests 章节补充完整的影响测试列表，至少包括上述两个测试。建议 grep 所有 `core_risk` + `accepted` 断言以确保完整覆盖。
- **修复风险（低/中/高）**: 低
- **严重程度（中）**: 中 - 可发现但增加 implementation 成本

### 003-未修复-中-角色值候选类型合并缺乏类型规格
- **位置**: Implementation Plan - Slice 1, "Add a private value-candidate dataclass"
- **问题类型**: 契约缺失
- **当前写法**: Plan 提议新增 `_CoreRiskRoleValueCandidate` dataclass（含 `output_path`, `role`, `value`, `anchor`, `source_field_path`），并让 `_select_core_risk_values()` 返回合并后的 dict。当前返回类型为 `tuple[dict[str, _RiskCharacteristicValueCandidate], set[str]]`。
- **反例/失败场景**: 合并后 dict 的值类型变为 `_RiskCharacteristicValueCandidate | _CoreRiskRoleValueCandidate`。`_build_core_risk_value()` 和 `_core_risk_emitted_output_paths()` 需要对两种类型做不同的处理（risk_characteristic_text 调用 `_build_risk_characteristic_text_value()`，roles 调用 `_build_core_risk_role_disclosure_value()`）。如果类型注解不明确，implementation agent 可能写出不安全的类型窄化代码。
- **为什么有问题**: Python 的 type checker（如 mypy/pyright）在 union type 上的 narrow 行为依赖于显式的 `isinstance` 或 `type: ignore`。当前代码已有一处 `# type: ignore[arg-type]`（processor.py:4779 使用无效 gap code），不应再引入更多类型不确定性。
- **直接证据**: processor.py:4665-4682 `_select_core_risk_values()` 返回类型；processor.py:385-392 `_RiskCharacteristicValueCandidate` 定义；plan 提议的 `_CoreRiskRoleValueCandidate` 字段与 `_RiskCharacteristicValueCandidate` 不同（多了 `role` 字段，少了 `output_path` 语义差异）。
- **影响**: Type checker 报告、implementation agent 类型处理错误。
- **建议改法和验证点**: 明确定义合并后的返回类型注解，例如 `tuple[dict[str, _RiskCharacteristicValueCandidate | _CoreRiskRoleValueCandidate], set[str]]`。在 `_build_core_risk_value()` 中使用明确的类型窄化（如 `isinstance` 检查或 output_path-based dispatch）。
- **修复风险（低/中/高）**: 低
- **严重程度（中）**: 中

### 004-未修复-中-core_risk_role_disclosure.v1 schema 定义位置未指定
- **位置**: Public Value Shape
- **问题类型**: 契约缺失
- **当前写法**: Plan 在 "Public Value Shape" 章节给出了 inline Python dict 示例，标注 schema_version 为 `"core_risk_role_disclosure.v1"`，但未指定该 schema 在代码中的定义位置、验证方式或类型定义。
- **反例/失败场景**: Implementation agent 可能：(a) 在 processor 内部用 dict literal 构造值而不定义 schema class；(b) 在 contracts.py 中定义 Pydantic/dataclass schema；(c) 仅在测试中通过 exact shape 断言隐式定义。不同选择影响后续维护和类型安全。
- **为什么有问题**: 当前 `risk_characteristic_text.v1` 的 shape 由 `_build_risk_characteristic_text_value()` 函数（processor.py 中）构造，没有独立的 schema class。但 plan 引入了新的 `core_risk_role_disclosure.v1` schema version，暗示这是一个可被消费方识别的 schema。如果没有明确的定义位置，后续 consumer（如 facade、renderer）无法可靠地依赖该 schema。
- **直接证据**: Plan "Public Value Shape" 章节的 inline dict 示例；当前代码中无 `core_risk_role_disclosure` 相关定义。
- **影响**: Implementation agent 设计选择不确定，可能产生不一致的 schema 定义。
- **建议改法和验证点**: 指定 `core_risk_role_disclosure.v1` 的定义位置（如 contracts.py 中的 dataclass，或 processor 内部的 builder 函数），并说明是否需要在 `FundFieldFamilyResult.value` 的 consumer 侧做 schema validation。
- **修复风险（低/中/高）**: 低
- **严重程度（中）**: 中

### 005-未修复-低-_core_risk_emitted_output_paths 更新逻辑未规格
- **位置**: Implementation Plan - Slice 1, "Update `_core_risk_emitted_output_paths()`"
- **问题类型**: 不可直接实施
- **当前写法**: Plan 说 "Update `_core_risk_emitted_output_paths()` to include emitted role output paths" 但未指定具体逻辑。
- **反例/失败场景**: 当前函数（processor.py:4713-4732）只检查 `risk_characteristic_text` key。更新后需要检查五个 key（risk_characteristic_text + 四个 roles），但 role key 的 output_path 命名约定未定义。如果 role 的 output_path 就是 role name（如 `"liquidation_or_scale_risk"`），则逻辑为检查 value dict 中是否包含对应 key；如果 output_path 是更复杂的路径（如 `"core_risk.liquidation_or_scale_risk.risk_disclosure_text"`），则需要额外的 mapping。
- **为什么有问题**: 该函数用于构造 `FundFieldFamilyResult.anchors`，anchor 的 `row_locator` 字段引用 output_path。如果 output_path 命名不一致，anchor 的可追溯性受损。
- **直接证据**: processor.py:4713-4732 `_core_risk_emitted_output_paths()` 当前实现；plan 未指定 role output_path 命名约定。
- **影响**: Implementation agent 需要自行设计 output_path 命名，可能与 plan 意图不一致。
- **建议改法和验证点**: 明确 role output_path 的命名约定（建议直接使用 role name，如 `"liquidation_or_scale_risk"`），并给出更新后的 `_core_risk_emitted_output_paths()` 伪代码。
- **修复风险（低/中/高）**: 低
- **严重程度（低）**: 低

## Open Questions

1. **`_CORE_RISK_MATCH_GROUPS` 复用范围**: Plan 说 "Reuse `_CORE_RISK_MATCH_GROUPS` for token/guard definitions but skip `risk_characteristic`"。但 `_CORE_RISK_MATCH_GROUPS` 的第一个 entry 就是 `risk_characteristic`（processor.py:681-685），跳过后剩余四个 entry 正好对应四个 deferred roles。这是否意味着 source-truth helpers 直接复用这四个 entry 的 strong_tokens/generic_tokens/guard_tokens？如果是，guard_tokens 是否需要调整以满足 source-truth 比 candidate 更严格的 admission 标准？

2. **Ambiguity resolution**: Plan 提到 "If a role has conflicting stable candidates, omit that role and emit `ambiguous_table_or_locator`"。当前 `_select_risk_characteristic_value()` 的 ambiguity 判定逻辑（processor.py:4035-4042）基于 output_path 去重后仍有多个候选。对于四个新 roles，ambiguity 判定是否使用相同逻辑？还是需要考虑同一 role 的 paragraph 和 cell 候选之间的 cross-source ambiguity？

3. **PR 34 base branch**: Plan 的 Slice 3 提到 docs sync 需要 downstream gate 授权。当前 PR 34 的 base 是 `funddisclosure-current-stage-source-truth`。实现此 plan 是否需要先 rebase 到最新 main，或在当前 base 上继续？

## Conclusion

`pass-with-risks`

Plan 的整体方向正确：四个 deferred roles 应成为 `core_risk.v1` 的 required public subvalues，不引入 `StructuredFundDataBundle.core_risk`，不扩展 `EvidenceSourceKind`，保持 facade no-projection。Status 语义变更（risk text only => partial）是合理的契约演进。Stop conditions 和 non-goals 定义清晰。

主要风险在于 source-truth 角色提取的实现规格不足（Finding 001）：plan 定义了高层的 allowed/disallowed sources 规则，但缺少具体的 guard context 组成、substantive 判定标准和 role-specific 提取示例。这会迫使 implementation agent 重新设计核心提取逻辑，增加跑偏风险。建议在 plan 中补充每个 role 的具体提取规则和示例。

次要风险包括 status 语义变更影响范围未完整枚举（Finding 002）、类型合并规格缺失（Finding 003）和 schema 定义位置未指定（Finding 004）。这些风险较低，可在 implementation review 阶段验证。

Blocker count: 0 (no true blockers; Finding 001 is high severity but has a clear fix path)
