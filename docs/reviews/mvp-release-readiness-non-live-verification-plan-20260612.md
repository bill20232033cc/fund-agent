## 7. Deterministic Non-live Verification Matrix

The later evidence gate should execute the following commands in order. The planning gate does not execute them.

| ID | Command | Evidence type | Success criteria | Failure classification |
|---|---|---|---|---|
| V1 | `git status --branch --short` | metadata | Branch and ahead/dirty state captured; no readiness claim derived from cleanliness alone | material residual if unrelated residue remains; blocker only if tracked source/test/runtime drift appears outside gate |
| V2 | `git status --short` | metadata | Current tracked/untracked state captured; accepted residue routes preserved | material residual if new unclassified residue family appears; non-blocking if previously accepted residue remains |
| V3 | `git diff --name-only` | metadata | Tracked modified files are limited to current gate artifact set before acceptance | blocker if source/tests/runtime/config/README/control docs changed before plan acceptance without authorization |
| V4 | `git diff --check` | whitespace/static | Exit 0 | blocker if fails in authorized files |
| V5 | `uv run ruff check fund_agent tests` | deterministic lint | Exit 0 | blocker if source/test lint fails; material residual if only unrelated pre-existing path outside touched scope is identified and documented |
| V6 | `uv run pytest tests/fund/test_source_provenance.py tests/fund/test_data_extractor.py tests/fund/test_extraction_snapshot.py tests/fund/test_extraction_score.py -q` | focused deterministic tests | Exit 0 | blocker if fails |
| V7 | `uv run pytest tests/fund/test_annual_evidence.py tests/fund/test_annual_period_report.py tests/services/test_multi_year_annual_analysis.py tests/ui/test_cli_annual_period.py -q` | focused deterministic product tests | Exit 0 | blocker if fails |
| V8 | `uv run pytest tests/services/test_llm_execution.py tests/host tests/agent -q` | deterministic boundary tests | Exit 0 | blocker if fails due current Service/Host/Agent boundary; material residual if unavailable path is unrelated and owner is documented |
| V9 | `uv run pytest -q` | broad local test pass | Exit 0 | blocker if fails in current product/readiness-critical scope; material residual if unrelated legacy failure has owner and does not affect this gate |
| V10 | `uv run pytest --cov=fund_agent --cov-report=term-missing --cov-fail-under=50 -q` | project coverage gate | Exit 0 | blocker if fails; material residual if command is too slow/interrupted and narrower commands passed, with explicit owner |

For V5 and V9 classification, a failure is unrelated only if the evidence artifact records the failing file/test path, the failure existed outside files modified since accepted checkpoint `c0b94bc` or is demonstrably outside the current deterministic product/readiness-critical surface, and a residual owner/next gate is assigned. Otherwise classify it as a blocker.

Passing V10 at the current project floor of 50% is a floor sanity check only. It does not prove coverage sufficiency, does not satisfy the AGENTS.md single-file 80% review target for newly or substantially modified modules, and must not be used as a standalone readiness claim.

If V10 exceeds 10 minutes or is manually interrupted after V6-V9 have passed, classify V10 as a material residual with explicit owner and command timing evidence. Partial coverage output is not accepted as a pass.

If a command attempts network/provider/live acquisition, the evidence gate must stop, record `blocking_question`, and not retry with live access.

If `uv run` dependency resolution fails because dependencies are missing locally, classify as `blocking_question` unless the user separately authorizes dependency/network repair.

`blocking_question` means the evidence gate cannot proceed without external resolution, such as dependency installation, user authorization, or a command attempting prohibited live behavior. It does not prove readiness or non-readiness by itself; the controller must route it to a named next gate or user decision.

No separate dependency pre-check is required. Missing-dependency failures from V5-V10 are sufficient evidence for the existing `blocking_question` classification unless a future controller judgment authorizes a dry-run dependency check.
