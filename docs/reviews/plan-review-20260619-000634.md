## Assumptions Tested

### A1: Default route preservation

**Claim**: `extract(fund_code, report_year, force_refresh=False)` with `disclosure_intermediate=None` remains byte-for-behavior equivalent.

**Test**: Traced the current code path. The new parameter is keyword-only with default `None`. The existing code path (no disclosure_intermediate passed) is gated by `if disclosure_intermediate is not None:` and would not be entered. All existing behavior — report loading, identity validation, NAV, fund type classification, and the active/non-active branch — is untouched.

**Result**: PASS. The default path is structurally preserved.

### A2: Explicit opt-in cannot become default

**Claim**: `fund_disclosure_document.v1` never preempts `parsed_annual_report.v1` by registry priority, default registry order, repository behavior, or automatic discovery.

**Test**: The registry priority is ActiveFundAnnualProcessor=100 > FundDisclosureDocumentProcessor=50. The plan's dispatch key for the disclosure route is constructed with `intermediate_kind="fund_disclosure_document.v1"`, so `ActiveFundAnnualProcessor.supports()` would return False (it checks for `parsed_annual_report.v1`). The disclosure route is only entered when `disclosure_intermediate is not None`. No automatic discovery mechanism exists.

**Result**: PASS. No path exists for `fund_disclosure_document.v1` to become the default.

### A3: Candidate boundary produces blocked, not usable output

**Claim**: Concrete `FundDisclosureDocument` with `candidate_boundary` returns `contract_status="blocked"` from the processor, and the facade raises RuntimeError.

**Test**: Traced the full path: `admit_disclosure_intermediate()` → candidate_boundary is not None → returns `admitted=True, contract_status="blocked"`. `FundDisclosureDocumentProcessor.extract()` → admission.contract_status="blocked" → `contract_status="blocked"`. Facade checks `result.contract_status in ("unsupported", "blocked")` → raises RuntimeError.

**Result**: PASS. Candidate input correctly fail-closed.

## Findings

### Finding 1 (minor): Bond risk evidence extraction not explicitly addressed for disclosure route

The plan says to reuse `_active_processor_result_to_bundle()` for bundle projection (Contract Decision 2, step 11). This function requires `bond_risk_evidence` and `drawdown_metric` as parameters. In the current code, these are extracted from `ParsedAnnualReport` inside `_extract_active_fund_via_processor()` (data_extractor.py:386-398).

For the disclosure route, the plan loads `ParsedAnnualReport` (step 1) and classifies fund type from it (step 4), so the parsed report is available. The implementation worker would need to also call `extract_bond_risk_evidence()` and `_load_drawdown_metric_for_bond_fund()` for the disclosure path. The plan doesn't explicitly state this.

**Severity**: Low. The function signature makes the requirement discoverable. The implementation worker cannot call `_active_processor_result_to_bundle()` without providing these parameters.

**Recommendation**: Implementation worker should add explicit bond_risk_evidence extraction step in the disclosure helper, using the loaded ParsedAnnualReport.

### Finding 2 (minor): Test location ambiguity for AST import guard

The proposed write set lists two possible locations for the AST import guard test: `tests/fund/documents/test_docling_no_consumption_guards.py` or `tests/fund/test_data_extractor.py`. The "or" creates ambiguity.

**Severity**: Low. Either location is acceptable; the test verifies the same invariant.

**Recommendation**: Pick `tests/fund/test_data_extractor.py` for colocation with other S5 facade tests, or make the choice explicit in the plan.

### Finding 3 (minor): `_active_processor_result_to_bundle()` naming may cause confusion

The function name suggests it's only for "active" fund results, but it's reused for the disclosure route (which is also active_fund). The name refers to fund_type, not intermediate_kind, so it's semantically correct. However, future readers may wonder why a disclosure route uses an "active" function.

**Severity**: Cosmetic. Does not affect correctness.

**Recommendation**: No action required for S5. Consider renaming in a future cleanup gate if it causes confusion.

### Finding 4 (informational): `index_profile` field provenance in disclosure bundle

The plan reuses `_active_processor_result_to_bundle()` which sets `index_profile` from `profile_result.index_profile` (from the ParsedAnnualReport), not from processor field families. This means even with all field families missing, `index_profile` will have a value from the parsed report. This is consistent with how the default route works and is noted in the code as "S2 residual" (data_extractor.py:575).

**Severity**: Informational. No action required.

## Open Questions

1. Should `bond_risk_evidence` extraction for the disclosure route use `report.key.year` or the `report_year` parameter? They are identity-validated to be equal (step 2), so either works. The implementation worker should use the parameter for consistency with the dispatch key.

2. The plan says NAV is loaded after identity validation (steps 2-3 say "before NAV"). Should the disclosure route load NAV at all when the processor returns all field families as missing? The NAV data is currently used in `StructuredFundDataBundle` but its consumer-facing utility with all-missing fields is nil. However, preserving the NAV load maintains interface consistency. The plan correctly preserves current NAV semantics.
