## Findings

### 001-unfixed-[medium]-redirect/render split rule missing domain parameter blocks domain-based test

- **位置**: §7.3 Split Rules, §7.4 Mapping Function Contract
- **问题类型**: 测试缺口 / 不可直接实施
- **当前写法**: §7.3 specifies a sub-condition "HTTP 3xx redirect to unrecognized non-EID domain" mapping to `identity_mismatch`. §7.4 defines `map_fund_disclosure_failure_to_source_category()` with parameters `(failure_code, *, http_status, error_type, retry_count, body_empty, body_is_html)` but no `redirect_domain` or `render_domain` parameter. §10.2 test `TestFailureMappingRedirectUnavailable` lists "non-EID domain→identity_mismatch" as a required test case.
- **反例/失败场景**: Implementation agent cannot test the domain-based redirect split because the function contract has no domain input. The agent must either (a) omit the domain-based test, violating the controller judgment's binding requirement for testable split rules, or (b) invent a domain parameter not in the plan, violating code-generation-ready contract.
- **为什么有问题**: Controller judgment §5 explicitly requires "testable split rules between temporary unavailable and stable render contract drift." The domain-based redirect is one of the specified split conditions. Without a domain parameter in the function contract, this split rule is specified but untestable.
- **直接证据**: §7.4 function signature lists only `http_status`, `error_type`, `retry_count`, `body_empty`, `body_is_html`; §7.3 sub-condition 2 requires domain comparison; §10.2 test expects "non-EID domain→identity_mismatch."
- **影响**: Implementation agent invents extra parameter or silently skips domain test; controller judgment requirement is violated.
- **建议改法和验证点**: Add `redirect_domain: str | None = None` (or equivalent) parameter to the mapping function contract. Update §10.2 test to pass explicit domain values. Verification: test list in §10.2 covers all six §7.3 sub-conditions with explicit parameter values.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: medium

### 002-unfixed-[low]-LocatorStability type already exists in candidates/models.py

- **位置**: §6.8 Locator Stability Classification
- **问题类型**: 最佳实践偏离
- **当前写法**: Plan defines `LocatorStability = Literal["strong", "usable", "weak", "blocked"]` as a new type in the new `fund_disclosure_document.py` module.
- **反例/失败场景**: `LocatorStability` already exists at `fund_agent/fund/documents/candidates/models.py` line 115 with identical values. Creating a duplicate type in the new module risks type identity divergence (two types with same values but different identity, breaking isinstance/Protocol checks) and forces every consumer to disambiguate imports.
- **为什么有问题**: The plan claims "reference only" for `candidates/models.py` (§3.3) but does not acknowledge that `LocatorStability` should be imported from the existing module rather than recreated. The existing `candidates/failures.py` already demonstrates the pattern of importing from sibling modules.
- **直接证据**: Code verification confirms `LocatorStability = Literal["strong", "usable", "weak", "blocked"]` at `candidates/models.py:115` and `CandidateLocatorStability` at `candidates/representation_models.py:17`. Both have identical values.
- **影响**: Implementation agent creates duplicate type; future maintenance burden; potential type identity issues.
- **建议改法和验证点**: Add explicit instruction: "Import `LocatorStability` from `fund_agent.fund.documents.candidates.models` (existing). Do not redefine." Verification: new module's import list includes `from fund_agent.fund.documents.candidates.models import LocatorStability`.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: low

### 003-unfixed-[low]-failure mapping module import pattern not specified

- **位置**: §5.1 Allowed Files, §7 Failure Mapping Rules, §8 Non-Consumption Guards
- **问题类型**: 契约缺失
- **当前写法**: §8 states "Candidate internals may reference `AnnualReportSourceFailureCategory` from `fund_agent/fund/documents/models.py` (for failure mapping only)." §5.1 lists `fund_disclosure_failure_mapping.py` as allowed. But the plan does not specify what the new failure mapping module should import from `documents/models.py`, nor does it reference the existing `candidates/failures.py` as an import pattern model.
- **反例/失败场景**: Implementation agent may not know that `AnnualReportSourceFailureCategory` is the import target, may import wrong types, or may create a mapping that doesn't align with the existing `failures.py` pattern.
- **为什么有问题**: The existing `candidates/failures.py` already imports `AnnualReportSourceFailureCategory` from `documents/models.py` and provides `map_candidate_failure_to_source_category()`. The new module should follow this established pattern but the plan doesn't reference it.
- **直接证据**: `candidates/failures.py` imports `AnnualReportSourceFailureCategory` and implements `_FAILURE_MAPPING`; §8 permits this import but doesn't specify the pattern.
- **影响**: Implementation agent may create inconsistent failure mapping structure.
- **建议改法和验证点**: Add explicit statement: "The new `fund_disclosure_failure_mapping.py` should import `AnnualReportSourceFailureCategory` from `fund_agent.fund.documents.models` and follow the pattern established by `fund_agent.fund.documents.candidates.failures`." Verification: new module imports match the specified pattern.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: low

### 004-unfixed-[low]-content_hash validation not covered by tests

- **位置**: §6.2 Identity, §10.1 Test File
- **问题类型**: 测试缺口
- **当前写法**: §6.2 `__post_init__` must enforce "`content_hash` is 64 hex chars if present." §10.1 lists `TestFundDisclosureDocumentIdentityConstruction` which should "reject invalid ... hash."
- **反例/失败场景**: The test description says "reject invalid ... hash" but does not specify the exact validation rule (64 hex chars). Implementation agent may write a weaker test (e.g., just checks non-empty) or skip hash format validation entirely.
- **为什么有问题**: `content_hash` format validation is a concrete data integrity guard. Without an explicit test specification, the implementation agent has to guess the expected test behavior.
- **直接证据**: §6.2 specifies "content_hash is 64 hex chars if present"; §10.1 test says "rejects invalid ... hash" without specifying the 64-hex-char rule.
- **影响**: Weak or missing hash validation in tests; hash format errors undetected.
- **建议改法和验证点**: Update §10.1 test description to include: "`content_hash` must be 64 hex chars when present; empty string or wrong length raises `ValueError`." Verification: test covers None (valid), 64-char hex (valid), non-hex or wrong-length (invalid).
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: low

### 005-unfixed-[low]-CandidateBoundaryStatus reuse not explicitly proposed

- **位置**: §6.10 Candidate Boundary Fields
- **问题类型**: 最佳实践偏离
- **当前写法**: §6.10 specifies five boundary fields enforced by `__post_init__` on the top-level `FundDisclosureDocument`. §6.10 says "This matches the existing `CandidateBoundaryStatus` in `contracts.py`." But does not explicitly propose reusing `CandidateBoundaryStatus` as a field type or base class.
- **反例/失败场景**: Implementation agent may re-implement the five boundary fields as separate dataclass fields with duplicate `__post_init__` validation, creating two independent enforcement paths for the same semantic constraint. If `CandidateBoundaryStatus` validation logic evolves, the new schema won't inherit the change.
- **为什么有问题**: The existing `CandidateBoundaryStatus` in `contracts.py` is a frozen dataclass with identical fields and validation logic. The plan acknowledges the match but doesn't propose reuse. The existing pattern (e.g., `CandidateArtifactIdentity` in `candidates/models.py`) uses `__post_init__` directly, which is the Docling pattern; but for the new EID candidate, reusing the existing dataclass is cleaner.
- **直接证据**: `CandidateBoundaryStatus` at `contracts.py:141` has `candidate_only`, `field_correctness_status`, `source_truth_status`, `parser_replacement_authorized`, `readiness_status` with identical validation; §6.10 says "matches" but doesn't propose reuse.
- **影响**: Duplicate validation code; maintenance divergence risk.
- **建议改法和验证点**: Add explicit instruction: "The top-level `FundDisclosureDocument` should carry `candidate_boundary: CandidateBoundaryStatus` as a single field (imported from `fund_agent.fund.processors.contracts`), or enforce the same five fields via `__post_init__` referencing the existing dataclass. Prefer the former to avoid duplication." Verification: test proves boundary fields match `CandidateBoundaryStatus` semantics.
- **修复风险（低/中/高）**: 低
- **严重程度（低/中/高/严重）**: low

## Conclusion

```text
VERDICT: pass-with-risks
```

The plan is substantially code-generation-ready and preserves all required constraints (candidate-only boundaries, no EvidenceSourceKind expansion, no repository/source/fallback changes, no facade integration, no field-family extraction). The allowed/forbidden file lists are safe and sufficient. Serialization and non-consumption test coverage is adequate to prevent boundary leakage.

The one medium-severity finding (001) is that the `redirect_unavailable` / `render_unavailable` mapping function contract lacks a domain parameter, making the domain-based redirect split rule specified in §7.3 untestable through the §7.4 function contract. This directly contradicts the controller judgment's binding requirement for testable split rules. The fix is low-risk: add a `redirect_domain` parameter to the function contract.

Four low-severity findings (002-005) identify best-practice deviations and minor test/contract gaps that should be fixed but do not block implementation. All residual risks are tracked to appropriate future gates.

The plan can proceed to implementation after addressing finding 001 (medium). Findings 002-005 (low) should be addressed but do not block.
