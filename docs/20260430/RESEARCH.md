## 一、Claude Code — Agent 工程最佳实践

### 1.1 核心架构

Claude Code 是 Anthropic 的官方 CLI Agent，其架构代表了当前 Agent 工程的最佳实践。

**Sub-Agent 探索机制**（最关键的设计）：
- 主 Agent 不直接读文件探索代码库
- 派 Sub-Agent 去探索，Sub-Agent 可以耗尽自己的上下文窗口
- Sub-Agent 返回 1,000-2,000 token 的精炼摘要给主 Agent
- 主 Agent 基于摘要直接行动，不浪费自己的上下文

> "By the time Claude is done just figuring out what's going on, maybe it's already blown through 70K-100K tokens, and it hasn't even started working yet." — Cal Rueb, Anthropic @ AWS re:Invent 2025

### 1.2 工具调用效率

| 策略 | 机制 |
|------|------|
| **批量并行调用** | 单个 turn 内派发多个并行工具调用，而非串行逐个读 |
| **指纹循环检测** | Hash `(tool_name, result_preview)`，连续 3 次相同指纹则强制换策略 |
| **工具输出截断** | 默认限制 25,000 token；read_file 只返回窗口而非全文 |
| **高级语义工具** | Anthropic 官方建议："构建 `schedule_event` 而非 `list_users + list_events + create_event`" |
| **渐进式披露** | 只加载必要的工具和上下文，不一次性全量加载 |

### 1.3 上下文管理

**Manus 团队生产数据**：
- 工具响应占总 token 的 **67.6%**，System prompt 仅 3.4%
- KV-cache 命中率是最重要的成本指标：缓存 token $0.30/M vs 非缓存 $3.00/M（10x 差异）
- **永远不要动态增删工具**（会破坏缓存前缀），用 logit masking 控制可用性
- "擦除失败就是擦除证据"——保留失败的工具调用，让模型看到什么行不通

**上下文压缩策略**：
1. **工具输出生命周期管理**（最高影响）：旧工具输出替换为摘要
2. **压缩（Compaction）**：在 80% 上下文上限时触发，模型生成压缩摘要
3. **Sub-Agent 隔离**：探索性工作卸载到独立上下文窗口

**上下文腐烂（Context Rot）**：
- Anthropic 研究表明，模型在 50K-100K token 时开始失去焦点
- 每个上下文中的 token 都有"超出 API 价格"的隐性成本
- 目标："最小的高信号 token 集合，最大化产生正确输出的概率"

### 1.4 Prompt 设计

**Goldilocks 原则**（Cal Rueb, Anthropic）：

| 过度 | 不足 | 最佳 |
|------|------|------|
| 32 页 SOP，if-else 伪代码 | 3 句话 | "最小但充分的指令" |

**关键建议**：从过于简略开始，测试后只为观察到的失败模式添加规则。如果 prompt 开始像伪代码，就过度了。

### 1.5 PonyForge 对照分析

PonyForge 运行 WIKI.md 生成任务的实际数据（2026-05-06）：

| 指标 | PonyForge 实际 | Claude Code 典型值 |
|------|---------------|-------------------|
| LLM 调用 | 35 次 | 10-20 次（同复杂度任务） |
| 重复调用率 | 57%（20/35） | < 10%（循环检测 + 缓存） |
| 上下文利用 | 线性增长至预算耗尽 | 压缩 + Sub-Agent 分流 |
| 最终产出 | 无（预算耗尽） | 成功完成 |

**PonyForge 的根本问题**：
1. 无 Sub-Agent 机制 → 主 Agent 上下文被工具输出塞满
2. 无上下文压缩 → 10 万 token 时模型已经在"上下文腐烂"中
3. 低级工具设计（单文件 read_file）→ 需要 25 轮才能收集完信息
4. 探索贪多，产出为零 → 缺少强制推进机制

**对 zhixing agent 的启示**：采用模板驱动后，这些问题大部分自然消失——模板规定了每步做什么，不需要 Agent 自由探索。

## 二、dayu-agent — 专业化投资 Agent 标杆

### 2.1 项目概况

| 指标 | 值 |
|------|-----|
| 作者 | 刘成岗（仓又加错-Leo） |
| 背景 | 小米前战略投资总监、20+ 年编程经验 |
| GitHub | github.com/noho/dayu-agent |
| Stars | 389 |
| 许可 | Apache 2.0 |
| 语言 | Python |
| 定位 | "每个投资者的助理分析师" |

### 2.2 核心创新："按图索骥"

刘成岗的核心洞察：传统做法是把整份财报丢给 AI，让 AI 自己大海捞针——输出质量不可控、幻觉频发。他的方案反转了这个流程：

**用结构化的定性分析模板驱动整个分析过程，让 AI 按图索骥。**

- 模板包含 10+ 章节，覆盖公司分析的完整维度
- 第 1-9 章为分析章节，按模板结构逐章生成
- 第 10 章为研究决策（投资结论）
- 第 0 章为总结概览（最后生成）
- 另附参考资料清单

这不是一个技术选择，而是一个**认知选择**——结构化的分析框架比通用 AI 能力更有价值。

### 2.3 技术架构

**4 层架构**：

```
UI Layer (cli, gui, web, wechat)
  → Service Layer (services, startup)
    → Host Layer (host, execution, process_lifecycle)
      → Agent Layer (engine, prompting, fins)
```

**核心引擎（AsyncAgent）**：
- 2 天用 AI 辅助完成
- 引擎状态机：`PrepareIteration → CallRunner → HandleToolBatch/ContinueAnswer/Finalize → PrepareNextIteration`
- 提供 AsyncAgent、ToolExecutor/ToolRegistry、StreamEvent、CancellationToken

**Scene 机制**：
每个场景（Scene）声明式定义：
- Prompt 组装规则
- 工具候选列表（哪些工具可用）
- 模型选择
- 执行参数

**上下文预算治理**：
- 预测性截断：预测上下文何时会溢出
- 压缩：总结较早的对话轮次
- 延续：跨多个 LLM 调用拆分处理
- 降级：预算紧张时优雅降低细节

**两层内存模型**：
- Pinned State：永久锚点（关键事实、处理结果），永不被驱逐
- Single Pool：最近的原始轮次 + 较早的压缩轮次 + 情节摘要

### 2.4 财报解析管线

刘成岗在雪球的评论中详细描述了技术选型过程：

> "给 AsyncAgent 添加财报读取能力，搞了十几天，总算有思路了。技术选型花了一半时间。"
> "美股用 edgartools，可以处理除 DEF 14 之外的 filings。港股 A 股用 Docling，保真度比 MinerU 稍微差一点点，但胜在开源免费。"

关键数据：
- 仅财报读取部分就写了 **~2 万行代码**
- 技术选型花了半个月
- 读取路径：`list_documents → get_document_sections → read_section` 等

> "不要用编程助手做复杂的事情，自己做好架构设计，一旦有屎山倾向立即小范围重构。"

### 2.5 项目演进路径

```
阶段1: 纯网页 ChatGPT 工作流

阶段4: 持续扩展
```

## 三、价值投资分析框架

### 3.1 标的筛选（Phase 0）

**量化硬指标**：

| 指标 | 阈值 | 依据 |
|------|------|------|
| P/E | < 15（或低于行业中位数） | 格雷厄姆经典筛选 |
| P/B | < 1.5（深度价值 < 1.0） | 资产安全边际 |
| EV/EBITDA | < 10（或低于同行中位数） | 资本结构中性比较 |
| Debt/Equity | < 0.5（重资产行业 < 1.0） | 财务稳定性 |
| ROE | > 12%，持续 5 年 | 巴菲特核心指标 |
| ROIC | > 10%，持续 | 真实经济盈利能力 |
| FCF Yield | > 5% | 现金创造能力 |
| Current Ratio | > 1.5 | 短期流动性 |

**筛选流程**：

```
全市场 → 硬指标筛选 → 行业/治理排除 → 观察名单 (20-50 只) → 深度分析
```

### 3.2 定性分析（Phase 1）

**商业模式分析**：收入模型、客户集中度、定价权、单位经济模型、可扩展性

**护城河评估**（巴菲特 4+1 护城河）：

| 护城河类型 | 指标 | 评分 |
|-----------|------|------|
| 无形资产 | 品牌认知、专利、牌照 | 1-5 |
| 转换成本 | 客户锁定、集成深度 | 1-5 |
| 网络效应 | 用户增长 reinforcing 价值 | 1-5 |
| 成本优势 | 规模经济、独特资源 | 1-5 |
| 高效规模 | 市场仅支撑 1-2 家 | 1-5 |

护城河耐久性评级：Wide（20+ 年）、Narrow（10-20 年）、None（< 10 年）

**管理层评估**：资本配置记录、内部人持股、诚实透明度、任期稳定性、薪酬结构

**公司治理红旗**（任一条即淘汰）：
- 关联交易 > 营收 10%
- 双重股权结构
- 审计师变更或保留意见
- 董事会独立性 < 50%
- 频繁 CFO 变动

### 3.3 定量分析（Phase 2）

**关键财务比率**：

| 类别 | 比率 | 目标范围 |
|------|------|---------|
| 盈利能力 | ROE > 12-15%, ROIC > 10-12%, 毛利率稳定或扩张 | |
| 杠杆 | Debt/Equity < 0.5, 利息覆盖 > 5x, Net Debt/EBITDA < 2.5x | |
| 效率 | 资产周转率、存货周转率、应收账款周转率 | 趋势改善 |
| 估值 | P/E < 15, P/B < 1.5, EV/EBITDA < 10, FCF Yield > 5% | |

**5-10 年趋势分析**：营收 CAGR、EPS CAGR、利润率趋势、FCF 增长轨迹、ROE/ROIC 稳定性

**同业对比**：选 3-5 家直接同行，比较标准化倍数，调整差异（增长、利润率、资产负债表）

### 3.4 估值定价（Phase 3）

**DCF 五步法**：

1. 预测 FCF（5-10 年）
2. 计算 WACC（CAPM: Rf + beta * ERP）
3. 终值计算（Gordon Growth 或 Exit Multiple）
4. 折现到现值
5. 桥接到每股价值

**敏感性分析**：WACC ±1%、终端增长率 ±1%、FCF ±5% → 熊/基准/牛三种情景

**安全边际**：

```
买入价 = 内在价值 × (1 - 安全边际%)
宽护城河稳定: 20-25%
窄护城河周期: 30-40%
无护城河反转: 40-50%
```

### 3.5 投资决策（Phase 4）

**仓位计算**：

| 方法 | 规则 |
|------|------|
| 信念驱动 | 高信念 8-10%，中 4-6%，低 2-3% |
| 集中度上限 | 单股不超 10% |
| 组合规模 | 10-20 个持仓 |

**建仓策略**：

| 策略 | 描述 | 适用 |
|------|------|------|
| 单次买入 | 全仓一次建仓 | 强信念 + 宽安全边际（>40%） |
| 三段建仓 | 1/3 初始，1/3 回调 5-10%，1/3 确认 | 标准策略 |
| 五段建仓 | 20% × 5 次，3-6 个月 | 大仓位或不确定市场 |

**卖出条件**（价值投资者不用价格止损）：

| 条件 | 触发 | 行动 |
|------|------|------|
| 逻辑止损 | 护城河被侵蚀、造假、管理层失败 | 全部卖出 |
| 时间止损 | 2-3 年不收敛到内在价值 | 减半，重新评估 |
| 估值上限 | 价格超内在价值 20%+ | 减半 |
| 极端估值 | 价格超内在价值 40%+ | 全部卖出 |

### 3.6 持仓追踪（Phase 5）

**季度检查清单**：
- 营收 vs 预期（偏差 >5% 调查原因）
- 利润率压缩 >2pp 评估结构性
- 自由现金流连续 2 季下滑深入分析
- 管理层语调变化

**护城河侵蚀信号**：
- 丧失定价权、市场份额下降 >2pp、关键人才流失
- 技术颠覆、监管侵蚀、品牌相关性下降

**监控频率**：

| 频率 | 活动 |
|------|------|
| 日常 | 仅价格提醒 |
| 每周 | 持仓新闻扫描 |
| 每季 | 完整季报审查 |
| 每半年 | 完整逻辑重评 |
| 每年 | 完整重估值 + 组合审查 |

## 四、笔记应用集成研究

### 4.1 Obsidian（集成难度：低）

Obsidian Vault 就是一个本地文件夹，纯 `.md` 文件，无任何专有数据库。

**读取方式**：

| 方式 | 适用场景 |
|------|---------|
| 直接文件系统读取 | CLI/本地使用（最简单） |
| obsidian-cli-rest 插件 | HTTP API，支持 MCP endpoint |
| Obsidian Plugin API | 需要在 Obsidian 内运行 |

**解析工具**（Python）：
- `python-frontmatter`：解析 YAML frontmatter
- `markdown-it-py`：Markdown AST 解析
- Wikilink 正则：`\[\[([^\]|#^]+)(?:#(\^[a-zA-Z0-9]+|[^\]|]+))?(?:\|([^\]]+))?\]\]`

### 4.2 flomo（集成难度：中）

**API 现状**：仅有 write-only webhook，**无读取 API**。

| 功能 | 支持 |
|------|------|
| 写入 | ✅ POST webhook |
| 读取 | ❌ 无 API |
| 导出格式 | HTML（通过应用内导出） |

**集成方案**：用户从 flomo 导出 HTML → 用 `markdownify`/`BeautifulSoup` 转换为 Markdown → 解析

### 4.3 NoNotes（集成难度：低）

自有产品，直接通过后端 API 读取用户笔记。微信小程序云开发或自建后端均可直接访问。

### 4.4 投资相关标签体系建议

```
#标的/茅台         — 提到具体投资标的
#护城河            — 护城河相关思考
#管理层            — 管理层评价
#行业/白酒         — 行业分析
#看多 / #看空       — 投资倾向
#估值              — 估值相关
#买入 / #卖出       — 交易记录
#反思              — 投资反思
```

## 五、财务数据源研究

### 5.1 A 股数据源

| 数据源 | 类型 | 覆盖 | 费用 |
|--------|------|------|------|
| **AKShare** | Python 库 | 股票/基金/期货/宏观 | 免费开源，无需 token |
| **Tushare Pro** | Python SDK | A 股全量财务数据 | 免费注册（积分制） |
| **巨潮信息网 (CNINFO)** | 官方披露平台 | 所有 A 股公告/年报 | 免费（网站） |
| **东方财富** | 金融门户 | 综合数据 | 免费（网页） |

**CNINFO 是 A 股的 SEC EDGAR 等价物**——所有上市公司年报、季报、公告都在此披露。

**推荐**：MVP 用 AKShare（零门槛），需要更深数据时加 Tushare Pro。

### 5.2 美股数据源

| 数据源 | 类型 | 覆盖 | 费用 |
|--------|------|------|------|
| **yfinance** | Python 库 | 行情/财务/基本面 | 免费开源 |
| **SEC EDGAR** | 官方 API | 10-K/10-Q/8-K | 免费 |
| **Alpha Vantage** | REST API | 股票/外汇/基本面 | 免费 25 请求/天 |
| **Finnhub** | REST API | 实时行情/财务 | 免费层可用 |

**推荐**：MVP 用 yfinance（零门槛）。

### 5.3 关键注意事项

- **A股会计准则**：中国公司使用 CAS（中国会计准则），与 US GAAP / IFRS 在收入确认、减值等方面有差异
- **报告语言**：A 股公告为中文，Agent 需要中文处理能力
- **市场结构**：A 股 T+1 交割、10% 涨跌停（创业板/科创板 20%）
- **行业分类**：使用证监会行业分类做同行对比，不用 GICS
